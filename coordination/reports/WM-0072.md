## Determinism, save, performance and security impact

Determinism: no runtime code changed. The plan requires fixed 30 TPS, seeded
streams, stable ordering, serializable anomaly/crisis state, content manifest
hashes and replayable command streams for M5.

Save: no save format, public compatibility promise, platform save UI, codec or
schema migration changed. ADR-0010 limits M5 save/replay to a focused scenario
harness unless a separate gate approves public changes.

Performance: no benchmark baseline changed. The M5 plan preserves the existing
10 percent warning and 20 percent blocking thresholds and adds required M5
metrics/invariants for WM-0083.

Security/authority: data mods remain schema-validated data only; arbitrary code
mods, network mods, platform API mods, executable scripts and unsafe paths
remain forbidden. UI, Pixi, React, Electron, Worker projections and read models
remain read-only.

## Decisions and deviations

- ADR-0010 is created during the planning package because WM-0072 explicitly
  required `ADR-0010 or equivalent`.
- The downstream M5 DAG starts with content schema/data-mod validation because
  runtime M5 systems must not consume unvalidated content definitions.
- WM-0083 combines benchmark/invariant evidence with M6 stop-sign reporting so
  M6 readiness can be evaluated without creating an M6 task.
- Spark was not used. WM-0072 is architecture/control-plane work and is not
  Spark-eligible.

## Remaining risks / follow-up tasks

- WM-0073 through WM-0083 remain proposed until WM-0072 is reviewed,
  integrated and closed by project-director workflow.
- Downstream owners must block on public protocol/save/schema/dependency needs,
  UI authority, hidden shared state, global scans, executable mods, unsafe
  content or M6 scope rather than coding through them.
- M6 remains unstarted; readiness is represented only by stop signs in this
  package.
