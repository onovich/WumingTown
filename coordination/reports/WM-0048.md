# WM-0048 Work report

## Outcome

Implemented the deterministic M3 needs owner store and derived urgency index in
`sim-core` only. The change adds fixed 0..1000 integer lanes for hunger, rest,
comfort, social, and safety; deterministic phase-bucketed scheduled updates;
structured last-change reason data; exact dirty actor/lane urgency rebuilds;
bounded urgency queries; and bounded urgency trace evidence.

Review repair: added deterministic per-phase scheduled-update cursors so
budgeted phase processing resumes where it stopped and wraps only after a full
phase pass. Added `NeedUrgencyIndex.markMutationDirty(result)` plus scheduled
`NeedDirtySink` support so owner mutations have an explicit exact-dirty
integration contract.

- `packages/sim-core/src/m3-needs.ts`: new `NeedStore`,
  `NeedUrgencyIndex`, `NeedUrgencyTraceStore`, constants, reason codes, metrics,

- `packages/sim-core/src/m3-needs.test.ts`: focused headless coverage for

- `docs/05_tech/03_performance_budget.md`: records the WM-0048 metrics surfaces

- `coordination/tasks/WM-0048.json`: taskctl claim/complete lifecycle updates.
- `coordination/reports/WM-0048.md`: this report.

## Acceptance evidence

- Need state uses 0..1000 integer lanes and deterministic scheduled
  phase-staggered updates: covered by `M3 NeedStore` tests for fixed lane reads,
  phase buckets, scheduled deltas, budgeted phase cursor progress, and clamp
  invariants.
- Urgency projections are derived/versioned from owner state: covered by
  rebuild, `markMutationDirty`, dirty mark, stale rejection, exact refresh, and
  source/index version assertions.
- Pawn thinking uses bounded indexed candidates and no all-actor/all-need scan:
  `queryUrgentActors` visits only requested lane urgency buckets up to
  `candidateCap`; tests assert 30 candidates produce only 24 visits and 12
  selected actors.
- Need changes produce structured reason data: tests assert last-change reason,
  source tick, source ids, previous/next values, delta, and owner version.
- Focused tests cover deterministic cadence, clamps, index rebuild, and no
  global scan behavior: `pnpm exec vitest run packages/sim-core/src/m3-needs.test.ts`
  passed with 9 tests after repair.

## Determinism, save, performance and security impact

Determinism: all new sim-core behavior uses explicit ticks, integer lanes,
stable actor/lane keys, fixed urgency thresholds, phase buckets, and stable
score/id ordering. No real time, ambient randomness, DOM, React, Pixi,
Electron, Node filesystem, Worker protocol, save format, schema, dependency, or
UI authority changes were added.

Save/replay: no public save or schema surface changed. `NeedUrgencyIndex` is
rebuildable from `NeedStore`, and rebuild behavior is tested as replay-style
deterministic evidence.

Performance: scheduled updates process one phase bucket with an explicit budget
and per-phase cursor progress. Urgency queries reject dirty backlog and
otherwise visit only requested indexed lane buckets up to explicit caps. Metrics
report scheduled update counts, dirty-backlog peak/final backlog, refreshed
rows, indexed row counts, and rebuild counts.

Security: no new dependency, platform API, file I/O, IPC, or renderer surface.

## Decisions and deviations

  needs owner state, urgency indexing, and typed trace evidence.
- The literal required check `pnpm test --filter m3-needs` is unsupported by the

  `pnpm exec vitest run packages/sim-core/src/m3-needs.test.ts`.
- `packages/sim-core/src/m3-needs.ts` exceeds the 400-line preference because
  the claimed public surface combines the owner store, derived urgency index,
  reason encoding, and trace store. It remains a single module to avoid adding
  extra task surfaces; reviewer may request a split into `*needs*` modules if
  desired.
