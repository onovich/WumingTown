# WM-0170 B2 Read-Only Coordinator Checkpoint

## Outcome

The exact-resume public API audit and repaired B1 checkpoint `f2912b6` passed
independent Canvas review. B2 now implements the protocol-neutral, read-only
`ResidentAutonomyCoordinator` body within the same five-file ownership. It
performs static-policy arbitration, bounded indexed owner ingress, raw Top-12,
full common scoring of retained rows only, common-score reordering and at most
four exact paths. It never reads a claim plan or calls reservation acquisition,
store transition/idle refresh, Job/world mutation, GameSession, protocol or
product surfaces.

Checkpoint `ca23f19` established the first real-owner pipeline and 19 focused
tests. Checkpoint `1319643` added the first Canvas A-E matrix but received
`changes_requested`. This repair closes all seven B2 findings: exact path
identity, KEEP slot semantics, causal rejection reasons, classified stale and
actual-work metrics, and a receiver-exact TypeScript declaration closure with
targeted structural splits. Canvas then returned a second B2 phase failure for
source-ability stale propagation and missing accessor/declaration audit edges;
both are now repaired without changing owners or entering B3. WM-0170 remains
`in_progress`; `taskctl complete`, B3, independent re-review, integration and
merge are deliberately not run in this phase.

## Repaired B1 design

- Candidate sources are numeric and fixed: NONE, FOOD, REST, MEDICAL, ORDINARY
  and WAIT. Current and durable-terminal rows persist both source and candidate
  id; offer-only autonomy identity is removed.
- Snapshot version 2 persists generic candidate id/owner/row/index/backlog plus
  the complete Food, Rest or Medical basis listed above. Non-applicable source
  fields must be zero. Generic fields must exactly map to the selected owner's
  public output, so an aggregate substitute or cross-source residue is rejected
  atomically. Source-specific success coherence is enforced identically for
  transition input and raw snapshot restore; supplemental uint32 versions are
  not required to be positive when the owner permits zero.
- NONE is valid only for the initial idle row. WAIT is valid only for idle.
  Claiming, moving, working, blocked and terminal states require Food, Rest,
  Medical or Ordinary. The transition table still rejects idle-to-idle; the
  narrow `refreshIdleInto` path persists the reviewed WAIT fallback without
  changing transition legality.
- Selection scratch owns exactly five named lanes in fixed Food, Rest, Medical,
  Ordinary, Wait slot order. Tie order is score, source row id, target id, then
  fixed slot. One shared mutable budget represents 24 visited, 12 retained and
  4 exact paths; the existing global limit remains 2 new decisions per tick.
  No source receives a private copy of those caps.
- Safety/health, active non-interruptible work, schedule, need, and
  capability/permission remain pre-gates. `AutonomyDecisionRequest` now carries
  only resident/tick/location/sequence/path-budget facts; callers no longer
  choose need lane, work type, definition, urgency, ability or threshold.
- Claim plans use caller-preallocated mutable header, owner/target/item refs,
  transaction, claims array and eight fixed slot bundles. Reset preserves all
  identities and clears standalone plus embedded refs independently, including
  malformed aliases. A zero-allocation validator requires transaction owner
  aliasing and every entity/item claim to alias its fixed slot ref. The binding
  helper rejects alias mismatch and only writes a reference to an existing
  fixed-slot claim object into the same claims array; it never creates a claim
  object. The transaction is structurally accepted by
  `ReservationLedger.acquireInto`.
- A successful plan must echo `header.pendingJobId === transaction.jobId`.
  Failed plans are not persisted. Promotion after acquire clears pending job.
  If post-acquire publication later fails, the coordinator must release the
  exact claim ids returned by that acquire before retrying.

## Final B2 pipeline and policy

- Policy is accepted only at coordinator construction. Actor/class/table lane
  lengths, enum and binary values, exact owner bucket ranges, ordinary count
  `1..8`, ability/need thresholds and every worst-case score product/sum are
  checked as safe integers. All scalar and typed lanes are defensively copied;
  later mutation of the caller's policy aliases cannot change behavior.
- Ordinary work uses at most eight compiled exact descriptors for each
  `(policyClass,scheduleCode)`. The selected descriptor is derived only from
  authoritative tick/fixed cadence and stable resident identity. Every
  continuously eligible descriptor is queried within eight ordinary decisions.
  `requestSequenceStart` is used only for exact-path request identity.
- Pre-gates validate resident/generation, source tick and capacity for schedule,
  job and wake lanes, schedule codes/flags/work mask/permission, active NeedStore
  actor plus all five need values/owner versions, consciousness and movement,
  active-job interruption policy, per-source ability threshold and exact owner
  permission/bucket. Safety emergency is a hard gate for non-medical sources and
  cannot be outweighed by score. Health facts come only from safety need,
  health-backed ability output and the medical owner's caregiver/patient facts.
- Enabled real sources share one deterministic visit budget. Their quotas are
  12 for N=1, 12+12 for N=2, 8 each for N=3 and 6 each for N=4. Every owner call
  receives `candidateCap === maxSelected === quota`; actual visits sum to at
  most 24. Food/Rest/Medical public APIs require physical 12-row caller-owned
  scratch arrays even for a 6/8 quota. Those arrays are compatibility buffers,
  not private source budgets; their active loop/cap remains the shared quota.
- Each eligible owner row receives only the checked cheap key
  `ownerRawScore + sourceHardPriority` and enters one coordinator-owned fixed raw
  Top-12. Distance, continuity and retry never affect cheap admission. Only the
  retained rows are full-scored, exactly once each, using raw score plus source
  base, need urgency, schedule and ability utility, candidate Manhattan
  distance, continuity bonus and retry penalty. The same lanes are reordered in
  place by `common desc, row asc, target asc, slot asc`.
- This is explicitly the bounded approximation
  `廉价过滤 -> Top-K 距离/效用 -> 精确路径`: raw rank 13 may have won after full
  scoring but is intentionally not scored. Metrics are unambiguous:
  `visitedCount <= 24`, `ingressCount` is every admitted owner row,
  `retainedCount <= ingressCount`, `scoredCount <= retainedCount`,
  `selectedCount` is final 0/1, and `exactPathCount <= 4`. Successful admission
  still has `retainedCount === scoredCount`; a mid-source failure preserves the
  actual work completed before scoring. Owner-local scored counters are not
  aggregated.
- Paths are attempted only in final common order. Sequence is start plus attempt
  ordinal, route scratch is cleared before each call, node expansions are
  checked and accumulated, and selected-route tail remains NONE. Stale rejection
  requires returned request sequence, start cell and goal cell to match the
  just-issued request, then compares the five authoritative path basis fields:
  map, navigation, region, room and region graph against both request and live
  values. `maxNodeExpansions` is a request budget, not a basis. B3 must
  revalidate these facts again before mutation.
