# WM-0074 Work report

## Outcome

Implemented the M5 anomaly roster owner surface and lifted the reviewed M4
borrowed-shadow rule as `core.anomaly.borrowed_shadow.v1`. The roster owns only
compiled immutable rule inputs: stable `defId`, `DefIndex`, roster version,
content manifest hash, validation basis, rule component, activation policy,
evidence mask and non-combat resolution mask. Mutable anomaly progress remains
in the existing borrowed-shadow crisis owner store.

Reviewer-requested repair: borrowed-shadow roster validation now enforces the
exact reviewed M4 rule basis for schema version, minimum activation score,
evidence mask, non-combat resolution mask, rule component, activation policy and
state owner kind. Roster loading also rejects duplicate `defId` values across
all compiled rows, not only duplicate borrowed-shadow rows.

## Acceptance evidence

- Roster validation: tests accept the borrowed-shadow row only when `DefIndex`
  order, kind, rule component, roster version, content hash and validation basis
  are valid. Missing borrowed-shadow, out-of-order rows, duplicate `defId`
  values, and tampered borrowed-shadow schema/threshold/masks fail with
  structured reasons before owner state is published.
- Borrowed-shadow lift: `createM5BorrowedShadowAnomalyDefinition()` exposes
  `core.anomaly.borrowed_shadow.v1` with the reviewed M4 thresholds/evidence
  masks. Tests feed the rostered activation basis into
  `M4BorrowedShadowCrisisStore` and confirm M4 activation behavior is unchanged.
- Bounded activation reads: candidates are stored in per-`DefIndex` lanes and
  queried with caller candidate/selected caps. Ordering is score desc, priority
  desc, stable owner id asc, stable sequence asc, candidate id asc. Metrics
  report visited rows and candidate-cap hits.
- Stale basis rejection: candidate writes and reads reject mismatched
  `contentManifestHash` or `rosterVersion` before publishing state or traversing
  lanes.
- M4 hash preservation: `pnpm sim:run -- --seed 4 --scenario
  m4-core-vertical-slice --ticks 36000` still reports content hash
  `0x698f2c41`, command stream hash `0x538d0e43`, final world hash
  `0xc201a925` and read-model hash `0xce261d9d`.

## File-size note

`packages/sim-core/src/m5-anomaly-roster.ts` is 667 lines. I kept it as one
file for this task because the typed-array lane ownership, atomic validation,
stable ordering and borrowed-shadow basis storage are tightly coupled and easier
to review together while the roster has a single lifted anomaly. The public
types are already split into `m5-anomaly-roster-types.ts`; if later M5 tasks add
more anomaly components, the next natural split is per-component definition
factories and shared numeric validators.
