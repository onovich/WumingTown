# WM-0092 Work report

## Outcome

WM-0092 establishes an automated input and accessibility baseline for the M6 Web and Windows product-gate surfaces. The shell now exposes visible non-color alert severity cues, includes a real bilingual title baseline (`Wuming Town / 无明镇`), and has Web/Desktop e2e coverage for pointer selection, keyboard pan/zoom, viewport scaling, text fit, reduced flashing, no audio-only cues and read-model-only UI behavior.

## Files and public contracts changed

- `packages/ui-react/src/shell-hud.ts`: adds visible `WARNING` / `STABLE` labels plus `data-alert-severity` on alert chips.
- `apps/web/src/product-gate-fixture.ts`: updates the product-gate settlement title to `Wuming Town / 无明镇` for real bilingual layout evidence.
- `apps/web/src/web-shell.e2e.test.ts`: adds Web input/a11y baseline coverage for desktop and compact viewports, mouse selection, keyboard pan/zoom, text fit, no document animations, no audio/video cues and non-color status cues.
- `apps/desktop-electron/src/desktop-shell.e2e.test.ts`: adds Electron product-surface baseline assertions for bilingual text, pointer selection, keyboard pan/zoom, non-color cues, no audio/video cues, no active document animations and no horizontal document overflow in dev and packaged launches.
- `docs/01_design/05_ui_ux_information_design.md`: normalizes the design note to ASCII English and records the WM-0092 baseline.
- `coordination/reports/WM-0092-plan.md`: execution plan for the task.
- `coordination/tasks/WM-0092.json`: claim state and workflow history.

No simulation, protocol, save, Worker, benchmark or authority contract changed.

## Acceptance evidence

- Keyboard and mouse basics work for product gate surfaces: Web e2e selects `lantern-keeper-shen` with a real canvas pointer event, then verifies keyboard `Equal` zoom and `ArrowLeft` pan labels. Desktop e2e performs the same pointer selection and keyboard `Equal` / `ArrowLeft` checks in both dev and packaged shells.
- UI scaling, long text, Chinese/English layout, reduced flashing and non-color danger/lamp cues are checked: Web e2e validates `1440 x 900` and `390 x 720` viewports, representative horizontal text fit, `Wuming Town / 无明镇`, `WARNING`/`STABLE` alert labels, storage `BLOCKED` text, no active document animations and no horizontal document overflow.
- Key audio cues have visual alternatives or documented blockers: the current M6 shell has no `audio` or `video` elements; status, alert and interoperability cues are visual/textual. Future audio cues must add visual alternatives before acceptance.
- React and Pixi remain read-model consumers: tests read debug payloads, dispatch pointer/keyboard input and inspect UI state only. They do not mutate authoritative simulation state, hashes, save format or Worker protocol.

## Decisions and deviations

- The first WM-0092 e2e attempt exposed that the shell did not actually contain Chinese baseline text. The fix uses the real product title `Wuming Town / 无明镇` instead of a synthetic test-only string.
- Changes-requested loop: independent review flagged missing Desktop Electron pointer and keyboard evidence. The fix extends the desktop baseline to select `lantern-keeper-shen` through canvas input and verify `Equal` zoom plus `ArrowLeft` pan labels in dev and packaged Electron launches.
- Verification environment note: e2e commands were run with `C:\Program Files\nodejs` first on `PATH` so spawned desktop build subprocesses use the repository-required `pnpm 11.8.0`.
- Current M6 has no audio cues, so WM-0092 records no-audio-currently rather than creating an audio system or placeholder.
- The design note was short and had mojibake in this workspace; it was normalized to ASCII English while preserving intent and adding WM-0092 evidence.

- Future M7 tutorial, public playtest, audio and full localization work must add broader coverage.
- Windows/Web save-container interoperability remains governed by WM-0088/WM-0091 follow-ups and is not solved here.
