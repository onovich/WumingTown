## Determinism, save, performance and security impact
Determinism: the scenario uses fixed fixture commands, integer lanes, stable
owner-store order and named random streams for director selection; tests compare
two runs and pin summary hashes.

Save: no save envelope, public save schema, resume harness or persisted
projection was added. Dawn-review rows and artifact data are diagnostic
evidence only.

Performance: normal scenario reads use the reviewed bounded owner APIs. The
summary reports lamp gap visits `2/2`, Chronicle evidence visits `4/8`,
obligation visits `2/4`, town-rule visits `2/8`, crisis candidate visits `1/1`
and director visits `1/1`. After the repair the crisis candidate cap is `1`
per branch, so no branch walks through candidates from another path.

Security and authority: sim-core remains free of DOM, React, Pixi, Electron,
Node filesystem, real time and ambient randomness. The director path emits a
lamp-repair opportunity descriptor and does not mutate lamp/evidence/obligation
or crisis facts directly.

## Decisions and deviations
ADR-0009 boundaries are followed: this is owner-store fixture composition, not a
director-authority workaround. The new scenario file is over 400 lines, exceeding
the 400-line preference. I kept it as one file for this integration task so the
fixture command stream, summary hashes and cross-store evidence remain auditable
in one place; it should be split into fixture builders and summary/hash helpers
only if WM-0068 or WM-0069 needs to reuse the same composition.
