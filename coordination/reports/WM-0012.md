## Determinism, save, performance and security impact

Determinism: no simulation code changed. The plan requires deterministic numeric/phase ADR work before map/path/job/reservation implementation and preserves 30 TPS, stable ordering, named random streams and versioned async rejection.

Save/replay: no save code changed. The plan makes Save Container v1 a visible proposed ADR gate before Job and parity tasks, and requires save/load index rebuild and uninterrupted-vs-resumed replay equality.

Performance: no hot path changed. The plan defines required M1 benchmark impacts: dirty map rebuild, 50k spatial/index pressure, 100 path requests, 10k logistics offers, reservation contention, hauling/building long run and no sustained queue/memory growth.

Security/runtime boundaries: no dependency, IPC, Worker protocol implementation, filesystem or UI authority changed. The plan keeps UI read-only and rejects hidden shared state, unversioned caches, global scans and UI-owned simulation.

## Decisions and deviations

No ADR was accepted and no product implementation started. Required architecture decisions are recorded as proposed task gates rather than hidden assumptions.

No human approval blocker was found under `docs/08_codex/05_human_approval_gates.md`; the plan stays within the approved roadmap and does not change platform, language, renderer, simulation-thread model, server/account/telemetry policy, code-mod policy, public release status or save compatibility commitments.

No Spark dispatch was used. WM-0012 and the ADR gates are not Spark-eligible because they own architecture, save format, Worker protocol, deterministic contracts and state ownership.
