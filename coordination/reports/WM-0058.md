# WM-0058 Work report

## Outcome
M3 ordinary-life sessions now run through the existing Simulation Worker protocol
surfaces and match Node headless M3 replay hashes for the six contract
checkpoints. Worker render snapshots, UI deltas, metrics, and save metadata are
derived read-only projections; no public Worker protocol, save format, schema,
dependency, UI authority, Electron, or M4 scope was introduced.

## Files and public contracts changed
- `packages/sim-worker/src/index.ts`: recognizes
  `m3.ordinary_life.injured_caregiver.v1`, consumes M3 advance command ids,
  projects M3 read-only snapshots/UI/metrics from sim-core, and returns focused
  M3 save metadata through the existing `SaveReady` payload.
- `packages/sim-worker/src/m3-worker-parity.test.ts`: adds the focused M3
  Worker/headless parity gate and diagnostics assertions.
- `packages/sim-worker/src/worker-smoke.e2e.test.ts`: keeps M1/M2 browser
  Worker coverage and adds a real browser module Worker M3 parity case.
- `tools/test-runner.mjs`: adds the `m3-worker-parity` unit filter.
- `docs/02_systems/14_explainability_debug_observability.md`: records the
  WM-0058 diagnostics/read-only projection boundary.
- `docs/05_tech/05_testing_observability_ci.md`: records the new test entry and
  browser Worker coverage.
- `coordination/tasks/WM-0058.json`: task state updated by taskctl claim and
  completion workflow.

## Acceptance evidence
- Browser Worker and Node headless consume the same M3 command stream:
  `pnpm test:e2e --filter worker-smoke` passed 10 tests, including M1, M2, and
  new M3 real browser module Worker hash checks.
- Node Worker and headless M3 hashes match:
  `pnpm test --filter m3-worker-parity` passed 2 tests. The M3 test compares
  ticks `0`, `3600`, `7200`, `12000`, `18000`, and `36000` against
  `runM3OrdinaryLifeReplay`.
- Read-only projections cannot mutate authority: M3 parity tests tamper with
  copied snapshot/read-model payloads and then assert final Worker world/read
  hashes still match headless replay; render and UI payloads carry
  `readOnly: true`.
- Diagnostics include seed, scenario, mismatch tick, snapshot/read-model size,
  and latency metrics: `M3ParityDiagnostics` records authoritative seed `46`,
  requested seed `3`, scenario id, checkpoint count `6`,
  `firstMismatchedCheckpointTick: null`, positive final snapshot/read-model byte
  sizes, and `6/6/6` render/UI/metrics projection-message counts.
- Existing M1/M2 worker-smoke coverage remains intact: the same E2E run passed
  the pre-existing M1 Node/browser Worker paths, M2 browser Worker path,
  lifecycle rejection tests, message-port test, and M1 save metadata test.
- No UI/React/Pixi/Electron authority or Worker protocol redesign: only
  `packages/sim-worker`, test runner routing, and docs changed; protocol types
  and client/Electron surfaces were not edited.

## Determinism, save, performance and security impact
Determinism: M3 Worker projections are generated from sim-core
`runM3OrdinaryLifeScenario` and compared to `runM3OrdinaryLifeReplay` checkpoint
hashes; no wall clock or ambient randomness was added.

Save: M3 focused save metadata is exposed through the existing Worker
`SaveReady` shape only. The task did not change the public save format, schema,
migration behavior, or platform save UI.

Performance: Worker projection diagnostics now count snapshot/UI/metrics
projection messages and positive snapshot/read-model byte sizes. Benchmark
baseline changes remain deferred to WM-0059 as planned.

Security/authority: UI, React, Pixi, Electron, platform save, and protocol
consumers remain read-only. No new dependencies or preload/platform APIs were
added.
