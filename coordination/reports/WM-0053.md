# WM-0053 Work report

## Outcome

Implemented the focused M3 medical care slice in sim-core: medical patient
requests are derived from health condition owner rows, caregiver eligibility is
derived from permission plus ability-cache state, and treatment jobs progress
through explicit serializable states with reservation cleanup, path failure
cleanup, integer condition deltas, stock consumption, and structured reasons.

Changes-requested repair: completion-time treatment basis failures now fail the
treatment job terminally and release held reservations; medical offer selection
stops traversal at `candidateCap`; treatment terminal reasons are preserved in
views and treatment-driver snapshots/restores; completed treatment records the
updated condition row version instead of the health store version.

## Files and public contracts changed

- `packages/sim-core/src/m3-medical-care.ts`: added bounded medical patient
  request/caregiver offer selection store with owner-version basis and metrics.
- `packages/sim-core/src/m3-treatment-jobs.ts`: added explicit treatment job
  driver over `JobCoreStore`, `ReservationLedger`, `ItemStackStore`, health,
  ability, and path basis checks. Repair added terminal cleanup for
  complete-time basis failures and treatment-driver snapshot/restore evidence.
- `packages/sim-core/src/m3-medical-care.test.ts`: added focused headless tests
  for selection, progression, cancellation, condition delta, cache invalidation,
  path/reservation cleanup, structured failures, candidate cap traversal,
  terminal reason restore, and condition-version basis.
- `packages/sim-core/src/index.ts`: exported the WM-0053 medical and treatment
  surfaces from the package public entrypoint.
- `docs/02_systems/05_needs_health_mood_social.md`: documented medical delta
  and ability invalidation ownership.
- `docs/02_systems/04_ai_jobs_reservations_pathing.md`: documented medical
  selection caps and treatment reservation/job cleanup behavior.
- `coordination/tasks/WM-0053.json`: task claim metadata from workflow.

## Acceptance evidence

- Medical care work offers are derived from patient condition owner state and
  caregiver ability/permission state: covered by
  `packages/sim-core/src/m3-medical-care.test.ts` selection test.
- Treatment jobs are explicit serializable state machines with reservation
  cleanup, pathing, and structured interruptions: covered by progression,
  cancellation, denied interruption/path failure cleanup paths in
  `M3TreatmentJobStore`; repair tests also cover complete-time stale/no-stock/
  ability failures releasing held claims.
- Condition changes from treatment are deterministic integer deltas and
  invalidate ability caches correctly: focused test reduces sprain severity
  `420 -> 260`, drains health invalidations, and verifies movement ability
  rebuild to `640`.
- Failures distinguish no patient, invalid condition, caregiver ability, path,
  reservation, permission, stale owner-state, and no stock: covered by focused
  tests and structured reason unions. Repair tests assert path, stale path,
  reservation, no-stock, ability, and stale owner-state terminal reasons in
  views; path failure is also round-tripped through treatment snapshot/restore.
- Focused tests cover treatment selection, job progression, cancellation,
  condition deltas, ability cache updates, and no reservation leaks: direct
  Vitest file passes.

## Determinism, save, performance and security impact

Determinism: integer-only severity deltas, stable bucket ordering by request id,
fixed candidate caps, no ambient randomness, no real time. Save/replay: no
public save/schema/Worker protocol changes; treatment views use serializable
numeric/entity-id fields only; repair adds an internal treatment-driver
snapshot/restore for its own serializable lanes, not a public save format.
Performance: no pawn think loop scans global patients; selection reads one
composite medical bucket and now stops traversal at `candidateCap`, reporting a
bounded cap-hit signal instead of full bucket size. Security/platform:
sim-core only; no DOM, React, Pixi, Electron, Node filesystem, package,
lockfile, or dependency changes.

supported by the current test runner, so the direct focused Vitest file plus
health+medical focused Vitest and supported `pnpm test:sim-core` were run.
