# WM-0182 Work report

## Outcome

WM-0182 implements exact ReservationLedger claim custody for the serial WM-0170
prerequisite. `ReservationTransactionRequest.jobGeneration` is optional for
source compatibility: omission normalizes to legacy generation `0`, while an
explicit value must be a positive uint32. Every active record now stores a
required `jobGeneration` and a positive `allocationEpoch` equal to the
successful transaction's post-acquire ledger version.

The public package-root contracts added by this task are:

- `ReservationReleaseIntoOutput`
- `ReservationClaimsIntoOutput`
- `ReservationLedger.releaseClaimsInto(...)`
- `ReservationLedger.readActiveClaimsInto(...)`

`ReservationClaimsIntoOutput` owns seven scalar result fields plus exactly
fourteen length-eight caller-owned lanes:
`channelCodes`, `ownerIndexes`, `ownerGenerations`, `jobIds`,
`jobGenerations`, `hasTargetFlags`, `targetIndexes`, `targetGenerations`,
`cellIndexes`, `slotIds`, `amounts`, `allocationEpochs`, `createdTicks`, and
`leaseExpiryTicks`. Tick lanes are `Float64Array`; no Tick is truncated to a
uint32 lane.

## Exact custody behavior

`releaseClaimsInto` resets its reusable output and validates the full caller
prefix before mutation: integer count `1..8`, both input lengths, duplicate
claim ids, exact owner `EntityId`, job id, job generation, positive allocation
epoch, current ledger version, and release-version headroom. Failure reports
the first deterministic `claimIndex`, `claimId`, and structured reason and
does not change claims, version, counts, or metrics. Success releases only the
exact prefix in reverse order, then commits one ledger-version bump and one
released-count update.

`readActiveClaimsInto` applies the same binding/version/epoch preflight, checks
all fourteen output lanes are exactly length eight, and writes nothing until
the complete prefix passes. It then copies active records in input order and
preserves caller-owned output and lane identities. Input tails are not read or
modified. Its `amounts` lane carries a record amount only for
`ITEM_QUANTITY` and `CAPACITY`; `ENTITY`, `CELL`, and `INTERACTION_SPOT` always
publish zero.

The focused matrix covers all five reservation channels, stale owner/job/
generation/epoch/version bindings, invalid first/middle/last ids, duplicates,
short inputs, malformed output lanes, no-partial-read behavior, full capacity,
reverse compensation order, recycled claim-id ABA, and reusable output
identity.

## Snapshot, hash, and version boundaries

`RESERVATION_LEDGER_SNAPSHOT_VERSION` is incremented from `1` to `2`.
Snapshot records, record views, restore validation, and canonical GameSession
owner hashes now include `jobGeneration` and `allocationEpoch`. Restore rejects
schema 1, missing/mixed schema-2 fields, invalid generations, zero/future/
wrapped epochs, and epochs later than the snapshot ledger version. It does not
synthesize compatibility defaults.

Claim-creating `acquire` and `acquireInto` reject current ledger versions
`0xfffffffe` and `0xffffffff` before mutation. An acquire at `0xfffffffd`
publishes epoch/version `0xfffffffe`; exact release may consume the final legal
bump to `0xffffffff`; no acquire, release, or epoch wraps. Recycled ids receive
the later post-acquire epoch. Canonical collision tests independently vary job
generation and allocation epoch and require both world and read-model hashes
to differ.

Legacy allocating acquire/release signatures remain available. Broad legacy
owner/job cleanup matches only normalized generation `0`, so it cannot release
a positive-generation autonomous claim. Entity lifecycle cleanup remains an
exact entity-destruction hook rather than owner/job matching.

## Allocation-free closure

`reservation-ledger-closure.test.ts` builds a real TypeScript Program and
TypeChecker and independently roots `releaseClaimsInto` and
`readActiveClaimsInto`. Every reachable project call must resolve to the exact
ReservationLedger/free-helper declaration; approved native receivers are
limited to the required typed-array fills, `Map` operations, and
`Number.isSafeInteger`.

The closure rejects unresolved or same-name foreign receivers, broad release
and materializing compatibility APIs, `new`, object/array literals, nested
functions/classes, `for-of`, unreviewed or unbounded loops, spread, regex,
template/constructed strings, and allocation-heavy calls. The only accepted
loops are the bounded claim prefix, reverse release, and O(8 squared) duplicate
checks.

## Determinism, save, performance, and security impact

- Determinism: allocation epochs derive only from the deterministic ledger
  version; exact prefixes use stable input order and reverse release order. The
  canonical hash changes are intentional and covered by collision tests and
  the updated Worker golden.
- Save: this is an internal ReservationLedger snapshot schema increment. Old
  shape restore fails closed. No public save compatibility or migration promise
  is introduced.
- Performance: the new exact read/release paths allocate no per-call object,
  array, closure, or string. They are bounded to at most eight claims and use
  caller-owned storage. No benchmark, baseline, admission, or threshold changed.
- Security/boundaries: no dependency, filesystem, network, DOM, protocol,
  Worker production, product, or package-internal import boundary changed.

## Limitations and follow-up

WM-0182 does not implement ResidentAutonomyStore coordination, claim-ready
domain-owner reads, JobCore token custody, historical driver adoption,
GameSession integration, Worker behavior, protocol, product UI, persistence
migration, or benchmarks. WM-0183 remains the second serial prerequisite and
must consume these package-root contracts after independent review and
integration. WM-0170 remains blocked until both prerequisites are verified,
integrated, and done.

Spark was not used: the task explicitly prohibits the Spark lane because it
changes authoritative ReservationLedger schema, ABA identity, snapshot/hash/
restore behavior, and public owner contracts. Implementation remained with the
simulation-engineer owner.
