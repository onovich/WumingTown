# WM-0027 Work report

## Outcome

Implemented the minimal build-site material delivery and construction path for the M1 road-lantern-frame scenario. Build sites now expose material demand/build work offers, accept reserved carried materials into a private site buffer exactly once, advance deterministic integer construction ticks, and complete one unlit pending-fuel road lantern frame through map/entity owner stores. The headless runner now supports `--scenario hauling-building`.

## Acceptance evidence

- Build sites register material demand and build work offers without broad production orders: `build-site.test.ts` verifies two material offers before delivery, one build offer after buffers are full, and no stale offers after completion.
- Delivered materials convert from reserved carried items into site inventory exactly once: `build-site.test.ts` verifies source decrement, JobCore carried cleanup, site buffer increment, reservation release and second delivery rejection without duplicate buffer increment.
- Construction progress uses deterministic integer ticks and creates the approved map/entity result through command phases: `build-site.test.ts` advances 120 integer build ticks, commits one map occupancy at `(12,7)`, and records `unlit_pending_fuel`.
- Failure reasons distinguish missing materials, blocked site, reservation conflict, path failure and invalid blueprint state: `build-site.test.ts` asserts `material.insufficient_required_amount`, `material.def_not_required`, `reservation.destination_capacity_conflict`, `path.no_route_to_destination`, `site.blocked` and `target.invalid_state`.
- Scenario end state: `pnpm sim:run -- --seed 1 --scenario hauling-building --ticks 100000` reports `scenarioId=m1.hauling_building.road_lantern_frame.v1`, `finalTick=100000`, `worldHash=0xf7815189`, `expectedTick2400Reached=true`, `longRunStable=true`.

## Determinism, save, performance and security impact

Determinism: simulation code uses integer quantities/progress, stable scripted scenario order, typed lanes and no ambient random/time. Save: no external save compatibility added; new state is serializable typed-lane owner state but no WM-0028 save/replay work was started. Performance: no pawn-side world scan was added; offer sync is state-change driven and the 100000-tick scenario is terminal no-op after completion. Security: sim-core remains free of DOM, React, Pixi, Electron, Node fs, real time and `Math.random`; CLI JSON output stays in tool layer.

## Decisions and deviations

ADR-0003 and ADR-0006 ownership rules were followed: `ItemStackStore` remains authoritative for source quantities, reservations claim but do not own quantities, and build-site buffers are a separate owner store rather than storage supply. WM-0027 does not implement WM-0028 Worker/headless parity, WM-0029 benchmark closeout, production chains, UI, light spread or social systems.

## Remaining risks / follow-up tasks

- The scenario runner is the Node/headless M1 surface; Worker parity and replay/save evidence remain WM-0028 scope.
- The build-site implementation is intentionally minimal for two required material lanes and the M1 road-lantern-frame completion path.
