Implemented a focused M4 save/replay/resume harness for
`m4.core_vertical_slice.borrowed_shadow_lamps.v1` / alias

## Determinism, save, performance and security impact
Determinism: replay uses the reviewed M4 scenario fixture, stable checkpoint
ticks, sorted records, integer lanes and deterministic hash helpers. Divergence
comparison reports the first mismatched checkpoint tick.

Save: this is a focused scenario harness only. It does not introduce a public
save container, compatibility promise, migration schema, codec dependency,
platform save UI or Worker protocol change. Rebuilt projections are diagnostic
hash surfaces and not persisted derived-cache authority.

Performance: load validation checks bounded M4 summary lanes and reports the
same bounded surfaces used by the scenario. The harness does not add pawn-think
global scans or normal-tick all-world traversal.

Security and authority: sim-core remains free of DOM, React, Pixi, Electron,
Node filesystem, real time and ambient randomness. The replay CLI writes
artifacts from `tools/`, not from sim-core.

## Decisions and deviations
`packages/sim-core/src/m4-save-replay.ts` is over the 400-line preference. I
kept the envelope types, validation and replay functions together for this
focused harness so reviewers can audit the scratch validation contract against
the saved section shapes in one file. A later reuse task can split validators
and projection hashing into focused helper files if the public surface grows.

## Remaining risks / follow-up tasks
- This does not provide public save compatibility or migration. Any public save
  format still needs its own architecture gate.
- Worker parity and broader benchmark gates remain out of scope for WM-0068.
