# WM-0005 Work report

## Outcome

Defined the M0 simulation Worker protocol spike with typed main-to-simulation and simulation-to-main envelopes, explicit `protocolVersion` / `schemaVersion` fields, boundary validation for untrusted messages, structured rejection reason codes, and a minimal authoritative Worker session state machine that owns lifecycle, session, and sequence state. The spike proves round-trip routing through both a direct Node harness and a browser Worker-compatible message-port adapter; it does not implement gameplay simulation, entity storage, maps, RNG/world hash, save format, or the WM-0006 headless runner.

- `packages/sim-protocol/src/index.ts`: public re-export entrypoint for the sim-protocol package.
- `packages/sim-protocol/src/constants.ts`: protocol/schema versions, message kinds, reason codes, and player command kinds.

## Acceptance evidence

- Main thread cannot mutate simulation state directly: `createSimulationWorker` keeps mutable lifecycle/session/sequence state inside the Worker closure; public callers can only submit `unknown` messages through `receive` or a Worker-like port and consume response envelopes.
- Protocol messages carry explicit schema/protocol version: all public envelopes include `protocolVersion`, `schemaVersion`, `sessionId`, `sequence`, `kind`, and `payload`; constants are `SIM_PROTOCOL_VERSION = 1` and `SIM_SCHEMA_VERSION = 1`.
- Unknown or stale commands fail with structured reason codes: validator and Worker tests cover `UnsupportedProtocolVersion`, `UnsupportedSchemaVersion`, `UnknownMessageKind`, `UnknownCommandKind`, `StaleSequence`, `StaleSession`, `InvalidPayload`, and `LifecycleError` paths.
- Round-trip integration test works in browser Worker and Node test harness: `pnpm run test:e2e` covers direct Node harness dispatch and a browser Worker-compatible `postMessage` / `addEventListener` port adapter.

## Determinism, save, performance and security impact

Determinism: no simulation core, RNG, real-time, or tick-runner behavior was introduced; the spike uses explicit sequence/session ordering only.

Save: no persistent save format was introduced. `SaveReady` is a protocol placeholder response for routing proof only.

Performance: validation runs at the untrusted message boundary only, not in hot internal loops. Snapshot payload is a zero-entity placeholder; no per-entity hot path exists in this task.

Security: main-thread input is accepted as `unknown` and validated before dispatch. Unsupported protocol/schema versions, unknown kinds, stale session/sequence, invalid payloads, and lifecycle misuse are rejected with stable reason codes.

## Decisions and deviations

- Followed ADR-0001 package boundaries; `sim-worker` imports only public `@wuming-town/sim-protocol`, `@wuming-town/sim-core`, and `@wuming-town/foundation` APIs.
- Followed ADR-0002 pinned tool policy; no new dependencies were added.
- Owner recheck required the original task JSON commands to pass literally. Added `tools/test-runner.mjs` and changed root `test` / `test:e2e` scripts to parse the supported filters before invoking Vitest, because Vitest itself does not support the task's `--filter` option.
- Split `packages/sim-protocol/src/index.ts` into focused same-package modules so no changed `packages/sim-protocol/src/*.ts` file exceeds the AGENTS.md 400-line review threshold. The public import contract remains `@wuming-town/sim-protocol`.
- Did not use Spark and did not create subagents, per task instruction.
- During validation, early parallel `pnpm` commands collided in Windows `node_modules` import/rename with `EPERM`; resolved by running `pnpm install --frozen-lockfile` once serially and running all subsequent pnpm commands serially.

## Remaining risks / follow-up tasks

- Browser coverage uses a Worker-compatible port adapter smoke in Vitest, not a real browser automation runtime. A future web/client task should add a true browser Worker smoke once the web app/test harness exists.
- Snapshot, UI delta, metrics, and save payloads are minimal placeholders. WM-0006 and later simulation tasks must replace them with real fixed-tick/headless data while preserving the protocol versioning and validation boundary.
- Protocol version is intentionally strict; no cross-version compatibility path exists until a migration task defines one.
