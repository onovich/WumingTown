# WM-0024 Work report

## Outcome

Implemented the first M1 derived `WorkOfferIndex` and bounded
`ReasonTraceStore` in `sim-core`. Work producers can register, update and
remove offers on owner-state changes; pawn thinking queries exact composite
buckets instead of scanning world entities or all offers. Selection visits at
most the caller-provided candidate cap, returns bounded top offers by integer
score and stable offer id, and records compact structured reason traces.

## Determinism, ownership, performance and security impact

Determinism: offers are linked in ascending offer id per exact bucket; selection
uses integer `scoreMilli` descending, then stable offer id. The implementation
uses typed numeric lanes and no `Math.random`, real time, platform scheduling or
implicit object identity for ordering.

Ownership: `WorkOfferIndex` is derived state only. It does not own job status,
item quantity, reservation claims or location membership. Later producer tasks
must update or remove offers from owner-store state changes.

Performance: lookup uses exact composite buckets and stored bucket counts.
Selection traverses at most `candidateCap` entries, with tests and benchmark
pinning the cap. Reason traces are bounded by fixed ring capacity and do not
grow with pawn or offer count.

Security: no new dependency, platform API, filesystem access or UI authority is
introduced. Benchmark filesystem writes remain inside the existing benchmark
CLI artifact path.

## Decisions and deviations

- Permission is an integer key for M1. Rich policy logic should compile to
  permission ids or producer-side offer updates, not query-time world scans.
- `WorkOfferIndex` uses exact buckets rather than wildcard query expansion.
  Later tasks may issue several bounded exact queries if a domain has reviewed
  caps for that expansion.
- Spark was not used because WM-0024 defines core selection and observability
  contracts used by later Job Driver and hauling work.
- Beacon implemented as simulation-engineer fallback in the WM-0024 worktree;
  independent `reviewer` / Mirror remains responsible for final verification.

## Remaining risks / follow-up tasks

- Job Driver and item/storage producers are future tasks. They must keep the
  WorkOffer index synchronized from authoritative owner changes.
- Path integration is not performed here. Later work should pass selected
  bounded offers into WM-0022 pathing without caching complete paths across
  incompatible versions.
- ReasonTrace currently stores compact work-offer selection fields. Scenario
  tasks may add domain-specific semantic reason classes as reviewed numeric
  codes instead of unbounded strings.
