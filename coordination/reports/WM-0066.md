# WM-0066 Work report

## Outcome

Implemented the narrow M4 director pressure and recovery-window owner surface
in `sim-core`. The director store owns aggregate pressure samples, incident and
recovery opportunity candidate rows, cooldown rows, recovery window rows,
selection traces/reasons and metrics. It does not own or directly mutate lamp,
evidence, obligation, town-rule, health, relationship, crisis or resource
facts.

## Files and public contracts changed

- `packages/sim-core/src/m4-director-types.ts`: public numeric constants,
  pressure sample, candidate, recovery window, selection, trace, metrics and
  structured reason types.
- `packages/sim-core/src/m4-director-pressure-store.ts`: typed-array
  `M4DirectorPressureStore` with bounded incident/recovery lanes, direct-index
  cooldown rows, active recovery window state and named-stream selection.
- `packages/sim-core/src/m4-director-pressure-store.test.ts`: focused WM-0066
  tests.
- `packages/sim-core/src/index.ts`: exports the WM-0066 public API.
- `tools/test-runner.mjs`: adds `--filter m4-director`.
- `docs/02_systems/11_story_director_incidents_quests.md` and
  `docs/05_tech/03_performance_budget.md`: record director authority and
  performance notes.

## Acceptance evidence

- Pressure sampling: tests assert aggregate lamp, evidence, obligation, active
  crisis, injury, mental risk and unresolved case pressure lanes with owner
  version basis fields and no source fact ownership.
- Bounded selection: tests assert candidate caps, stable score/id ordering,
  candidate visit metrics and cap-hit reasons without scanning source stores.
- Named random streams: tests run two stores with the same named stream and
  seed, then assert identical selected results and stream snapshots.
- Cooldowns: tests assert selected incident candidates write cooldown rows and
  later selection returns `director_cooldown_active`.
- Recovery windows: tests assert an active recovery window suppresses incident
  selection, uses per-type recovery lanes for the active recovery type, rejects
  duplicate window opens without overwriting rows, and resumes incident
  selection after expiry.
- No direct fact mutation: tests reject an incident candidate that tries to use
  a recovery repair command and assert recovery candidates expose descriptors
  only.
- Replay-shaped state: tests assert bounded numeric trace rows with selected
  candidate, caps, counts, reason and owner version.

## Repair notes

- Recovery selection now uses separate recovery candidate lanes per recovery
  type. A lamp-repair window traverses only lamp-repair candidates, so
  higher-scoring evidence-review, obligation-settlement or rest-care rows cannot
  consume the active window's `candidateCap`.
- `openRecoveryWindow` rejects duplicate opens for an already-present
  `windowId` with `director_recovery_window_already_open`, preserving the
  existing row and metrics.
- Untimed `readRecoveryWindow` remains a save-shaped row view and no longer
  claims runtime activity. `isRecoveryWindowActive(windowId, tick)` is the
  minimal tick-aware query for debug/tests and returns false after expiry.

## Determinism, save, performance and security impact

Determinism: all director-owned state is numeric typed-array data. Candidate
lanes sort by score descending, priority descending and candidate id ascending.
Top-K choice uses `NamedRandomStreams` through an explicit stream-name query
field. No `Math.random`, real time, UI state, hidden singleton or ambient
shared state participates in selection.

Save: no save format, schema, migration, Worker protocol or replay envelope was
added. Pressure samples, recovery windows, cooldowns, candidates and trace views
are save-shaped numeric records for later focused save/replay work.

Performance: normal selection walks only the active incident lane or the active
recovery lane for the active recovery type with caller candidate and selected
caps. It does not scan lamp, evidence, obligation, town-rule, health,
relationship, crisis, resource, resident or map owner stores. Cooldown rows are
direct-indexed by key.

Security/authority: no DOM, React, Pixi, Electron, Node filesystem, new
dependencies, ambient randomness or ambient time was added to `sim-core`.

## File length note

`packages/sim-core/src/m4-director-pressure-store.ts` is 870 lines. I kept it
as one file for WM-0066 because the pressure sample ring, candidate lanes,
cooldown rows, recovery window state and selection trace ring share private
typed-array invariants and atomic owner-version ordering. Splitting this first
slice would either expose internal lane helpers or duplicate validation logic.
If the surface grows in a later task, the natural split is pressure samples,
candidate lane selection and recovery/cooldown windows.

## Residual risks

- Direct integration with WM-0062/63/64/65 owner stores is intentionally not
  implemented. The director consumes aggregate numeric samples with owner
  versions, and later integration must ensure those samples are produced only
  from approved owner surfaces.
- Descriptor outputs are not routed into Worker or save protocols in this task.
  Downstream tasks must carry them through existing legal command paths without
  granting the director direct mutation authority.
