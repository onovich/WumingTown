# WM-0165 Work report

## Outcome

The default Web gameplay route now starts the authoritative PR-1 GameSession
through the public Simulation Worker browser session, waits for the exact
schema-v3 / GameSession projection-v1 Ready contract, and renders only coherent
same-basis RenderSnapshot and UiDelta pairs. Residents, resources, alerts,
generic job markers, lamp/build facts, and selected-object detail no longer
come from `WEB_PRODUCT_GATE_READ_MODEL` or WM-0150 playable overlays. Normal
time advances through the WM-0164 Worker scheduler without Web advance/drain
requests. Malformed or mismatched data fails closed to an empty lifecycle
surface without fixture fallback.

## Files and public contracts changed

- `apps/web/src/simulation-worker-session.ts`: adds Web-local package-root
  GameSession start/read functions while retaining WM-0150 helpers only for
  historical focused tests/tools.
- `apps/web/src/playable-worker-projection.ts`: replaces the default playable
  overlay mapper with a coherent GameSession render/UI assembler, read-only
  `WorldReadModel` adapter, lifecycle placeholder, and basis evidence.
- `apps/web/src/shell-bootstrap.ts`: switches the default route to
  `initGameSession()`, removes fixture and WM-0150 command/drain authority,
  requests selection detail by projected entity id, and clears gameplay on
  fatal.
- `apps/web/src/simulation-worker-session.test.ts`: covers exact Init/Ready,
  package-root projection reads, malformed fatal, coherent basis mapping,
  selection identity, and empty fail-closed lifecycle state.
- `apps/web/src/product-gate-harness.test.ts`: mechanically proves the default
  bootstrap has no fixture/reviewed-playback/playable-clock import and the Web
  E2E has no canvas `dispatchEvent` gameplay helper.
- `apps/web/src/web-shell.e2e.test.ts`: replaces WM-0152 default-route command
  evidence with real PR-1 automatic scheduler progression and real Playwright
  mouse input; updates responsive/storage/accessibility expectations to the
  GameSession route.
- `docs/05_tech/12_integrated_gamesession_architecture.md`: records the
  implemented Web route, same-basis policy, failure closure, save boundary,
  and residual fixture inventory.
- No package public contract, Worker implementation, protocol schema, React,
  Pixi, Electron, persistence, dependency, or save format changed.

## Acceptance evidence

1. Default gameplay calls `startWebGameSession(workerSession)`, which delegates
   to the public root `session.initGameSession()` and sends PR-1 catalog/seed
   with exact `{ kind: "game_session", version: 1 }`. Focused test asserts the
   complete InitSession envelope and `initializing -> active` only after Ready.
2. `shell-bootstrap.ts` has no `WEB_PRODUCT_GATE_READ_MODEL`,
   `reviewed-playable-session`, WM-0150 start, advance, wait, projection, or
   drain import. `product-gate-harness.test.ts` source-locks this boundary.
3. Web keeps only selected entity id, hover, camera/Pixi state, build-mode
   placeholder state, locale, UI scale, diagnostics visibility, and lifecycle
   transport state. It does not create commands, jobs, reservations, counts,
   resident positions, build progress, reasons, or outcomes.
4. `createWebGameSessionProjectionAssembler()` publishes only same-snapshot
   pairs accepted by public-root
   `validateCoherentGameSessionProjectionPair()`. Debug evidence exposes the
   single projection basis: tick, snapshot sequence, scenario id, content hash,
   world hash, read-model hash, and owner/index versions.
5. Entity positions and map dimensions are read only from RenderSnapshot.
   Residents, exact resource quantities, structured alerts, job state/progress,
   lamp/build facts, and selection detail are read only from the paired
   UiDelta. The mapper never derives position from `resident.cellIndex`.
6. Job projection v1 has no public job-kind field, so Web does not guess one.
   It renders generic marker state/progress/reason at the same-basis projected
   owner/target position instead of claiming lamp/build semantics.
7. Selection is a local id only. A real mouse click sends public
   `requestUiDetail({ subject: { kind: "entity", entityId } })`; focused tests
   reject stale detail identity and E2E observes resident detail from a later
   authoritative frame.
8. A newer malformed negotiated payload makes the public browser session
   `fatal` and terminates its Worker. An incoherent Web pair is rejected and
   clears the frame. Loading/fatal read models contain zero entities,
   resources, alerts, markers, positions, build facts, or outcomes.
9. The default route never calls WM-0160 advance/drain. Web E2E observes tick,
   snapshot sequence, and world hash advance from the real module Worker after
   only initialization; it sends no canvas synthetic event or time command.
10. The player surface exposes 22 render entities, 8 residents, 4 selectable
    resource entities with exact quantities, 8 generic job markers, and
    selection detail from one GameSession. Lamp/build commands remain disabled
    because PR-1 exposes no command contract.
11. GameSession save/load stays unsupported. The diagnostics-only OPFS envelope
    restores local shell selection/evidence only and cannot restore runtime
    tick, stores, jobs, reservations, resources, projections, or scheduler.

## Determinism, save, performance and security impact

- Determinism: Web remains read-only and sends no normal-time tick request.
  Worker scheduler and core hashes remain authoritative; Worker smoke passes.
- Save: no GameSession save bytes, migration, public UI, or compatibility
  promise. Existing shell evidence remains explicitly non-authoritative.
- Performance: each publication maps bounded protocol lanes. No per-frame
  fixture merge, terrain synthesis, command drain loop, or UI timer remains.
- Security: no dependency, network, telemetry, account, Electron preload,
  storage permission, or package-boundary expansion. All simulation imports use
  package roots.

## Decisions and deviations

- ADR-0017 remains authoritative. WM-0165 consumes, but does not modify, the
  schema-v3 / GameSession-v1 contract implemented by WM-0164.
- No presentation interpolation was introduced. Pixi snaps to each accepted
  RenderSnapshot position; therefore Web cannot mutate authoritative cell/path
  progress between snapshots.
- `playable-worker-projection.ts` is retained as the task-approved filename but
  now contains the PR-1 mapper. At roughly 470 formatted lines it exceeds the
  preferred 400-line guideline; the WM-0165 task grants only exact existing
  files and does not allow a new adapter file. The previous file was 741 lines,
  and this change materially reduces it. `shell-bootstrap.ts` similarly drops
  from 838 to roughly 620 lines. Further splitting needs a task with explicit
  new-file ownership.

## Remaining risks / follow-up tasks

- `product-gate-fixture.ts` remains exported by the app root and
  `smoke-read-model.ts`, consumed by the product-gate unit test and historical
  `reviewed-playable-session.ts`. These are explicit smoke/test/history entry
  points; the default bootstrap source-lock prevents product use.
- `product-gate-harness.ts` and its M5/M4 hashes remain in the explicit
  `wmDiagnostics=1` release-gate overlay, local diagnostic package, and shell
  evidence envelope. They do not populate gameplay state.
- `reviewed-playable-session.ts` remains the WM-0151 browser-local playback
  module with no default-shell import.
- WM-0150 playable start/advance/wait/drain exports remain in
  `simulation-worker-session.ts` for focused regression tests/tools. The
  default bootstrap imports none of them.
- Existing first-play copy still describes the historical WM-0152 lamp/build
  chain. PR-1 HUD buttons are correctly disabled because GameSession exposes no
  such command contract. Rewriting that package-owned guidance is a later
  reviewed UI/localization task, not a reason to synthesize commands here.
- GameSession projection v1 has no job-kind field and no terrain lane. Web shows
  generic authoritative job markers and an unsynthesized map background rather
  than guessing either fact.
- The legacy top-bar resource row requires a trend enum although GameSession v1
  has no trend. WM-0165 leaves that legacy row empty and exposes selectable
  resource entities with exact available/reserved/total quantities instead of
  inventing a trend. A later protocol/UI cleanup should add an explicit
  no-trend GameSession resource surface.
