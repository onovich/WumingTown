# WM-0025 Work report

## Outcome

Implemented the first explicit serializable Job Driver core in `sim-core`.
`JobCoreStore` stores job owner, kind, target, status, current driver step,
interruption policy, integer progress, carried state and structured failure
reason in numeric lanes. Driver movement is expressed through explicit methods
for create, enter, tick, complete, fail, cancel, interrupt, snapshot and
restore; no Promise, Generator, coroutine, closure or UI execution position is
stored.

## Determinism, save, performance and security impact

Determinism: job state uses integer lanes and stable job-id iteration. Progress
uses unsigned Q16 integer fields. Interruption and terminal handling are
explicit method calls, not async completion order.

Save: `JobCoreStore` exposes `JOB_CORE_SNAPSHOT_VERSION = 1`, stable ascending
records and fail-closed restore through a scratch store. `createJobCoreHashFields`
provides canonical fields for replay/world-hash comparisons.

Performance: no scheduler or pawn loop is introduced. Job core storage is typed
lane based, and terminal cleanup releases reservations through the existing
owner/job index rather than scanning unrelated claims.

Security: no new dependency, platform API, filesystem access or UI authority is
introduced.

## Decisions and deviations

- Carried item state is represented as explicit job lanes for cleanup auditing.
  Authoritative item quantity conservation remains WM-0026 scope.
- This task adds the core driver state surface only; it does not implement
  hauling, building, production orders or broad job catalogs.
- Spark was not used because WM-0025 defines core job execution and save/restore
  contracts.
- Beacon implemented as simulation-engineer fallback in the WM-0025 worktree;
  independent `reviewer` / Mirror remains responsible for final verification.
