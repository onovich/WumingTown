# WM-0007 Work report

## Outcome
Implemented generation-checked entity handles, bounded `Int32` component storage, stable ascending iteration, and a fixed-capacity structural command buffer with explicit commit. Structural command conflicts resolve deterministically by command priority, entity index, and insertion sequence. `taskctl complete` has been run after project-director owner review.

- `packages/benchmarks/src/index.ts`: added `entity-store` benchmark target exercising registry/store/buffer allocation, mutation, commit, and iteration paths.
- `tools/test-runner.mjs`: added literal `--filter entity-store` support.
- `coordination/reports/WM-0007-plan.md`: recorded execution plan and verification plan.
- `coordination/tasks/WM-0007.json`, `coordination/thread-registry.json`: updated by `taskctl claim`.

## Acceptance evidence
- Destroyed entity handles cannot alias newly allocated entities: covered by `prevents destroyed handles from aliasing reused slots`; same slot is reused with incremented generation and stale handle fails validation.

- Structural changes are deferred to explicit commit phases: covered by test that queued attach/destroy has no effect until `commit`.

## Determinism, save, performance and security impact
Determinism: entity iteration is ascending by index; structural commit order is explicit and deterministic. No ambient randomness, real time, DOM, React, Pixi, Electron, or filesystem access was added to `sim-core`.

Save: no save format or persistence changes. `EntityId` uses explicit `index` and `generation` fields suitable for future persisted references with generation validation.

Performance: registry, component store, and command buffer use bounded TypedArrays and explicit capacities. The command buffer uses an in-place insertion sort over bounded command slots during commit. After the owner-requested lightweight split, `packages/sim-core/src/structural-commands.ts` is 333 lines and `packages/sim-core/src/structural-command-types.ts` is 95 lines; the public `@wuming-town/sim-core` export contract is unchanged.

Security: no new runtime dependency, IPC, filesystem, Electron, or network surface.

## Decisions and deviations
- Stable structural command ordering is destroy, detach, attach, set, allocate; same priority uses entity index and then insertion sequence. This is intentionally exposed through per-command result sequences for testability.
- Destroy commit removes any component at the destroyed entity index only after generation validation succeeds, preventing stale destroy commands from clearing a new generation's component.
- Spark was not used. Implementation was performed by the assigned `simulation-engineer`; no nested subagents were created.
- `taskctl complete` moved WM-0007 to `review_requested` after all owner checks passed.

## Remaining risks / follow-up tasks
- The initial component store is a foundational single-lane `Int32` SoA store. Future typed component schemas can build on the same active bitmap/generation pattern.
- Commit sorting is bounded by command buffer capacity and is suitable for this foundation; if future command buffers become large, add benchmark thresholds before changing the algorithm.
- No persistence round-trip coverage was added because save format is out of WM-0007 scope.
