# WM-0091 Work report

## Outcome
WM-0091 audits and locks the Electron security posture for the M6 Windows product gate. The Electron window remains `nodeIntegration=false`, `contextIsolation=true` and `sandbox=true`; the preload bridge is now backed by a typed allowlist and desktop-shell e2e verifies the runtime bridge exactly matches that allowlist. No storage bridge, diagnostics bridge, generic host API, dependency, installer, signing, updater, store or M7 work was added.

## Files and public contracts changed
- `apps/desktop-electron/src/preload-contract.ts`: defines the typed M6 preload allowlist for top-level bridge keys, mod methods, save-store methods, host facts, unavailable error and forbidden host API families.
- `apps/desktop-electron/src/desktop-shell.e2e.test.ts`: asserts the runtime preload bridge exactly matches the typed allowlist, exposes no forbidden bridge keys, keeps placeholder unavailable results and preserves renderer sandbox facts.
- `docs/05_tech/04_persistence_mods_security.md`: records the WM-0091 Electron preload audit status and future diagnostics expansion rule.
- `docs/06_engineering/06_dependency_security_policy.md`: normalizes the short dependency/security policy to ASCII English and records that WM-0091 adds no dependency or release pipeline scope.
- `coordination/reports/WM-0091-plan.md`: execution plan for the task.
- `coordination/tasks/WM-0091.json`: claim state and workflow history.

`apps/desktop-electron/src/preload.cjs` and `apps/desktop-electron/src/main.ts` were inspected but not changed.

## Acceptance evidence
- Electron settings are audited and locked: `main.ts` already sets `contextIsolation: true`, `nodeIntegration: false` and `sandbox: true`; e2e reads the host bridge facts and verifies the renderer has no `process` or `require`.
- Preload surface is minimal and typed: `preload-contract.ts` defines allowed top-level keys `host`, `mods`, `saveStore`; allowed mod methods `importArchive`, `listPackages`; and allowed save-store methods `list`, `read`, `remove`, `writeAtomic`.
- Storage and diagnostic APIs expose only whitelisted operations: desktop-shell e2e compares runtime keys against the typed allowlist. Diagnostics are intentionally absent in WM-0091 and must update the contract plus e2e in WM-0093 before acceptance.
- No generic fs, shell or arbitrary IPC exposure exists: e2e rejects forbidden bridge keys including `fs`, `shell`, `ipcRenderer`, `process`, `require`, clipboard, dialog, native-image and open-external families; no IPC or host filesystem API was added.

## Determinism, save, performance and security impact
Determinism impact: none to simulation stepping, seeds, hashes, Worker protocol or replay behavior.

Save impact: no desktop save bridge was implemented. Placeholder unavailable save-store operations remain, so the WM-0088 Windows/Web save-container interoperability blocker remains.

Performance impact: no benchmark threshold, package size or runtime code path changed beyond e2e audit assertions.

Security impact: positive audit-only impact. The typed allowlist and e2e make future preload expansion explicit. No new dependency, lockfile change, IPC, generic host API, secret handling or path exposure was introduced.

## Decisions and deviations
- The preload implementation remains CommonJS because sandboxed Electron preload scripts must stay compatible with the existing packaged runtime. The new TypeScript contract is a reviewed allowlist used by tests, not a runtime import.
- `docs/06_engineering/06_dependency_security_policy.md` was short and previously contained mojibake in this workspace; it was normalized to ASCII English while preserving the policy meaning and adding WM-0091 evidence.
- WM-0091 records diagnostics as absent rather than implementing them. WM-0093 owns the crash/diagnostic package path.
