# WM-0179 Work report

## Outcome

Implemented the bounded structural-command ordering repair at checkpoint
`0882988`. The product change replaces inversion-dependent insertion shifting
with nine fixed stable counting/radix passes over constructor-owned typed
storage and preserves the reviewed priority, signed Int32 index and insertion
sequence contract. All pre-admission functional checks passed, but WM-0179 is
blocked before its authorized benchmark because the only two admission windows
were executed without retaining their CPU-threshold output. The benchmark
invocation count is exactly zero.

- `packages/sim-core/src/structural-command-order.ts`: adds an internal fixed-pass
  ordering owner with one ordered-slot array, one scratch-slot array and one
  256-bucket array allocated in its constructor.
- `packages/sim-core/src/structural-commands.ts`: delegates queued-slot ordering
  to the constructor-owned internal helper; public queue, commit and result APIs
  are unchanged.
- `packages/sim-core/src/entity-store.invariants.test.ts`: adds exact mixed
  priority, signed invalid-index, reverse-index, same-index stability, capacity
  and consecutive-reuse evidence.
- `coordination/tasks/WM-0179.json`: claim and structured blocker state only.
- `coordination/reports/WM-0179.md`: this implementation and blocker evidence.

No public export, protocol, save, snapshot, benchmark, baseline, dependency,
`entity-id.ts` or `component-store.ts` contract changed.

## Acceptance evidence

- Ordering is fixed at nine stable passes: four Uint32 insertion-sequence byte
  passes, four signed Int32 index byte passes using
  `index ^ 0x80000000`, then one canonical command-priority pass.
- `StructuralCommandOrder` allocates its two slot arrays and bucket array only
  from the `StructuralCommandBuffer` constructor. Its `order` path only clears,
  counts, prefix-sums, distributes and swaps local typed-array references. It
  contains no `Array.sort`, comparison sort, insertion shifting, closure or
  per-commit array/object/typed-array/string construction.
- Focused test evidence is 11/11 passing. The exact historical result sequence
  for mixed commands is `[6, 2, 3, 7, 8, 4, 5, 9, 0, 1]`; the consecutive
  reverse/reuse commits retain `[3, 2, 1, 0]` and `[6, 4, 5, 7]`.
- M1, M2 and M3 100000-tick checks retain world hashes `0xf7815189`,
  `0x6943dad4` and `0x7eb81a69` with their printed invariants clean.
- Forbidden file inspection against HEAD reported no diff for
  `packages/sim-core/src/entity-id.ts`,
  `packages/sim-core/src/component-store.ts` or `packages/benchmarks/**`.
- Performance acceptance is not satisfied. The admission script executed two
  consecutive six-second `Get-Process` delta windows with a clean tree before
  and after, but its `Measure-Window` function emitted evidence through
  `Write-Output` while each invocation was assigned to a variable. PowerShell
  captured every window line in the process-local result arrays instead of
  returning it to the tool. Those arrays were lost when the process exited.
  Worse, each non-empty array was truthy, so exit code zero cannot prove either
  CPU threshold. The only retained line is `GIT_CLEAN_AFTER=True`.
- Per the two-window/no-retry policy, no additional admission sample was taken,
  no prewarm occurred and `corepack pnpm bench --filter entity-store` was not
  invoked. There is therefore no WM-0179 benchmark artifact or artifact hash.
