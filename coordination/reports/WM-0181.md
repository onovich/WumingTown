# WM-0181 Work report

## Outcome

WM-0181 adds the seven approved caller-owned owner roots required by WM-0170
without mirroring food, rest or medical truth into generic WorkOffer rows:

1. `M3FoodAvailabilityStore.selectCandidatesInto`
2. `M3FoodAvailabilityStore.readPortionInto`
3. `RestCandidateIndex.selectCandidatesInto`
4. `RestSleepStore.readFixtureInto`
5. `M3MedicalCareStore.selectTreatmentRequestsInto`
6. `M3MedicalCareStore.readPatientRequestInto`
7. `M3MedicalCareStore.readCaregiverStateInto`

All roots return `void`, reset caller-owned output on entry and keep the legacy
allocating APIs behavior-compatible. The new selection roots reject caps above
24 candidates or 12 retained rows, require every aligned scratch lane to have
capacity at least 12, and use bounded loops with no callback sort.

Implementation checkpoints are `e42e38c`, `de9ac0f`, `03bb466`, `1c9933f`,
`fce8b72`, `c46dace` and `2601b35`. The approved plan is in
`coordination/reports/WM-0181-plan.md`.

## Owner and version bindings

### Food

- The selection scratch binds complete retained portion facts, including
  stack/source-row id, score, target, item owner version, meal-window version,
  permission/schedule facts, amount and linked state.
- The flat result binds exact bucket/visited/selected counts, both cap facts,
  the complete first selected row, `foodAvailabilityVersion` and the exact
  same-call dirty backlog.
- `markStackDirty` does not advance the food owner version, so any non-zero
  backlog fails closed. Every retained row is re-read from typed owner lanes
  before success.
- Ordering is score descending and stack id ascending. Target id remains the
  third WM-0170 merge key and is mechanically a no-op after the unique row id.

### Rest

- `readFixtureInto` flattens entity index/generation and all fixture facts; it
  does not materialize an `EntityId` or fixture view.
- Selection binds both the index-cached fixture version and current
  `RestSleepStore` row owner version, plus store/source/index versions and the
  exact dirty backlog.
- Environment basis is explicit: schedule-window id and
  `scheduleWindowVersion`, fixture weather exposure and outdoor-work decision,
  `weatherVersion` and `weatherSourceVersion`. The aggregate environment
  version is not accepted as a substitute.
- Final revalidation compares all retained rows and every environment version.
  Regressions cover each environment version changing after entry.
- The missed-dirty regression mutates fixture A, marks only fixture B dirty,
  refreshes B until `sourceVersion` catches up, and proves the stale cached A
  row still rejects.
- The new root has no trace-store or trace-input path, so the legacy trace
  object construction remains outside the hot closure.

### Medical

- A private per-bucket counter is maintained by request link/unlink. The new
  output reports the exact bucket total before bounded traversal. The legacy
  `bucketCandidateCount` intentionally remains its historical visited count.
- Patient rows bind patient/condition ids, condition and actor-condition
  versions, cached/current health store basis, treatment/stock facts, target,
  score, severity and clue facts.
- Caregiver output binds id, region, permission, ability lane, minimum/value,
  actor-condition version, base-ability version, valid/allowed flags and the
  medical store version.
- Selection calls only `readPatientRequestInto`, `readCaregiverStateInto`,
  `queryAbilityInto`, `health.storeVersion` and
  `health.actorConditionVersion`. It never calls `health.readCondition`,
  `abilities.queryAbility`, or either materializing medical read.
- Above-threshold base-ability skew proves the complete cached/current tuple is
  compared even when the medical version does not advance. A second regression
  changes the base-ability version after patient collection and proves final
  caregiver revalidation clears and rejects the result.
- Ordering is score descending and request id ascending; target is the third,
  mechanically inactive key for a unique request id.
