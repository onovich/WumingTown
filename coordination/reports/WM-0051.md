# WM-0051 Work report

## Outcome

Implemented the focused M3 food/eating logistics slice in `sim-core`. Hungry actors can select edible portions through a bounded derived food index, run exact pathing only for retained Top-K candidates, reserve item quantity plus interaction spots atomically, pick up one integer portion into carried lanes, consume it exactly once into consumed lanes, and apply a hunger delta through `NeedStore`.

Repair update after `changes_requested`: `M3EatingJobDriverStore` now rejects `hungerRestore` values outside the non-negative int32 need-delta contract at job creation, preflights the stored delta again in `consume()`, applies the need delta before `JobCoreStore.completeJob()` can release reservations, and returns structured validation reasons instead of throwing for expected need-delta failures.

## Files and public contracts changed

- `packages/sim-core/src/m3-food.ts`: added `M3FoodAvailabilityStore`, bounded food candidate selection, dirty stack refresh, path Top-K evidence, and food metrics.
- `packages/sim-core/src/m3-eating-jobs.ts`: added `M3EatingJobDriverStore`, explicit eating job states, atomic reservation/pickup/consume/cancel/fail transitions, structured reasons, conservation helper, and repaired int32 need-delta validation/consume ordering.
- `packages/sim-core/src/m3-food-eating.test.ts`: added focused headless tests for selection, path evidence, reservation conflicts, consumption, cancellation, conservation, failure reasons, and the repaired no-split need-delta failure path.
- `packages/sim-core/src/index.ts`: exported the new food/eating APIs and types from the package public entry.
- `docs/02_systems/04_ai_jobs_reservations_pathing.md`: documented WM-0051 food candidate and eating job-driver behavior.
- `docs/02_systems/05_needs_health_mood_social.md`: documented hunger delta ownership and conservation lanes.
- `coordination/tasks/WM-0051.json`: taskctl claim metadata changed from `ready` to `in_progress` before implementation.

## Acceptance evidence

- Hungry actors select edible resources through bounded indexed candidates and exact pathing only for Top-K entries: `m3-food-eating.test.ts` asserts 30 indexed candidates, 24 visited, 12 selected, 4 exact paths, candidate cap hit, and exact path cap hit.
- Eating jobs reserve source item quantity interaction spots and actor/job state atomically before consumption: focused test pre-reserves an interaction spot and verifies the eating reservation fails with `reservation.interaction_spot_conflict` without adding an item reservation or moving quantity.
- Food quantity conservation holds across storage carried and consumed lanes with no negative resources or duplicate consumption: focused tests use `calculateM3FoodConservationTotal` across stacks, carried lanes, and consumed lanes during pickup, consume, duplicate consume rejection, and cancellation.
- Failures emit structured reasons for no food permission path reservation ability schedule and stale owner state: focused tests assert `food.rejected_no_available_portion`, `food.rejected_permission`, `food.rejected_schedule`, `food.rejected_ability`, `food.rejected_stale_owner`, `path.no_route_to_food`, and reservation conflict reasons.
- Focused tests cover deterministic selection consumption cancellation and conservation: `pnpm exec vitest run packages/sim-core/src/m3-food-eating.test.ts` passes 7 tests.
- Changes-requested repair evidence: focused tests assert `hungerRestore: 0xffffffff` returns `eating_need_delta_invalid` before creating job or reservation state, and a consume-time need validation failure leaves the eating driver `picked_up`, `JobCoreStore` running, reservations active, and storage/carried/consumed conservation coherent.

## Determinism, save, performance and security impact

Determinism: no real time, ambient randomness, promises, generators, coroutines, DOM, React, Pixi, Electron, or filesystem APIs were added to `sim-core`. Ordering is score descending then stable stack id, and job progress is explicit typed-array state.

Save/replay: no public save format, schema, Worker protocol, or migration code changed. New stores are serializable lane-based surfaces but are not wired into public save sections in WM-0051.

Performance: normal food availability updates use exact dirty stack ids and bounded candidate buckets; selection metrics record candidate count, visited count, selected count, cap hits, exact path count, and node expansions. Actor selection does not scan all stacks or map cells. The new files are over the preferred 400-line guideline because the focused typed-array lanes, state transitions, reasons, and tests are kept together for this isolated slice; no broad refactor was added.

Security/scope: no dependencies, package/lock files, UI/apps, sim-worker, tools, Worker protocol, save/schema, broad economy/catalog production, medical/mood/relationship implementation, or M4 surfaces changed.
