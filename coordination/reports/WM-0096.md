# WM-0096 Product Gate Decision Report

## Decision Summary

M6 product-gate verdicts from reviewed WM-0087 through WM-0095 evidence:

- Web tier verdict: `demo-only`
- Windows external-test build verdict: `ready-for-controlled-external-test`
- M7 readiness input: `not-ready-until-M6-closeout-records-blockers-and-next-scope`

This is not a Web cancellation decision. Web remains a gated target in the
Roadmap, but the current M6 evidence does not support `same-spec`,
`lower fast-forward` or `lower cap`. Owner approval is still required before
cancelling Web as a formal target, public release upload, store submission,
signing, paid services or M7 execution.

## Web Tier Verdict

Verdict: `demo-only`

Reasoning:

- Same-spec is not proven. WM-0095 records that the current browser path is a
  product-gate shell plus Worker projection path, not a measured 30 TPS /
  20k-entity browser authority runtime.
- Lower fast-forward is not proven. Current evidence does not measure a real
  product-scale browser authority path at a reduced speed.
- Lower cap is not proven. Current evidence shows a 40-visible-actor shell and
  product-gate fixture, but not a measured lower population/map authority tier.
- Demo-only is supported. Chrome Stable and Edge Stable load and interact with
  the product-gate shell, Web save/export/import and diagnostics are covered,
  SharedArrayBuffer-unavailable fallback is verified, and runtime deliverable
  size is far under the 150 MB compressed budget.

Evidence:

- WM-0086 Web build harness established the reviewed
  `wm-0086-web-product-gate` target.
- WM-0087 measured Chrome Stable and Edge Stable and explicitly blocked a
  same-spec claim from shell-only evidence.
- WM-0088 verified Web OPFS write/read/export/import and quota recovery.
- WM-0089 verified the non-cross-origin-isolated Worker fallback path when
  SharedArrayBuffer is unavailable.
- WM-0092 verified input/accessibility baseline coverage.
- WM-0093 verified local Web diagnostics download and redaction behavior.
- WM-0094 verified Web launch, save/export/import, input, diagnostics and M5
  product-gate surface coverage in e2e.
- WM-0095 consolidated current Web evidence: Chrome shell-ready `480 ms`,
  Edge shell-ready `456 ms`, interaction P95 `17.1 ms` / `17.4 ms`, JS heap
  delta `0.684 MB` / `0.691 MB`, and runtime estimated gzip `273561` bytes.

Web blockers:

- No measured 30 TPS / 20k-entity browser authority runtime.
- No evidence for a lower fast-forward or lower-cap browser authority tier.
- Current main-thread interaction P95 remains above the 12 ms Web target even
  on the shell path.
- Chrome/Edge evidence is adequate for a demo shell, not for same-spec product
  parity.

## Windows External-Test Build Verdict

Verdict: `ready-for-controlled-external-test`

Meaning:

- Ready means a local unsigned unpacked Windows directory build can be shared
  for controlled external testing of launch, shell rendering, input,
  sandbox/preload boundaries, product-gate surfaces and local diagnostic
  blocker behavior.
- It does not mean installer, signing, updater, Steam/store, public release,
  crash upload, telemetry, or paid service readiness.

Evidence:

- WM-0090 produces `dist/desktop/win-unpacked/WumingTown.exe` and
  `dist/desktop/wm-desktop-package-report.json`.
- WM-0091 verifies Electron security boundaries: `contextIsolation=true`,
  `nodeIntegration=false`, `sandbox=true`, and whitelisted preload API only.
- WM-0092 verifies Windows shell input/accessibility baseline via packaged
  Electron e2e.
- WM-0093 records Windows host diagnostic package writing as an explicit
  blocker instead of adding broad host access.
- WM-0094 verifies packaged launch from `file:///`, clean unpacked artifact
  path, M5 product-gate fixture visibility, sandbox, input and diagnostics.
- WM-0095 consolidates Windows package evidence: `376084065` bytes, `91`
  files, digest
  `79e7f734fc6741f09070921dcdbc992db3b612619537271faae2774dc4cbeb60`,
  package kind `windows-unpacked-directory`, and simulation authority
  `simulation-worker-or-headless`.

Windows blockers and boundaries:

- Windows/Web save-container interoperability is not proven because the
  reviewed Desktop preload still exposes unavailable save ports.
- Windows host-side diagnostic package writing remains blocked until a reviewed
  narrow diagnostics bridge exists.
- Build is unsigned and unpacked; installer, signing, updater, store and public
  release work remain out of M6 scope without owner approval.
