# WM-0160 Implementation Report

## Outcome

Implemented the public playable Worker drain / terminal wait helper at the
package-root `@wuming-town/sim-worker` surface, with a thin Web adapter in
`apps/web/src/simulation-worker-session.ts`.

The implementation keeps `browser-worker-entry.ts` internal, uses the existing
`advancePlayableCommandScenarioToTick` / `PlayerCommandBatch` path, waits on
authoritative `UiDeltaPayload.playable`, and does not add a new Worker message
family, timer pump, global session singleton, app-side terminal authority, or
product UI feature work.

## Review Follow-up

Keystone requested changes for a concurrency cleanup blocker: live iteration of
session message/lifecycle listener arrays could be mutated by helper cleanup.
With two concurrent public wait/drain helpers, the first rejection during
`destroy` or `shutdown` could splice the same listener array being emitted and
skip the second listener, leaving a pending promise alive after session end.

Fix:

- `emitMessage`, `emitReliableMessage`, and `emitLifecycleMessage` now snapshot
  listeners before dispatching.
- Public helper APIs are unchanged.
- `playable-drain` cleanup remains local to each helper invocation, but
  unsubscribe-during-emit is now safe for message, reliable, and lifecycle
  delivery.

Focused coverage added:

- Two concurrent waits both fulfill from the same authoritative playable
  `UiDelta` even though the first fulfillment unsubscribes during message emit.
- Two concurrent waits both reject on `destroy`.
- Two concurrent drains both reject on `shutdown`.

## Drain Contract

`waitForPlayableProjectionAtOrBeyondTick` posts one explicit public advance
request and resolves with the first authoritative playable `UiDelta` whose
`PlayableProjectionV1.basis.tick` is at or beyond the requested target.

`drainPlayableCommandsToTerminal` accepts requested command ids,
`maxTargetTick`, `stepTicks`, and optional `AbortSignal`. It keeps at most one
in-flight advance target per invocation. The next target is posted only after a
prior authoritative playable `UiDelta` is received for the previous target.

Result payloads return:

- final `UiDeltaMessage`
- final `PlayableProjectionV1`
- final `targetTick`
- `postedMessages`
- `status: "terminal" | "max_target_reached"`
- `terminalCommandIds`
- `activeCommandIds`
- `missingCommandIds`

Terminal judgement is derived only from `PlayableProjectionV1.orders[].markerState`.
Terminal states are:

- `blocked`
- `completed`
- `failed`
- `canceled`

Active states are:

- `queued`
- `claimable`
- `claimed`
- `moving`
- `working`

Missing command ids are not considered terminal; they remain in
`missingCommandIds` and lead to `max_target_reached` if no terminal projection
arrives before the caller's maximum target tick.

## Cancellation And Cleanup

Both helpers support `AbortSignal`. Cancellation rejects the pending promise
with `signal.reason` when it is an `Error`, otherwise with a deterministic
cancel error.

Both helpers subscribe temporarily to normal session messages and lifecycle
events. On resolve, reject, cancellation, `shutdown`, or `destroy`, they remove
message listeners, lifecycle listeners, and abort listeners. No timeout,
interval, animation frame, background loop, global Worker/session singleton, or
hidden real-time authority is introduced.

Listener emission snapshots the current listener set before callbacks run, so
helper cleanup can unsubscribe during message or lifecycle delivery without
skipping other concurrent helpers.

Reliable `CommandResult`, `SaveReady`, and `FatalSimulationError` delivery
remains separate through the existing `subscribeReliable` path.

## WM-0152 Resume Path

WM-0152 should continue using only `apps/web/src/simulation-worker-session.ts`
as the Web-side adapter. Product code should:

1. Start WM-0150 through `startWebPlayableWorkerScenario`.
2. Send `PrioritizeLampWork` / `QueueSimpleBuild` through
   `sendWebPlayableCommandBatch`.
3. Await command progress with `drainWebPlayableCommandsToTerminal`, passing
   the known command ids, `maxTargetTick`, `stepTicks`, and optional cancellation
   signal.
4. Render from authoritative `readWebPlayableProjection(message)` output and
   reliable `CommandResult` messages.

WM-0152 must not import `@wuming-town/sim-core`, deep-import sim-worker
internals, parse `wm0150.advance` ids, parse summaries/debug payloads as
gameplay state, or synthesize world/job/pawn/resource/build authority in Web.

## Save / Load Stance

No public save compatibility promise is introduced. Drain helpers are
active-session deterministic coordination around authoritative Worker outputs
only. In-progress playable command persistence remains unsupported until a
future versioned save/load task adds focused sections, migration rules, and
fixture evidence.
