## Determinism, save, performance and security impact

Determinism: the ledger uses explicit integer ticks, integer amounts, stable
claim ids and deterministic per-owner/per-target cleanup lists. No
`Math.random`, real time or platform scheduling enters `sim-core`.

Save: `ReservationLedger` exposes a v1 snapshot/restore surface with version,
capacity, entity/cell bounds, ledger version, active count and sorted records.
This is not the full Save Container v1 implementation; it is the reservation
subsection needed by later `JobsReservations` work.

Performance: normal acquire and cleanup use target indexes and owner/target
linked lists instead of full-ledger scans. Full record iteration is limited to
snapshot/debug-style paths. `pnpm bench --filter reservations` records accepted
and rejected transactions, cleanup releases, active claims, conflict count and
checksums.

Security: no new dependency, platform API, filesystem access or UI authority is
introduced. External save bytes remain future loader scope; this task only
round-trips already typed ledger snapshots.

## Decisions and deviations

- ADR-0003: `ReservationLedger` is the sole owner of active claims; item
  quantities and future location/item commits remain owned by their stores.
- ADR-0005: snapshot fields align with the future `JobsReservations`
  reservation subsection without finalizing the full container.
- ADR-0006: transaction validation and release use deterministic integer
  ordering and structured reason codes.
- Spark was not used because WM-0023 touches core transaction semantics and
  save-facing state.
- Two spawned `simulation-engineer` subagents stalled before implementation.
  Beacon took over the implementation in the WM-0023 worktree and will still
  route the branch to independent `reviewer` / Mirror for final verification.

## Remaining risks / follow-up tasks

- `ItemStackStore`, `JobCoreStore` and full Save Container v1 are still future
  tasks. Current item quantity and capacity limits are provided by callers.
- Lease expiry remains recovery metadata only; automated lease recovery should
  be added only when the job/interruption runtime exists.
- WorkOffer and hauling tasks must use the ledger through public exports and
  must not infer reservation ownership from cached availability.
