# WM-0159 Architecture report

## Outcome

Verified the third WM-0152 blocker. WM-0157 and WM-0158 provide the legal
browser Worker session and public playable projection, but there is still no
reviewed public browser path for Web to advance the authoritative WM-0150
playable runtime after accepted commands.

I created ADR-0015 and the ready precursor task `WM-0159` for a narrow public
root `@wuming-town/sim-worker` deterministic advance helper/session API.

## Evidence

| Question | Evidence |
| --- | --- |
| Can Web send real WM-0150 commands? | Yes. WM-0157/WM-0158 expose `createWebSimulationWorkerSession`, `sendWebPlayableCommandBatch`, and `readWebPlayableProjection`. |
| Does `SetSpeed` / `Pause` advance playable time? | No. `packages/sim-worker/src/index.ts` handles those messages by updating speed/paused state and returning `CommandResult` / `MetricsSample`; no playable runtime `advanceTo` call occurs. |
| Is there an internal advance path? | Yes. `applyPlayableCommands` advances only when a `Noop` command id parses through `parsePlayableAdvanceCommandId`. |
| Is that path legal for `apps/web` today? | No. `createPlayableAdvanceCommandId` is exported by `@wuming-town/sim-core`; WM-0152 must not import `sim-core` or hardcode `wm0150.advance.<tick>`. |
| Did a mechanical check reproduce frozen tick? | Yes. Init -> `SetSpeed(1)` -> `Pause(false)` -> `RequestUiDetail` kept `UiDeltaPayload.playable.basis.tick` at `0`. |
| What unblocks WM-0152? | `coordination/tasks/WM-0159.json` implementing ADR-0015. |

## Decision

ADR-0015 chooses the narrow helper/session route over autonomous
`SetSpeed`/`Pause` ticking:

- expose a root public `@wuming-town/sim-worker` helper or session method;
- accept an explicit logical `targetTick`;
- send through the existing `PlayerCommandBatch` path;
- hide the internal `wm0150.advance.<tick>` string from Web;
- keep Worker/headless as the only authority for time, jobs, pawns, resources,
  build progress and blocked reasons;
- keep reliable messages separate.

No product UI implementation was changed.

## Deterministic clock and save/load stance

The proposed helper is not a wall-clock runtime. It is an explicit deterministic
logical tick input sent to the Worker in message order. Worker authority must
not use `Date.now()`, `performance.now()`, `setInterval` or
`requestAnimationFrame` to mutate simulation state.

No public save compatibility, migration, in-progress command persistence,
release promise, telemetry, account or public feedback surface is introduced.
