# WM-0159 Work report

## Outcome

Implemented a public deterministic WM-0150 playable advance helper on the
package-root browser Worker session surface. Web can now request an explicit
logical target tick through `@wuming-town/sim-worker` without importing
`sim-core`, importing worker internals, or hardcoding the internal
`wm0150.advance.<tick>` command-id convention.

## Files and public contracts changed

- `packages/sim-worker/src/browser-session.ts`: added
  `BrowserSimulationWorkerSession.advancePlayableCommandScenarioToTick` and
  root helper `advancePlayableCommandScenarioToTick`.
- `packages/sim-worker/src/index.ts`: root-exported
  `advancePlayableCommandScenarioToTick`.
- `apps/web/src/simulation-worker-session.ts`: added thin Web adapter
  `advanceWebPlayableWorkerScenarioToTick`.
- `apps/web/src/simulation-worker-session.test.ts`: verifies the Web adapter
  posts a normal `PlayerCommandBatch` with `Noop` kind without naming the
  internal command id.
- `packages/sim-worker/src/worker-smoke.e2e.test.ts`: verifies the public
  session helper advances WM-0150 to tick 45 in a harness and advances the real
  Chromium module Worker path through lamp/build completion.

Exact public API names:

```ts
interface BrowserSimulationWorkerSession {
  advancePlayableCommandScenarioToTick(targetTick: number): MainToSimulationMessage;
}

function advancePlayableCommandScenarioToTick(
  session: BrowserSimulationWorkerSession,
  targetTick: number,
): MainToSimulationMessage;
```

The helper accepts a non-negative safe integer `targetTick`, posts a normal
`PlayerCommandBatch` with one Worker-owned playable advance `Noop` command, and
returns the posted `MainToSimulationMessage`.

## WM-0152 resume path

Use only the existing Web adapter:

```ts
import {
  WEB_PLAYABLE_WORKER_SCENARIO_ID,
  advanceWebPlayableWorkerScenarioToTick,
  createWebSimulationWorkerSession,
  readWebPlayableProjection,
  sendWebPlayableCommandBatch,
  startWebPlayableWorkerScenario,
  type WebPlayableProjection,
} from "./simulation-worker-session";
```

Recommended flow:

1. Create the session with `createWebSimulationWorkerSession`.
2. Start WM-0150 with `startWebPlayableWorkerScenario`.
3. Send `PrioritizeLampWork` / `QueueSimpleBuild` through
   `sendWebPlayableCommandBatch`.
4. Request deterministic progress with
   `advanceWebPlayableWorkerScenarioToTick(session, targetTick)`.
5. Render only `readWebPlayableProjection(message)` output and reliable
   `CommandResult` messages.

WM-0152 must not import `@wuming-town/sim-core`, must not deep import
`@wuming-town/sim-worker`, and must not construct or parse
`wm0150.advance.<tick>` strings.

## Determinism, save, performance and security impact

Determinism: advance uses an explicit integer target tick and the existing
ordered `PlayerCommandBatch` stream. No real-time source, timer, random source,
or autonomous loop was added.

Save: no public save compatibility, migration, or in-progress playable command
persistence was added. Advance requests are transient active-session inputs.

Performance: no pawn loop, world scan, polling loop, or UI cache was added. The
helper allocates one protocol command per explicit advance request.

Security: no Electron, preload, filesystem, telemetry, account, paid-service,
release, or public feedback surface changed.

## Decisions and deviations

- Followed ADR-0015 option 5: a public root `@wuming-town/sim-worker`
  helper/session method hides the internal advance id and sends a normal
  `PlayerCommandBatch`.
- Kept `SetSpeed` and `Pause` semantics unchanged. They still do not imply
  autonomous browser ticking for this precursor.
- Did not add a new top-level Worker message family or protocol schema field.
- Did not use Spark/rapid-implementer; this task touches a public Worker API.

## Remaining risks / follow-up tasks

- WM-0152 still needs to wire product HUD/renderer flow to the public
  `advanceWebPlayableWorkerScenarioToTick` helper and render
  `UiDeltaPayload.playable`.
- Continuous wall-clock play remains out of scope until a later ADR defines
  timer ownership, throttling, backpressure, lifecycle and replay semantics.
- Public save/load for in-progress playable command state remains unsupported.
- `coordination/thread-registry.json` was already dirty from project-director
  thread registration before implementation and was preserved as control-plane
  state.
