## Acceptance evidence

- Relationship graph stores ordinary-life edges with stable ids and integer
  kinship/care/trust/gratitude/resentment lanes:
  `m3-relationships` tests cover stable edge ids, kinship, care, trust,
  gratitude, resentment, and `-1000..1000` integer lane storage.
- Social event facts are structured data:
  event rows store stable event id, source tick, actor ids, kind, lane, delta,
  source event id, source owner version, reason, edge id, and applied value.
- Relationship queries use bounded indexed candidate lanes:
  `selectRelationshipEdges` and `selectRecentSocialEvents` read only
  per-actor/per-lane linked buckets into caller-owned buffers with candidate
  cap `16`, selected cap `8`, stale-basis rejection, and deterministic ordering.
- Scenario-facing explanations distinguish relationship fact source, trust, and
  recent events:
  `createExplanation` returns source event id/tick, graph version, current
  trust, bounded recent-event count, newest event id, candidate total, visited
  count, candidate cap, cap-hit status, and structured reason.
- Mood/thought integration remains data-only:
  `createMoodThoughtInputForEvent` materializes `M3MoodThoughtInput` with
  source kind `social`; it does not mutate `MoodThoughtMemoryStore`.
- Replay evidence is deterministic:
  `createSnapshot()` materializes edge and event rows in stable id order, and
  `createHash()` covers owner versions, edge lanes, event facts, and source
  fields. The focused replay test compares two independent runs.

## Determinism, performance, and security impact

Determinism: all owner ids are stable integers; edge iteration, event
materialization, candidate selection, traces, snapshots, and hashes use stable
ordering. No `Math.random`, real time, platform clocks, DOM, React, Pixi,
Electron, Node filesystem, Promise/coroutine execution state, or ambient
entropy is used.

Performance: normal candidate reads use exact per-actor/per-lane indexes and
caller-owned buffers. They do not scan all actors, all relationships, all
events, jobs, world cells, or content. Metrics report graph version, active
edge count, event count, event applications, candidate queries, visited rows,
cap hits, and selected rows. Explanation reads share the same candidate metrics
instead of walking target-sparse buckets without evidence.

Security and boundaries: no new dependencies, package metadata, Worker
messages, public save/schema surfaces, renderer authority, preload APIs,
Chronicle evidence, town rules, anomaly/crisis/director systems, or M4 state
were added.

## Decisions and deviations

ADR-0008 is the controlling decision. WM-0055 implements only the
`RelationshipGraphStore` owner slice and structured social event facts for M3
ordinary-life consequences. Relationship event kind defaults are intentionally
small fixture values: `care_received` -> gratitude, `meal_shared` -> trust,
`work_burden_shifted` -> care, and `care_delayed` -> resentment.

The requested `pnpm test --filter m3-relationships` command is unsupported by
the existing test wrapper, so the direct Vitest file and supported `sim-core`
wrapper check were run.

`m3-relationships.ts` exceeds the preferred 400-line guideline because it keeps
one focused owner store, its derived indexes, trace store, metrics, snapshot,
and hash helpers together, matching the existing M3 owner-store pattern. The
scope remains within one sim-core relationship surface and its tests.
