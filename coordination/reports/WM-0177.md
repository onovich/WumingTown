# WM-0177 Blocker Report

## Outcome

Blocked during the mandatory pre-code public-invariant audit. No sim-core
product file changed.

The proposed four-owner repair is insufficient to create an allocation-free
`GridPathfinder.findPathInto`: the pathfinder's required MapGrid public reads
allocate result/cell objects per request and per expanded neighbor, while
`map-grid.ts` is outside WM-0177's write ownership.

## Evidence

- `GridPathfinder.findPath` calls `MapGrid.isCellPassableByIndex` for start and
  goal validation.
- `GridPathfinder.tryRelaxNeighbor` calls
  `MapGrid.canMoveBetweenCardinalNeighbors` and `MapGrid.readCellByIndex` for
  each candidate neighbor.
- Each of those MapGrid methods returns object literals; `readCellByIndex`
  additionally calls `readCellAtIndex`, which materializes the cell view.
- The required exact path may visit many nodes, so these are hot-path
  allocations, not cold compatibility-wrapper allocations.

Removing only the final `Uint32Array` path allocation would satisfy neither
WM-0177's full focused allocation proof nor the unchanged WM-0170 audit. Using
private MapGrid storage through a cast would violate package/owner boundaries.

## Required repair decision

Either revise WM-0177's reviewed ownership or add a strict predecessor that
provides additive caller-owned/primitive MapGrid surfaces for:

1. indexed passability;
2. cardinal-neighbor movement permission;
3. indexed walk cost (and any other scalar path fact needed by A-star).

The legacy MapGrid object-returning APIs, snapshots, versions and hashes must
remain unchanged. WM-0177 should resume only after those surfaces are in
scope, with `map-grid.ts`, its focused tests and package-root exports explicitly
owned.
