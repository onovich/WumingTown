# WM-0075 Work report

## Outcome

Implemented `core.anomaly.third_knock.v1` as a rostered M5 anomaly rule with a
dedicated sim-core owner store. The implementation keeps immutable anomaly rule
metadata in the M5 roster and keeps mutable third-knock activation/crisis
progress in `M5ThirdKnockCrisisStore`.

The third-knock owner store models invitation/debt activation candidates,
activation prevention, low-risk evidence, escalation, non-combat containment or
temporary-policy resolution, failure and accident-review rows as explicit
serializable numeric state.

## Acceptance evidence

- Serializable owner store: crisis state is numeric and explicit:
  `activated -> trace -> escalated -> resolved|failed`. Crisis views include
  activation, trace, escalation, terminal reason, resolution method and owner
  version fields.
- Versioned activation basis: candidate writes require matching roster version
  and content manifest hash plus positive threshold, Chronicle evidence,
  town-rule, obligation, guesthouse policy and lodging-register versions.
- Structured prevention: activation is rejected with specific reasons when the
  third knock was not answered, invitation/debt score is below threshold, or a
  confirmed rule/temporary policy is known.
- Low-risk evidence: tests record prior knocks, witness disagreement, threshold
  marks, lodging-register mismatch and obligation pressure as structured trace
  rows before escalation.
- Non-combat resolution: tests cover threshold containment and temporary-policy
  resolution. Terminal resolution/failure writes accident-review rows with
  crisis id, tick, method, terminal reason, evidence count, invitation score,
  obligation pressure and source owner versions.
- Bounded reads: activation candidates are queried with caller candidate and
  selected caps. Tests assert stable Top-K ordering and candidate-cap metrics.
- No UI/coroutine authority: tests assert exposed crisis state has no UI,
  Promise, coroutine or closure progress fields and round-trips through JSON.

## Review repair evidence

Reviewer requested changes on four issues. The repair adds focused regression
coverage and code guards for each:

- Required basis ids now reject the `M5_THIRD_KNOCK_NONE` sentinel for resident,
  guest, door, threshold, Chronicle case, Chronicle hypothesis and stable owner
  ids before any candidate is published.
- The store pins its roster/content basis on first successful registration and
  rejects later candidates or queries using a different basis. Candidate reads
  now return the candidate's original content manifest hash.
- Resolution input now validates method, flags and integer score ranges before
  resolving a terminal state.
- Low-risk evidence count now fails closed at the `Uint16` limit with
  `third_knock_low_risk_evidence_cap_reached` instead of wrapping.

## File-size note

`packages/sim-core/src/m5-third-knock.ts` is 1040 lines and
`packages/sim-core/src/m5-third-knock.test.ts` is 433 lines. I kept the store as
one file for this task because the typed-array lanes, activation prevention,
state transitions, trace ring and accident-review ring are tightly coupled and
mirror the already-reviewed borrowed-shadow crisis store shape. The public
types are split into `m5-third-knock-types.ts`; the next natural split, if M5
adds more shared anomaly helpers, is common numeric validation/ring-buffer
helpers after at least two M5 crisis stores converge on identical code.
