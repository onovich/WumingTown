## Current UI / Interaction / I18n Audit

### Default UI

The current code has a localized start surface, display settings and a player
HUD path, but it still imports the active read model from
`WEB_PRODUCT_GATE_READ_MODEL` via `apps/web/src/smoke-read-model.ts`. This
means the shell still has product-gate fixture identity in its data basis even
when default diagnostics are hidden.

Diagnostics are gated by query parameter and debug overlay state, but product
gate wording still exists in runtime/debug surfaces and can leak into the
player perception through fixture labels or reports unless the remediation
renames/reframes the default player read-model basis.

### Mouse Interaction

Pixi already supports entity hit testing and updates `selectedEntityId` through
`onSelectionChange`. The current renderer listens for `pointerdown`, `wheel`
and keyboard movement. However:

- owner screenshots show practical selection/clickability was not discoverable;
- HUD panels and overlay layout must be retested for pointer-event interception;
- empty-tile selection feedback is only an input label, not a localized player
  affordance;
- UI buttons are mostly menu/settings/storage buttons, not gameplay actions.

### Camera

Wheel zoom and keyboard pan exist in `pixi-world-renderer.ts`, but mouse drag
pan and reset controls are missing. This directly matches the owner blocker.

### Player Commands

`packages/sim-protocol/src/constants.ts` exposes only `Noop` and `Echo` player
command kinds. Therefore the minimum playable command task must either:

1. route a meaningful action through an already reviewed non-authoritative
   shell adapter with clear trace/feedback; or
2. create/review a narrow protocol task before claiming simulation authority.

The DAG forbids faking world mutation in UI.

### I18n

The locale contract and runtime already support `zh-CN` and `en`, system
language detection, manual override and missing-key validation. The remaining
risk is not absence of infrastructure, but mixed player-visible text from
read-model labels and fixture strings that are not localized keys. WM-0134
therefore includes hardcoded player-facing string inventory and default HUD
coverage.

### Responsive

M8 tests cover a viewport matrix, but owner screenshots at windowed and
fullscreen sizes show the actual perceived layout remains problematic: dense
panels obscure the map, important controls are not arranged as a game command
surface, and screenshots still read like a product-gate harness. WM-0135 and
WM-0140 must include screenshot/DOM evidence, not only unit assertions.

### Visual Identity

The current theme uses Wuming-like colors, but the layout still feels like
stacked cards over a debug map instead of a cohesive management HUD with a
bottom command bar, resident rail, selected-object inspector and semantic art
slots. WM-0131 and WM-0133 address this without requiring final cut-in art.

### Playability

Current practical playability remains blocked because a player can inspect
state but cannot complete a meaningful chain such as selecting a lamp, issuing
a traceable action, seeing a queued/accepted result and receiving localized
feedback. WM-0138 makes this a blocking gate.
