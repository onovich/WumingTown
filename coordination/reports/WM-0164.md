`@wuming-town/sim-protocol` now exports:

```ts
import {
  GAME_SESSION_PROJECTION_VERSION,
  SIM_SCHEMA_VERSION,
  sameGameSessionProjectionBasis,
  validateCoherentGameSessionProjectionPair,
  validateGameSessionAlertsV1,

  validateGameSessionRenderProjectionV1,
  validateGameSessionUiProjectionV1,
  type GameSessionAlertV1,

  type GameSessionProjectionContractV1,
  type GameSessionProjectionRequestV1,
  type GameSessionRenderEntityV1,
  type GameSessionRenderProjectionV1,
  type GameSessionSelectionDetailV1,
  type GameSessionStructuredReasonV1,
  type GameSessionUiProjectionV1,
} from "@wuming-town/sim-protocol";
```

`SIM_SCHEMA_VERSION` is `3`; `SIM_PROTOCOL_VERSION` remains `1`; projection
version is `1`. `InitSessionPayload.projectionRequest` negotiates
`{ kind: "game_session", version: 1 }`, and
`ReadyPayload.projectionContract` echoes it. The versioned payload fields are:

- `RenderSnapshotPayload.gameSession: GameSessionRenderProjectionV1`
- `UiDeltaPayload.gameSession: GameSessionUiProjectionV1`
- `AlertBatchPayload.gameSession: readonly GameSessionAlertV1[]`

`GameSessionProjectionBasisV1` is shared exactly across each render/UI pair and
contains projection/scenario/content identity, tick, current/previous snapshot
sequence, world/read-model hashes, and map/reservation/job/derived-index
versions. Render rows carry stable entity refs, kind/def id, Q16 position,
facing, animation and flags. UI rows carry pause/speed/daylight, eight residents
with five needs and job facts, four semantic resources, job markers, structured
alerts, lamp/build progress and versioned requested resident/resource/structure
selection detail.

## Public Worker/session API

WM-0165 should consume only the package root:

```ts

session.initGameSession({ seed: PR1_GAME_SESSION_DEFAULT_SEED });
```

The session moves `created -> initializing -> active` only after a matching
schema-v3 `Ready.projectionContract`. WM-0165 should subscribe to package-root
`RenderSnapshot.gameSession`, `UiDelta.gameSession`, reliable `AlertBatch`,
`CommandResult`, `SaveReady` and fatal delivery; it must render projection facts
and use `requestUiDetail()` for resident/resource/structure details. It must not
import sim-core, sim-worker subpaths, package source internals, static product
fixtures or call playable advance/drain as the default clock.

## Worker scheduling and backpressure

- Worker timer quantum: 100 ms outside sim-core.
- Requested speeds `1/2/3`: 3/6/9 integer core ticks per quantum, effective
  `30/60/90` TPS.
- Pause or speed `0`: zero ticks. Pause preserves requested speed.
- `SetSpeed` and `Pause` publish current authority immediately but affect only
  future scheduling quanta; no retroactive ticks are synthesized.
- Render/UI/Metrics slots are latest-wins in the Worker outbox. Replaced render
  snapshots increment `droppedSnapshots`.
- `CommandResult`, structured `AlertBatch`, `SaveReady` and
  `FatalSimulationError` use a reliable FIFO and are emitted in Worker sequence
  order. Fatal clears stale presentation slots, not reliable entries.

The browser `postMessage` transport has no consumption ACK in protocol v1.
Therefore the outbox can bound pending Worker-side publication slots and avoid
dropping reliable messages, but it cannot measure or retract snapshots already
accepted into the browser task queue. Adding an ACK/message family was outside
ADR-0017 and remains a future transport decision, not hidden compatibility.

## Fail-closed evidence

Focused tests cover schema-v2 envelopes, unknown projection versions,
mismatched/missing Ready contracts, projection before Ready, missing negotiated
render/UI payloads, malformed/duplicate rows, malformed fresh Q16 positions,
invalid selection detail, wrong session/order and incoherent basis. These close
and terminate the browser session with structured `FatalSimulationError` and do
not fall back to legacy/static state. Lower stale render snapshots are droppable
from a safe outer snapshot sequence without dereferencing malformed nested
payloads.

Legacy M0-M8 same-build modes may omit the projection request under schema 3;
they do not negotiate `GameSession` and are mechanically ineligible for the
default PR-1 route.

## Residual PR-1 blockers

- WM-0165 must route the default Web shell/Pixi/HUD to these exact projections
  and quarantine static product fixture authority. WM-0164 intentionally does
  not modify apps or UI.
- WM-0163 currently accepts deterministic Noop/Echo command ordering only.
  Integrated hauling/build/lamp/needs gameplay commands are future reviewed
  runtime work; WM-0164 structurally rejects unrelated typed playable commands
  rather than copying WM-0150 authority.
- General GameSession persistence and cross-build compatibility do not exist.
- Protocol v1 has no browser consumption ACK; already-posted task-queue
  snapshots cannot be retracted under severe main-thread stalls.
