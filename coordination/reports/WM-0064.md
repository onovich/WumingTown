# WM-0064 Work report

## Outcome

Implemented the M4 obligation and town-rule owner-store slice in `sim-core`.
The new stores own deterministic numeric facts for lampkeeper oil duty, lodging
witness duty, name confirmation and night-knock compliance, with bounded
candidate reads and structured reasons. Anomaly/crisis, director, Worker
protocol, save/schema, UI, package dependencies, broad governance/faction work
and M5 were not implemented.

## Files and public contracts changed

- `packages/sim-core/src/m4-obligation-types.ts`: public obligation constants,
  inputs, views, results, metrics and shared integer validators.
- `packages/sim-core/src/m4-obligation-store.ts`: authoritative obligation
  typed-array owner store, due lanes, fulfillment and violation transitions.
- `packages/sim-core/src/m4-town-rule-types.ts`: public town-rule constants,
  compliance context, results and metrics.
- `packages/sim-core/src/m4-town-rule-store.ts`: authoritative town-rule
  typed-array owner store, compliance buckets and structured compliance reasons.
- `packages/sim-core/src/m4-obligation-store.test.ts`,
  `packages/sim-core/src/m4-town-rule-obligation.test.ts` and
  `packages/sim-core/src/m4-town-rule-compliance.test.ts`: focused WM-0064
  tests.
- `packages/sim-core/src/index.ts`: exports the WM-0064 public API.
- `tools/test-runner.mjs`: adds `--filter m4-obligations`.
- `docs/02_systems/09_obligations_pacts_factions.md`,
  `docs/02_systems/10_ordinances_governance_identity.md` and
  `docs/05_tech/03_performance_budget.md`: record authority and performance
  notes.

## Acceptance evidence

- Obligation facts: tests assert creditor, debtor, type, condition, due window,
  visibility, inheritance, action, consequence, source event and state for oil,
  witness and name-confirmation obligations.
- Town-rule facts: tests assert subject scope, time, region, trigger, action,
  exception, enforcement, cost, legitimacy, penalty and state for name
  confirmation and night-knock compliance.
- Structured reasons: tests assert fulfillment, violation, due-index,
  candidate-cap, trigger/action-filtered compliance, stored exception,
  scan-cap, rejected context and enforcement-cost reasons.
- Bounded reads: due queries walk debtor due lanes with caps and deterministic
  due-end/id order, with `candidateCap` limiting returned due-window matches
  and `scanCap` limiting total inspected debtor rows. Compliance queries walk
  subject-scope/region/trigger/action buckets with candidate/scan caps and
  deterministic rule-id order; out-of-time rows do not consume candidate cap.
- Atomic failed mutations: tests assert invalid obligation/rule inputs,
  duplicate obligation registration and invalid terminal/compliance ticks do
  not publish owner version, active counts, due indexes or compliance indexes.
- No hidden mechanical text: tests assert obligation and town-rule views expose
  numeric structured fields rather than prose/presentation keys.

## Determinism, save, performance and security impact

Determinism: all fact lanes are integer typed arrays. Due lanes sort by
due-end/id; compliance lanes sort by rule id. No random, real-time or
presentation-only state participates in owner decisions.

Save: no save format, schema, migration, Worker protocol or replay envelope was
added.

Performance: actor-facing reads use caller caps and typed output buffers.
Obligation reads walk only the debtor due lane with explicit candidate and scan
caps. Town-rule reads walk only the subject-scope/region/trigger/action bucket
with explicit candidate and scan caps. Metrics expose actual inspected row
counts and indexed counts for review.

Security/authority: no DOM, React, Pixi, Electron, Node filesystem, ambient
time, package dependencies, UI or Worker changes were added.

## Decisions and deviations

- ADR-0009 is followed for the M4 obligation/town-rule owner-store slice only.
- `m4-town-rule-store.ts` is 472 formatted lines after the second review repair. It
  remains unified because the typed-array fact lanes, trigger/action bucket
  links, scan-cap/candidate-cap read loop, retire unlink path, context
  validation and exception evaluation share one owner-version/index invariant.
  The public types and focused compliance tests are split out; splitting the
  store now would add cross-file helper surface without reducing the review
  risk.
- The town-rule compliance context intentionally accepts numeric downstream
  inputs for trigger, action, knowledge, need, relationship, fear, enforcement
  risk, emergency, confirmed identity and obligation pressure because the
  dependent systems are not all integrated in WM-0064.

## Repair notes after changes_requested

- Compliance context now carries explicit trigger and action lanes. Registered
  rules are indexed into subject-scope/region/trigger/action buckets, so a
  name-confirmation rule cannot consume a night-knock query cap.
- Emergency and confirmed-identity context lanes now honor stored exception
  masks. Rows with `exception = 0` are still evaluated normally under emergency
  or confirmed-identity context.
- Obligation due queries now count only due-window matching rows against
  `candidateCap`, so earlier non-overlapping due rows cannot hide a later
  matching obligation.

## Repair notes after second changes_requested

- Obligation due queries now require `scanCap`, report actual inspected debtor
  rows in `visitedCount`, expose `scanCapHit`, and return
  `obligation_due_scan_cap_reached` for incomplete reads instead of
  `obligation_due_no_candidate`.
- Town-rule compliance context now requires `scanCap`; out-of-time rows are
  counted as inspected but do not consume candidate cap. Incomplete compliance
  reads return `town_rule_scan_cap_reached`.
- Added regressions for many non-matching due rows before a tail match,
  bounded no-match due reads, and an out-of-time town rule before a later
  in-time rule with `candidateCap = 1`.

## Remaining risks / follow-up tasks
