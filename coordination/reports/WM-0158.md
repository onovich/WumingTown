# WM-0158 Work report

## Outcome

Implemented a public, protocol-owned WM-0150 playable projection delivered on
Worker `UiDelta` messages. The projection is produced from the authoritative
`sim-core` playable read-model inside `sim-worker` and exposed to `apps/web`
through the reviewed package-root Worker session path. Web does not import
`sim-core`, does not deep import worker internals, and does not parse summaries
as gameplay state.

Reliable `CommandResult`, `SaveReady`, and `FatalSimulationError` delivery stays
separate from projection delivery.

## Public projection contract

Exact public field:

```ts
UiDeltaPayload.playable?: PlayableProjectionV1;
```

Exact public protocol types exported from `@wuming-town/sim-protocol`:

- `PlayableProjectionV1`
- `PlayableProjectionBasisV1`
- `PlayableCommandTemplateV1`
- `PlayableTargetActionProjectionV1`
- `PlayablePlacementProjectionV1`
- `PlayableOrderJobProjectionV1`
- `PlayablePawnProjectionV1`
- `PlayableBuildProjectionV1`
- `PlayableLampProjectionV1`
- `PlayableResourceRequirementProjectionV1`
- `PlayableResourceCountProjectionV1`
- `PlayableResourceProjectionV1`

Supporting command/basis types now exported for Web command construction:
`CommandBasis`, `CommandTargetRef`, `CommandBlockedReason`,
`CommandBlockedReasonCode`, `PlayerCommandPayload`, `PlayerCommandResult`,
`PrioritizeLampWorkPayload`, `QueueSimpleBuildPayload`, `ProtocolEntityRef`,
`CellRef`, `ResourceRequirementRef`, `StaleBasisRef`, and `PolicyRef`.

Projection fields:

- `playableCommandReadModelVersion`
- `basis.tick`, `basis.snapshotSequence`, `basis.worldHash`,
  `basis.readModelHash`, `basis.contentManifestHash`, `basis.targetVersion`,
  `basis.mapVersion`, `basis.reservationVersion`, `basis.jobVersion`,
  `basis.commandBasis`
- `targets[].target`, `targets[].targetState`, `targets[].targetVersion`,
  `targets[].actions[]`, `targets[].blockedReason`
- `placements[].blueprint`, `placements[].anchorCell`,
  `placements[].orientation`, `placements[].orientationOptions`,
  `placements[].footprint`, `placements[].interactionCells`,
  `placements[].valid`, `placements[].command`, `placements[].blockedReason`
- `orders[]`, `pawns[]`, `build`, `lamps[]`, `resources.materials[]`,
  `alerts[]`

`PlayableCommandTemplateV1` carries `commandKind`, `commandBasis`, `payload`,
`available`, and optional `disabledReason` so WM-0152 can send commands without
inventing authority.

## WM-0152 resume path

Use only the Web adapter and package-root public APIs:

```ts
import {
  WEB_PLAYABLE_WORKER_SCENARIO_ID,
  createWebSimulationWorkerSession,
  readWebPlayableProjection,
  sendWebPlayableCommandBatch,
  startWebPlayableWorkerScenario,
  type WebPlayableProjection,
} from "./simulation-worker-session";
```

Runtime path for WM-0152:

1. Create the session with `createWebSimulationWorkerSession`.
2. Start WM-0150 with `startWebPlayableWorkerScenario`.
3. Read projection data from incoming Worker messages with
   `readWebPlayableProjection(message)`.
4. Render HUD, placement, command chains, pawn/job/build/resource state from
   `message.payload.playable`.
5. Build `PrioritizeLampWork` and `QueueSimpleBuild` commands from
   `projection.basis.commandBasis` and the reviewed
   `projection.targets[].actions[].payload` /
   `projection.placements[].command.payload`.

Do not parse `summaries` for gameplay state. Do not import `sim-core`. Do not
deep import `@wuming-town/sim-worker` internals.

## Save/load stance

The playable projection is derived, read-only Worker output. WM-0158 adds no
public save compatibility, migration promise, in-progress command persistence,
release promise, telemetry, account, or public feedback surface.

## Limitations and follow-up

- WM-0158 does not implement product HUD, placement UI, renderer-pixi,
  ui-react, localization, or shell-bootstrap wiring.
- The projection is intentionally focused on the WM-0150 playable command slice.
- WM-0152 should resume by replacing reviewed playback reads with
  `readWebPlayableProjection` and command templates from the Worker projection.
- `coordination/thread-registry.json` was already dirty from project-director
  thread registration before implementation and was preserved as control-plane
  state.
