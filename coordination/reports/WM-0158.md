# WM-0158 Architecture report

## Outcome

Verified the second WM-0152 blocker. WM-0157 made Worker transport legal, but
the public Worker protocol still publishes only compact `RenderSnapshot` /
`UiDelta` hashes and summaries for the playable slice. The authoritative
WM-0150 runtime has an internal `PlayableReadModel`, but Web cannot legally
consume it without a public protocol projection. I created ADR-0014 and the
ready precursor task `WM-0158` to expose that projection.

## Files and public contracts changed

- `coordination/decisions/ADR-0014.md`: proposed the public playable projection
  contract over the existing Worker `UiDelta` path.
- `coordination/tasks/WM-0158.json`: created the implementation precursor task.
- `coordination/reports/WM-0158.md`: this architecture report.

No product UI code, WM-0152 task state, Worker runtime code, protocol code or
save code was changed.

## Determinism, save, performance and security impact

Determinism: no runtime behavior changed. ADR-0014 requires future projection
data to be derived from authoritative Worker/headless owner stores with stable
hashes and basis versions.

Save: no public save compatibility or migration promise was added. The
projection is derived read-only output; in-progress playable command
persistence remains unsupported unless a later versioned save task adds focused
sections and fail-closed migration evidence.

Performance: no runtime path changed. ADR-0014 requires the projection to stay
bounded for the focused slice, avoid global scans, and preserve reliable
message delivery even when projection payloads grow.

Security: no Electron, preload, filesystem, telemetry, account, store, release
or signing surface changed. The decision forbids Web deep imports and UI-owned
authority.

## Decisions and deviations

- Chose a protocol-owned `PlayableProjectionV1` carried by existing `UiDelta`
  instead of a new message family. This is the narrowest path because
  `CommandResult` remains reliable through the WM-0157 bridge, while `UiDelta`
  is already the read-only UI projection channel.
- Deferred a broad `WorldReadModel` Worker contract because it would couple the
  protocol to current Web presentation.
- Rejected parsing `UiDelta.summaries` and rejected direct Web access to
  `sim-core`.

## Remaining risks / follow-up tasks

- WM-0158 must choose final protocol type/field names and report the exact

- WM-0152 remains blocked until WM-0158 is reviewed and integrated.
- WM-0152 will still need to convert the public projection into Web/Pixi/HUD
