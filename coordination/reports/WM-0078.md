# WM-0078 Work report

## Outcome

Implemented `M5SeasonEventPoolStore` for the M5 first-season event pool. The
store owns event/recovery candidates, cooldown rows, recovery windows,
per-theme freshness, precondition failure diagnostics and pool owner version.
It schedules legal command/opportunity descriptors only; source owner facts
remain in anomaly, faction, governance, season, resource, Chronicle and other
stores.

## Acceptance evidence

- Owner store: `M5SeasonEventPoolStore` owns active event candidates, recovery
  candidates, cooldown rows, recovery windows, freshness and precondition
  failure diagnostics with an explicit owner version.
- Versioned basis: candidates carry anomaly, faction, governance, season,
  resource and recovery basis versions by value, plus source event def ids and
  stable ordering keys.
- Bounded stable selection: selection walks only the incident lane or exact
  active recovery-type lane with caller `candidateCap` and `selectedCap`.
  Stable order is score, priority, stable owner id, stable sequence and
  candidate id before deterministic random stream choice from the selected
  Top-K output.
- Legal commands only: candidates must map to reviewed command/opportunity ids.
  The store returns descriptors and never mutates source owner facts.
- Required content coverage: tests register resource pressure, registration
  pressure, market night, bridge-route pressure, archive damage risk and
  recovery opportunity candidates.
- Recovery boundaries: recovery windows suppress incidents and wrong-type
  recovery candidates remain in separate lanes.

## Review repair evidence

- Repaired active recovery-window lookup so selection first honors the newest
  active window, then falls back through the bounded recovery-window capacity
  when a newer overlapping window has expired but an earlier window remains
  active.
- Reordered selection reasons so concrete rejection diagnostics such as
  `m5_season_event_precondition_failed` take priority over the wrong-type
  recovery fallback.
- Added focused regressions for both reviewed findings: overlapping resource
  and archive recovery windows at tick 50, and a matching recovery candidate
  that fails its precondition inside an active recovery window.
