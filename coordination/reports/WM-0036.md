# WM-0036 Work report

## Outcome

Implemented bounded multi-pawn WorkOffer scoring for the focused M2 selection
surface. `WorkOfferIndex` can now score 20 actor query lanes from indexed
composite buckets with explicit candidate and selected-offer caps, stable
score/offer-id ordering, compact per-pawn output lanes and shared ReasonTrace
evidence. WorkOffer rows remain derived diagnostics and do not mutate jobs,
reservations, items, save state or Worker/UI surfaces.

## Acceptance evidence

- 20 actors query indexed buckets within caps: `work-offers.test.ts` selects
  across 20 pawn lanes with `candidateCap = 24`, `selectedCap = 12` and
  `ReasonTraceStore` capacity `64`; the direct benchmark reports 20 pawns,
  600 offers, 600 bucket candidates, 480 visited/scored rows and 240 selected
  rows.
- Deterministic stable ordering: tests register offers in descending order and
  assert equivalent high scores select lower offer ids first for every pawn;
  benchmark reports `equivalentScoreStable = true` and checksum `4117442084`.
- ReasonTrace evidence: traces record candidate totals, visited, scored, caps,
  selected offer id/score, candidate-cap rejections, selected-cap rejections,
  rejection mask and semantic reason. Tests assert newest trace fields; artifact
  reports `storedTraceCount = 20`, candidate-cap hits `20` and selected-cap
  hits `20`.
- No full-world scans: the selector reads exact composite buckets and stops at
  `candidateCap`; tests assert 480 total visits are less than the 12,000
  actor-by-all-offers scan equivalent. No map/entity scan code was added.
- Fixture-only content scope: benchmark/test data use synthetic fixture ids and
  do not add production work catalogs, economy rules, town-life policy or M3
  behavior.

## Determinism, save, performance and security impact

Determinism: selection order remains score descending with offer id tie-breaks.
The multi-pawn helper processes pawn lanes in caller-provided index order and
uses integer counters, typed arrays and deterministic checksum evidence.

Save/replay: no save schema, snapshot format, persisted cache, Worker protocol
or public compatibility promise changed. WorkOffer rows and traces remain
rebuildable diagnostics.

Performance: per-pawn visits are bounded by `candidateCap`; selected output is
bounded by `maxSelectedOffers`; trace storage is fixed capacity. The new
multi-pawn helper uses caller-owned typed scratch/output buffers. Some helper
methods exceed the 60-line guideline because they keep validation and aggregate
selection in one typed-lane pass without introducing extra hot-loop allocations;
future broader integration can split them if behavior expands.

Security/boundaries: `sim-core` remains free of DOM, React, Pixi, Electron,
Node filesystem, real time and ambient randomness. Node filesystem and timer
usage are confined to benchmark code, matching existing benchmark patterns.

## Decisions and deviations

ADR-0007 is followed: WorkOffer rows are derived, bounded, versioned candidates
and cannot own job, quantity, reservation, path, save or UI state.

The required `m2-work-offer-selection` test filter and
`m2-work-offer-20-pawns` benchmark filter are unsupported by the existing
registry CLIs. Adding those registry entries would require editing files outside
WM-0036 allowed paths (`tools/test-runner.mjs`, `packages/benchmarks/src/benchmarks.ts`
and `packages/benchmarks/src/cli-lib.ts`). The substitution is therefore the
repo-native `work-offers` focused test/benchmark plus the exported direct
WM-0036 benchmark artifact helper.

`coordination/artifacts/WM-0036/**` is outside explicit allowedPaths, but the
artifact was generated for reproducible review evidence. Older benchmark
artifacts created by required checks were restored to avoid dirtying older task
evidence.
