# WM-0120 Work report

## Outcome

Added a shell-local UI scale setting for the M8 Web/Desktop player shell,
expanded display settings to cover both language and scale, tightened
readability/non-color-only semantics, and extended automated evidence for
contrast, bilingual compact layouts, scroll reachability, and persisted display
preferences. The change stays inside the React shell layer and does not alter
Simulation Worker/headless authority, world state, or Pixi authoritative world
behavior.

## Files and public contracts changed

- `packages/ui-react/src/shell-ui-scale.ts`: adds shell-local UI scale
  preference state, storage key, persistence diagnostics, and scale factors.
- `packages/ui-react/src/shell-store.ts`: extends shell state with `uiScale`
  and adds shell display-setting actions.
- `packages/ui-react/src/shell-settings-panel.ts`: turns the panel into Display
  settings with localized language + UI scale controls, persistence status, and
  an explicit shell-only boundary note.
- `packages/ui-react/src/localization.ts`: adds zh-CN/en copy for UI scale,
  display-setting descriptions, and updated settings/onboarding copy.
- `packages/ui-react/src/shell-hud.ts`: applies shell-only display scaling,
  adds `data-ui-scale` markers, improves long-text wrapping, and adds aria/text
  signals for risk/status surfaces.
- `packages/ui-react/src/shell-main-menu-surface.ts`: routes settings actions
  through the expanded display settings panel and keeps compact layout bounds
  stable under scaled shell chrome.
- `apps/web/src/shell-locale.ts`: replaces locale-only controller logic with a
  combined shell settings controller that manages locale and UI scale as local
  preferences.
- `apps/web/src/shell-bootstrap.ts`: hydrates UI scale into shell state/debug
  payload and syncs display settings without touching world authority.
- `apps/web/src/web-shell.e2e.test.ts`: adds UI scale persistence/readability
  coverage, contrast checks, tall-settings reachability, and extends the Web
  responsive matrix timeout to match the heavier accessibility assertions.
- `apps/desktop-electron/src/desktop-shell.e2e.test.ts`: mirrors the new
  display-setting persistence, contrast, and compact viewport coverage in the
  sandboxed Electron shell.
- `packages/ui-react/src/shell-ui-scale.test.ts`: adds focused unit coverage for
  UI scale preference load/write behavior.
- `packages/ui-react/src/shell-hud.test.ts`: extends SSR shell coverage for the
  new display-setting copy.
- `coordination/reports/WM-0120-plan.md`: records the pre-implementation plan
  required by workflow.

Workflow-only files updated by `taskctl claim`:

- `coordination/tasks/WM-0120.json`
- `coordination/thread-registry.json`

## Acceptance evidence

- Text remains readable in zh-CN and en at required viewports:
  `apps/web/src/web-shell.e2e.test.ts` and
  `apps/desktop-electron/src/desktop-shell.e2e.test.ts` keep the full M8
  viewport matrix active, validate `zh-CN`/`en`, and assert compact/tall
  settings panels remain reachable rather than clipped.

- Important state is not color-only:
  Alerts, resident watch entries, night-risk badges, and settings persistence
  now expose explicit text/aria signals in addition to color. Web/Desktop
  accessibility baselines assert visible severity text such as `WARNING`,
  `STABLE`, and blocked/persistence status copy.

- UI scale or font-scale behavior is available and tested:
  `packages/ui-react/src/shell-ui-scale.ts` adds the shell-local scale model and
  `ShellSettingsPanel` exposes it as localized display settings. Web/Desktop
  E2E verify default `standard`, manual `large`, persistence across reload or
  restart, and compact-viewport usability after scale changes.

- Keyboard/mouse basics, contrast, long text and scroll regions are validated:
  Existing keyboard/mouse smoke remains intact in Web/Desktop accessibility
  baselines. New contrast checks validate representative shell surfaces. Long
  text now uses explicit wrapping styles and the tests assert tall scrollable
  settings/inspector regions remain reachable instead of requiring full-height
  fit.

- Does not degrade simulation authority or existing E2E:
  No `sim-core`/`sim-worker` files changed. UI scale is stored in local shell
  settings only and applied to the React HUD layer. Required Web/Desktop E2E
  still pass after the new assertions. `coordination/artifacts/WM-0118/**`
  generated during test runs were restored to `HEAD` and are not part of the
  WM-0120 submission.

## Determinism, save, performance and security impact

Determinism: none. No simulation tick logic, worker protocol, random stream, or
authoritative read-model generation changed.

Save: low and shell-local only. UI scale persists in a new local shell setting
key and does not change save schema, evidence envelopes, or Windows/Web save
interoperability guarantees.

Performance: low runtime risk in product code, moderate test-runtime impact.
The shell adds one extra local display preference and a scaled HUD wrapper.
Web responsive-matrix E2E became materially slower, so its timeout was raised
to preserve stable coverage rather than weaken assertions.

Security: none. Electron sandbox/preload constraints remain unchanged. No new
IPC, filesystem, telemetry, account, store, or paid-service surface was added.

## Decisions and deviations

- Chose a shell-local UI scale preference instead of touching Pixi/world zoom.
  This preserves the authoritative boundary and keeps the setting limited to
  player-facing React shell chrome.
- Kept `coordination/artifacts/WM-0118/**` out of WM-0120 scope. Tests
  regenerated those artifacts, but they were restored to `HEAD` because
  WM-0120 allowedPaths do not include them.
- No ADR required. No task-path expansion was needed.

## Remaining risks / follow-up tasks

- Web responsive-matrix coverage is now significantly slower. If more shell
  accessibility assertions are added later, the matrix may need splitting or a
  dedicated artifact-generation task to keep E2E wall-clock time predictable.
- UI scale currently targets the React shell layer only. If future work needs
  player-controlled world/canvas zoom accessibility, that should be a separate
  scoped task because it crosses different UX and input contracts.
