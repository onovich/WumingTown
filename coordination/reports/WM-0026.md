## Determinism, save, performance and security impact

Determinism: all new sim-core state uses integer lanes, stable ids and explicit
state machine steps. No real time, ambient randomness, Promise, Generator,
coroutine or closure execution state is used in `sim-core`.

Save/replay: no save container section is added in WM-0026. New stores expose
typed state through public APIs but persistence remains future Save Container
work.

Performance: storage refresh uses exact dirty queues and WorkOffer indexed
selection; pawn thinking does not scan item stacks or storage slots. Benchmark
`logistics-10k` records candidate caps, dirty backlog, final active claims and
quantity checksums.

Security: no runtime dependency, DOM, React, Pixi, Electron, Node filesystem or
Worker protocol surface is added to `sim-core`.

## Decisions and deviations

- ADR-0003: `ItemStackStore` is the only quantity owner; storage and WorkOffer
  availability are derived.
- ADR-0006: item quantities, reservation amounts and hauling steps are integer
  deterministic state.
- WM-0026 stays scoped to minimal hauling and storage. Build-site material
  sockets, build progress and completion remain WM-0027.
- File length guideline deviation: `hauling-jobs.ts` is 504 lines and
  `storage-logistics-index.ts` is 404 lines. They were kept cohesive for this
  first minimal owner/index/job slice so the public API remains narrow; a later
  cleanup can split helper validation/types after WM-0027 clarifies build-site
  buffers.

## Remaining risks / follow-up tasks

- `StorageLogisticsIndex` is a minimal one-stack-per-slot M1 surface. Broader
  storage filters, stack merging and production-order matching remain later
  logistics/economy work.
- Save/load snapshots for item/storage/hauling stores remain future
  Save Container work.
- Destination storage currently becomes supply when it contains delivered
  material; WM-0027 may need build-site material buffers that are not general
  supply offers.
