# WM-0176 Work report

## Outcome

Documented a fail-closed, approval-gated recovery procedure for an abandoned
per-task `taskctl` lock after the same transition fails twice. The procedure
keeps active locks authoritative, forbids automatic expiry and wildcard
deletion, and requires exact-path inspection plus explicit user approval.

## Files and public contracts changed

- `.agents/skills/wuming-town-agent-workflow/SKILL.md`: routes repeated task
  lock failures to the reviewed protocol procedure.
- `.agents/skills/wuming-town-agent-workflow/references/protocol.md`: defines
  diagnosis, rejection, approval, exact-file removal and retry rules.
- `coordination/tasks/WM-0176.json`: records the scoped documentation task and
  its workflow history.

No `taskctl` implementation or task-transition contract changed.

## Acceptance evidence

- The protocol requires two identical lock failures separated by a wait before
  recovery is considered.
- It requires all task agents and transitions to be idle, an exact resolved
  path inside `coordination/tasks`, a zero-byte current-format lock, timestamps
  predating both failures, a 60-second quiet interval, the expected task state
  and a clean scoped worktree.
- It requires explicit user approval and an exact `Remove-Item -LiteralPath`
  command. Wildcards, recursion, path traversal, symlinks, multi-lock removal
  and automatic expiry are forbidden.
- It preserves fail-closed behavior: uncertainty blocks recovery, and a failed
  post-removal retry requires a separate reviewed workflow repair rather than
  manual state mutation.

## Decisions and deviations

No ADR was required. The task deliberately documents manual recovery instead
of adding automatic stale-lock eviction because wall-clock age alone cannot
prove that a transition is abandoned.

## Remaining risks / follow-up tasks

depends on external agent/transition inspection. A future implementation change
to add owner metadata requires its own reviewed coordination task and tests.
