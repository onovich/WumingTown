# WM-0038 Work report

## Outcome

Expanded the focused storage/hauling core beyond the M1 fixture by adding
derived per-def storage supply and demand candidate lanes, M2-focused hauling
tests, explicit hauling failure/interruption cleanup, and a direct M2 logistics
hauling benchmark helper. Item quantities remain owned by `ItemStackStore` or
the carried job lanes; storage indexes remain derived from owner-state dirty
refresh.

## Files and public contracts changed

- `packages/sim-core/src/storage-logistics-index.ts`: added derived per-def
  supply/demand candidate lanes, bounded selection helpers, indexed candidate
  metrics, and exact-slot dirty synchronization.
- `packages/sim-core/src/hauling-jobs.ts`: added explicit `failed` hauling
  terminal state, fail/interruption APIs, terminal cleanup guards and rollback
  when returned carried items cannot be terminalized by JobCore.
- `packages/sim-core/src/hauling-jobs.test.ts`: added focused
  `m2-storage-hauling` tests inside the existing hauling suite for
  multi-source/multi-demand indexed selection, dirty drain, fail/interruption
  cleanup and cancel rollback.
- `packages/benchmarks/src/m2-logistics-hauling-benchmark.ts`: added direct
  focused benchmark helper for 12 source slots, 8 destination/demand slots and
  20 hauling jobs across delivery/cancel/fail/interruption terminal paths.
- `packages/benchmarks/src/index.ts`: exported the direct WM-0038 benchmark
  helper and report types.
- `docs/02_systems/04_ai_jobs_reservations_pathing.md`: documented WM-0038
  derived storage candidate lanes and hauling terminal cleanup rules.
- `docs/02_systems/06_economy_logistics_production.md`: documented storage
  owner/derived-index boundaries and exact-once hauling conservation.
- `docs/05_tech/03_performance_budget.md`: documented the M2 logistics hauling
  benchmark metrics and unsupported CLI-filter substitution.
- `coordination/reports/WM-0038.md`: this report.

No Worker protocol, save format, schema, package-boundary import, dependency,
UI authority, broad economy, crafting, balance table or production content
catalog change was introduced.

## Acceptance evidence

1. Multiple source stacks, storage destinations and demand slots update indexes
   from owner-state changes only: `StorageLogisticsIndex.refreshDirty` rebuilds
   derived supply/demand candidate membership from `ItemStackStore` and
   `ReservationLedger` reads. `hauling-jobs.test.ts` verifies wood and stone
   supply/demand selection across six slots and dirty drain after reservation,
   pickup and delivery.
2. Hauling jobs reserve source amount, destination capacity and interaction
   spots before pickup: existing reservation test still passes and WM-0038 did
   not weaken `reserveBeforePickup`; it atomically claims item quantity,
   capacity and two interaction spots before pickup.
3. Pickup, delivery, cancellation, fail and interruption paths conserve integer
   quantities exactly once: tests cover delivery, cancel after pickup,
   fail-after-pickup, interruption-after-pickup, and cancellation rollback when
   JobCore rejects terminalization. The M2 benchmark reports wood `48 -> 48`,
   stone `48 -> 48`, and zero final active claims.
4. Storage supply/demand dirty queues drain without sustained backlog growth
   under the M2 fixture: tests assert `dirtyBacklog: 0`; the direct M2 benchmark
   reports `dirtyBacklogPeak: 20` and `finalDirtyBacklog: 0`.
5. No broad economy, crafting, balance or content catalog expansion is
   introduced: changes are limited to allowed storage/hauling code, focused
   tests, benchmark helper, and docs.

## Review repair

The first independent review requested one blocking repair: if
`HaulingJobStore.reserveBeforePickup` acquired source quantity, destination
capacity and interaction spot claims, but `JobCoreStore.enterStep` rejected the
reserve transition, those newly acquired claims were not released before the
failure returned. The repair now releases the exact acquired claim ids, marks
the affected slots dirty, and keeps item quantities and carried lanes unchanged.
`hauling-jobs.test.ts` includes a regression that uses a tick accepted by
`ReservationLedger` but rejected by `JobCoreStore` to prove active claims return
to zero and acquired/released counts match `4/4`.

## Determinism, save, performance and security impact

Determinism: candidate lanes are ordered by ascending slot id within each def;
selection uses caller-provided caps and output buffers. No ambient randomness,
real time, Promise/Generator/coroutine job progress, or unstable ordering was
added to `sim-core`.

Save/replay: no save section, snapshot schema, public compatibility promise or
Worker protocol changed. The new storage candidate lanes are derived and
rebuildable from owner stores.

Performance: storage selection reads per-def linked lanes and stops at the
caller cap. Dirty work remains exact-slot queued. The direct benchmark records
candidate-cap hits, final dirty backlog 0 and conservation checksums. No pawn
think loop scans the full world.

Security/boundaries: `sim-core` remains free of DOM, React, Pixi, Electron,
Node filesystem, real time and ambient randomness. Node `performance` and
memory APIs are used only in `packages/benchmarks`, matching existing benchmark
patterns.

## Decisions and deviations

ADR-0007 is followed: `ItemStackStore` and `JobCoreStore` own quantities,
`ReservationLedger` owns claims, and `StorageLogisticsIndex` owns only derived
availability/candidate membership.

The task requested `m2-storage-hauling` and `m2-logistics-hauling` filters, but
the current repo registries do not support those names. Adding support would
require editing `tools/test-runner.mjs` and benchmark CLI registry files outside
WM-0038 allowed paths. Substitutions are documented above and use only allowed
paths.

No `coordination/artifacts/WM-0038/**` artifact is kept. The required
`logistics-10k` benchmark dirtied the older WM-0026 artifact, and that generated
side effect was restored before completion.
