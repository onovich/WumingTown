# WM-0163 Work report

## Outcome

`@wuming-town/sim-core` now exports a protocol-neutral authoritative
`GameSessionRuntime`, PR-1 `ScenarioDefinition` initializer and headless scenario
entry. The runtime owns fixed 30 TPS integer advancement, seed/named random
streams, stable command ordering, the minimum PR-1 owner graph, bounded indexed
work/path probes, one real bounded JobCore/Reservation lifecycle, exhaustive
canonical owner-fact world/read-model hashes, read-only projection basis builders,
metrics and checkpoint-aware structured conservation diagnostics. The fixed tick
path does not call allocating metrics or scheduled-needs result APIs. The exact
`pr1-integrated-gamesession` headless alias delegates to that package-root entry.

## Acceptance evidence

- Package-root runtime/initializer: `game-session.test.ts` creates the runtime
  exclusively through `initializeGameSessionRuntime` imported from `./index`.
- Initial world: tests verify 64x64, every cell passable, eight residents/eight
  need rows/eight beds, four item/storage lanes, one lamp, one build site, 26
  live entities, 21 map memberships and four container memberships.
- Fixed phases: every tick records exactly `GAME_SESSION_PHASE_ORDER`; phase 4
  creates/acquires and phase 5 enters, advances, completes and releases the
  minimal lifecycle. 100000 ticks report zero phase violations/global Pawn scans.
- Determinism: equal seed/command streams produce equal world/read-model hashes
  and metrics; same-tick command reordering produces a different world hash.
  Equal-version comfort-vs-social, resident activity/reason, food-vs-wood and
  resident-position mutations produce different world and read-model hashes.
- Projection: render and UI builders are read-only, share tick/snapshot/world/
  read-model/owner-version basis and expose 22 stable render rows/eight resident
  UI rows/four resource rows without protocol imports.
- Lifecycle: at tick 1, JobCore serializes `running/interact`, ReservationLedger
  serializes one cell claim and resident 0 is `working/currentJob=0`. At tick 3,
  JobCore is terminal completed, the claim is released and resident 0 is idle.
  Observational diagnostics accept the legal active state; terminal diagnostics
  alone classify it as a leak checkpoint.
- Conservation: the 100000-tick test queues commands at ticks 0 and 99999, records
  active job/reservation peaks `1/1`, terminal count `1` and acquired/released
  counts `1/1`, then ends with zero active jobs/reservations, pending commands/
  paths and every derived backlog; all four resources have delta zero. A forced
  food mutation reports
  `game_session.resource_conservation_failed` with tick, def id, expected and
  actual quantities.
- Allocation strategy: preallocated typed/path/lamp/job/reservation input scratch
  is created once. The unconditional per-tick path calls no result-object metrics
  API; dirty/work/path/job/environment APIs run only on bounded events. The
  100000-tick metrics report `unconditionalResultObjectCallCount=0`,
  `needScheduledUpdateCallCount=0` and 2757 event result calls. A dedicated quiet
  tick test proves the event-result count does not increase after dirty queues
  drain and no scheduled boundary is due. The runtime no
  longer calls `NeedStore.processScheduledUpdates`; the existing NeedStore remains
  the only needs owner and is changed only by explicit future gameplay events.
- CLI boundary: focused tests accept only `pr1-integrated-gamesession`, reject an
  extended unknown alias and call `runPr1IntegratedGameSessionScenario` from the
  sim-core package root. `cli.ts` remains unchanged.

## Determinism, save, performance and security impact

- Determinism: fixed 30 TPS integer ticks, stable numeric initialization,
  non-decreasing command ticks with insertion sequence tie-break, named seeded
  streams, versioned path basis and owner-fact canonical hashes. Hash publication
  includes map lanes, known locations, resident five-lane needs/activity/job/reason,
  item/storage, JobCore/Reservation records, WorkOffers, rest, environment, lamp,
  build, command entries and stream states. No real clock,
  `Math.random`, Node global, Promise/coroutine job state or hidden singleton.
- Performance: normal work selection touches one resident and one exact indexed
  bucket every 300 ticks, with one queued exact path and a 256-node cap. The
  100000-tick run reports 334 visits/paths, 6853 node expansions and zero global
  Pawn scans. No allocating `createMetrics` or scheduled-needs result call occurs
  on the unconditional tick path. Publication-time projection/hash allocation is
  outside hot tick loops.
- Save/load: no snapshot API, public save container, schema, migration or
  compatibility promise is added. Future save authority remains owner stores,
  tick, random streams and command tail; derived indexes/projections remain
  rebuild-only/read-only.
- Security/boundaries: sim-core imports no DOM, React, Pixi, Electron, Node file
  system, Worker protocol or Web code. The tool adapter imports only the sim-core
  package root.

## Decisions and deviations

- Follows `ADR-0017`; no protocol/schema/message/save change was required.
- Keystone changes request `MSG-MREUXRSP-CD16AE` is closed by the explicit
  JobCore/Reservation lifecycle, owner-fact hashing/collision suite and event-only
  allocation strategy. Diagnostics now distinguish observational from terminal
  checkpoints.
- Keystone's blocker fix `231956f` added only the exact headless registry/test
  paths. Implementation did not modify `cli.ts`, package metadata or other tools.
- The default build anchor is `(24,0)` so the existing focused `BuildSiteStore`
  anchor-cell convention remains coherent without modifying that protected
  historical store. General arbitrary-Y build placement is intentionally not
  claimed by this scaffold.
- An intermediate CLI run exposed Node strip-only rejection of TypeScript
  parameter properties. The two internal constructors were rewritten with
  explicit fields; final CLI and all gates pass.
- `packages/sim-core/src/runner.ts` and historical runner fixtures were left
  unchanged because the new package-root scenario entry advances the exact same
  `GameSessionRuntime` surface directly.

## Remaining risks / follow-up tasks

- Gameplay command semantics beyond deterministic `noop` ordering are not
  connected; the queue is a fixed 128-entry PR-1 scaffold, not a save contract.
- Hauling, build delivery/construction, lamp refuel, rest/eating/medical drivers
  and resource production/consumption are not scheduled. The one-shot cell-claim
  lifecycle proves owner integration without consuming resources or claiming a
  PR-2 gameplay driver.
- Health, ability, mood/thought, relationships, schedules beyond the minimum
  environment owner, Chronicle/evidence/obligation/town rules/crises/director,
  M5 content and M8 faction/endgame remain regression-only/unconnected.
- Normal wall-time scheduling, schema-v3 protocol mapping, reliable Worker
  delivery/backpressure and Web rendering are WM-0164/WM-0165 work. Sim-core
  intentionally owns no timer.
- No GameSession load/save implementation exists. Public compatibility remains
  owner-gated.
