# WM-0052 Work report

## Outcome

Implemented the focused M3 health condition owner store and derived ability
cache invalidation surface in sim-core. The slice covers deterministic injury
and illness condition rows, exact actor/ability dirty lanes, rebuildable
consciousness/movement/manipulation/sight/communication/stamina cache rows,
structured reason classes, deterministic hash evidence, and metrics for
condition updates, invalidations, queries, rebuilds, stale basis rejects, dirty
backlog and rebuild row visits. The changes-requested repair adds atomic dirty
queue overflow rejection for add/update/age/remove, terminal state range
validation, `ability.value_out_of_range` for invalid thresholds, and
`ability.cache_hit` for successful cache hits.

## Files and public contracts changed

- `packages/sim-core/src/m3-health.ts`: new pure sim-core owner/cache module for
  `M3HealthConditionStore`, `M3AbilityCacheStore`, constants, reason classes,
  metrics and deterministic hashes.
- `packages/sim-core/src/m3-health.test.ts`: focused headless tests for injury
  and illness rows, exact invalidation, stale cache prevention, update/remove,
  atomic overflow rejection, terminal state validation, invalid threshold
  reasons, deterministic hashes and no per-query full condition scan.
- `packages/sim-core/src/index.ts`: exports the new M3 health/ability API.
- `docs/02_systems/05_needs_health_mood_social.md`: documents WM-0052 owner
  store and ability cache behavior.
- `docs/02_systems/14_explainability_debug_observability.md`: documents
  structured reasons and metrics exposed by the slice.
- `coordination/tasks/WM-0052.json`: workflow claim/complete metadata only.

## Acceptance evidence

- Condition owner stores represent focused injury and illness instances:
  `m3-health.test.ts` stores left-leg sprain and mild cough rows with def, kind,
  body part, severity, age, source, clue/counterevidence, terminal state and
  versions.
- Derived ability cache covers all six M3 ability lanes:
  `M3_ABILITY_LANE_COUNT` and exported lane constants cover consciousness,
  movement, manipulation, sight, communication and stamina.
- Condition changes invalidate affected ability cache entries without every
  query scanning all conditions: condition writes enqueue exact actor/ability
  dirty keys; cache rebuilds use the actor condition linked lane. Tests assert a
  second valid query returns `ability.cache_hit` with `visitedConditionCount:
0`, while rebuilds visit only Yao's one condition even when another actor has
  a condition.
- Ability failures and condition changes emit structured reasons:
  tests assert `condition.injury_applied`, `condition.illness_applied`,
  `condition.updated`, `condition.removed`,
  `condition.dirty_queue_overflow`, `condition.terminal_state_out_of_range`,
  `ability.cache_invalidated`, `ability.cache_rebuilt`, `ability.cache_hit`,
  `ability.value_out_of_range` and `ability.rejected_below_threshold`.
- Focused tests cover add/update/remove, cache invalidation, deterministic hash
  and no stale ability state: covered by direct Vitest and
  `pnpm test --filter sim-core`.

## Determinism, save, performance and security impact

Determinism: all health and ability state uses integer lanes, stable condition
ids, stable per-actor linked ordering, explicit versions and deterministic hash
helpers. No real time, ambient randomness, DOM, React, Pixi, Electron or Node
filesystem APIs were added.

Save: no public save format, schema, migration or Worker protocol changes.
`createHash()` surfaces are focused replay/debug evidence only.

Performance: ability cache queries hit valid rows without condition visits and
rebuild from exact actor condition lanes after invalidation. Metrics expose
condition update count, ability invalidation count, ability query count, cache
hit/rebuild/stale counts, dirty backlog/peak and condition rows visited during
rebuild. Condition mutations preflight dirty capacity before owner writes, so
overflow rejects do not leave partial state or partial invalidation queues. The
implementation avoids full-world or full-condition scans in ability query logic.

Security: no dependencies, platform APIs, renderer paths or IPC surfaces were
added.

## Decisions and deviations

- ADR-0008 was followed: `HealthConditionStore` owns condition facts and
  `AbilityCacheStore` owns rebuildable cache rows only.
- Spark was not used because the task packet marks Spark eligibility false:
  deterministic simulation architecture work.
- `packages/sim-core/src/m3-health.ts` is 938 lines after formatting. This is
  larger than the preferred single-file guideline, but it keeps the tightly
  coupled first health/cache slice together and follows the existing typed-array
  store style used by larger sim-core modules such as `work-offers.ts`. A
  follow-up can split constants/types/store/cache if the reviewer wants smaller
  files.

## Remaining risks / follow-up tasks

- Medical care, treatment jobs, mood effects, read models, save/replay and
  Worker parity integration are intentionally not implemented in WM-0052.
- Public content definitions for condition defs/body parts are still numeric
  lanes in this focused sim-core slice; schema/content integration belongs to
  later approved tasks.
