# WM-0039 Work report

## Outcome

Added the minimal M2 build/order scaffold on top of the existing build-site and
JobCore state machines. Build orders now have a serializable derived order view,
reservation rollback is explicit when JobCore rejects post-acquire transitions,
policy denial has a structured reason, and a focused M2 work/logistics scaffold
scenario proves four deterministic build orders over a 20-actor fixture.

## Files and public contracts changed

- `packages/sim-core/src/build-site.ts`: added `BuildSiteOrderView`,
  `readBuildOrder`, `BuildSiteInterruptionInput`, policy-denial handling, and
  checked rollback of acquired reservation claim ids when JobCore transition
  rejects delivery/build reservation.
- `packages/sim-core/src/build-site.test.ts`: added focused
  `m2-build-production-orders` test names for order view/offer lifecycle,
  reservation rollback, policy denial, and direct M2 scaffold scenario.

  `m2.work_logistics.lantern_yard.v1`.
- `docs/02_systems/04_ai_jobs_reservations_pathing.md`: documented order
  state, post-acquire rollback and interruption policy boundaries.
- `docs/02_systems/06_economy_logistics_production.md`: documented the scoped
  build/order scaffold and owner-store quantity boundaries.
- `docs/02_systems/14_explainability_debug_observability.md`: documented the
  structured reason classes exposed by WM-0039.
- `coordination/reports/WM-0039.md`: this report.

No app, Worker, package manifest, lockfile, production catalog, balance table,
UI construction tool, ritual or investigation production system was changed.

## Acceptance evidence

1. Build/order demand registers and removes WorkOffers from owner-state changes:
   `BuildSiteStore.syncOffers` remains owner-state driven, and
   `readBuildOrder` exposes material/build offer active flags derived from the
   site lanes. `pnpm test --filter building` verifies material offers appear
   while sockets are empty, build offers appear after sockets fill, and all
   offers are removed after deterministic completion.
2. Required inputs are reserved before work starts and serializable progress is
   explicit: delivery reserves item quantity, build-site material capacity and
   interaction spots before pickup; build jobs reserve site entity and
   interaction spot before ticking. Build progress remains in integer
   `buildProgressTicks` and JobCore progress lanes; `BuildSiteOrderView`
   materializes those lanes as serializable numbers/booleans.
3. Completion commits deterministically through existing command/tick phases:
   the direct M2 scenario completes four build orders with final world hash
   `0xf2a38685`, zero active reservations, zero active offers and zero running
   jobs at tick `20000`.
4. Structured reasons distinguish required failure classes:
   focused tests and scenario summary cover
   `material.insufficient_required_amount`, `target.invalid_state`,
   `site.blocked`, `reservation.destination_capacity_conflict`,
   `path.no_route_to_destination` and `policy.interruption_denied`.
5. Scope remains limited to the M2 vertical slice: the scenario consumes wood
   and stone sockets for four road-lantern frame orders and preserves a
   binding-paper stack as decoy material. Full source-to-depot, three-material
   build order and long-run benchmark closure remain later M2 work.

## Determinism, save, performance and security impact

Determinism: all new scenario state is fixed by integer constants and explicit
tick values. No ambient randomness, real time, Promise/Generator/coroutine job
position or unstable ordering was added.

Save/replay: no save format, Worker protocol or public compatibility promise
changed. `BuildSiteOrderView` is a derived read view suitable for later save
diagnostics; it is not an authority or persisted derived cache.

Performance: the scaffold does not add pawn thinking loops or global work
selection. Offer sync remains explicit per owner site and the M2 helper uses a
bounded four-site fixture.

Security/boundaries: `sim-core` remains free of DOM, React, Pixi, Electron,
Node filesystem, real time and ambient randomness. No new dependencies were
introduced.

## Decisions and deviations

ADR-0007 remains the authority: JobCore owns driver state, ReservationLedger
owns claims, ItemStackStore owns source quantities, BuildSiteStore owns
delivered site buffers/progress, and WorkOffer rows are derived.

The task-required `m2-build-production-orders` test filter and
`m2-work-logistics` headless CLI scenario are not registered in files allowed by
WM-0039. This task therefore keeps the runtime helper and focused tests in
allowed paths and documents the runner substitutions for a later task whose
scope permits runner/CLI registry edits.
