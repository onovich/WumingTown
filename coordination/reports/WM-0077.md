# WM-0077 Work report

## Outcome

Implemented M5 faction and governance hooks as deterministic sim-core owner
stores:

- `M5FactionFactStore` owns decomposed faction fact lanes for legal stance,
  trade stance, debt stance, legitimacy, fear/memory, memory events and known
  claims.
- `M5GovernanceHookStore` owns policy hook lanes for council post authority,
  temporary policy authority, enforcement capacity, legitimacy source and risk
  flags.

Both stores use typed arrays, owner versions, versioned source facts, caller
caps and stable lane ordering. They expose numeric query results for downstream
event legality, town-rule pressure and anomaly context without mutating
TownRule, Obligation, Chronicle, UI or text-template authority.

## Acceptance evidence

- Faction facts remain decomposed: tests register legal, trade, debt,
  legitimacy, fear/memory, memory-event and known-claim rows, assert per-lane
  metrics and assert exported views have no `mood`, `attitude`, UI, text or
  diplomacy authority fields.
- Versioned source facts: faction rows require source event, source owner,
  Chronicle owner and obligation owner versions; governance rows require
  town-rule, obligation, Chronicle and source owner versions.
- Bounded versioned queries: faction queries walk only one faction lane and
  governance legality queries walk only one policy lane. Both reject stale
  expected owner versions before traversal and report candidate/scan cap
  reasons.
- Governance legality remains a hook output: `evaluatePolicyHooks` returns
  selected hook ids, enforcement capacity, legitimacy score, policy pressure,
  risk flags and structured legality reasons. It does not mutate town rules,
  obligations, Chronicle facts or UI state.
- No monolithic political director: stores expose explicit owner facts and
  metrics only; strategy/event systems consume numeric query results later.

## Review repair evidence

Gate requested changes because faction and governance query cap detection used
"any later lane row exists" as a candidate-cap hit, even when later rows did
not match the query mask, subject, score or active window. The repair keeps
query traversal inside caller `scanCap` after filling the selected output; it
reports `candidateCapHit` only when another matching row is found, and reports
`scanCapHit` when the scan cap is exhausted before that can be proven.

Regression coverage: `does not report cap hits from nonmatching tail rows`
builds a faction lane with one matching legal row followed by a nonmatching
debt row, and a governance policy lane with one matching enforcement hook
followed by a nonmatching risk hook. Both queries use `candidateCap: 1` and
assert they return indexed/allowed results rather than false cap-hit reasons.

## File-size note

`packages/sim-core/src/m5-governance-store.ts` is 487 lines and
`packages/sim-core/src/m5-faction-governance.test.ts` is 446 lines. I kept the
governance store as one file in this task because policy-lane ownership,
legality selection, pressure/risk aggregation, stale-basis rejection and
metrics are tightly coupled and mirror the existing owner-store style. The
focused test keeps faction and governance hook behavior together because the
review repair covers matching-row cap-hit semantics across both stores. The
shared public contracts are split into `m5-faction-governance-types.ts`; the
next natural split is a shared M5 lane/helper module if later M5 season/event
stores reuse the same query and metrics mechanics.
