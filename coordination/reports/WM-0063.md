# WM-0063 Work report

## Outcome

Implemented the first authoritative M4 Chronicle evidence fact surface in
`sim-core`: case files with version history, structured sources/evidence/
hypotheses/contradictions/confirmed rules, bounded deterministic support-tier
evaluation, exact dissemination dirty keys, resident automatic-action gates and
compact evidence reason traces. Obligations, town rules, anomaly/crisis,
director, Worker protocol, save/schema, UI, new dependencies and M5 work were
not implemented.

  append-only version history.
- `packages/sim-core/src/m4-evidence-facts.ts`: structured evidence,
  hypothesis, contradiction and confirmed-rule owner store.
- `packages/sim-core/src/m4-evidence-dissemination.ts`: resident knowledge

## Acceptance evidence

- Case and evidence ownership: tests create a case file, source rows,
  hypotheses and version history, then assert version sequences and owner
  metrics.
- Structured evidence tiers: support tiers are computed from integer evidence
  lanes and deterministic ordering. Tests assert replay-equivalent support
  results and trace rows.
- Confirmation gates: tests prove confirmed rules require configured support,
  independent evidence classes and no unresolved fatal contradiction; resolving
  the contradiction then permits confirmation.
- Resident action boundary: tests prove hypothesis exposure is not actionable,
  while known confirmed rules and explicit temporary policies are actionable.
- Dissemination dirty keys: tests drain exact resident/subject dirty keys in
  stable sequence order.
- Bounded candidate reads: tests register 40 evidence rows and assert support
  evaluation visits 32 candidates with a 32-row cap, reports cap hits and does
  not scan every row.
- Repair coverage after reviewer changes: tests assert failed Chronicle
  recording leaves no local owner rows/counts/dirty keys/status/confirmed rules
  or support metrics for case open/close, source, hypothesis, evidence,
  contradiction, contradiction resolution, confirmation and dissemination
  grants. Tests also assert `evidence_case_mismatch` for cross-case evidence and
  contradiction contamination, plus invalid resolve/confirmation ticks without
  a recorder.

## Determinism, save, performance and security impact

Determinism: support ordering is stable by integer score, evidence class and
stable evidence id. Confirmation uses integer support thresholds, independent
class counts and fatal-contradiction counters.

Save: no public save format, schema, codec, migration, platform save UI or M4
focused save/replay harness was added.

Performance: normal support reads walk only a hypothesis evidence lane with
caller caps. Dissemination changes enqueue exact resident/subject dirty keys.
Metrics expose evidence row counts, support candidate visits, confirmed-rule
count and dirty dissemination backlog.

Repair note: confirmation now computes support without publishing support
metrics or traces until Chronicle recording and local owner-version capacity
preflight succeed. Failed recorder paths leave `lastSupportCandidateVisits`,
`totalSupportCandidateVisits`, confirmed-rule rows and trace rows unchanged.

Security/authority: no apps, UI, Electron, Worker protocol, sim-worker, Node
filesystem, real time or ambient randomness were added to `sim-core`.

## Decisions and deviations

- ADR-0009 is followed for the Chronicle evidence owner-store slice only.
- `m4-evidence-facts.ts` is 725 formatted lines after the atomicity repair. It remains a single file
  because the typed-array lanes, mutation ordering, support gate and
  contradiction counters share tight invariants; public types and scoring
  helpers were already split out, and repair-specific tests were split into
  `m4-chronicle-atomic.test.ts`. Further splitting would obscure the owner-store
  transaction order during this first gate review.
- Added `evidence_case_mismatch` as a public structured reason so cross-case
  source/hypothesis/evidence contamination is rejected without prose-only
  interpretation.
- Spark was not used because WM-0063 is marked not Spark-eligible.

## Remaining risks / follow-up tasks

- The stores are not yet integrated into Worker read models, focused
  save/replay, anomaly/crisis, director, UI or M4 scenario orchestration; those
  are explicitly downstream tasks.
- Version history is append-only owner evidence for this slice, not a public
  save/migration format.
- Broader benchmark baselines remain for WM-0070; WM-0063 adds metrics and
  focused tests only.
