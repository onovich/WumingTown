# WM-0167 Work report

## Outcome

The PR-1 authoritative lifecycle now exposes deterministic non-instant `moving -> working -> idle` state across the unchanged 100 ms / 3-tick Worker publication quantum. The implementation keeps `JobCoreStore` and `ReservationLedger` as the owners of execution and claims, and uses the existing serializable `step`, `stepEnteredTick`, `stepTickCount`, and Q16 progress fields rather than a second cursor or wall-clock state in `sim-core`.

The Worker scheduler now converts monotonic elapsed wall time into integer 100 ms quantum debt. It dispatches at most 10 quanta per interval callback, retains remaining debt up to a 60000 ms cap, records discarded excess time for diagnostics, and flushes catch-up outputs through the existing latest-wins/reliable outbox. The public protocol, schema 3, GameSession projection v1, save stance, 100 ms interval, 3-tick quantum, and speed semantics were not changed.

## Runtime architecture repair

- Tick 0 acquires the existing work offer and reservation, creates JobCore job 0, enters `path_to_source`, and keeps resident 0 in authoritative `moving` activity with structured `game_session.job_reserved` reason.
- Movement advances with `JobCoreStore.tickJob(jobId, tick, 0)`. At publication tick 3 the serialized movement count is 2; at tick 6 it is 5. Tick 6 then enters `interact`, resetting the owner-controlled step counter without adding hidden state.
- Interaction advances six integer Q16 work ticks. Publication tick 9 has `stepEnteredTick=6, stepTickCount=2`; tick 12 has count 5. Tick 12 completes the job, releases the reservation, and publication tick 15 is terminal `idle` with `game_session.job_completed`.
- The fixed phase order remains unchanged: phase 4 acquires the selected job/reservation and phase 5 advances the explicit lifecycle. No coroutine, Promise, closure-only cursor, real clock, global scan, or UI authority was added.
- `globalThis.performance.now()` is read only in `game-session-scheduler.ts`. The hot callback mutates scalar counters and invokes the pre-existing callback; diagnostics allocate only when `createDiagnostics()` is explicitly requested.

## Scheduler and backpressure policy

- Constants remain `GAME_SESSION_SCHEDULER_QUANTUM_MS=100` and `GAME_SESSION_TICKS_PER_QUANTUM_AT_SPEED_1=3`.
- Normal cadence dispatches one quantum per callback. A delayed callback dispatches no more than `GAME_SESSION_MAX_CATCH_UP_QUANTA_PER_CALLBACK=10`; undispatched integer debt remains for later callbacks.
- Debt is bounded by `GAME_SESSION_MAX_WALL_TIME_DEBT_MS=60000`. The focused sequence 550 ms -> 3050 ms -> 100000 ms dispatches 5, 10, and 10 quanta, leaves 59000 ms debt, records 38500 ms discarded excess, and reports 25 total dispatched quanta.
- A passive fake-monotonic 600000 ms normal-speed run dispatches 6000 scheduler quanta and advances the real `GameSessionWorkerHost` to tick 18000 without command/drain/advance helpers. This is deterministic scheduler evidence, not the real Chromium 600-second evidence owned by WM-0168.
- Paused cadence is consumed without advancing the world. After 1000 ms paused, resume advances only the next 100 ms quantum; requested speed remains host-owned and retains 0/30/60/90 TPS semantics.
- Catch-up quanta are enqueued before one flush. A maximum 10-quantum focused batch observes 11 stale render drops (including the pre-catch-up command publication), exactly one retained render and UI projection, a bounded reliable depth of 6, and strictly increasing worker sequence across CommandResult/Alert and projection delivery. Existing pause tests retain CommandResult, SaveReady, and FatalSimulationError; accepted Shutdown and idempotent disconnect clear scheduling exactly once. A simulated callback during `clearRepeating` is ignored after the scheduler first marks itself stopped.
