# WM-0082 Work report

## Outcome

Implemented M5 Worker read-only projections for
`m5.alpha_content_framework.first_season.v1` and proved Node Worker/browser
Worker parity against the full M5 headless replay oracle at the reviewed
checkpoint stream. The Worker reuses the existing protocol message families
and emits M5 content, anomaly, faction/governance, season, validation, review
and projection basis metadata as read-only summaries without adding UI repair
authority, platform save UI, public protocol redesign, dependencies or M6
scope.

## Acceptance evidence

- Node Worker and browser Worker consume the same M5 checkpoint command stream:
  `0`, `3600`, `7200`, `12000`, `18000`, `36000`. `pnpm test --filter
  m5-worker-parity` compares Node Worker snapshots to full headless replay
  checkpoint hashes, and `pnpm test:e2e --filter worker-smoke` repeats the
  stream through a real browser module Worker.
- Read-only projection metadata includes content manifest hash `0xe55d3015`,
  anomaly roster, faction/governance, season-events, content-validation,
  review, read-model and projection basis summaries. The final authoritative
  world hash is `0x9a4a905c`, authoritative read-model hash is `0x57eba2b7`
  and final projection hash is `0xc6420cb1`.
- Stale projection basis is rejected with structured reason
  `InvalidPayload` and detail `stale M5 projection basis: content manifest hash
  does not match active Worker content basis`. The test requests UI detail
  after stale rejection and confirms the Worker remains read-only with the
  authoritative read-model hash unchanged.
- No new Worker message family or public protocol field was added. M5 data is
  carried through existing `RenderSnapshot`, `UiDelta`, `MetricsSample` and
  `SaveReady` optional metadata already used by focused Worker scenarios.
- `coordination/artifacts/WM-0082/m5-worker-parity-diagnostics.json` records
  final snapshot bytes `179`, final read-model bytes `1073`, final projection
  bytes `1960`, `firstMismatchedCheckpointTick: null`, 18 projection messages
  and 17 basis summaries.

## Determinism, save, performance and security impact

- Determinism: Worker checkpoint tables are pinned to the full M5 headless
  replay oracle for requested seed `5` / authoritative seed `155`. Tests
  compare world/read-model hashes at every checkpoint and canonical
  projection/detail/surface hashes at the final checkpoint.
- Save: Worker `SaveReady` exposes focused M5 save metadata only: scenario id,
  checkpoint tick and world hash. No public save format, platform save UI or
  migration promise was added.
- Performance: diagnostics record projection byte sizes and message counts.
  Benchmark baseline updates remain WM-0083 scope.
- Security: UI and browser Worker surfaces stay read-only. The content-schema
  loader repair avoids bundling static Node filesystem access into browser
  Workers while preserving Node-only file loading behavior behind explicit
  async loader calls.

## Decisions and deviations

- ADR-0010 remains followed: mutable M5 state is not made client-authoritative;
  Worker output is a read-only projection over reviewed M5 content and owner
  surfaces.
- Public Worker protocol was not changed. The implementation reuses existing
  message kinds and optional payload metadata.
- Scope clarification: the task allowed paths were expanded to include
  `packages/content-schema/src/content-fixtures.ts` and
  `packages/content-schema/src/m5-content-pack-files.ts` after the required
  browser Worker smoke exposed static `node:fs/promises` imports in browser
  bundles. The repair is limited to moving those imports into Node-only async
  loader functions.
- `packages/sim-worker/src/m5-worker-parity.test.ts` is 451 lines because it
  keeps the focused command builders, diagnostics writer, stale-rejection
  assertion and save metadata assertion together for reviewer reproduction.
  Product code added by WM-0082 stays below the 400-line warning threshold.
- No Spark lane was used; WM-0082 is protocol-adjacent Worker parity work and
  the task packet marks it ineligible for Spark.
