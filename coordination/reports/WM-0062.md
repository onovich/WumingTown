# WM-0062 Work report

## Outcome

Implemented the first M4 authoritative lamp rule-field surface in sim-core:
linked lamp groups, integer maintenance/fuel/wick/damage/`humanClaim`/
`shadowGap` lanes, exact lamp dirty keys, read-only visual projection basis
validation, active lamp-gap indexing, bounded Top-K risk reads, metrics, and
structured reason codes. Chronicle, obligations, anomaly/crisis, director,
Worker protocol, save/schema, UI authority, dependencies, and M5 scope were not
implemented.

- `packages/sim-core/src/m4-lamp-network.ts`: new authoritative lamp owner
  store, dirty queue, read-only visual projection, metrics and structured
  reasons.
- `packages/sim-core/src/m4-lamp-gap-index.ts`: new active lamp-gap candidate
  index, bounded query path, trace rows and metrics.
- `packages/sim-core/src/m4-lamps.test.ts`: focused WM-0062 tests for dirty
  propagation, stale projections, reason codes, indexed Top-K reads and
  no-full-map normal dirty work.
- `packages/sim-core/src/index.ts`: exports the M4 lamp store/index public
  sim-core API.
- `tools/test-runner.mjs`: adds `--filter m4-lamps`.
- `docs/02_systems/03_map_space_rooms_lanterns.md`: records WM-0062 lamp
  authority and projection boundary.
- `docs/05_tech/03_performance_budget.md`: records WM-0062 lamp dirty and gap
  index metrics.
- `coordination/reports/WM-0062.md`: this report.

## Acceptance evidence

- Lamp network authority: `M4LampNetworkStore` owns group membership,
  maintenance, fuel, wick, damage, `humanClaim`, and `shadowGap` typed integer
  lanes. `pnpm test --filter m4-lamps` asserts owner reads and deterministic
  replay-equivalent mutations.
- Read-only visuals: `createVisualProjection()` returns read-only derived
  brightness with owner-version basis; `rejectVisualProjectionMutation()` and
  `validateProjectionBasis()` reject UI/projection authority and stale bases.
- Exact dirty keys: lamp mutations enqueue exact lamp, group, cell, room, chunk
  and projection keys. Tests assert deterministic dirty drain output and
  bounded drain metrics.
- Indexed risk reads: `M4LampGapIndex` refreshes exact dirty lamp ids and
  queries active gap candidates with candidate/selected caps. Tests assert
  Top-K ordering, dirty-backlog rejection, and no all-lamp query path.
- No full-map normal diffusion: tests assert one changed lamp drains one
  lamp/cell key, `fullMapDiffusionCount` stays `0`, and load/debug rebuild
  scans are reported separately from normal dirty refresh.

## Determinism, save, performance and security impact

Determinism: dirty queues and gap index ordering are stable by integer score and
lamp id. Tests compare replay-equivalent mutation and dirty-drain output.

Save: no public save format, schema, codec, migration, platform save UI, or
focused save/replay harness was added.

Performance: normal lamp dirty work drains caller-bounded exact lamp ids.
Active gap queries use candidate and selected caps. Metrics expose dirty
backlog peak/final, drained key counts, active gap count, query visits, refresh
counts, and load/debug rebuild scan counts.

Security/authority: no apps, UI, Electron, Worker protocol, or sim-worker path
was edited. Visual projections are derived and cannot mutate lamp rule fields.

## Decisions and deviations

- ADR-0009 is followed for the lamp owner-store slice only. Chronicle,
  obligations, anomaly/crisis, director, Worker protocol and save/schema
  surfaces remain downstream work.
- Project-director pre-review hardening tightened owner-store transaction
  safety after owner completion: dirty-queue capacity and owner-version
  exhaustion are checked before state mutation, dirty drain writes caller-
  preallocated output rows in place, and the focused tests assert queue-full
  failures do not partially register a lamp. `pnpm typecheck`,
  `pnpm test --filter m4-lamps`, and `git diff --check` were re-run after this
  hardening.
- Reviewer changes requested on WM-0062 and the fixes were applied in the same
  allowed M4 lamp files:
  - gap-index dirty queue overflow now marks the derived index as requiring a
    rebuild; `refreshDirty()` no longer publishes the store owner version as
    current after a missed dirty key, and queries reject with
    `lamp_gap_index_rebuild_required` until `rebuildFromStore()` runs;
  - `M4LampGapIndex` now maintains global, group, room, and group-room sorted
    candidate buckets, so scoped actor/anomaly-style reads apply candidate caps
    inside the requested scope instead of filtering a global capped list;
  - selected-cap-only truncation is recorded through `selectedCapHit` and
    `trace.selected_cap_reached`.
  Focused tests now cover all three reviewer findings.
- `M4LampGapIndex.rebuildFromStore()` may scan lamp capacity for load/debug
  rebuild and reports `fullRebuildLampScans`; normal dirty refresh and risk
  queries do not use that path.
- `m4-lamp-network.ts` and `m4-lamp-gap-index.ts` exceed the usual 400-line
  preference because WM-0062 keeps each typed-array owner surface self-contained
  with its public types, validation, metrics, and deterministic link helpers.
  Further splitting is deferred until a second M4 consumer appears, to avoid
  obscuring ownership during the first gate review.
- Spark was not used because WM-0062 is explicitly not Spark-eligible in the
  task packet.
