# WM-0152 Work report

## Outcome
WM-0152 remains blocked after resuming against the new public Worker projection path. The Web shell now issues real WM-0150 lamp/build commands through the public Worker bridge and receives authoritative `CommandResult` acceptance/rejection, but the authoritative lamp/build chain cannot progress past `claimed` / `moving` / `0%` because the current public browser Worker surface does not provide a legal, documented way for `apps/web` to advance the playable scenario clock.

The investigation confirmed that `SetSpeed` and `Pause` are not sufficient: after command acceptance, the Worker projection remains at `tick: 0` indefinitely. The only existing advancement path in `sim-worker` is an internal convention where a `PlayerCommandKind.Noop` command with command id `wm0150.advance.<tick>` triggers `scenario.runtime.advanceTo(tick)`. That helper exists in `@wuming-town/sim-core`, not in the allowed/public surfaces for this task, and the task explicitly forbids importing `sim-core`. Implementing that magic string directly from `apps/web` would bypass the missing public package boundary instead of resolving it.

## Decisions and deviations
- I did not implement a browser-local fake runtime or import `sim-core`, per task constraints.
- I did not hardcode the internal `wm0150.advance.<tick>` `Noop` command-id convention into `apps/web`, because that convention is currently only exposed through `@wuming-town/sim-core` internals/tests and is not a documented/public API for the allowed package boundaries.
- I verified that `pause(false)` and `setSpeed(1)` are not enough. In `packages/sim-worker/src/index.ts`, those messages only mutate Worker state and emit `CommandResult` / `MetricsSample`; they do not advance playable tick or emit progressive `UiDelta` snapshots on their own.

## Remaining risks / follow-up tasks
- Exact missing public surface: a legal public way for `apps/web` to advance authoritative WM-0150 playable time after command submission.
- Recommended option: expose a root-level public helper or session method from `@wuming-town/sim-worker` or `@wuming-town/sim-protocol`, for example `createPlayableAdvanceCommandId(tick)` or `advancePlayableScenarioToTick(session, tick)`, so Web can pump authoritative time without importing `sim-core` or copying undocumented string formats.
- Option 2: make the public browser Worker session honor `setSpeed` / `pause` with an actual browser-side authoritative tick loop. Impact: larger `packages/sim-worker` behavior change; requires explicit ownership of clock semantics and test coverage.
- Option 3: formally document the `PlayerCommandKind.Noop` playable-advance contract in `sim-protocol` and export the helper there. Impact: smaller surface than a session clock, but still a public protocol decision that needs architect approval.
