# WM-0152 Work report

## Outcome

Blocked before product-code changes. WM-0152 requires a real authoritative
command path from `apps/web` to the Simulation Worker, but the current reviewed
public surfaces do not provide one. `apps/web` still uses the WM-0151 reviewed
projection harness, and the only browser Worker entry for `sim-worker` is an
internal file rather than a public package export. Per ADR-0012 and the task
packet, I stopped instead of faking shell-local authority.

  and blocker threshold.

## Determinism, save, performance and security impact

Determinism: none changed. Blocking preserves the existing Worker/headless-only
authority model.

Save: no new save/load promise was added. Existing M6 shell save behavior
remains unchanged and is explicitly insufficient for authoritative playable
command persistence.

Performance: none changed. No new UI polling loop, Worker bridge, or renderer
object churn was introduced.

Security: blocking avoids an illegal deep import or UI-owned authoritative
runtime. Electron, preload, network, telemetry, accounts and store surfaces are
unchanged.

## Decisions and deviations

- Followed `coordination/decisions/ADR-0012.md`: React/Pixi/Web may build
  commands from read-model basis and render read-only outcomes, but may not
  invent authority or use local adapters as a substitute.
- Followed the task packet's critical instruction: WM-0151 ended as reviewed
  projection playback, not live Worker integration, so this task must stop if a
  legal public Worker path is absent.
- Deviation from the ideal implementation path: no product-code wiring was
  attempted because the missing boundary decision occurs before UI work.

Required public decision:

1. Recommended: expose a reviewed public browser Worker client/entrypoint for
   the authoritative session and explicitly allow `apps/web` to consume it.
   Impact: creates the narrowest truthful path for WM-0152 and later Web/Electron
   command tasks; requires systems-architect approval and a follow-up task that
   can touch the relevant package manifests/exports.
2. Alternative: expose the same capability through an already-allowed package
   such as `@wuming-town/platform` so `apps/web` does not depend on
   `@wuming-town/sim-worker` directly.
   Impact: keeps `apps/web` boundaries smaller, but still requires a reviewed
   new public adapter contract and follow-up implementation task.
3. Alternative: split a precursor task that adds the public Worker/session
   bridge plus focused save/load stance first, then resume WM-0152 on top of
   that bridge.
   Impact: slower, but keeps architecture and product wiring separately
   reviewable.

## Remaining risks / follow-up tasks

- If `apps/web` deep-imports `packages/sim-worker/src/browser-worker-entry.ts`
  or locally recreates the runtime, it will violate package-boundary and
  authority rules.
- Even after a Worker bridge is approved, save/load truth still needs an
  explicit stance for authoritative in-progress command state instead of the
  current M6 gate envelope.
- Build placement UX still needs a reviewed read-model/action-availability
  surface for hover validity and blueprint placement basis data; that should be
  carried by the same public session bridge decision.
