# WM-0152 Work report

## Outcome

Blocked again before product-code changes. WM-0157 solved the first WM-0152
blocker by exposing a public browser Worker session bridge from
`@wuming-town/sim-worker`, so `apps/web` can now send real WM-0150 commands
without deep imports. The resumed investigation found a second blocker: the
public Worker protocol still does not deliver a full authoritative playable
projection object that Web can render. Per ADR-0012 and the task packet, I
stopped instead of replacing WM-0151 reviewed playback with another shell-local
projection layer.

## Determinism, save, performance and security impact

Determinism: none changed. Blocking preserves the existing Worker/headless-only
authority model and avoids UI-side projection authority.

Save: no new save/load promise was added. Existing M6 shell save behavior
remains unchanged and is explicitly insufficient for authoritative playable
command persistence.

Performance: none changed. No new UI polling loop, Worker bridge, or renderer
object churn was introduced.

Security: blocking avoids both UI-owned authoritative runtime behavior and a new
unreviewed protocol shim. Electron, preload, network, telemetry, accounts and
store surfaces are unchanged.

## Decisions and deviations

- Followed `coordination/decisions/ADR-0012.md`: React/Pixi/Web may build
  commands from read-model basis and render read-only outcomes, but may not
  invent authority or use local adapters as a substitute.
- Followed the task packet's critical instruction: WM-0151 ended as reviewed
  projection playback, not live Worker integration. WM-0157 fixed transport,
  but not public authoritative projection delivery, so this task still cannot
  replace the playback honestly.
- Deviation from the ideal implementation path: no product-code wiring was
  attempted because the missing reviewed protocol surface occurs before HUD,
  placement, and save/load UI work.

Required public decision:

1. Recommended: extend the public Worker protocol/session surface so Web
   receives an authoritative playable projection object for WM-0150, including
   basis fields, job markers, pawn states, build progress, resources/alerts,
   and placement/action availability.
   Impact: unlocks WM-0152 honestly and matches ADR-0012's intended UI wiring,
   but requires a reviewed `sim-protocol` + `sim-worker` follow-up task outside
   WM-0152 allowed paths.
2. Alternative: keep the top-level message families but add a reviewed
   `RequestUiDetail`/`UiDelta` payload variant for the focused playable slice.
   Impact: narrower than a broad render snapshot contract, but still requires
   `sim-worker` implementation work and explicit protocol review.
3. Alternative: create a separate precursor task that exposes the authoritative
   playable read-model to Web, then resume WM-0152 on top of that projection
   surface.
   Impact: slower, but keeps protocol/runtime review separate from product UI
   work.

## Remaining risks / follow-up tasks

- If `apps/web` now synthesizes motion/progress from the old reviewed harness
  while presenting Worker command acceptance as live authority, it will violate
  the task packet and ADR-0012 even though transport is real.
- Even after protocol projection is added, save/load truth still needs an
  explicit stance for authoritative in-progress command state instead of the
  current M6 gate envelope.
- Build placement UX still needs a reviewed authoritative action-availability
  and basis surface for hover validity and blueprint placement data.
