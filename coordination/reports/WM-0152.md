## Determinism, save, performance and security impact
- Determinism: the Web shell now uses the legal public deterministic advance adapter instead of guessing the internal `wm0150.advance` format.
- Save: explicit non-support for authoritative runtime restore remains in place.
- Performance: the remaining blocker is related to sustained live Worker pumping / browser-automation stability, not to UI-owned world mutation.
- Security: unchanged; no `sim-core` import was added to Web, and no deep Worker internal import was used.

## Decisions and deviations
- I did not import `sim-core` or hardcode `wm0150.advance`, per task constraints.
- I kept the live Worker projection path and used only the WM-0159 public advance adapter.
- I paused the deterministic pump while build placement mode is active so the player does not issue stale-basis build commands during targeting.
- I added page-hide teardown for the Worker/pump resources, but that was not sufficient to make the required E2E complete reliably.

## Remaining risks / follow-up tasks
- Exact remaining blocker: the public browser Worker/Web adapter surface still lacks a reliable authoritative “terminal/drain” contract for live playable command progression. Web currently has to poll debug/read-model state while pumping advance requests, and under Playwright that loop still causes the required command-flow test to time out.
- Recommended option: expose a public `sim-worker` session helper that advances the playable scenario and resolves when a new authoritative `UiDelta` or terminal command state is ready, or a public terminal/idle event that Web/E2E can await without hot polling.
- Option 2: expose a public authoritative “command terminal” reliable message for playable orders so HUD/E2E can stop pumping based on protocol events rather than debug payload inspection.
- Option 3: expose a public session “drain/quiesce” or graceful shutdown contract so browser tests can stop the advance loop deterministically before context/browser close.
