# WM-0150 Work report

## Outcome

Implemented the authoritative headless/Simulation Worker command-to-job slice
for reviewed lamp-priority and simple-build commands. The slice now validates
typed playable commands, creates explicit serializable order/job/pawn state in
sim-core, advances by deterministic ticks, returns reliable per-command
results, and projects read-only job markers, pawn state, build progress,
structured blocked reasons, replay hashes and performance counters.

Changes requested by independent review were addressed: rejected duplicate
lamp commands no longer mutate authoritative/read-model state, reviewed basis
validation now rejects stale snapshot/target/map/reservation/job versions, and
schema-v2 batch rejection payloads now expose `batchAccepted:false`,
`batchReason`, and `commandResults`.

## Determinism, save, performance and security impact

Determinism: fixed 30 TPS evidence, no real time or ambient randomness, stable command log hashing and deterministic tick progression. The Worker advances WM-0150 only through explicit Noop advance ids or typed commands.

Save: no public save compatibility or migration is claimed. The focused runtime records replay evidence in summary output; save/load support remains a future reviewed task if UI integration needs persisted playable command tails.

Performance: pawn/job selection is bounded to the focused two-pawn candidate set; the read model records candidate visits, cap hits, path requests, dirty backlog and projection bytes. No pawn think loop scans the world.

Security/UI authority: no React, Pixi, Electron, preload, filesystem, telemetry, account, store, signing or public-release surface changed. UI remains a read-model consumer and command sender only.

## Decisions and deviations

- Implemented schema version 2 because ADR-0012 required a schema bump for incompatible playable command/result payloads.
- Kept Noop/Echo support for existing reviewed deterministic advance/parity tests; playable commands require basis and payload validation.
- Kept the first focused runtime in one sim-core module to make the vertical slice auditable end-to-end. The file exceeds the normal 400-line preference; split it into `types`, `runtime` and `scenario` modules if this surface grows beyond WM-0150.
- No Spark/rapid-implementer lane was used; this task touches protocol and Worker behavior and is outside the Spark classifier.

## Remaining risks / follow-up tasks

- UI/Pixi integration must consume these read-only projections and must not recreate job state locally.
- Public save migration/compatibility for playable command tails is still not implemented or promised.
- The focused slice has minimal build balance only; material economics, lamp repair/refill balance and larger worker selection remain future reviewed tasks.
- WM-0150 implements bounded focused candidates, not a general town-wide work scheduler.
