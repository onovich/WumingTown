# WM-0134 Work report

## Outcome
WM-0134 hardened the current Web shell player-facing localization slice without changing simulation authority, Worker payloads, protocol contracts, or save schema. The default player HUD and the main visible fixture identity fields now render through a narrow UI-side presentation adapter for the current Web shell fixture/read-model, `shell-hud.ts` no longer hardcodes player-visible connector/input prose, and localization validation plus tests now fail if the current audited fixture strings leak back into `zh-CN` unchecked.

## Files and public contracts changed
- `packages/ui-react/src/shell-read-model-localization.ts`: added a non-authoritative bilingual presentation adapter and validator for the current Web shell fixture/read-model fields plus localized player input-feedback formatting.
- `packages/ui-react/src/shell-hud.ts`: localized player-visible fixture-backed HUD strings, removed hardcoded connector prose from player surfaces, and routed player input feedback through localization keys.
- `packages/ui-react/src/shell-main-menu-surface.ts`: localized the visible fixture-backed settlement/cycle/phase identity shown on the default start surface.
- `packages/ui-react/src/localization.ts`: added player-surface composition/input keys used by the HUD instead of hardcoded prose.
- `packages/ui-react/src/index.ts`: exported the fixture localization validator utilities.
- `packages/ui-react/src/shell-hud.test.ts`: added regression coverage for zh-CN fixture-backed HUD localization and localized input feedback using a local minimal read-model slice, without importing app-internal fixtures across workspace boundaries.
- `apps/web/src/web-shell.e2e.test.ts`: added zh-CN HUD English-leak assertions and updated start-surface expectations to the locale-specific settlement-name presentation.
- `tools/validate-localization.mjs`: extended localization validation to audit the current Web shell fixture/read-model presentation mapping.
- `coordination/reports/WM-0134-plan.md`: execution plan for this task.

No sim/worker/protocol/save public contract changed. The new adapter is explicitly presentation-only and limited to the current Web shell fixture/read-model slice.

## Acceptance evidence
- Chinese browser/system candidates default to `zh-CN`; non-Chinese candidates default to `en`:
  confirmed by existing locale-resolution coverage in `packages/ui-react/src/localization.test.ts` and by `apps/web/src/web-shell.e2e.test.ts` `defaults locale by browser language and persists manual display overrides across reload`.
- Manual language switching persists without save schema or Worker mutation:
  confirmed by the same `web-shell` e2e locale-persistence case; no save/protocol code changed.
- Player-visible strings in default HUD/menu/settings/actions/feedback/onboarding use localization keys or an explicit locale adapter:
  `shell-hud.ts` now routes current fixture/read-model player prose through `shell-read-model-localization.ts`, and player-visible HUD connector/input strings go through `localization.ts` keys.
- Missing zh-CN/en player keys fail tests; English diagnostics remain isolated behind debug surfaces:
  `tools/validate-localization.mjs` now audits both the shared catalogs and the current Web shell fixture adapter; `wmDiagnostics=1` debug overlay remains the only intentional English diagnostics surface.
- Hardcoded player-facing string audit is updated and reviewer-verifiable:
  this report documents the player-facing fixture/read-model fields now covered by the UI adapter, plus the remaining future-content gaps listed below.

## Hardcoded player-facing string audit
- Covered in the current UI-side fixture adapter:
  settlement name, phase/cycle/speed/map labels, alert labels/details, resource labels/units, entity display names, role labels, summaries, current jobs, current steps, mood/health labels, last-decision text, explainers, thoughts, need labels, and player input-feedback patterns such as `Canvas inspect`, `Canvas select`, `Wheel zoom`, `Keyboard ...`, `Ready`, and `Loaded ...`.
- Kept intentionally outside player localization scope for this task:
  debug overlay / diagnostic storage surfaces behind `wmDiagnostics=1`, release-gate copy, and other `dev.*` English diagnostics.
- Still left for later broader content/read-model i18n work:
  future Worker/read-model payload fields that are not part of the current Web shell fixture, any new content prose added outside the audited fixture map, and a long-term shift from source-string adapter matching to structured content ids/reason codes.

## Determinism, save, performance and security impact
Determinism: none. No simulation logic, tick order, random stream, or authoritative data path changed.

Save: none. Locale preference remains app-shell state only; save schema, export/import payloads, and Worker state were not modified.

Performance: low and bounded UI-only impact. The new adapter runs only in the React presentation layer for the current fixture-backed shell surfaces; no sim hot path or Pixi world update loop changed.

Security: none. No Electron preload/API surface, IPC, filesystem exposure, telemetry, accounts, or hosted-service boundary changed.

## Decisions and deviations
- Followed `docs/05_tech/12_m8_i18n_locale_contract.md` and `ADR-0011` by keeping localization as presentation state only.
- Implemented the allowed narrow UI-side non-authoritative adapter instead of changing protocol/read-model schema.
- Extended the validation gate to the current fixture-source audit instead of pretending this solves all future content localization.
- Review fix: removed the reverse `@wuming-town/ui-react` test dependency on `apps/web/src/product-gate-fixture` and replaced it with a local minimal fixture-like read-model slice inside `packages/ui-react/src/shell-hud.test.ts`, so the regression stays in package ownership and boundary checks pass.
- Verification refreshed WM-0118 artifacts during `web-shell` e2e (`coordination/artifacts/WM-0118/web/player-hud-en-2560x1369.png`, `coordination/artifacts/WM-0118/web/start-surface-zh-CN-1424x861.png`, `coordination/artifacts/WM-0118/web/debug-overlay-en-1424x861.png`, `coordination/artifacts/WM-0118/web-responsive-layout.json`); all were restored to `HEAD` because they are outside WM-0134 allowedPaths.
- The task packet requires `corepack pnpm test --filter web-shell`, but the repository’s current `test` script is unit-only and rejects `web-shell` as an unsupported filter. I recorded that failure and ran the actual supported `web-shell` e2e entry (`node tools/test-runner.mjs e2e --filter web-shell`) instead so reviewer evidence still exists.

## Remaining risks / follow-up tasks
- The new adapter is intentionally limited to the current Web shell fixture/read-model strings. It is not a substitute for future structured content ids or broader content-catalog localization.
- New fixture/read-model prose added later will need to be added to the adapter or replaced by structured ids; the validator now blocks silent drift for the current fixture only.
- Debug overlay / release-gate / storage-diagnostic English remains intentionally outside player-surface localization scope; if any of that copy moves onto default player surfaces later, it must be localized then.
- The repository-level `corepack pnpm test --filter web-shell` mismatch should be cleaned up separately so task packets can reference a supported command without manual substitution.
