## Acceptance evidence

- Node headless and browser Worker run the same hauling/building command stream to the same periodic world hash: `worker-smoke.e2e.test.ts` compares Worker snapshots at ticks `0`, `2400` and `100000` against `runM1HaulingBuildingReplay`.
- Save at an interruption point, load, rebuild indexes and resume produces the same final hash as uninterrupted replay: `m1-save-replay.test.ts` saves at tick `2400`, validates/rebuilds, resumes to `100000`, and compares final world/read-model hashes.
- Snapshot and UI detail remain read-only projections and cannot mutate authoritative state: projections contain copied hash/detail payloads only; `m1-save-replay.test.ts` mutates an external UI projection copy and verifies resumed authority hashes are unchanged.
- Diagnostics output seed, scenario id, first divergent tick and artifact paths on failure: `compareM1ReplayRuns` and `pnpm sim:replay-test` report these fields; generated artifacts include `coordination/artifacts/WM-0010/m1-save-replay/summary.json`.

## Determinism, save, performance and security impact

Determinism: checkpoint hashes are derived from the M1 scenario runner with stable tick lists and deterministic command ids. Save: this is a minimal WM-0028 M1 envelope, not a full Save Container v1 compatibility promise. Load validates unknown input fail-closed before returning rebuilt derived indexes. Performance: no normal tick save container work or pawn scan was added; replay/save operations are harness boundary work. Security: sim-core remains free of DOM, React, Pixi, Electron, Node fs, real time and `Math.random`; artifact writing stays in the tool layer.

## Decisions and deviations

ADR-0004 was followed by keeping snapshots/UI detail as read-only projections produced by Worker/headless authority. ADR-0005 was scoped to a minimal M1 envelope with the ADR section names needed for hauling-building parity; full save compatibility, section migrations, platform save UI and public save slots remain out of scope.

## Remaining risks / follow-up tasks

- Worker M1 mode is a focused parity harness, not the full future Worker runtime for all simulation systems.
- Full Save Container v1, migration matrix, benchmark closeout and M1 closeout remain WM-0029/WM-0030 or later scope.
