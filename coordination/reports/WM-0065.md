# WM-0065 Work report

## Outcome

Implemented the narrow M4 borrowed-shadow anomaly/crisis owner surface in
`sim-core`. The slice owns activation candidates, explicit crisis state
machine progress, low-risk evidence traces, escalation, non-combat resolution
and terminal failure reasons. Worker protocol, save/schema, UI, director,
broad anomaly catalog, broad combat/security, package dependencies and M5 were
not implemented.

## Files and public contracts changed

- `packages/sim-core/src/m4-borrowed-shadow-types.ts`: public numeric
  constants, activation basis inputs, crisis views, trace views, mutation/query
  results, metrics and structured reason type.
- `packages/sim-core/src/m4-borrowed-shadow-crisis.ts`: typed-array owner
  store for borrowed-shadow candidates, crisis state machine rows and bounded
  trace ring.
- `packages/sim-core/src/m4-borrowed-shadow-crisis.test.ts`: focused WM-0065

- `docs/02_systems/08_anomalies_rules_investigation.md` and
  `docs/05_tech/03_performance_budget.md`: record authority and performance
  notes.

## Acceptance evidence

- Activation depends on explicit lamp gap and Chronicle identity basis fields:
  tests assert low lamp gap and confirmed identity evidence both prevent
  activation while preserving candidate state.
- Crisis progress is serializable and explicit:
  `activated -> trace -> escalated -> resolved|failed` is represented by
  numeric state, tick, terminal reason, resolution method and owner-version
  fields.
- Low-risk evidence is recorded before escalation: tests reject escalation
  before trace evidence and then accept escalation only after a low-risk
  evidence row.
- Non-combat resolution is supported through containment and negotiation
  commands with numeric lamp-gap, identity and score inputs.
- Candidate reads are bounded and versioned: tests assert deterministic
  score/id order, caller caps, visited counts, cap-hit reason and owner version.
- Terminal cleanup is explicit: tests assert failed crises leave active count at
  zero and reject later progress with a structured terminal-state reason.
- Save-shaped state: tests assert crisis views are numeric JSON-stable records
  without coroutine, Promise, closure or UI-local progress fields.

## Repair notes

- `resolveCrisis` now accepts resolution only from
  `M4_BORROWED_SHADOW_STATE_ESCALATED`; resolution attempts from `ACTIVATED`
  return `borrowed_shadow_resolution_requirements_unmet` and leave state,
  counts and owner version unchanged.
- `failCrisis` now rejects irreversible `M4_BORROWED_SHADOW_TERMINAL_HARM`
  unless the crisis has reached `M4_BORROWED_SHADOW_STATE_ESCALATED`. HARM from
  `ACTIVATED` or `TRACE` leaves state and counts unchanged, while
  `M4_BORROWED_SHADOW_TERMINAL_ABORTED` remains available as cleanup from an
  activated crisis.
- Activation-basis validation now rejects version fields outside positive
  uint32 range, identity support tiers outside the approved Chronicle evidence
  tier range, and identity independent class counts outside integer `0..4`
  before any typed-array row is published.

## Determinism, save, performance and security impact

Determinism: all owned state is numeric typed-array data. Candidate ordering is
stable by lamp gap score descending, then candidate id ascending. No random,
real time, prose probability or presentation-only state participates in owner
decisions.

Save: no save format, schema, migration, Worker protocol or replay envelope was
added. The crisis and trace views are deliberately save-shaped numeric records
for future schema work.

Performance: normal activation reads consume explicit candidate rows only and
walk the sorted candidate lane with caller `candidateCap` and `selectedCap`.
They do not scan all lamps, evidence rows, obligations, residents or map cells.
Crisis progress writes O(1) state rows and bounded trace-ring entries.

Security/authority: no DOM, React, Pixi, Electron, Node filesystem, ambient
randomness or ambient time was added to `sim-core`.

## File length note

`packages/sim-core/src/m4-borrowed-shadow-crisis.ts` is 816 lines. I kept it as
one file for WM-0065 because it is one cohesive typed-array owner store: the
candidate index, crisis state machine, trace ring, validation and reason
encoding share private invariants and atomic mutation ordering. Splitting now
would either expose extra internal helpers or duplicate validation logic. If the
surface grows in a later task, the natural split is candidate index, crisis
state transitions and trace ring modules.

## Residual risks

- Direct coupling to WM-0062/63/64 stores is intentionally avoided. The crisis
  store consumes their approved facts as numeric basis structs with owner
  versions; future integration must ensure those structs are produced only by
  approved owner surfaces.
- Low-risk evidence is emitted as crisis trace/evidence request state, not
  Chronicle authority. A future task must explicitly bridge accepted requests
  into Chronicle mutations if needed.
