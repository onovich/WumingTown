# WM-0076 Work report

## Outcome

Implemented `core.anomaly.old_bridge_guest.v1` as a rostered M5 anomaly rule
with a dedicated sim-core owner store. The implementation keeps immutable
anomaly rule metadata in the M5 roster and keeps mutable old-bridge crossing
progress in `M5OldBridgeGuestCrisisStore`.

The old-bridge owner store models route/prepared-item activation candidates,
safe-passage prevention, low-risk evidence, escalation, non-combat reciprocity,
reroute or obligation-settlement resolution, failure, review rows and terminal
cleanup as explicit serializable numeric state.

## Acceptance evidence

- Serializable owner store: crisis state is numeric and explicit:
  `activated -> trace -> escalated -> resolved|failed`. Crisis views include
  bridge window, prepared-item/logistics basis, route basis, terminal reason,
  cleanup pending, resolution method and owner version fields.
- Source authority boundary: activation rows carry route, prepared item,
  logistics, bridge ledger, Chronicle, obligation, faction and season owner
  versions by value. The store does not import or mutate logistics, route,
  item, obligation, faction or season owner stores.
- Safe-passage checks: activation is prevented when the bridge window is active,
  route basis is passable and a prepared item exists for another actor rather
  than a self-serving toll. Unsafe crossings activate through structured
  reasons without scanning source logistics.
- Stale basis rejection: candidate writes and reads reject mismatched roster
  version or content manifest hash; required route/prepared/logistics/Chronicle/
  obligation/faction/season basis versions must be positive.
- Low-risk evidence: tests record bridge ledger, missing prepared goods, route
  delay, merchant testimony and old-family oral record trace rows before
  escalation.
- Non-combat resolution: tests cover reciprocity, reroute and obligation
  settlement. Terminal resolution/failure writes review rows and appends crisis
  ids to a bounded terminal-cleanup lane.
- Bounded reads: activation candidates and terminal cleanup are queried with
  caller caps. Tests assert stable Top-K ordering, candidate-cap metrics and
  terminal-cleanup cap-hit metrics.
- No UI/coroutine authority: tests assert exposed crisis state has no UI,
  Promise, coroutine, closure or logistics-scan progress fields and round-trips
  through JSON.

## Review repair evidence

Gate requested changes because `drainTerminalCleanup` unlinked pending terminal
cleanup rows before owner-version preflight, so `old_bridge_version_exhausted`
could return after mutating the cleanup lane. The repair makes terminal cleanup
two-phase: it first walks the lane read-only to count selected rows and detect a
cap hit, then obtains the next owner version, and only then clears output,
unlinks rows and updates metrics. If the owner version is exhausted, the method
returns without changing the output buffer, cleanup-pending flags or cleanup
cap-hit metrics.

Regression coverage: `preserves terminal cleanup rows when owner version is
exhausted` creates two resolved crises, forces owner-version exhaustion, drains
with cap 1 and asserts both cleanup rows remain pending, the caller output
buffer remains unchanged and the cap-hit metric is not incremented.

## File-size note

`packages/sim-core/src/m5-old-bridge.ts` is 1224 lines and
`packages/sim-core/src/m5-old-bridge.test.ts` is 485 lines. I kept the store as
one file for this task because activation basis validation, safe-passage
prevention, crisis transitions, trace/review rings and terminal cleanup lanes
are tightly coupled and mirror the reviewed M5 crisis-store pattern. The public
types are split into `m5-old-bridge-types.ts`; the next natural split is shared
M5 anomaly ring-buffer and numeric-validation helpers once old-bridge,
third-knock and later M5 owners converge on identical code.
