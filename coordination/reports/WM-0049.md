# WM-0049 Work report

## Outcome

Implemented deterministic M3 day/night and basic weather owner state inside
`sim-core`. The change adds tick-derived schedule windows, named-stream or
command-forced weather, exact environment dirty keys, versioned read-only
projections, structured reason codes, focused replay evidence, and owner
metrics. It does not wire these facts into Worker protocol, save format,
schemas, UI, M4 systems, or downstream needs/mood/work systems.

Changes-requested repair: day/night and scheduled weather now catch up elapsed
transition boundaries deterministically when a caller jumps across multiple
ticks. Weather changes also dirty `schedule.rest` for rest/sleep consumers.

- `packages/sim-core/src/m3-environment.ts`: `DayNightStore`,

## Acceptance evidence

- Day/hour derive only from tick and documented constants:
  `m3-environment.test.ts` asserts tick `0`, `3_000`, `16_500`, `19_500`, and
  `TICKS_PER_DAY` map to dawn/daytime/evening/night/next-day dawn without
  changing time constants.
- Weather basics use named seeded streams and stable scheduling:
  tests assert `weather:m3-ordinary-life`, matching snapshots for same seed,
  four scheduled draws by tick `12_000`, and no stream draw for a command-forced
  weather transition at tick `3_000`.
- Environment context outputs schedule, need, mood, work, and explanation
  inputs only through owner-state projections: tests assert
  `needRateModifierMilli`, `moodContextCode`, outdoor-work eligibility, and
  structured reasons.
- Projections are read-only and versioned: tests assert frozen environment,
  day/night, weather, and severity projections, plus version composition from
  schedule and weather owner versions.
- Focused replay and structured reasons: `runM3EnvironmentReplay` produces
  identical results for the same seed and command stream and rejects unsorted
  command streams with `environment.command_stream_not_sorted`.
- Changes-requested regression coverage: tests assert jumped day/night
  advancement matches stepped advancement for final projection, schedule
  version, transition count, and dirty keys; jumped scheduled weather matches
  stepped scheduled weather including `transitionTick`; and a weather-only
  change drains `schedule.rest`.

## Determinism, save, performance and security impact

Determinism: day/hour/window derive from integer ticks and constants;
scheduled weather uses `NamedRandomStreams.nextInt` with
`weather:m3-ordinary-life`; command weather is explicit scenario input. No
`Math.random`, real-time clocks, platform timers, DOM, React, Pixi, Electron,
or Node filesystem APIs are used.

Save/replay: no public save format, schema, migration, or Worker protocol was
changed. The focused replay helper is in-memory evidence only and returns
versioned read-only projections.

Performance: owner metrics report day/night updates, window transitions,
scheduled weather updates, command transitions, stream draws, projection
builds, dirty backlog, and dirty backlog peak. The focused test evidence
records 4 scheduled weather draws at tick `12_000`, zero stream draws for the
command-forced primary scenario weather path, and a peak dirty backlog of 6
exact environment keys when schedule and weather context change at tick
`3_000`. The catch-up loops process only elapsed schedule/weather boundaries,
not entities, map cells, or downstream consumers. Baseline benchmark updates
remain deferred to WM-0059.

Security and authority: UI/read-model consumers receive frozen projections and
cannot create authoritative weather or time state through this API. No new
dependency or platform surface was added.

## Decisions and deviations

- Implemented the ADR-0008 WM-0049 owner stores only; no downstream needs,
  mood, schedule eligibility index, work selection, Worker, save, visual
  weather, M4, schema, dependency, or lockfile scope was added.
- Weather commands at the same tick as a scheduled weather boundary apply as
  explicit scenario authority without first consuming a named stream draw.
- Scheduled weather transitions record their scheduled boundary tick, not the
  caller's target tick when a tick batch jumps across intervals.
- `pnpm test --filter m3-environment` is unsupported by the repository wrapper,
  so direct Vitest file execution was used as the valid focused test command.
- Spark was not used; WM-0049 is marked not Spark-eligible because it touches
  deterministic simulation and random-stream design.
