# WM-0093 Work report

## Outcome

WM-0093 adds a local M6 diagnostic package path for the Web and Windows
product-gate shells without telemetry, upload, private paths, secrets or full
save contents. Web can download `wuming-town-m6-diagnostics.json`; Windows
Electron records an explicit `windows_host_diagnostics_bridge_blocked` blocker
instead of exposing broad filesystem, shell, dialog or arbitrary IPC authority.

## Files and public contracts changed

- `packages/platform/src/diagnostics.ts`: adds the M6 diagnostic package
  builder, JSON encoder and redaction rules for paths, secrets and
  full-save-like fields.
- `packages/platform/src/diagnostics.test.ts`: covers package evidence shape,
  privacy flags, Windows blocker state and redaction.
- `packages/platform/src/index.ts`: exports the diagnostics API from the
  platform public entry point.
- `apps/web/src/diagnostic-package-gate.ts`: adds Web shell diagnostic package
  assembly, local download, renderer error capture and diagnostic status dock.
- `apps/web/src/shell-bootstrap.ts`: wires diagnostics into the Web shell debug
  payload without making UI or Electron authoritative.
- `apps/web/src/web-shell.e2e.test.ts`: downloads and inspects the package,
  verifying M5/M4 hashes, safe logs, structured errors and redaction.
- `apps/desktop-electron/src/desktop-shell.e2e.test.ts`: verifies Electron dev
  and packaged shells expose the safe diagnostic blocker and keep telemetry and
  network upload disabled.
- `tools/test-runner.mjs`: registers the required `diagnostics` test filter.
- `docs/05_tech/05_testing_observability_ci.md`: records WM-0093 diagnostics
  behavior and privacy rules.
- `coordination/tasks/WM-0093.json`: claim state and narrow tool-runner path
  ownership update.
- `coordination/reports/WM-0093-plan.md`: execution plan.

No simulation authority, Worker protocol, save format, preload bridge, runtime
dependency, lockfile, telemetry, account, public feedback or M7 scope changed.

## Acceptance evidence

- Local diagnostic package captures build, platform, scenario id, hashes,
  recent structured errors and safe logs: Web e2e downloads the package and
  verifies `WM-0093`, platform host data, fixture/scenario id, M5 command and
  content hashes, M5 read-model hash, M4 regression read-model hash, a captured
  structured renderer error and two safe logs.
- Crash/recovery path works for Web and records Windows blocker: Web produces a
  local browser download; Electron dev and packaged shells expose
  `windows_host_diagnostics_bridge_blocked` with `windowsHostPackageStatus:
blocked` until a reviewed narrow diagnostics bridge exists.
- No private paths, secrets or full save contents are written: platform unit
  tests and Web e2e inject Windows paths and token-like values and assert the
  package contains redaction markers instead of raw values. Full-save-like
  detail keys are replaced with `[redacted-unsafe-field]`.
- No telemetry, network upload or public feedback system is introduced:
  package privacy flags are `telemetry=false` and `networkUpload=false`; no
  network/upload APIs, account flow or public feedback path were added.

## Determinism, save, performance and security impact

Determinism impact: none to simulation stepping, random streams, hashes,
Worker protocol or replay behavior.

Save impact: no save schema or compatibility promise changed. Diagnostics may
record save slot counts, checksums and structured status codes only; full save
contents are excluded.

Performance impact: no benchmark threshold changed. The Web bundle grows by a
small diagnostics module and remains covered by existing release-gate report
generation.

Security impact: no dependency, preload bridge, IPC or Electron setting changed.
The package is generated locally in the renderer and downloaded by explicit
user action. Windows host file diagnostics remain blocked until a reviewed
narrow bridge exists.

## Decisions and deviations

- WM-0093 required `pnpm test --filter diagnostics`, but the test runner did
  not have that filter. I added the narrow `tools/test-runner.mjs` registration
  and recorded it in the task allowed paths.
- I did not expand Electron preload for diagnostics. This keeps WM-0091's
  allowlist intact and records the Windows host-file path as a product-gate
  blocker instead of adding broad host access.
- Diagnostic UI lives in `apps/web/src/diagnostic-package-gate.ts` rather than
  `packages/ui-react` because WM-0093 file ownership did not include the shared
  UI package.

## Remaining risks / follow-up tasks

- Windows host-side diagnostic package writing remains blocked until a reviewed
  narrow bridge defines exact path, payload and audit behavior.
- WM-0094 external smoke should exercise the Web diagnostic download and carry
  the Windows blocker forward into the product-gate smoke evidence.
- Future public feedback, crash upload or account-linked diagnostics remain out
  of scope and require owner approval plus new tasking.
