## Determinism, save, performance and security impact

Determinism: no runtime behavior changed. The proposed design requires accepted
player commands to become deterministic inputs ordered by main sequence,
command ordinal, kind priority, target id and command id.

Save: no save code or public save compatibility changed. ADR-0012 says future
implementation must version focused command-tail fields or regenerate focused
fixtures with explicit evidence; public save migration remains owner-gated.

Performance: no runtime hot path changed. The design requires indexed target
resolution, WorkOffer/Top-K caps, bounded placement validation, exact dirty
read-model projection and metrics for candidate visits, cap hits, path
requests, stale rejects and projection bytes.

Security: no Electron, preload, filesystem, network, telemetry, account,
signing, store, paid-service or public-release surface changed. The design
continues to forbid broad renderer authority and generic platform access.

## Decisions and deviations

- Created ADR-0012 instead of only a loose design note because the task covers
  public Worker/player command and read-model contracts.
- Proposed reusing `PlayerCommandBatch` and `CommandResult` message families
  with a schema bump, rather than adding a new message family for the first
  slice.
- Proposed separating batch acceptance from per-command accepted/rejected
  results so one valid batch can still contain a blocked or rejected command.
- No Spark/rapid-implementer path was used; this is architecture/protocol
  design, explicitly outside the Spark lane.

## Remaining risks / follow-up tasks

- WM-0152 must remove player-facing reliance on shell-local command feedback
