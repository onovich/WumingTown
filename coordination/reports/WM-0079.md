# WM-0079 Work report

## Outcome

Implemented the M5 alpha content catalog fixture and validation evidence. The
catalog is data-only, contains 20 localization-ready catalog entries plus 10
supporting M5 definitions, and is consumed by both schema validation and
compiler tests without adding bespoke runtime code.

## Files and public contracts changed

- `packages/content-schema/src/m5-alpha-content-catalog-fixtures.ts`: accepted
  WM-0079 catalog pack, approved owner-surface allowlist, review notes and
  rejected fixture variants.
- `packages/content-schema/src/index.ts`: exports the WM-0079 fixture helpers
  and constants for compiler/tests.
- `packages/content-schema/src/m5-content-validation.test.ts`: validates the
  accepted catalog and rejected building, tag, anomaly, faction, season,
  localization and unsafe-data fixtures.
- `packages/content-compiler/src/m5-content-validation.test.ts`: compiles the
  same accepted catalog and checks deterministic DefIndex rows/counters/hash.
- `coordination/artifacts/WM-0079/alpha-content-catalog-summary.json`: compact
  reviewer summary of accepted counts, blocked rows and rejected fixture kinds.
- `docs/04_content_balance/08_sample_content_catalog.md`: documents the M5
  alpha catalog fixture counts and blocked entry.
- `docs/04_content_balance/00_content_design_guide.md`: documents the
  WM-0079 catalog review rule and fail-closed fixture scope.
- `docs/03_world/07_cultural_research_policy.md`: records the WM-0079 cultural
  and fairness review boundary.
- `coordination/tasks/WM-0079.json`: taskctl claim/history/state only.

## Acceptance evidence

- Catalog size: `createM5AlphaContentCatalogPack()` contains 30 accepted
  definitions, including 20 `m5.catalog_entry` rows.
- Owner mapping or block: every catalog entry declares non-empty owner surfaces;
  `core.catalog.market_contract_board.v1` is explicitly blocked because M5 has
  no reviewed market-contract owner surface.
- Review repair: tests now require every accepted owner surface to match the
  WM-0079 approved owner-surface allowlist or use a `blocked.owner_surface.*`
  value with `blockedReason`.
- No bespoke runtime code: every accepted definition carries
  `contentBudget.bespokeRuntimeComponents: 0`; no sim-core, sim-worker, apps,
  package or lockfile paths were touched.
- Deterministic validation/compile order: compiler tests assert contiguous
  DefIndex values, counters and content manifest hash shape for the shared
  accepted fixture.
- Review notes: each catalog entry records system value alternatives,
  faction/governance/anomaly hooks and a culture/fairness review note.
- Rejected fixtures: tests cover building owner-surface failure, tag
  owner-surface failure, anomaly evidence failure, faction hook lane failure,
  season-event cooldown failure, missing localization and unsafe data.

## Determinism, save, performance and security impact

- Determinism: compiler test covers stable DefIndex order and stable hash shape
  from the shared accepted fixture.
- Save: no save format, Worker protocol or public schema compatibility changes.
- Performance: no runtime hot path touched; catalog size and validation counts
  are test/artifact evidence only.
- Security: unsafe rejected fixture covers traversal and executable data paths;
  accepted fixture is data-only.

## Decisions and deviations

- ADR-0010 is followed: content definitions are validated data inputs and do
  not become authoritative runtime state.
- `packages/content-schema/src/m5-alpha-content-catalog-fixtures.ts` is 659
- `packages/content-schema/src/m5-alpha-content-catalog-fixtures.ts` is 677
  lines and remains intentionally centralized. It keeps accepted/rejected
  packs, approved owner-surface constants and review notes in one place so they
  cannot drift during WM-0079 review. A future content volume task can split
  catalog data from rejected fixture builders if the fixture grows.
- Existing mojibake-rendered Chinese docs were not rewritten; WM-0079 English
  sections were appended only.

## Remaining risks / follow-up tasks

- Downstream integration must decide whether blocked catalog data such as
  `core.catalog.market_contract_board.v1` receives a reviewed owner surface or
  remains excluded from runtime use.
- Real localization copy can replace placeholder test strings when content
  production moves beyond validation fixtures.
