# WM-0020 Work report

## Outcome

Implemented `sim-core` location ownership and derived spatial indexes for map
cells, chunks, and regions. Location mutations now enforce one map/container
membership per entity, update `MapGrid` occupancy through explicit lifecycle
paths, reject stale generations, and expose reservation cleanup hook counts for
future WM-0023 integration.

- `packages/sim-core/src/location-store.ts`: new `LocationStore` owner store,
  lifecycle cleanup API, metrics, and map/container membership mutations.
- `packages/sim-core/src/spatial-index.ts`: new deterministic per-cell,
  per-chunk, and per-region linked-bucket index with caller-owned query buffers.
- `packages/sim-core/src/spatial-index.test.ts`: focused lifecycle/index tests.
- `packages/sim-core/src/index.ts`: public exports for location and spatial
  index contracts.

- `packages/benchmarks/src/{benchmarks,baseline,cli-lib}.ts` and
  `packages/benchmarks/baseline.json`: registered spatial-index benchmark,
  baseline parsing, comparison, and WM-0020 artifact routing.
- `coordination/artifacts/WM-0020/benchmarks/benchmark-results.json`: benchmark
  evidence artifact.
- `docs/02_systems/02_entities_defs_components.md`,

## Acceptance evidence

- Exactly one map/container membership: `spatial-index.test.ts` covers map to
  container to map transfer, occupied-cell rejection, and membership counters.
- Indexed spatial queries: `SpatialIndex` exposes only cell/chunk/region bucket
  query APIs using caller-provided buffers; tests assert stable chunk/region
  order and `visitedCount` equal to bucket size, not world capacity.
- Despawn/destroy cleanup: tests cover `clearLocation(..., "despawn")` and
  `destroyAndCleanup`, including index removal, occupancy clear, registry
  generation advance, and reservation cleanup hook counts.
- Stale generation rejection: direct destroy plus slot reuse test returns
  `location_entity_generation_mismatch`.
- 50k inert/indexed pressure: `pnpm bench --filter spatial-index` passed with
  50,000 indexed entities, 145 queries, 1,024 moves, 512 cleanups,
  `finalBacklogCount: 0`, `queryChecksum: 1919598032`,

## Determinism, save, performance and security impact

Determinism: query order is by ascending entity index inside indexed buckets;
mutations use integer coordinates, typed-array lanes, generation validation, and
structured reason codes. No ambient randomness, real time, DOM, React, Pixi,
Electron, or Node filesystem dependencies were added to `sim-core`.

Save: no save container format is introduced. `LocationStore` and
`SpatialIndex` are explicit owner/derived surfaces that future save/load tasks
can rebuild from owner lanes.

Performance: normal spatial queries walk only indexed cell/chunk/region bucket
members and use caller-owned output buffers. The benchmark records zero derived
backlog after 50k placement, movement, and cleanup. `location-store.ts` is over
400 lines because it keeps the owner-store lifecycle API, counters, and
structured result types in one file for review; no separate runtime dependency
or cross-package boundary was introduced.

Security: no new runtime dependencies, filesystem access, worker protocol,
renderer authority, or UI mutation path.

## Decisions and deviations

- ADR-0003: implemented `LocationStore` as the map/container authority and
  `SpatialIndex` as a derived index; reservation cleanup is an explicit hook
  surface only because ledger semantics remain WM-0023 scope.
- ADR-0006: kept integer coordinates, stable ordering, explicit lifecycle
  mutations, and no completion-order or time-based behavior.
- Benchmark CLI now routes filtered `spatial-index` artifacts to
  `coordination/artifacts/WM-0020` by default to avoid rewriting WM-0010
  artifacts.
- No changelog file exists in this worktree; documentation updates are in the
  system and performance docs listed above.

## Remaining risks / follow-up tasks

- Direct `EntityRegistry.destroy` can leave old location/index data because it
  bypasses lifecycle cleanup; systems must use `LocationStore.destroyAndCleanup`
  or the future world lifecycle coordinator. The stale generation path is tested
  and rejects old handles.
- Reservation release is a counted hook surface only. WM-0023 must connect the
  real ledger and add reservation-owned leak tests.
- Region ids are read from `MapGrid` at placement time. WM-0021 region/room
  rebuild work must define how region changes invalidate or refresh existing
  location-derived region buckets.
