# ADR-0008: M3 ordinary-life architecture gate

- Status: Proposed
- Date: 2026-06-25
- Owners: systems-architect
- Related tasks: WM-0047, WM-0048, WM-0049, WM-0050, WM-0051,
  WM-0052, WM-0053, WM-0054, WM-0055, WM-0056, WM-0057, WM-0058,
  WM-0059, WM-0060

## Context

M3 composes the M2 work/logistics foundation into
`m3.ordinary_life.injured_caregiver.v1`: needs, rest, food, injury and illness,
medical care, abilities, mood, relationships, day/night, and weather basics.
The scenario must prove that one injury can affect work capacity, medical care,
logistics, mood, and relationships while preserving deterministic save/replay
and Worker/headless parity.

This ADR is the architecture gate required before M3 implementation tasks can
run. It records state ownership, derived-index boundaries, candidate caps, job
driver boundaries, save/replay scope, Worker read-model limits, migration,
rollback, test, and performance implications. It does not implement runtime
behavior, tests, benchmarks, public Worker protocol, public save compatibility,
schemas, package-boundary changes, new dependencies, or M4 systems.

Failure modes this gate must prevent:

- UI, Pixi, React, Electron, protocol, read-model, or platform save code
  mutating authoritative ordinary-life state.
- A monolithic pawn "life brain" owning facts through hidden shared state or
  scanning all facts every decision cycle.
- Normal ticks scanning the full map, all entities, all work rows, all
  conditions, all food, all rest fixtures, all medical offers, or all
  relationships.
- Need, food, medical, mood, relationship, ability, weather, read-model, or
  save caches becoming unversioned or authoritative by accident.
- Rest, eating, or treatment jobs storing execution position in Promise,
  Generator, coroutine, closure, async completion order, UI state, real time, or
  thread-local state.
- Focused save/replay or Worker parity work turning into a public compatibility,
  protocol, schema, dependency, or package-boundary change without a separate
  reviewed gate.
- M4 lamp, Chronicle, obligation, anomaly, crisis-chain, story director, or
  dawn replay product work entering M3 ordinary-life implementation.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered options

- Monolithic per-pawn life brain that scans all facts each decision cycle:
  rejected. It is easy to prototype, but it hides state ownership, encourages
  `Pawn x AllFacts` scans, makes derived cache invalidation implicit, and gives
  save/replay no stable section boundaries.
- UI-owned life/status state: rejected. UI status panels can present ordinary
  life, but any UI-owned need, mood, relationship, medical, weather, or sleep
  authority violates Worker/headless ownership and cannot satisfy replay parity.
- Persist derived caches and read models for faster resume: rejected for M3.
  Derived indexes, ability caches, WorkOffer rows, path caches, read models,
  and UI projections are valid only for named owner-store versions. Load must
  rebuild or invalidate them from owner stores before resumed ticks.
- Allow each M3 slice to adjust public Worker protocol, save sections, schemas,
  dependencies, or package boundaries as needed: rejected. Public contract drift
  must be reviewed as its own architecture task, not hidden inside ordinary-life
  implementation.
- Store-owned facts with versioned derived indexes and bounded selection:
  proposed. Every mutable ordinary-life fact has one owner store; derived
  indexes are scratch or versioned; actor thinking queries bounded candidate
  buckets and Top-K results; focused save/replay and Worker parity remain gates.

## Decision

Keep Simulation Worker or Node headless as the only authoritative world writer.
M3 implementation tasks must extend the existing owner-store model without
moving authority into UI, public protocol adapters, platform storage, or
unversioned caches. `sim-core` remains pure: no DOM, React, PixiJS, Electron,
Node filesystem, real time, or ambient randomness.

M3 introduces the architecture gate contract
`m3OrdinaryLifeArchitectureGateVersion = 1` for reports and future focused
harness metadata that need to cite this decision.

### Derived indexes, dirty queues, and caps

All M3 derived surfaces are versioned, scratch, or rebuilt from owner stores:

- `NeedUrgencyIndex` buckets by actor id, need lane, urgency bucket, schedule
  phase, and `NeedStore.version`.
- `RestCandidateIndex` buckets rest fixtures by region, fixture kind, schedule
  window, weather exposure, permission, and owner versions.
- `FoodPortionIndex` buckets edible portions by def, region/storage, meal
  window, availability, reservation basis, and owner versions.
- `MedicalOfferIndex` buckets patient and caregiver work by region, condition
  kind, urgency, skill/ability requirement, stock need, and owner versions.
- `AbilityCacheDirtyQueue` names exact actor/ability lanes invalidated by
  condition or base ability changes.
- `MoodThoughtDirtyQueue` names exact actors whose thought/mood inputs changed.
- `RelationshipEventIndex` buckets recent social events by actor pair, lane,
  and source event version.
- `ScheduleWindowIndex` and `WeatherExposureIndex` provide bounded context for
  work/rest/need/mood selection; they do not own time or weather facts.
- `WorkerReadModelDirtyQueue` names exact actor, job, condition, relationship,
  weather, and metric projections that need rebuilt read-only output.

Dirty queues carry exact owner keys and source versions. They may be drained in
deterministic budgets during `DerivedRebuildBudget`; remaining backlog is an
observable metric. They must not repair stale state by scanning all entities or
all rows during normal ticks.

Default M3 caps apply unless a downstream task narrows them with tests and
benchmark evidence:

| Selection surface | Candidate visit cap | Retained Top-K | Exact path cap | Notes |
| --- | ---: | ---: | ---: | --- |
| Need urgency per scheduled lane | 24 actors | 12 actors | 0 | Actor self-read of its fixed five need lanes is O(1); scheduled index passes are capped. |
| Rest fixture selection | 24 fixtures/offers | 12 | 4 | Exact pathing is only for retained Top-K candidates. |
| Food/eating selection | 24 portions/offers | 12 | 4 | Quantity reservation and stock mutation happen after path/reservation basis validation. |
| Medical patient/caregiver selection | 24 offers | 12 | 4 | Patient, caregiver, stock, interaction cell, and condition basis must all validate. |
| Ordinary work selection reused by M3 | 24 WorkOffer rows | 12 | 4 | Preserves the M2 WorkOffer/pathing cap contract unless a task documents a lower cap. |
| Social/relationship candidate selection | 16 events/edges | 8 | 0 | Relationship writes are bounded by affected actor ids and event source. |
| Mood/thought generation per actor pass | 16 inputs | 8 retained/new outputs | 0 | Scenario retention remains at most 24 thought rows unless a later task lowers it. |

`ReasonTraceStore` capacity for the focused M3 scenario is `64` retained rows,
matching the scenario contract. Cap hits must be recorded through structured
reasons such as `trace.candidate_cap_reached`; raising caps requires a task
report and performance evidence, not an incidental code change.

Top-K ordering is deterministic: score descending, then domain priority, then
stable target id, then request/enqueue sequence. Random jitter is forbidden
unless a later reviewed decision adds a named stream draw site with replay tests.

### Rest, eating, and treatment job-driver boundaries

Rest, eating, and treatment jobs use `JobCoreStore` plus job-kind-specific
driver stores. Driver state is serializable numeric/entity-id data. It cannot
use Promise, Generator, coroutine, closure, async stack, UI state, real time, or
thread completion order as execution position.

`RestJobDriverStore` owns fields for actor id, rest fixture id, requested rest
kind, reservation claim ids, target cell or interaction spot, schedule/weather
basis, recovery target, step-entered tick, per-step tick count, interruption
policy, and structured terminal reason. Valid states are explicit, for example
`created`, `reserved_fixture`, `pathing_to_fixture`, `resting`, `sleeping`,
`complete`, `failed`, and `cancelled`. Interruptions at unsafe points either
return `job.interruption_denied` or defer to a safe point before cleanup.

`EatingJobDriverStore` owns fields for actor id, food stack/entity id, food
def, amount, meal window, reservation claim ids, optional carried portion lanes
owned with `JobCoreStore`, target cell or interaction spot, consume-once flag,
step-entered tick, progress, interruption policy, and structured terminal
reason. It may not decrement item quantity until the reservation, stack version,
meal window, and job step basis validate. Cancellation after pickup returns or
cleans carried state exactly once through owner-store transactions.

`TreatmentJobDriverStore` owns fields for caregiver id, patient id, condition
id, treatment def, medical stock def/amount, diagnosis clue ids,
counterevidence ids, reservation claim ids, treatment cell or interaction spot,
ability/path/condition basis, progress, applied-delta flag, step-entered tick,
interruption policy, and structured terminal reason. It applies a condition
delta exactly once and only after patient state, caregiver ability, stock,
reservations, and path basis still match. Missing stock, path, ability,
reservation, patient-state, and stale-basis failures are distinct reasons.

### ReasonTrace ownership

`ReasonTraceStore` is the shared source for player explanation, debug panels,
tests, and benchmark diagnostics. Each retained M3 row includes:

- scenario id, seed, tick, and system id;
- actor id when applicable;
- decision or job id when applicable;
- candidate total, visited count, scored count, cap values, and cap-hit counts;
- selected target id or terminal target id when applicable;
- reason class as structured data, not prose;
- source owner-store version basis for relevant need, condition, ability,
  food, rest, medical, relationship, day/night, weather, reservation, job,
  path, and read-model inputs;
- stable trace sequence for deterministic materialization.

Trace text is presentation only. Tests assert reason classes, source ids,
counts, versions, and terminal states.

### Save/replay and Worker parity boundary

M3 save/replay is a focused scenario harness, not a public save container or
compatibility promise. The focused M3 checkpoint may include:

- scenario id, seed, content hash, command stream hash, current tick, and
  checkpoint tick;
- owner-store snapshots for M3 owner facts named in this ADR;
- `NamedRandomStreams` state for the scenario streams;
- `JobCoreStore`, rest/eating/treatment driver state, and `ReservationLedger`
  owner state;
- command log tail and periodic hash diagnostics;
- compact ReasonTrace and metrics summaries for diagnostics only;
- read-only projection hashes for comparison only.

Load validates shape, versions, integer ranges, ids, generation handles,
condition ids, stock quantities, reservation claims, job states, random stream
positions, and command sequence continuity in scratch state before mutating a
live world. Load then rebuilds need, rest, food, medical, ability, mood,
relationship, schedule, weather, WorkOffer, path, read-model, and trace
materialization surfaces before the first resumed tick.

Worker parity must compare browser Worker and Node headless authoritative
hashes for the M3 checkpoint ticks defined by the scenario. Worker read models
are read-only projections. If M3 evidence requires a new public Worker message
family, schema version, save section, migration function, dependency, package
boundary, or UI mutation path, the implementation task blocks until a separate
gate is reviewed.

### Downstream block conditions

Downstream M3 implementation must remain proposed or be marked blocked when any
of these conditions apply:

- A mutable ordinary-life fact has no named owner store in this ADR or a
  reviewed superseding ADR.
- A normal tick, pawn thinking pass, cleanup path, cache repair, path selection,
  reservation acquisition, medical selection, food/rest selection, mood update,
  social update, or read-model production needs a full-map, all-entity,
  all-offer, all-condition, all-food, all-rest-fixture, all-relationship, or
  all-job scan.
- A derived index, dirty queue, ability cache, WorkOffer row, path result,
  read model, save diagnostic, or UI projection cannot name its owner versions
  or be rebuilt from owner stores.
- A task needs public Worker protocol changes, public save compatibility
  changes, public schema/migration changes, new runtime dependencies, package
  boundary exceptions, or public import path changes.
- A task moves authoritative mutation into UI, Pixi, React, Electron, preload,
  protocol adapters, renderer state, platform save code, or read-model
  consumers.
- A task uses real time, platform timers, `Date.now`, `performance.now`,
  `Math.random`, ambient entropy, visual-only randomness, thread scheduling, or
  completion order to drive authority.
- Rest, eating, treatment, medical, mood, social, or work jobs depend on
  Promise, Generator, coroutine, closure, async stack, UI state, or hidden
  singleton state for execution position.
- Save/load cannot rebuild derived indexes before resumed ticks, or cannot
  produce focused uninterrupted-vs-resume hashes and structured divergence
  diagnostics.
- Worker and Node headless cannot run the same command stream with matching
  authoritative hashes without UI repair.
- The task expands into M4 lamp network, Chronicle evidence gameplay,
  obligations, anomaly rules, crisis chains, story director, dawn replay
  productization, M4 task creation, M4 task promotion, or M4 implementation.

## Consequences

### Positive

- M3 implementers have explicit owner stores before adding ordinary-life
  behavior.
- Derived indexes, caches, and read models can be optimized without becoming
  hidden authority or save payloads.
- Rest, eating, and treatment jobs compose with existing reservation and job
  rules instead of inventing implicit execution state.
- Reviewers can reject global scans, unversioned caches, UI authority, public
  contract drift, real-time/random authority, and M4 scope with direct gate
  language.

### Negative

- Downstream tasks must create explicit owner versions, dirty queues, and
  rebuild paths before visible ordinary-life behavior is complete.
- Focused save/replay and Worker parity tasks may need extra diagnostics to
  prove M3 without public protocol or save-format expansion.
- Some speed optimizations, such as persisted ability caches or read models,
  are delayed until metrics justify a separate save/protocol gate.

### Risks and mitigations

- Risk: M3 owner stores become too many small surfaces. Mitigation: each store
  owns a coherent fact family, and derived indexes can compose them through
  versioned basis fields without owning facts.
- Risk: candidate caps skip plausible behavior in larger scenarios. Mitigation:
  cap hits are ReasonTrace and benchmark evidence; caps can be adjusted only
  with explicit tests, metrics, and stable Top-K ordering.
- Risk: load-time rebuild hides expensive work. Mitigation: WM-0057 and
  WM-0059 must report rebuild time, queue backlog, rebuilt row counts, and first
  resumed hash.
- Risk: ability cache is mistaken for authority. Mitigation: the cache owns
  validity and cached values only; source conditions and base abilities remain
  the owner facts and cache rows are rebuilt or rejected on stale basis.
- Risk: medical clues drift into M4 Chronicle gameplay. Mitigation: M3 medical
  clues and counterevidence are local health facts only and cannot create
  Chronicle evidence graph, testimony gameplay, dawn replay, or case-board
  state.

## Contracts introduced

- M3 architecture gate contract: `m3OrdinaryLifeArchitectureGateVersion = 1`.
- Authoritative mutation remains only in Simulation Worker or Node headless.
- M3 owner stores are `NeedStore`, `RestSleepStore`, `FoodAvailabilityStore`,
  `HealthConditionStore`, `AbilityCacheStore`, `MedicalCareStore`,
  `MoodThoughtMemoryStore`, `RelationshipGraphStore`, `DayNightStore`,
  `WeatherStore`, `RandomStreamStore` / `NamedRandomStreams`, `JobCoreStore`,
  `RestJobDriverStore`, `EatingJobDriverStore`, `TreatmentJobDriverStore`,
  `ReservationLedger`, `ReasonTraceStore`, `M3FocusedSaveReplayHarness`, and
  `WorkerReadModelStore` / read-model builders.
- `ItemStackStore` remains the authoritative quantity owner for food, water,
  and medical stock.
- Ability cache rows are versioned derived cache state, not source ability or
  health authority.
- ReasonTrace rows are bounded diagnostic evidence shared by UI, debug, tests,
  and benchmarks; they are not job execution state or unbounded logs.
- Rest, eating, and treatment jobs use explicit serializable driver states with
  all reservation claims owned by `ReservationLedger`.
- Focused M3 save/replay may snapshot owner stores and deterministic streams
  for the scenario harness only. Public save compatibility, public schema
  changes, and broad migration promises are non-decisions here.
- Worker read models remain read-only projections with basis metadata. UI and
  renderer layers cannot mutate or repair authoritative world state.

## Performance implications

- Normal M3 ticks must not allocate objects, arrays, closures, or strings in
  loops proportional to actors, offers, conditions, food stacks, rest fixtures,
  relationships, jobs, reservations, paths, or read-model rows.
- Normal M3 ticks must not use full-world scans, unbounded sort, regex, string
  concatenation, `map`, `filter`, `reduce`, or `flatMap` in hot loops.
- Required M3 counters include scheduled need updates, need/rest/food/medical/
  social candidate totals, visited rows, selected rows, Top-K cap hits, exact
  path requests, path stale rejects, reservation attempts, cleanup releases,
  active claims, condition updates, ability invalidations, ability query cost,
  thought generation/retention, relationship events/edges, day/night and
  weather update cost, dirty queue peak/final backlog, ReasonTrace capacity use,
  snapshot byte size, rebuild time, command log size, Worker latency counts,
  read-model sizes, checkpoint hashes, and long-run queue/cache drift.
- Existing benchmark policy remains: key benchmark P95 regression over 10
  percent requires explanation, and over 20 percent blocks merge by default.
- Load-time full scans are allowed only for validation and rebuild before ticks
  resume. They must report rebuild counts, time, backlog, and first resumed
  hash comparison.
- Raising candidate caps or trace retention requires explicit task evidence.
  It must not be hidden inside implementation.

## Rollback / migration

Before review acceptance, rollback is deleting
`coordination/decisions/ADR-0008.md`, reverting the narrow doc/index/report/task
edits that cite it, and leaving downstream M3 tasks proposed or blocked.

If accepted before implementation, migration from M2 to M3 is incremental:

1. Keep M2 `WorldState`, owner stores, explicit jobs, `ReservationLedger`,
   `WorkOfferIndex`, versioned pathing, focused save/replay, and read-only
   Worker boundaries.
2. Add M3 owner stores only in their owning downstream tasks.
3. Add exact dirty queues and owner version fields before derived indexes or
   caches can influence selection.
4. Add focused tests that rebuild derived indexes from owner stores before
   save/replay, Worker parity, or integrated scenario evidence can pass.
5. Treat any public Worker protocol, public save/schema, dependency,
   package-boundary, UI authority, real-time/random authority, Promise/coroutine
   job, or M4 need as a blocker and request a separate reviewed gate.

If a downstream implementation task fails review because it violates this gate,
rollback is reverting that task branch. Do not merge temporary global scans,
unversioned caches, persisted derived indexes, UI-authoritative fixes,
real-time/random authority, Promise-based job progress, stale path commits,
public protocol/save/schema drift, or M4 scaffolding as bridges.

If this gate proves too broad or too narrow, supersede ADR-0008 with a reviewed
ADR before M3 implementation resumes. Downstream M3 tasks remain proposed or
blocked until the replacement gate is verified.
