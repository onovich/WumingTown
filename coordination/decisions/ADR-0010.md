# ADR-0010: M5 alpha content framework architecture gate

- Status: Proposed
- Date: 2026-06-26
- Owners: systems-architect
- Related tasks: WM-0072, WM-0073, WM-0074, WM-0075, WM-0076,
  WM-0077, WM-0078, WM-0079, WM-0080, WM-0081, WM-0082, WM-0083

## Context

M5 expands the reviewed M4 core vertical slice into the alpha content
framework. The milestone must prove that new anomalies, faction/governance
hooks, season events, building/content definitions and data mods can be added
mainly through data and reusable rule components without weakening M0-M4
authority, determinism, save/replay, Worker projection, content validation or
benchmark gates.

This ADR is a planning gate. It does not implement runtime behavior, tests,
benchmarks, public Worker messages, public save compatibility, content schema
implementation, package-boundary changes, new dependencies, M6 product work or
UI authority.

Failure modes this gate must prevent:

- A broad content push that hides bespoke runtime code inside each anomaly or
  event.
- Content definitions, data mods, UI panels or director queues becoming
  authoritative mutable world state.
- Normal ticks scanning all content, anomalies, factions, policies, buildings,
  events, residents or projections.
- Unversioned caches, compiled definition rows, read models or save diagnostics
  surviving schema, content-manifest, owner-store or projection changes.
- Data mods reaching runtime without schema, reference, semantic,
  localization, path-safety and size validation.
- Focused save/replay or Worker parity becoming public compatibility or public
  protocol promises without separate review.
- M6 platform/product gates being started before M5 content-framework evidence
  exists.

## Decision Drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered Options

- Bespoke code per anomaly/event: rejected. It can ship content quickly but
  defeats the content-framework milestone, makes tests and save/replay ad hoc,
  and creates unbounded ownership risk.
- Monolithic content director: rejected. A single director owning anomaly
  truth, faction state, governance, season queues and event outcomes would
  bypass owner stores and invite global scans.
- UI-authored mod/content state: rejected. UI may author or inspect data, but
  validation and simulation authority remain in tools/simulation boundaries.
- Public save/protocol/schema redesign as part of M5: rejected. M5 may define
  focused harness expectations; public compatibility needs separate gates.
- Data-only content framework with owner stores, versioned compiled defs,
  bounded derived indexes and focused gates: proposed.

## Decision

Keep Simulation Worker or Node headless as the only authoritative world writer.
`sim-core` remains free of DOM, React, PixiJS, Electron, Node filesystem, real
time and ambient randomness. M5 introduces the architecture gate contract
`m5AlphaContentFrameworkArchitectureGateVersion = 1`.

### M5 Ownership Table

| Surface | Authoritative owner | Derived or non-owner surfaces | Version, rebuild and mutation rule |
| --- | --- | --- | --- |
| Content manifests and compiled defs | Content compiler owns validated manifest identity, schema versions, dependency graph, stable DefIndex order and content hashes. Runtime owner stores own mutable facts created from defs. | Source files, editor forms, UI catalogs, localized strings, diagnostic reports and compiled preview views. | External content is parsed as `unknown`, validated, compiled deterministically and fail-closed before runtime. Compiled rows carry schema/content version and cannot mutate world state by themselves. |
| Anomaly roster | `M5AnomalyRosterStore` owns active anomaly definitions by compiled def id, rule-component handles, activation policy, rule-state allocation basis and roster version. | Third-knock anomaly | `ThirdKnockCrisisStore` owns invitation/debt state, threshold basis, knock count, target/door ids, evidence requests, escalation stage, resolution and terminal reason. | Door overlays, guesthouse UI, policy suggestions, Chronicle materialization and incident summaries. | Progress is an explicit serializable state machine. It uses town-rule knowledge, obligations and evidence basis versions before mutation. |
| Old-bridge guest anomaly | `OldBridgeGuestCrisisStore` owns bridge crossing windows, prepared-item basis, route/trade obligation state, evidence traces, resolution and terminal reason. | Route UI, merchant summaries, bridge event previews and director opportunities. | Logistics and obligation facts remain in their existing owners. | Faction campaign facts | `FactionFactStore` owns faction ids, standing lanes, pressure lanes, known claims, legal stance, trade stance, debt stance, memory events and source versions. | Faction UI, diplomacy text, event weights, relationship summaries and debug reports. | Facts are integer/versioned owner lanes. Derived attitudes and text cannot own consequences. |
| Governance hooks | `GovernanceHookStore` owns council post assignment, temporary policy authority, legitimacy source, enforcement capacity, corruption/risk flags and policy hook version. | Ordinance UI, position badges, director candidate weights and season review rows. | Governance hooks feed town-rule/event legality through bounded queries. ### Required Anomaly Roster

M5 must include at least:

- `core.anomaly.borrowed_shadow.v1`
- `core.anomaly.third_knock.v1`
- `core.anomaly.old_bridge_guest.v1`

Each anomaly must declare activation, perception, movement/scope where relevant,
preference, constraints, traces, exchange/handling, propagation or inheritance
when relevant, resolution, consequences, evidence classes, common misread,
non-combat handling and accident-review parameters.

### Derived Indexes, Dirty Queues And Caps

All M5 derived surfaces are versioned, scratch or rebuilt from owner stores:

- anomaly activation lanes carry anomaly def id, owner state id, source owner
  versions, candidate score and stable sequence;
- faction/governance hook lanes carry faction/governance owner versions,
  policy context and bounded candidate caps;
- season/event pool lanes carry event def id, cooldown/recovery basis, source
  owner versions and precondition failure reason;
- content validation rows carry schema version, content manifest hash,
  dependency order, reference basis and failure code;
- Worker projection dirty queues name exact anomaly, faction, governance,
  event, content, Chronicle, obligation, town-rule and review keys.

Default M5 caps apply until a downstream task narrows or raises them with tests
and benchmark evidence:

| Selection surface | Candidate visit cap | Retained Top-K | Exact path cap |
| --- | ---: | ---: | ---: |
| Anomaly activation per rule pass | 24 candidates | 12 | 4 |
| Anomaly transition per active crisis | 16 transitions | 8 | 0 |
| Faction/governance hook query | 24 facts/hooks | 12 | 0 |
| Season/event candidate pass | 20 events | 8 | 0 |
| Content validation semantic references per def | 64 references | 0 | 0 |
| Worker projection dirty drain per sample | 64 keys | 0 | 0 |
| Season review retained causal rows | 96 rows | 48 rows | 0 |

Top-K ordering is deterministic: score descending, domain priority, stable
owner id, then stable enqueue sequence. Random choice requires named stream
draws with replay tests.

### Data-Mod And Content Schema Policy

M5 content is data-only. Allowed content pack inputs are manifests, JSON/JSON5
defs, localization resources, balance data, references and scenario fixtures.
The validation pipeline must check path safety, file size, manifest shape,
schema version, dependency graph, references, semantic constraints,
localization coverage, deterministic compile order and scenario fixtures.

Forbidden inputs include arbitrary JavaScript/TypeScript, WebAssembly, native
code, executable scripts, install hooks, network access, platform APIs, remote
URLs, unsafe paths, recursive archive expansion and oversized assets. Invalid
content fails closed with structured reasons before runtime.

Public schema changes or schema implementation must happen only in the owning
downstream task. WM-0072 does not implement schemas.

### Save/Replay And Worker Scope

M5 focused save/replay may snapshot:

- scenario id, seed, content manifest hash, command stream hash, tick and
  checkpoint tick;
- M0-M4 owner stores already approved by earlier gates;
- M5 owner stores named in this ADR;
- named random stream state and draw counts;
- command log tail and periodic hash diagnostics;
- compact validation/anomaly/faction/event ReasonTrace diagnostics;
- read-only projection hashes for comparison.

Public save compatibility, platform save UI, public schema migrations, codec
dependencies and cross-version guarantees are non-decisions here.

Worker projections remain read-only. Any need for new public Worker message
families, schema versions or client-side authority blocks for a separate gate.

## Consequences

### Positive

- M5 content work has clear ownership before broad content production begins.
- Data mods can be validated and benchmarked without granting code execution.
- Faction/governance/event hooks can influence scenario legality without a
  monolithic political director.
- Save/replay, Worker parity and benchmark tasks can compare content-manifest
  basis rather than hidden UI/editor state.

### Negative

- The first M5 tasks must build schema/validation and owner-version surfaces
  before content volume feels productive.
- Some desirable content may block if it cannot map to reusable rule
  components.
- Focused save/replay remains narrower than future public save compatibility.

## Risks And Mitigations

- Risk: content authors request bespoke code for every anomaly. Mitigation:
  every anomaly must name reusable rule components or block for architecture
  review.
- Risk: faction standing collapses to a single mood number. Mitigation:
  `FactionFactStore` keeps decomposed legal, trade, debt, fear/legitimacy and
  memory facts.
- Risk: season events hide direct fact mutation. Mitigation: event pool rows
  schedule legal commands only and emit structured rejections.
- Risk: mod validation misses unsafe paths or huge files. Mitigation: path and
  size checks run before schema parsing; invalid content never reaches runtime.
- Risk: content volume masks performance regressions. Mitigation: benchmark
  thresholds remain unchanged and M0-M4 regression gates stay active.
- Risk: M5 drifts into M6. Mitigation: M6 readiness is stop-sign evidence only;
  no M6 task can be created by this package.

## Contracts Introduced

- M5 architecture gate contract:
  `m5AlphaContentFrameworkArchitectureGateVersion = 1`.
- Authoritative mutation remains only in Simulation Worker or Node headless.
- M5 owner surfaces are `M5AnomalyRosterStore`,
  `ThirdKnockCrisisStore`, `OldBridgeGuestCrisisStore`, `FactionFactStore`,
  `GovernanceHookStore`, `M5SeasonEventPoolStore`,
  `ContentValidationPipeline`/compiled manifest outputs,
  `M5FocusedSaveReplayHarness`, Worker read-model builders and
  `M5InvariantSuite`.
- Existing M0-M4 owner stores remain authoritative and must not be bypassed.
- Data mods are schema-validated data only; code/network/platform mods are
  forbidden.
- Focused M5 save/replay does not approve public save compatibility.
- Worker projections are read-only and must carry basis metadata and hashes.
- Benchmark thresholds remain 10 percent warning and 20 percent blocking.
- M6 readiness stop signs do not create, promote, claim or implement M6.

Future implementation verification required by this ADR:

- Content validation tests reject invalid manifests, schema versions, unsafe
  paths, oversized files, unknown references, missing localization, semantic
  rule violations and unsupported mod capabilities.
- Anomaly roster tests prove stable DefIndex order, bounded activation,
  explicit state machines, non-combat resolution and structured traces.
- Third-knock and old-bridge tests prove evidence, obligation, town-rule,
  faction/governance and season basis validation before mutation.
- Faction/governance tests prove decomposed fact lanes, bounded queries and no
  prose-only authority.
- Season/event tests prove preconditions, cooldowns, recovery windows,
  freshness, structured rejections and legal command scheduling only.
- Save/replay tests prove content-manifest validation, rebuilt derived
  surfaces before resume and uninterrupted/resume hash equality.
- Worker parity tests prove Node and browser Worker read-only projections match
  or reject stale basis identically.
- Invariant tests fail on M0-M4 regression, content validation drift, unsafe
  mod acceptance, anomaly leaks, event queue growth, save/replay divergence,
  Worker mismatch, threshold weakening, UI authority and M6 task creation.

## Performance Implications

Normal M5 ticks must not allocate objects, arrays, closures or strings in loops
proportional to actors, map cells, content defs, anomalies, faction facts,
governance hooks, events, jobs, reservations, paths or projections.

Normal M5 ticks must not use full-world scans, unbounded sorts, regex, string
concatenation, `map`, `filter`, `reduce` or `flatMap` in hot loops.

Required M5 metrics include anomaly roster counts, active anomalies,
candidate-cap hits, transition counts, faction/governance candidate visits,
event candidate/rejection counts, cooldowns, recovery windows, validation
failure counts, compiled DefIndex stability, save bytes, load/rebuild time,
Worker projection bytes, checkpoint hashes, M0-M4 regression and M6 stop-sign
verdict.

Existing benchmark policy remains unchanged: key benchmark P95 regression over
10 percent requires explanation, and over 20 percent blocks merge by default.

## Migration

If accepted before implementation, migration from M4 to M5 is incremental:

1. Keep M0-M4 owner stores, fixed tick, named streams, Worker authority,
   focused save/replay discipline and benchmark baselines intact.
2. Add schema/validation outputs before runtime consumes M5 content.
3. Add M5 owner stores only in their owning downstream tasks.
4. Add exact dirty queues and owner-version fields before derived indexes or
   read models can influence choices.
5. Add focused tests and metrics for each owner surface before integrating the
   M5 scenario.
6. Treat public Worker protocol, public save/schema, dependency, package
   boundary, UI authority, hidden shared state, global scans or M6 needs as
   blockers.

## Rollback

Before downstream implementation, rollback is deleting
`coordination/decisions/ADR-0010.md`, reverting the M5 scenario/roadmap/report
files, proposed WM-0073 through WM-0083 task packets and project-state marker,
then restoring M4 closed / M5 unstarted.

If a downstream task violates this gate, rollback is reverting that task
branch. Do not merge temporary global scans, unversioned caches, UI-authority
fixes, code-mod execution, direct director mutation, persisted derived caches,
public contract drift, threshold weakening or M6 scaffolding as bridges.
