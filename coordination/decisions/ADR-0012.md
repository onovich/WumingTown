# ADR-0012: Authoritative playable command and job protocol vertical slice

- Status: Proposed
- Date: 2026-06-30
- Owners: systems-architect
- Related tasks: WM-0149, WM-0150, WM-0151, WM-0152, WM-0154

## Context

Post-M8 product review reopened the playable-product bar. The first accepted
player slice must show a real command-to-job loop for dusk lamp work and simple
building: the player issues a command, the Simulation Worker validates it,
the world records an authoritative order/job, a pawn visibly claims and moves,
work progresses, and the target reaches a completed or structured blocked
state.

Current public protocol facts:

- `PlayerCommand` is only `{ commandId, kind }`.
- Supported player command kinds are `Noop` and `Echo`.
- `CommandResultPayload` is batch-level only and has no per-command order/job
  result surface.
- `RenderSnapshot` and `UiDelta` are read-only projections, but the current
  UI model mostly carries counts, hashes and summary strings.
- `BuildSiteStore`, `JobCoreStore`, `ReservationLedger`, `WorkOfferIndex`,
  `ItemStackStore`, `MapGrid` and `LocationStore` already provide owner-store
  primitives for delivery/build jobs, reservations, materials, path failures
  and build-site read views.

This ADR designs the first reviewed Worker/player command contract for the
lamp-priority and simple-build vertical slice. It does not implement protocol
types, validators, Worker behavior, sim-core runtime behavior, UI wiring, save
code, migration code or product features.

Failure modes this decision must prevent:

- React, Pixi, Electron or shell-local adapters mutating world state directly.
- A command result that says "accepted" while the job state exists only in UI.
- Magic command ids, summary strings or diagnostic prose becoming command
  authority.
- Rejections such as no work, path failed or condition unmet without structured
  player-facing reason classes.
- Save/replay drift because accepted player commands are not versioned and
  logged as deterministic inputs.
- Pawn work selection scanning the whole map or all offers when a player
  prioritizes a target.
- Public Worker protocol, read-model schema or save-tail changes sneaking into
  implementation without version and migration review.

## Decision Drivers

- Preserve Simulation Worker / Node headless authority.
- Produce a player-visible command-to-job chain for the first lamp/build slice.
- Keep command, result and read-model shapes versioned and validateable.
- Reuse existing owner-store concepts instead of introducing UI-owned state.
- Keep job execution as explicit serializable state machines.
- Keep work discovery indexed, capped and deterministic.
- Keep save/replay and Worker/headless parity testable.

## Considered Options

1. Keep the WM-0138 shell-local adapter and make it look player-facing.
   Rejected. It can teach the UI interaction, but it does not mutate the
   Simulation Worker world, does not create authoritative jobs, and cannot
   satisfy replay or Worker/headless parity.

2. Encode real commands as `Noop` command ids such as
   `prioritize-lamp:<target>`.
   Rejected. It hides schema in strings, bypasses payload validation, weakens
   versioning and makes command results hard to audit.

3. Let the UI select a pawn and directly create a job state.
   Rejected. Pawn assignment, reservations, path basis and job progress are
   authoritative simulation decisions. UI can request intent; it cannot own
   worker selection or job execution position.

4. Add a new Worker message family only for playable command results.
   Rejected for the first slice. The existing `PlayerCommandBatch` and
   `CommandResult` families can carry the contract with a schema bump. A new
   family is justified only if later reliable streams need different
   backpressure behavior.

5. Extend `PlayerCommandBatch`, `CommandResult`, and read-model projections
   with versioned command/job payloads.
   Proposed. This keeps one command entry point, adds per-command result
   evidence, and keeps UI/renderer consumers read-only.

## Decision

Introduce `playableCommandContractVersion = 1` behind a public protocol schema
bump. The top-level message families remain `PlayerCommandBatch` and
`CommandResult`, but their payload schemas change incompatibly, so
`SIM_SCHEMA_VERSION` must advance before implementation. `SIM_PROTOCOL_VERSION`
can remain unchanged if no message-family semantics change beyond schema.

The contract has four surfaces:

1. Typed player command payloads.
2. Per-command result payloads.
3. Shared structured blocked/rejection reasons.
4. Read-model projections for targets, actions, jobs, pawns and progress.

### Command Payloads

Every playable command extends the existing command envelope:

```ts
interface PlayerCommandV2 {
  readonly commandId: string;
  readonly kind: PlayerCommandKindV2;
  readonly payload: PlayerCommandPayloadV2;
  readonly basis: CommandBasisV1;
}

interface CommandBasisV1 {
  readonly playableCommandContractVersion: 1;
  readonly basisTick: number;
  readonly basisSnapshotSequence: number;
  readonly basisReadModelHash: string;
  readonly contentManifestHash: string;
  readonly targetVersion?: number;
  readonly mapVersion?: number;
  readonly reservationVersion?: number;
  readonly jobVersion?: number;
}
```

`basis` is not authority. It is the read-model basis the player acted on. The
Worker validates it against current authoritative owner versions. A mismatch
returns `stale_command`; UI must request a fresh projection instead of patching
the world.

The first slice introduces these command kinds:

```ts
type PlayerCommandKindV2 =
  | "PrioritizeLampWork"
  | "QueueSimpleBuild";
```

#### `PrioritizeLampWork`

Player intent: "make this lamp gap, lamp, or lamp-related build site the next
important work target."

```ts
interface PrioritizeLampWorkPayloadV1 {
  readonly target:
    | { readonly kind: "lamp"; readonly entity: ProtocolEntityRefV1 }
    | { readonly kind: "lamp_gap"; readonly gapId: string; readonly anchorCell: CellRefV1 }
    | { readonly kind: "build_site"; readonly siteId: number; readonly site: ProtocolEntityRefV1 };
  readonly requestedAction:
    | "auto"
    | "refill_lamp"
    | "repair_lamp"
    | "complete_lamp_build_site";
  readonly priorityBand: 1 | 2 | 3;
}
```

Worker responsibilities:

- Resolve the target through authoritative owner stores and indexed target
  lanes, not through UI state.
- Validate the target is still active and relevant.
- Choose or expose candidate work through `WorkOfferIndex` and bounded Top-K
  selection.
- Create or reprioritize an authoritative order/work intent only inside the
  Simulation Worker or headless runtime.
- Let authoritative work selection and job drivers choose the pawn; UI does
  not assign the worker for this command.

#### `QueueSimpleBuild`

Player intent: "place or activate the approved simple build target for the
slice."

```ts
interface QueueSimpleBuildPayloadV1 {
  readonly blueprint:
    | { readonly kind: "simple_lamp_post"; readonly blueprintDefId: number }
    | { readonly kind: "simple_repair_frame"; readonly blueprintDefId: number };
  readonly anchorCell: CellRefV1;
  readonly orientation: 0 | 1 | 2 | 3;
  readonly priorityBand: 1 | 2 | 3;
}
```

Worker responsibilities:

- Validate content manifest, blueprint id, cell bounds, occupancy, policy and
  reachable interaction cells.
- Allocate or activate the build-site/order through authoritative owner stores
  only.
- Emit or update material/build offers from owner-store facts.
- Return an order reference and the current blocked/ready state. The command
  does not instantly build the structure.

Shared protocol refs:

```ts
interface ProtocolEntityRefV1 {
  readonly index: number;
  readonly generation: number;
}

interface CellRefV1 {
  readonly x: number;
  readonly y: number;
  readonly cellIndex: number;
}
```

No command payload may contain localized prose, functions, callbacks, object
references, renderer ids, Pixi object ids, React component ids, file paths,
wall-clock timestamps or random values.

### Result Payloads

`CommandResultPayload` becomes batch plus per-command evidence:

```ts
interface CommandResultPayloadV2 {
  readonly inReplyToSequence: number;
  readonly batchAccepted: boolean;
  readonly batchReason?: ProtocolRejection;
  readonly commandResults: readonly PlayerCommandResultV1[];
}

type PlayerCommandResultV1 =
  | PlayerCommandAcceptedResultV1
  | PlayerCommandRejectedResultV1;

interface PlayerCommandAcceptedResultV1 {
  readonly commandId: string;
  readonly kind: PlayerCommandKindV2;
  readonly status: "accepted";
  readonly acceptedTick: number;
  readonly committedTick: number;
  readonly target: CommandTargetRefV1;
  readonly order?: CommandOrderRefV1;
  readonly job?: CommandJobRefV1;
  readonly initialState: "queued" | "claimable" | "claimed" | "blocked" | "completed";
  readonly blockedReason?: CommandBlockedReasonV1;
}

interface PlayerCommandRejectedResultV1 {
  readonly commandId: string;
  readonly kind: PlayerCommandKindV2;
  readonly status: "rejected";
  readonly rejectedTick: number;
  readonly reason: CommandBlockedReasonV1;
}
```

`batchAccepted=false` is reserved for envelope, lifecycle, version, session,
sequence or payload validation failure where the Worker cannot safely inspect
individual commands. If the batch envelope is accepted, each command receives
its own accepted or rejected result.

Accepted commands may still produce `initialState: "blocked"` when the target is
valid but no executable work can currently be claimed. That blocked state must
also appear in read models until resolved or superseded.

```ts
interface CommandOrderRefV1 {
  readonly orderKind: "lamp_priority" | "simple_build";
  readonly orderId: string;
  readonly targetVersion: number;
}

interface CommandJobRefV1 {
  readonly jobId: number;
  readonly jobKind:
    | "lamp_refill"
    | "lamp_repair"
    | "build_site_delivery"
    | "build_site_construction";
  readonly owner?: ProtocolEntityRefV1;
}
```

The result payload is reliable. It must not be dropped for render backpressure.

### Structured Blocked And Rejection Reasons

Use one reason shape for immediate rejections and later job/order blocked
projections:

```ts
type CommandBlockedReasonCodeV1 =
  | "missing_resource"
  | "no_path"
  | "no_worker"
  | "invalid_target"
  | "stale_command"
  | "rule_policy_denial";

interface CommandBlockedReasonV1 {
  readonly code: CommandBlockedReasonCodeV1;
  readonly source:
    | "command_validation"
    | "work_selection"
    | "reservation"
    | "pathing"
    | "job_driver"
    | "policy";
  readonly target?: CommandTargetRefV1;
  readonly actor?: ProtocolEntityRefV1;
  readonly requirement?: ResourceRequirementRefV1;
  readonly basis?: StaleBasisRefV1;
  readonly policy?: PolicyRefV1;
  readonly candidateCounts?: CandidateCountsV1;
}
```

Required code meanings:

| Code | Meaning | Minimum structured parameters |
| --- | --- | --- |
| `missing_resource` | Required fuel/material/tool/stock is absent, insufficient or fully reserved. | `target`, `requirement.defId`, `requiredAmount`, `availableAmount`, `reservedAmount` |
| `no_path` | No valid route exists to source, destination, interaction cell or target under the command basis. | `target`, optional `actor`, `basis.mapVersion`, `candidateCounts` |
| `no_worker` | No capable, available and policy-allowed worker can claim the order. | `target`, `candidateCounts`, optional missing role/ability/schedule ids through `requirement` |
| `invalid_target` | Target id, cell, blueprint, lamp gap or build site is absent, destroyed, complete, occupied, out of range or not actionable. | `target`, `basis` when version-related |
| `stale_command` | The command acted on an old session, sequence, content hash, read-model hash, owner version, target generation, map version, reservation version or job version. | `basis.expected*`, `basis.observed*`, `target` |
| `rule_policy_denial` | Town rule, safety policy, work permission, interruption policy, forbidden zone, curfew or scenario policy denies the action. | `target`, optional `actor`, `policy.policyId`, `policy.reasonCode` |

Existing core reasons map into the shared reason classes during projection:

- `material.insufficient_required_amount`, reservation insufficient amount or
  capacity -> `missing_resource`.
- `path.no_route_to_source`, `path.no_route_to_destination` -> `no_path`.
- invalid site id, invalid entity, invalid blueprint, completed target, blocked
  placement or impossible target state -> `invalid_target`.
- stale session, stale sequence, stale content basis, stale map/navigation/job
  or target owner version -> `stale_command`.
- `policy.interruption_denied`, permission failure, forbidden zone or town-rule
  conflict -> `rule_policy_denial`.
- no selected worker after indexed candidate scoring, schedule/ability filters
  or worker caps -> `no_worker`.

No result or projection may return only a free-form string such as "no work",
"path failed" or "condition unmet". Localized text is produced by UI from reason
codes and named parameters.

### Read-Model Projection Requirements

The playable read model must expose enough data for React, Pixi and Electron to
render the action chain without becoming authority.

Every projection involved in this slice includes:

- `playableCommandReadModelVersion: 1`
- `basisTick`
- `basisSnapshotSequence`
- `basisWorldHash`
- `basisReadModelHash`
- relevant owner versions for targets, jobs, reservations, paths and content
- command basis fields needed to build the next command

Required projection groups:

1. Target/action projection
   - selected lamp, lamp gap, build site or build cell;
   - target state, target version and action availability;
   - command payload template fields for `PrioritizeLampWork` or
     `QueueSimpleBuild`;
   - disabled/blocked reason using `CommandBlockedReasonV1`.

2. Order/job projection
   - order id, command id, target, priority band and owner version;
   - job marker state:
     `queued`, `claimable`, `claimed`, `moving`, `working`, `blocked`,
     `completed`, `failed`, `canceled`;
   - `jobId`, `jobKind`, `owner`, `target`, `step`, `progressQ16` or tick
     progress, required work and terminal reason.

3. Pawn projection
   - actor id/generation, display id, tile/cell and current state:
     `idle`, `moving`, `working`, `blocked`, `completed`, `failed`;
   - current job/order reference;
   - intent path summary or path target basis for visible path feedback;
   - structured blocked reason when idle or blocked by command-related facts.

4. Build/lamp projection
   - build-site id/entity, active/completed state, blueprint id, anchor cell,
     interaction cells, required material defs/amounts, delivered amounts,
     reserved amounts, remaining demand, build progress and lantern state;
   - lamp target or lamp gap id and reason why it matters for the objective.

5. Resource and alert projection
   - concise material/fuel counts relevant to the selected command;
   - reservations that explain why available stock differs from total stock;
   - at most the ranked alert(s) needed for this slice.

UI consumers may store selection, hover, camera and locale preference locally.
They may not store or mutate order state, job state, pawn world position,
resource counts, build progress, blocked reasons or command outcomes as
authority.

### Authority And State Ownership

The command flow is:

```text
React/Pixi/Electron input
  -> app command builder using read-model basis
  -> PlayerCommandBatch
  -> Simulation Worker boundary validation
  -> authoritative sim-core owner stores
  -> read-only CommandResult/RenderSnapshot/UiDelta/MetricsSample
  -> React/Pixi/Electron render only
```

Authoritative owners:

- `BuildSiteStore` owns simple build-site/order material buffers, build
  progress, lantern completion state and build-site job views.
- `JobCoreStore` owns generic job id, owner, kind, status, step, progress,
  carried state, terminal reason and interruption policy.
- `ReservationLedger` owns claims for entity, item quantity, capacity and
  interaction spots.
- `ItemStackStore` owns material/fuel quantities.
- map, location and path/navigation owner stores own target cells, placement
  legality and route basis.
- Worker/headless runtime owns command sequence, session, deduplication window,
  deterministic tick phase and read-model publication.

Non-owners:

- React, Pixi and Electron are read-model consumers and command senders only.
- Protocol adapters validate and route; they do not mutate simulation facts.
- Renderer job markers are projections, not jobs.
- Local shell adapters are development aids only and cannot satisfy this ADR.

### Determinism, Replay And Save

Accepted commands are deterministic inputs. The Worker/headless runtime must
commit them in total order:

1. main message sequence;
2. command ordinal inside the accepted batch;
3. command kind priority if a domain queues follow-up domain commands;
4. target stable id;
5. command id as a final deterministic tie-breaker.

Each accepted command must enter the focused command log tail with:

- schema version and `playableCommandContractVersion`;
- session id and main sequence;
- command ordinal and command id;
- command kind and validated payload;
- basis fields used for validation;
- accepted/rejected status for diagnostics;
- committed tick and resulting order/job refs when accepted.

Rejected commands do not mutate world state. They may be retained in a bounded
diagnostic command-result ring and command tail for replay diagnostics, but the
authoritative replay input for world mutation is the accepted command stream.

Save/load for the focused slice must restore command sequence, command dedupe
window if used, owner stores, job states, reservation claims and random stream
positions before resuming. Derived read-model state, job markers and UI action
availability are rebuilt from owner stores after load.

No public save compatibility promise is introduced. If implementation adds or
changes focused save sections or command-tail fields, it must add section
versions, fail-closed validation and focused migration or fixture regeneration
evidence. Public migration remains owner-gated under the M7/M8 save policy.

### Schema And Versioning

Implementation of this ADR requires a protocol/schema gate before product code
changes:

- bump `SIM_SCHEMA_VERSION` for the incompatible `PlayerCommand` and
  `CommandResult` payload shapes;
- add `playableCommandContractVersion = 1`;
- add `playableCommandReadModelVersion = 1`;
- update shared protocol types, validators, protocol tests and Worker parity
  tests;
- reject unsupported schema/contract/read-model versions with structured
  reasons;
- update save/replay command-tail shape only through versioned focused
  sections or focused fixture regeneration.

No cross-version hot connection is supported. A v1 Worker rejects v2 command
payloads; a v2 Worker rejects unsupported old playable command schemas unless
the task explicitly implements a compatibility adapter.

## Consequences

### Positive

- The first player action can be reviewed through a concrete Worker command,
  job/order, pawn and blocked-reason chain.
- UI remains thin: it builds commands from read-model basis and renders
  read-only outcomes.
- Per-command results make batch success different from individual command
  success.
- Blocked states become localized, structured and testable instead of prose.
- Save/replay can treat accepted commands as deterministic inputs.

### Negative

- WM-0150/WM-0152 cannot use the existing v1 `PlayerCommand` shape unchanged.
- Protocol validators and Worker tests need real expansion before UI wiring.
- Read models must carry more basis/version data than the current summary-only
  surfaces.
- Existing command-result consumers must migrate to batch plus per-command
  semantics when schema v2 is implemented.

## Risks And Mitigations

- Risk: command acceptance is confused with immediate job completion.
  Mitigation: result status distinguishes accepted/rejected from
  `initialState`, and read models carry ongoing job state.
- Risk: no-worker or no-path states become UI-invented explanations.
  Mitigation: reason codes and parameters originate from Worker/headless
  validation, work selection, pathing, reservation or job-driver code.
- Risk: priority commands force a full map scan.
  Mitigation: target resolution, WorkOffer lookup and pawn selection must use
  indexed lanes, candidate caps and Top-K metrics.
- Risk: stale commands are accepted against changed targets.
  Mitigation: commands include basis fields and the Worker rejects mismatched
  owner/content/map/job/reservation versions.
- Risk: save/replay cannot reproduce command consequences.
  Mitigation: accepted command payloads, basis, sequence and resulting order/job
  refs enter the focused command tail.
- Risk: protocol churn breaks existing M1-M8 parity tests.
  Mitigation: schema bump, explicit validators and compatibility tests must
  land before client wiring.

## Contracts Introduced

- `playableCommandContractVersion = 1`.
- `playableCommandReadModelVersion = 1`.
- Player commands in scope: `PrioritizeLampWork` and `QueueSimpleBuild`.
- Batch-level command acceptance is separate from per-command results.
- Required structured blocked reason codes:
  `missing_resource`, `no_path`, `no_worker`, `invalid_target`,
  `stale_command`, `rule_policy_denial`.
- Read-model projections must expose target action availability, order/job
  markers, pawn states, build/lamp progress, resources/alerts and basis fields.
- React, Pixi and Electron may send commands and render projections only. They
  must not mutate world state, create jobs, reserve materials, move pawns,
  complete builds or repair stale projections.

## Performance Implications

- Command validation must be O(1) or bounded by explicit target/candidate caps.
- `PrioritizeLampWork` must resolve through lamp/build target indexes and
  WorkOffer buckets, not by scanning every map cell, lamp, worker or job.
- `QueueSimpleBuild` placement validation may inspect the blueprint footprint
  and bounded interaction cells only.
- Worker selection uses indexed work-offer, ability/schedule and path
  candidate lanes with task-defined caps and stable Top-K ordering.
- Read-model projection drains exact dirty target/job/pawn/resource keys.
  It must not rebuild the whole player UI model every tick.
- Metrics for the implementation slice must report command count, rejected
  reason counts, candidate visits, candidate-cap hits, exact path requests,
  no-worker counts, material reservation conflicts, stale command rejects,
  read-model dirty backlog, command-result queue depth and projection bytes.
- Existing 10 percent warning and 20 percent blocking benchmark thresholds are
  unchanged.

## Migration

Implementation migration from the current protocol is staged:

1. Add schema v2 protocol types, constants and validators while keeping message
   family names.
2. Add Worker handling that rejects unsupported old/new schema combinations
   fail-closed.
3. Add command-tail version fields for focused save/replay, or regenerate
   focused fixtures with explicit report evidence if no public migration is
   promised.
4. Add read-model basis/action/job/pawn projections.
5. Wire UI command builders to the new read-model templates.
6. Remove or quarantine shell-local WM-0138-style feedback from the default
   player path once authoritative command wiring lands.

No public save migration is required before this slice because public save
compatibility remains owner-gated. If any implementation task tries to support
older focused saves, it must do so through section-owned migration tests rather
than normal runtime conditionals.

## Rollback

Before implementation, rollback is deleting this ADR and removing its index and
protocol-document references.

After protocol implementation but before UI wiring, rollback is reverting the
schema v2 protocol/Worker task branch and keeping the old v1 protocol as the
only supported runtime.

After focused save/replay fixtures include command-tail v2 data, rollback must
either revert the fixture-producing task or provide a fail-closed loader for
unsupported command-tail versions. Do not add UI-local authority, magic `Noop`
strings, unversioned caches, global scans, Promise-based job progress or
renderer-owned job state as a rollback bridge.
