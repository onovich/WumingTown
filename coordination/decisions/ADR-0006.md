# ADR-0006: M1 deterministic numeric and phase contracts

- Status: Proposed
- Date: 2026-06-24
- Owners: systems-architect
- Related tasks: WM-0018, WM-0019, WM-0021, WM-0022, WM-0023, WM-0024,
  WM-0025, WM-0026, WM-0027, WM-0028, WM-0029

## Context

M1 expands the accepted M0 deterministic foundation into a first authoritative
gameplay kernel. M0 already provides fixed 30 TPS ticks, explicit seed input,
named deterministic random streams, serializable RNG state, canonical world
hashing, generation-checked entity ids, stable ascending iteration, and a
deferred structural command commit buffer. ADR-0006 proposes the numeric,
phase, command-order, random-stream, and async-version contracts that downstream
M1 map, pathing, reservation, job, hauling, building, save, and Worker parity
tasks must use.

This ADR is a Proposed architecture decision only. It does not implement a new
PRNG, does not change `TICKS_PER_SECOND = 30`, does not change accepted M0
determinism behavior, does not add runtime dependencies, and does not accept the
final M1 contract without independent review.

Failure modes this proposal must prevent:

- Floating accumulation or platform timing entering authoritative world state.
- Local subsystem phases that race with command, job, reservation, or map
  updates.
- Random draws whose order changes when UI, renderer, pathing, or unrelated
  systems are added.
- Async pathing or read-model helpers overwriting newer authoritative state.
- Promise, Generator, coroutine, closure, thread completion order, or UI state
  becoming a job execution position.
- Stable-looking behavior that actually depends on JavaScript object key order,
  default sort behavior, unbounded scans, or platform scheduling.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Alternatives

- Floating-point authoritative state: easy for progress bars, path heuristics,
  scores, and animation values, but rejected. Repeated floating accumulation
  makes save/replay comparisons fragile, invites `NaN`/infinity boundary bugs,
  and hides rounding policy inside individual systems.
- BigInt or arbitrary precision decimal for all simulation math: precise, but
  rejected for M1 because it is slower, awkward with typed arrays and
  Transferable buffers, and unnecessary for the M1 scale if each lane declares
  integer ranges and overflow tests.
- Subsystem-local immediate writes: simple for a single feature, but rejected
  because reservations, jobs, location, map dirty queues, and read models would
  observe different partial worlds depending on call order.
- Async result applies when its Promise resolves: rejected. Browser Worker,
  Node, and future helper threads can complete in different orders; completion
  timing cannot decide authoritative state.
- Existing M0 primitives plus explicit numeric scales, tick phases, stable
  ordering, named stream usage, and versioned async result commits: proposed.
  This keeps the accepted M0 foundation and narrows the missing M1 contracts.

## Decision

Use the accepted M0 fixed-tick and named-random-stream foundation as the base
contract, then add M1-specific integer/fixed-point scales, phase boundaries,
command ordering, async result rejection, and stable ordering rules.

### Inherited M0 constraints

- `Tick` remains a non-negative safe integer.
- `TICKS_PER_SECOND` remains `30`; changing it requires a separate accepted ADR
  and migration plan.
- The current `NamedRandomStreams` facility remains the M1 PRNG surface. This
  ADR defines stream use, ordering, and persistence expectations; it does not
  replace the concrete M0 PRNG implementation.
- Headless snapshots and Save Container v1 persist deterministic random stream
  seed and stream positions, not `Math.random()` state or platform entropy.
- Existing structural command priority from WM-0007 remains valid unless a
  later accepted ADR supersedes it: destroy, detach, attach, set, allocate; ties
  use entity index, then command insertion sequence.

### Numeric conventions

All authoritative numeric state uses safe integers and integer typed-array
lanes. External data is parsed as `unknown`, range-checked, scale-checked, and
rejected before mutating owner stores. Diagnostic timing can use real elapsed
time outside authoritative state, but timing values are excluded from world
hashes, read-model hashes, save authority, and command decisions.

| Surface           | Proposed convention                                                                                                                                                                                  | Rounding and invariants                                                                                                                                                                              |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Map coordinates   | Cell coordinates are signed safe integers. Chunk/cell traversal order is by integer chunk key, then `y`, then `x`, unless a narrower task defines a reviewed key order.                              | Coordinates never use floating point in authority. Render projection may expose Q16 lanes as read models only.                                                                                       |
| Map cost          | `PathCostMilli`: non-negative integer cost units where `1000` is one neutral orthogonal cell step. Cell base costs fit in `Uint32`; accumulators such as `g`, `h`, and `f` are safe integers.        | Impassable cells use an explicit blocked flag, not infinity. Heuristics must be integer and admissible for the movement set in use. Overflow or unsafe accumulator range is a structured path error. |
| Job progress      | Duration-only steps use integer tick counters. Work-amount steps use unsigned Q16.16 integer progress where `65536` is one work unit, with `requiredWorkQ16` and `workPerTickQ16` integer fields.    | Completion is `progressQ16 >= requiredWorkQ16`. Excess progress may be carried only when the job driver records it explicitly. No Promise, coroutine, or closure stores execution position.          |
| Item quantities   | `ItemQuantity`: non-negative integer in the smallest content-defined unit for the item definition. Stack capacity, storage capacity, reservation amount, pickup amount, and delivered amount use it. | Quantity deltas are all-or-nothing per transaction. Owner quantity cannot go below zero or above capacity. Display packs, weights, and fractional UI labels are read-model conversions only.         |
| Scores            | `ScoreMilli`: signed integer score units where `1000` is one whole score point. | Probabilities     | Authoritative chances use integer ratios such as `numerator` / `denominator`, with positive denominator and bounded numerator.                                                                       | A roll compares a named stream `Uint32` against an integer threshold derived from the ratio. Floating probability accumulation is forbidden.                                                         |
| Versions/counters | Store versions, command sequences, request sequences, RNG draw counts, and metric counters are non-negative safe integers unless a narrower typed lane is explicitly documented.                     | Wraparound is not allowed silently. Exhaustion is a structured failure or a save/migration blocker.                                                                                                  |

Runtime division is allowed only when the receiving lane declares the rounding
policy. Non-negative gameplay quantities, costs, and progress use floor
division. Signed score normalization uses truncation toward zero. Content
compilation should precompute scaled integer values when possible so hot loops
avoid division.

### Tick phases

Each authoritative tick runs these phases in order. A phase may enqueue commands
for a later phase, but it may not mutate owner state outside its allowed commit
surface.

1. `TickStart`: establish the current `tick`, deterministic context, and phase
   counters. No real time is read.
2. `ExternalCommandCommit`: drain validated external command batches whose
   `targetTick` equals the current tick. Commands are processed by ascending
   main-thread command sequence, then batch payload ordinal. Stale session,
   stale sequence, future-schema, past-target-tick, or malformed commands are
   rejected with structured reasons.
3. `VersionedAsyncResultCommit`: inspect completed async helper results, sorted
   by original request sequence and stable target id, not by completion order.
   Each result must carry the input versions it used. Results whose basis no
   longer matches current owner-store, map, navigation, job, reservation,
   content, or session versions are rejected and counted as stale.
4. `ScheduledSystemIntent`: scheduled systems read the committed state and
   derived indexes. They iterate stable owner/index orders, choose bounded
   candidates, draw random numbers only at reviewed draw sites, and enqueue
   domain commands or transactions. They do not rely on Promise chains,
   coroutine frames, closures, UI state, or thread completion order.
5. `DomainTransactionCommit`: apply domain queues in this fixed order:
   lifecycle cancellation/failure/completion cleanup, reservation release,
   reservation acquire/renew, item quantity transfer, job progress/status
   update, location/map occupancy update, map dirty/version update, and
   WorkOffer/index dirty marking. Each transaction either commits completely or
   leaves no owner-state partials.
6. `StructuralCommit`: apply entity/component structural commands through the
   accepted WM-0007 structural buffer order. Domain tasks that need an allocated
   entity id before dependent writes must make that dependency explicit rather
   than assuming allocate can precede same-buffer attach/set.
7. `DerivedRebuildBudget`: process deterministic dirty queues for derived
   indexes within the task-defined budget. Remaining backlog is observable
   metric state, not hidden work.
8. `HashAndSnapshotBasis`: compute deterministic hash inputs and read-model
   bases for this tick after all authoritative commits. Diagnostic elapsed time
   and platform queue latency are excluded.
9. `TickEnd`: increment to the next tick.

Fast-forward and batched Worker execution may run multiple ticks before yielding
to the message loop, but it must run every phase for every tick. It may reduce
read-model snapshot cadence; it may not skip authoritative phases.

### Command commit order

External and internal command queues must expose a total order before commit:

1. `targetTick`
2. phase bucket
3. command family priority defined by the owning domain contract
4. target kind priority, if the command can target multiple owner stores
5. target stable id, such as map key, entity index/generation, job id,
   reservation id, or content def index
6. source command sequence or enqueue sequence
7. payload ordinal inside the original command batch

Every domain that adds command families must document its priority table in the
implementation task or a later ADR. Default JavaScript object property order,
`Map`/`Set` insertion order, Promise resolution order, and default `Array.sort`
without a total comparator are not command order contracts.

### Stable iteration and ordering

- Entity iteration is ascending `EntityId.index`, validating generation before
  reads or writes.
- Map/chunk iteration uses explicit integer keys and a documented coordinate
  order. Save, replay, world hash, and read-model builders must use the same key
  order for the same surface.
- Content definitions use compiled `DefIndex` order from the content compiler,
  not source file order unless the compiler makes that order explicit.
- WorkOffer, path, and reservation candidate selection must use indexes and
  bounded Top-K selection. Unbounded global scans and unbounded sorts are
  forbidden in normal ticks.
- Bounded Top-K ordering is score descending, then domain priority, then stable
  target ids, then enqueue/request sequence. Top-K capacity is part of the
  domain contract and benchmark output.
- Sorting is allowed outside hot paths for load validation, debug tooling,
  tests, and bounded candidate sets only when a total comparator is supplied and
  the maximum size is documented.

### Deterministic random stream use

- Authoritative systems use `NamedRandomStreams.nextUint32` or approved derived
  helpers with explicit non-empty stream names.
- Stream names are domain-owned and stable, such as `story-director`,
  `world-generation`, `incident:<incidentId>`, or
  `anomaly:<entityIndex>:<generation>`. Names may include stable ids, not object
  references, memory addresses, UI ids, or thread ids.
- Random draws occur only after candidates are in stable order. Adding,
  removing, or moving a draw site is a gameplay determinism change and requires
  replay/hash tests that pin the new behavior.
- Random draws must not be used to break ties when deterministic stable keys are
  sufficient.
- Visual-only randomness belongs in renderer/UI state and must not consume
  authoritative streams or affect world hashes.

### Async result version checks

Any async helper result that can affect authoritative state or authoritative
read-model bases must carry:

- `requestSequence`
- `issuedTick`
- `sessionId`
- `contentManifestHash`
- relevant owner-store versions
- relevant entity ids and generations
- map/chunk/navigation/room/region versions when pathing or map queries are
  involved
- job and reservation versions when work execution or claims are involved
- requested candidate cap or budget, when the result depends on one

The authoritative runtime validates every basis field in
`VersionedAsyncResultCommit`. A mismatch produces a structured stale reason,
for example `StaleNavigationVersion`, `StaleEntityGeneration`,
`StaleReservationVersion`, `StaleJobVersion`, `StaleContentManifest`, or
`StaleSession`. Stale results are discarded; they are never patched by UI code,
never applied by a background thread, and never retried silently inside the same
command.

Async helpers may compute candidate data, paths, or read-model payloads. They
may not own job state, save/load state, reservation claims, item quantities, map
occupancy, or command cursors. A job waiting for an async path stores an
explicit serializable state such as `WaitingForPath(requestSequence, basis...)`,
not a Promise or coroutine frame.

### Explicit prohibitions

Authoritative state and authoritative decisions must not rely on:

- `Math.random()`
- `Date.now()`, `performance.now()`, timers, frame time, sleep duration, or wall
  clock
- Promise, Generator, coroutine, async stack, closure, or UI state as execution
  position
- worker thread completion order, OS scheduling, browser task timing, or
  platform latency
- unversioned caches, hidden singleton stores, or module-level mutable world
  state
- full-map or all-entity scans in normal pawn thinking, work selection, path
  selection, reservation acquisition, cleanup, or cache repair
- default object key traversal or default sort behavior as an authoritative
  ordering rule

### Explicit non-decisions

- This ADR does not implement fixed-point helpers, command phases, pathing,
  reservations, jobs, map stores, read models, or Worker protocol changes.
- This ADR does not change the M0 xorshift-based `NamedRandomStreams`
  implementation.
- This ADR does not define save container bytes; ADR-0005 owns Save Container
  v1 and its section versions.
- This ADR does not define Worker snapshot encoding; ADR-0004 owns snapshot and
  read-model payload versions.
- This ADR does not accept an M1 final decision. It remains Proposed until
  reviewed and accepted through the project workflow.

## Consequences

### Positive

- Numeric behavior becomes reviewable before map, pathing, jobs, reservations,
  and logistics independently choose incompatible scales.
- Every tick has a single partial-order contract, which makes Worker/headless
  parity and replay diagnostics easier to localize.
- Async pathing can be introduced without making browser or thread scheduling
  authoritative.
- Save Container v1 can persist compact integer state and stream positions
  instead of reconstructing behavior from floating values or hidden callbacks.

### Negative

- Downstream implementation tasks must define typed lanes, scale constants, and
  overflow behavior before feature behavior is visible.
- Explicit phases and transaction queues add upfront ceremony for simple
  gameplay actions.
- Candidate ordering and Top-K caps must be tested, benchmarked, and documented
  for each domain rather than relying on general-purpose collection behavior.

### Risks and mitigations

- Risk: fixed-point scales are too small for later content. Mitigation:
  section/schema versions include scale versions, and later milestones can
  migrate through explicit N-to-N+1 transforms.
- Risk: command phases become too rigid for feature work. Mitigation:
  downstream tasks may add domain command families inside the existing phase
  buckets with documented priority tables and tests.
- Risk: async stale rejection causes excessive recomputation. Mitigation:
  benchmarks must report stale rejection counts, queue depth, and accepted
  result counts so caps and invalidation scopes can be tuned without relaxing
  correctness.
- Risk: random stream names proliferate. Mitigation: stream names are
  domain-owned, persisted, and reviewed with replay tests; visual randomness
  stays outside authoritative streams.

## Contracts introduced

- Proposed deterministic contract version: `deterministicNumericPhaseVersion =
1` for M1 tests, reports, and future save/read-model metadata that need to
  name this contract.
- Authoritative numeric lanes use integers or documented fixed-point scales.
  `NaN`, infinity, floating accumulation, and ambient time-derived values are
  forbidden in owner state.
- Map path costs use `PathCostMilli` with `1000` as one neutral orthogonal cell
  step.
- Job work progress uses unsigned Q16.16 integer progress, or integer tick
  counters for pure durations.
- Item quantities use non-negative integer smallest units.
- Candidate scores use signed `ScoreMilli`.
- Tick execution follows `TickStart`, `ExternalCommandCommit`,
  `VersionedAsyncResultCommit`, `ScheduledSystemIntent`,
  `DomainTransactionCommit`, `StructuralCommit`, `DerivedRebuildBudget`,
  `HashAndSnapshotBasis`, and `TickEnd`.
- External and internal commands expose total ordering keys before commit.
- Async helper results are data with basis versions. Stale results are rejected
  inside the authoritative runtime and never applied by completion order.
- Random draws use explicit named streams, stable draw sites, and stable
  candidate order.
- Job execution position is explicit serializable data only.

- `git diff --check`

Future implementation verification required by this ADR:

- Numeric tests validate scale constants, min/max ranges, overflow rejection,
  division rounding, map cost accumulation, job Q16.16 progress completion, item
  quantity transaction bounds, and `ScoreMilli` ordering.
- Phase tests prove that a fixed command stream produces identical owner-store
  state and hashes when run in Node headless and browser Worker.
- Command ordering tests cover command family priority, target id tie-breaks,
  payload ordinals, and existing WM-0007 structural ordering.
- RNG tests pin stream names, draw counts, save/restore positions, and stable
  draw order after candidate selection.
- Async tests create path/read-model results with stale navigation, entity,
  job, reservation, content, session, and request versions, then prove the
  results are rejected with structured reasons.
- Job-driver tests prove waiting states are serializable data and no Promise,
  Generator, coroutine, closure, or UI state is needed to resume.
- Replay tests compare first-divergence output at deterministic
  `HashAndSnapshotBasis` ticks.

## Performance implications

- Integer and fixed-point lanes are compatible with typed-array owner stores,
  Transferable read-model buffers, compact save sections, and canonical hashes.
- Normal ticks must not allocate objects, arrays, strings, or closures in loops
  proportional to entities, cells, jobs, reservations, offers, or candidates.
- Normal ticks must not use unbounded `sort`, `map`, `filter`, `reduce`,
  `flatMap`, regular expressions, string concatenation, or full-world scans in
  hot loops.
- Bounded Top-K selection is allowed when the cap is explicit, the comparator is
  total, and benchmarks report candidate count, accepted count, rejected count,
  and cap hit count.
- Async version validation is constant per result basis field and must report
  accepted, stale rejected, invalid rejected, and queued result counts.
- Benchmarks that compare deterministic behavior must take measurements at
  phase boundaries, preferably after `HashAndSnapshotBasis`, so timing probes do
  not change the world being measured.
- Diagnostic elapsed time, Worker latency, and queue wait are performance
  metrics only. They are excluded from authoritative state and hashes.

## Migration

If accepted before implementation, migration is from the current M0 foundation
to M1 domain contracts:

1. Keep existing `Tick`, `TICKS_PER_SECOND`, `NamedRandomStreams`,
   generation-checked `EntityId`, stable iteration, structural command order,
   and canonical hashing behavior.
2. Add public scale constants, validators, and type aliases behind package
   public exports before domain tasks persist or hash new numeric lanes.
3. Add domain command family priority tables as each domain introduces command
   queues.
4. Add map/navigation/job/reservation/content version fields before any async
   helper result can affect authority.
5. Add tests that pin fixed-point ranges, phase order, command order, stream
   draw counts, and stale rejection before enabling Worker/headless parity gates.
6. Persist scale and contract versions through ADR-0005 sections only after
   their owner-store section versions exist.

Changing any accepted scale after saves or replay fixtures exist requires a
section-owned migration with old fixture input, new fixture output, hash
expectations, and explicit failure behavior for unsupported versions.

## Rollback

Before acceptance, rollback is deleting
`coordination/decisions/ADR-0006.md` and removing its ADR index link. No
runtime code, dependency, save file, or accepted decision depends on this
Proposed ADR.

If accepted before implementation and the contract proves too broad, rollback
is a superseding ADR that narrows numeric scales, phase buckets, or async basis
fields before downstream implementation tasks merge.

If an implementation task based on this ADR fails review before a save
compatibility promise exists, rollback is reverting that task branch and keeping
ADR-0006 Proposed or superseded. Do not merge temporary floats, global scans,
unversioned caches, UI-owned simulation repair, Promise-based job state, or
completion-order async commits as a bridge.

After M1 save/replay fixtures or Save Container v1 sections exist, rollback of
numeric scales or phase ordering requires migration tests and replay fixtures
that prove old saves either migrate deterministically or fail closed with
structured reasons.
