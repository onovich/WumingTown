# ADR-0016: Public playable Worker drain and terminal wait helper

- Status: Proposed
- Date: 2026-07-01
- Owners: systems-architect
- Related tasks: WM-0150, WM-0152, WM-0157, WM-0158, WM-0159, WM-0160

## Context

WM-0157 exposed the public browser Simulation Worker session. WM-0158 exposed
`UiDeltaPayload.playable?: PlayableProjectionV1`. WM-0159 exposed a legal
public helper for explicit WM-0150 playable tick advancement through the
existing `PlayerCommandBatch` path.

WM-0152 can now send authoritative lamp/build commands, receive reliable
`CommandResult` messages, read authoritative playable projections, and advance
the scenario without importing `sim-core` or hardcoding internal advance ids.
The remaining blocker is coordination of progress, not authority.

Current facts:

- `BrowserSimulationWorkerSession` exposes callbacks for raw messages and
  reliable messages, but no public one-shot helper to await the next
  authoritative playable `UiDelta`.
- Web has to run its own timer loop, send advance requests, read debug payloads
  and infer when command progression is terminal enough to stop.
- The focused WM-0152 web-shell E2E still times out after `120000ms` under that
  live loop, even though direct repro reaches accepted, moving, working and
  terminal lamp/build projection states.
- A longer Playwright timeout would not create a reviewed public contract. It
  would only make product tests depend on debug polling and browser timing.

WM-0152 needs a public, deterministic, awaitable drain surface that is owned by
the Worker/session boundary and based only on authoritative protocol
projections.

## Decision Drivers

- Preserve Simulation Worker / headless authority.
- Keep Web as a command sender and read-model consumer only.
- Avoid deep imports, `sim-core` dependency in Web, summary parsing, debug
  polling and shell-local gameplay authority.
- Reuse the existing `PlayerCommandBatch` and `UiDelta` message families where
  possible.
- Keep reliable `CommandResult`, `SaveReady` and fatal/error delivery separate
  from projection delivery.
- Avoid hidden timers, background clock loops, global Worker/session singletons
  and real-time simulation authority.
- Give WM-0152 a deterministic way to wait for command terminal/quiescent state
  without implementing product UI in the precursor.

## Considered Options

1. Increase the web-shell E2E timeout or keep debug polling in tests.
   Rejected. It does not create an architectural contract and leaves Web/E2E
   coupled to diagnostic payload timing.

2. Let Web compute terminal state from debug payloads or summary strings.
   Rejected. Debug and summaries are diagnostics, not authoritative public
   gameplay contracts.

3. Let Web import `@wuming-town/sim-core` terminal helpers.
   Rejected. It violates ADR-0001/ADR-0013 boundaries and moves simulation
   interpretation into the app.

4. Make `SetSpeed` / `Pause` start autonomous browser ticking.
   Deferred. Continuous play needs lifecycle, throttling, backpressure, replay
   and shutdown semantics. It is wider than the WM-0152 unblock.

5. Add a new top-level Worker message such as `DrainPlayable` or
   `PlayableTerminal`.
   Deferred. A protocol message may be justified later, but the existing Worker
   already emits the authoritative projection needed to derive terminal state.

6. Expose a public root `@wuming-town/sim-worker` drain/wait helper that uses
   existing explicit advance requests and resolves on authoritative playable
   `UiDelta` terminal/quiescent state.
   Chosen. It keeps the contract in the reviewed Worker/session package, avoids
   new message families, and lets Web await authoritative projections without
   debug polling.

## Decision

`@wuming-town/sim-worker` may expose public root browser session helpers for
awaitable playable projection progress. The exact names may vary during
implementation, but the public contract must include:

- a helper that posts or delegates to the existing explicit
  `advancePlayableCommandScenarioToTick` path and resolves when a playable
  `UiDelta` at or beyond that target tick arrives;
- a helper that repeatedly advances by explicit logical tick steps until either
  requested command ids reach terminal projection state or a maximum target tick
  is reached;
- exported result types that return the authoritative `UiDelta` message,
  `PlayableProjectionV1`, target tick and status such as `advanced`,
  `terminal` or `max_target_reached`;
- optional caller cancellation through explicit lifecycle, such as an
  `AbortSignal`, with all temporary subscriptions removed on resolve, reject or
  cancellation;
- no hidden Worker/session singleton, interval, timeout, animation frame,
  wall-clock simulation step or background loop.

Implementation-equivalent shape:

```ts
interface PlayableDrainRequest {
  readonly commandIds: readonly string[];
  readonly maxTargetTick: number;
  readonly stepTicks: number;
  readonly signal?: AbortSignal;
}

interface PlayableDrainResult {
  readonly status: "terminal" | "max_target_reached";
  readonly targetTick: number;
  readonly projection: PlayableProjectionV1;
  readonly uiDelta: Extract<
    SimulationToMainMessage,
    { readonly kind: typeof SIMULATION_TO_MAIN_MESSAGE_KIND.UiDelta }
  >;
  readonly postedMessages: readonly MainToSimulationMessage[];
}
```

The helper must classify terminal state only from protocol-owned
`PlayableProjectionV1` data. For requested command ids, terminal order states
are `blocked`, `completed`, `failed` and `canceled`. Active states are
`queued`, `claimable`, `claimed`, `moving` and `working`. Build completion,
pawn state and lamp state may be included as supporting evidence, but Web must
not be required to read debug payloads or synthesize terminal authority.

The helper is a deterministic session drain, not a gameplay clock. It may post
the next explicit target tick only after observing the prior authoritative
playable `UiDelta`. It must maintain at most one in-flight advance target per
helper invocation and unsubscribe all listeners when done.

`apps/web` may expose a thin adapter from `apps/web/src/simulation-worker-session.ts`
to this root helper. Product HUD, Pixi rendering and `shell-bootstrap` wiring
remain WM-0152 work and must not be implemented by WM-0160.

## Deterministic Clock Stance

Drain requests use explicit integer target ticks and step sizes. The Worker
remains the only authority that advances simulation state. The helper must not
call `Date.now()`, `performance.now()`, `setInterval`,
`requestAnimationFrame`, or `setTimeout` to decide simulation progress. Tests
may use their own timeouts as test guards, but real time is not part of the
public simulation contract.

## Save/Load Stance

No public save compatibility, migration or in-progress playable command
persistence is introduced. Drain requests are transient active-session client
coordination around authoritative Worker outputs. Persisting playable command
state still requires a separate versioned save/load task with fixtures and
migration evidence.

## Verification

Implementation must prove:

- `apps/web` imports only package roots and does not import
  `@wuming-town/sim-core` or worker internals.
- The drain helper uses existing `advancePlayableCommandScenarioToTick` /
  `PlayerCommandBatch` and existing `UiDeltaPayload.playable`; no new top-level
  Worker message family is added unless this ADR is superseded.
- The helper waits for a new authoritative playable `UiDelta` without debug
  payload polling.
- Accepted `PrioritizeLampWork` and `QueueSimpleBuild` command ids followed by
  the drain helper reach terminal lamp/build projection state in a real
  Chromium module Worker smoke.
- Reliable `CommandResult`, `SaveReady` and fatal/error delivery remains
  separately delivered.
- Destroy/cancel paths unsubscribe listeners and do not leave hidden sessions,
  timers or global state.
- No public save/release/telemetry/account promise is introduced.

## Performance Implications

The helper should replace hot debug polling with message-driven waiting. It
should allocate only per request/subscription, keep one in-flight advance target
per invocation, and avoid global scans or unbounded queues. It must not make
projection delivery block reliable message delivery.

## Rollback / Migration

Before WM-0152 consumes the helper, rollback is removing the helper, tests and
thin Web adapter.

After WM-0152 consumes it, rollback must return WM-0152 to a blocked state or
to a clearly labeled reviewed playback harness. It must not fall back to
debug-payload polling, summary parsing, app-owned terminal state, magic command
ids, deep imports, unversioned caches or public save compatibility.
