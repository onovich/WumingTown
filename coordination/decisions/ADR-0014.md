# ADR-0014: Public playable projection through Worker protocol

- Status: Proposed
- Date: 2026-06-30
- Owners: systems-architect
- Related tasks: WM-0150, WM-0152, WM-0157, WM-0158

## Context

WM-0157 added the public package-root browser session bridge, so `apps/web` can
now start the authoritative Simulation Worker and send WM-0150 playable
commands without deep imports. WM-0152 is still blocked because the public
Worker output does not carry a renderable authoritative playable projection.

Current public message facts:

- `RenderSnapshotPayload` carries `snapshotSequence`, `tick`, `entityCount`,
  optional `scenarioId`, `worldHash`, `readModelHash`, and `readOnly`.
- `UiDeltaPayload` carries `tick`, `summaries`, optional `scenarioId`,
  `readModelHash`, `detailHash`, and `readOnly`.
- The WM-0150 authoritative runtime has an internal `PlayableReadModel` with
  job markers, pawn state, build state, resource counts, alerts, basis hashes
  and owner versions.
- `packages/sim-worker/src/index.ts` currently collapses that internal
  projection to hashes, entity count and summary strings before publishing it.
- `apps/web` may consume `@wuming-town/sim-worker` and
  `@wuming-town/sim-protocol`, but it must not import `@wuming-town/sim-core`
  or package internals.

WM-0152 cannot truthfully render the lamp/build command chain from summaries or
from the WM-0151 shell-local projection harness. The missing surface must be
public, versioned, read-only, and owned by the Worker protocol.

## Decision drivers

- Preserve Simulation Worker / headless authority.
- Keep React, Pixi and Web as read-model consumers and command senders only.
- Reuse the existing Worker session and message families unless a new family is
  required.
- Keep command results reliable and separate from any latest-wins projection
  delivery.
- Avoid exposing `sim-core` types or internals directly to Web.
- Keep save/load promises explicit and fail-closed.
- Give WM-0152 enough data for HUD, Pixi job markers, pawn progress, placement
  basis and structured blocked reasons without adding product UI in the
  precursor.

## Considered options

1. Parse `UiDeltaPayload.summaries` into UI state.
   Rejected. Summary strings are diagnostics, not a stable schema. This would
   make prose into authority and lose structured blocked-reason parameters.

2. Let `apps/web` import `@wuming-town/sim-core` or internal Worker files.
   Rejected. This violates ADR-0001 package boundaries and makes the browser
   shell a simulation/projection owner.

3. Add a new top-level `PlayableProjection` Worker message family.
   Deferred. It may be justified when projection streams need different
   backpressure or reliability semantics, but the first focused slice can use
   existing `UiDelta` delivery with an explicit versioned payload field.

4. Publish a broad `WorldReadModel` directly from the Worker.
   Deferred. The current product `WorldReadModel` is UI-facing and partly
   product-gate-specific. A broad contract would couple simulation protocol to
   current Web presentation too early.

5. Add a protocol-owned `PlayableProjectionV1` field to `UiDeltaPayload`.
   Chosen. It extends the existing read-only UI delta path, keeps
   `CommandResult` reliability unchanged, and exposes only the focused
   authoritative projection needed by WM-0152.

## Decision

Add a public, protocol-owned `PlayableProjectionV1` shape and carry it on
`UiDeltaPayload` for the WM-0150 playable scenario:

```ts
interface UiDeltaPayload {
  readonly tick: number;
  readonly summaries: readonly string[];
  readonly scenarioId?: string;
  readonly readModelHash?: string;
  readonly detailHash?: string;
  readonly readOnly?: true;
  readonly playable?: PlayableProjectionV1;
}
```

The exact interface names may vary during implementation, but the public
contract must include these groups:

- `playableCommandReadModelVersion: 1`;
- basis fields: tick, snapshot sequence, world/read-model hash, content hash,
  target/map/reservation/job versions, and command basis fields needed for the
  next command;
- target/action projection for `PrioritizeLampWork` and `QueueSimpleBuild`,
  including action availability, disabled/blocked reason, and command payload
  template data;
- placement projection for the focused simple build slice, including anchor
  cell, orientation domain, valid/invalid state, footprint/interactions when
  available, and structured reason for invalid placement;
- order/job projection with order id, command id, job id/kind, target, owner,
  marker state, progress, required work and terminal/blocked reason;
- pawn projection with actor ref, display id, cell/tile, state, current
  job/order reference, path/intent target and structured blocked reason;
- build/lamp projection with build-site id/entity, blueprint id, anchor cell,
  material requirements/delivered/reserved/remaining amounts, build progress,
  lamp or lamp-gap state and lantern completion state;
- resource and alert projection with focused fuel/material counts,
  reservation-aware availability and structured blocked reasons.

`sim-worker` must map or copy from authoritative runtime/read-model data into
the protocol shape. `apps/web` must receive only the protocol shape and may
adapt it into existing renderer/HUD view models without inventing authority.
The adapter may compute presentation labels, coordinates and localization, but
it must not synthesize command acceptance, jobs, pawn motion, resource counts,
build progress or blocked reasons.

No new top-level message family is introduced by this decision. `UiDelta` is a
read-only latest projection. Reliable families such as `CommandResult`,
`SaveReady` and fatal/error messages must remain separately delivered through
the WM-0157 reliable path and must not be dropped because projection payloads
grow.

## Consequences

### Positive

- WM-0152 can render live Worker-owned command chains instead of the WM-0151
  playback harness.
- The projection contract becomes versioned and testable in `sim-protocol`.
- Web remains a consumer of `@wuming-town/sim-protocol` and
  `@wuming-town/sim-worker` only.
- Existing Worker session and browser smoke infrastructure can verify the path.

### Negative

- `UiDeltaPayload` grows for the playable slice.
- The protocol must now carry focused gameplay read-model types, so future
  changes require schema/version review.
- `sim-worker` needs an explicit mapping boundary from internal runtime data to
  public protocol data.

### Risks and mitigations

- Risk: projection payload becomes a hidden mutable cache in Web.
  Mitigation: define it as read-only, include hashes/versions, and keep command
  handling exclusively in the Worker.
- Risk: UI creates missing placement/action availability locally.
  Mitigation: require target/action and placement availability in the protocol
  projection with structured reasons.
- Risk: projection delivery blocks reliable command results.
  Mitigation: keep `CommandResult` in the WM-0157 reliable subscription path
  and add smoke coverage proving command results still arrive alongside
  playable projections.
- Risk: public save compatibility is implied.
  Mitigation: projection is derived and not a save contract. Save/load remains
  unsupported for in-progress playable command state unless a later task adds
  versioned focused save sections and migration evidence.

## Contracts introduced

- Public playable projection version: `playableCommandReadModelVersion: 1`.
- Public projection transport: `UiDeltaPayload.playable` or an equivalent
  versioned field on the existing `UiDelta` message.
- Public projection owner: Simulation Worker/headless runtime.
- Public consumers: Web, React and Pixi may render and build commands from the
  projection basis; they may not mutate world state.
- Public forbidden path: `apps/web` must not import `@wuming-town/sim-core`,
  `packages/sim-worker/src/*`, or parse summary strings as gameplay state.

## Verification

Implementation must run:

- protocol tests for the public playable projection types and optional
  `UiDeltaPayload.playable` shape;
- sim-core tests proving projection fields are derived from owner stores;
- sim-worker tests proving `UiDelta` carries the playable projection after
  init, command batches and `RequestUiDetail`;
- real Chromium worker-smoke proving WM-0150 commands yield both reliable
  `CommandResult` and renderable `UiDeltaPayload.playable`;
- package boundary checks proving Web imports only public roots.

Required command gates:

- `corepack pnpm typecheck`
- `corepack pnpm exec vitest run packages/sim-core/src/playable-command-slice.test.ts packages/sim-worker/src/playable-command-worker.test.ts packages/sim-protocol/src/protocol-validation.test.ts apps/web/src/simulation-worker-session.test.ts`

## Rollback / migration

Before WM-0152 consumes the projection, rollback is removing the new protocol
types, `UiDelta` field and Worker mapping.

After WM-0152 consumes it, rollback must return WM-0152 to a blocked state or
to explicitly labeled WM-0151 projection playback. It must not fall back to
summary parsing, UI-owned job state, deep imports, unversioned caches or public
save compatibility.

No public save migration is introduced. If a later task persists playable
command state, it must add versioned save sections, fail-closed loaders and
fixture or migration evidence.
