# ADR-0009: M4 core vertical slice architecture gate

- Status: Proposed
- Date: 2026-06-25
- Owners: systems-architect
- Related tasks: WM-0061, WM-0062, WM-0063, WM-0064, WM-0065, WM-0066,
  WM-0067, WM-0068, WM-0069, WM-0070, WM-0071

## Context

M4 is the first Wuming Town milestone that turns the three product pillars
into gameplay: lamps, Chronicle evidence, and obligations. It composes the M3
ordinary-life baseline with a focused borrowed-shadow crisis chain, director
recovery windows, save/replay, Worker projections, and benchmark gates.

This ADR is a planning gate. It does not implement runtime behavior, tests,
benchmarks, public Worker messages, save compatibility, schemas, package
boundary changes, new dependencies, M5 content, or UI authority.

Failure modes this gate must prevent:

- Treating lamps as visual-only UI state instead of authoritative integer rule
  fields.
- Treating Chronicle evidence as probability truth, a prose-only log, or a UI
  case board that owns facts.
- Letting obligations, town rules, or crisis state become hidden shared state
  inside the director, jobs, React, Pixi, Electron, Promise chains, closures, or
  thread-local callbacks.
- Letting the director repair or escalate the world by directly mutating facts
  outside normal owner-store commands.
- Persisting unversioned derived caches, read models, projections, evidence
  summaries, or dawn review text as authority.
- Weakening M0-M3 deterministic, save/replay, Worker/headless, and benchmark
  evidence to make M4 pass.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered options

- Monolithic story director brain: rejected. It is easy to stage a crisis, but
  it hides ownership, encourages global scans, makes recovery windows
  non-replayable, and invites direct fact mutation outside worker authority.
- UI-owned investigation board: rejected. UI can show cases and hypotheses, but
  evidence, knowledge dissemination, rule confirmation, and dawn review facts
  must be produced by authoritative simulation data.
- Visual-only lamp network: rejected. M4 needs lamp state that affects paths,
  safety, anomaly activation, obligations, and evidence. Visual brightness can
  be derived, but `humanClaim` and lamp gaps are simulation facts.
- Persisted projections and evidence summaries for fast load: rejected for M4.
  Focused save/replay may persist owner-store facts and deterministic streams;
  derived indexes, Worker projections, dawn review text, and UI summaries
  rebuild after validation.
- Owner-store M4 facts with versioned derived indexes, bounded selection, and a
  constrained director: proposed. Each mutable fact has one owner. Derived
  surfaces name their source versions. Actor and director decisions use indexed
  candidate buckets and Top-K limits.

## Decision

Keep Simulation Worker or Node headless as the only authoritative world writer.
`sim-core` remains free of DOM, React, PixiJS, Electron, Node filesystem, real
time, and ambient randomness. M4 introduces the architecture gate contract
`m4CoreVerticalSliceArchitectureGateVersion = 1`.

### Derived indexes, dirty queues, and caps

M4 derived surfaces are versioned, scratch, or rebuilt from owner stores:

- `LampNetworkDirtyQueue` names exact lamp ids, link groups, cells, rooms, and
  chunks; normal dirty drain is budgeted and reports backlog.
- `LampGapIndex` buckets active `shadowGap` cells by region, room, dusk/night
  window, lamp group, and network version.
- `EvidenceCandidateIndex` buckets observations, testimony, traces, sources,
  hypotheses, contradictions, confirmed rules, and dissemination rows by case
  and owner version.
- `ObligationDueIndex` buckets due obligations by debtor, creditor, role,
  region, due window, visibility, and owner version.
- `TownRuleComplianceIndex` buckets residents and rules by known rule,
  region/time scope, enforcement basis, exception flags, and owner versions.
- `CrisisTransitionIndex` buckets anomaly transition candidates by active
  crisis id, stage, lamp gap, target identity, evidence status, and owner
  versions.
- `DirectorIncidentIndex` buckets legal incident and recovery opportunity
  candidates by pressure theme, cooldown, recovery window, and aggregate owner
  versions.
- `M4WorkerProjectionDirtyQueue` names exact lamp, case, obligation, rule,
  crisis, director, resident, and dawn-review projection keys.

Default caps apply until a downstream task narrows them with tests and metrics:

| Selection surface | Candidate visit cap | Retained Top-K | Exact path cap |
| --- | ---: | ---: | ---: |
| Lamp dirty drain per tick | 64 changed lamp/cell keys | 0 | 0 |
| Lamp-gap risk candidates per actor pass | 24 gaps | 12 | 4 |
| Evidence support candidates per case update | 32 rows | 16 | 0 |
| Dissemination/compliance candidates per resident pass | 24 rules/facts | 12 | 0 |
| Obligation due candidates per role/window pass | 24 obligations | 12 | 0 |
| Crisis transition candidates per crisis tick | 16 transitions | 8 | 0 |
| Director incident/recovery candidates per pressure pass | 12 candidates | 6 | 0 |
| Dawn review retained causal rows | 64 rows | 32 rows | 0 |

Top-K ordering is deterministic: score descending, domain priority, stable
owner id, then stable enqueue sequence. Random choice requires named stream
draws with replay tests.

### Save/replay and migration scope

M4 save/replay is focused to the scenario harness. It may snapshot:

- scenario id, requested seed, authoritative seed, content hash, command stream
  hash, tick, checkpoint tick, and architecture gate version;
- M0-M3 owner stores already approved by earlier gates;
- M4 owner stores named in this ADR;
- named random streams and draw counts;
- `JobCoreStore`, M4 job/crisis driver fields when present, and
  `ReservationLedger`;
- command log tail and periodic hash diagnostics;
- compact ReasonTrace and metric summaries for diagnostics only;
- read-only projection hashes for comparison.

Load must validate magic, versions, id ranges, integer lanes, sorted records,
entity generations, obligation states, evidence support tiers, crisis states,
director recovery windows, random stream positions, command continuity, and
projection hashes in scratch state before mutating a live world. It then
rebuilds lamp, gap, evidence, dissemination, obligation, town-rule, crisis,
director, WorkOffer, path, read-model, dawn-review, and metric surfaces before
the first resumed tick.

Public save compatibility, public schema migration, platform save UI, codec
dependency, package-boundary changes, or cross-version guarantees are
non-decisions here and block downstream work until a separate gate is reviewed.

### Director boundaries

The director may read aggregate pressure, active anomaly count, unresolved
case count, due obligation count, lamp-oil days, injury/mental-risk metrics,
recent deaths, weather windows, and recovery cooldowns. It may choose legal
incident or recovery opportunity commands from a bounded candidate set.

The director may not:

- mutate lamp fuel, evidence truth, obligations, town rules, health,
  relationships, crisis state, deaths, or resources directly;
- create pressure by bypassing normal job, reservation, item, or path rules;
- erase evidence, forgive debts, or suppress consequences during recovery;
- schedule unrelated high-pressure incidents inside a recovery window unless a
  separate task documents a critical invariant emergency;
- use real time, UI state, thread scheduling, or ambient randomness.

## Consequences

### Positive

- M4 implementers get named owner surfaces before touching runtime behavior.
- Lamps, evidence, obligations, crisis state, and director windows can be
  saved, replayed, benchmarked, and explained through shared structured facts.
- Reviewers can reject hidden shared state, global scans, UI authority,
  unversioned caches, public contract drift, and director overreach using one
  gate.

### Negative

- M4 requires several owner stores and indexes before the first visible crisis
  feels complete.
- Focused save/replay and Worker parity tasks must rebuild more derived
  surfaces than M3 did.
- Some desirable product UI, such as a rich case-board editor, remains blocked
  until read-only projections and owner facts are stable.

### Risks and mitigations

- Risk: evidence support feels opaque. Mitigation: the scenario proves one
  rule with named evidence classes and dawn review causal rows.
- Risk: lamp updates become map-scale diffusion. Mitigation: lamp dirty queues,
  linked groups, chunk/room keys, budgeted drain, and benchmark counters are
  required before broad use.
- Risk: obligations become vague narrative flags. Mitigation: obligation rows
  name creditor, debtor, type, due window, fulfillment, violation, source, and
  structured state.
- Risk: director windows hide failures. Mitigation: recovery decisions emit
  candidate/rejection traces and cannot mutate owner facts directly.
- Risk: M4 drifts into M5 content. Mitigation: downstream packets forbid broad
  anomaly roster, faction campaign, season event pool, data-mod production, and
  M5 task creation.

## Contracts introduced

- M4 architecture gate contract:
  `m4CoreVerticalSliceArchitectureGateVersion = 1`.
- Authoritative mutation remains only in Simulation Worker or Node headless.
- M4 owner stores are `LampNetworkStore`, `ChronicleCaseFileStore`,
  `EvidenceFactStore`, `KnowledgeDisseminationStore`, `ObligationStore`,
  `TownRuleStore`, `AnomalyRuleStore`, `BorrowedShadowCrisisStore`,
  `DirectorPressureStore`, `M4ReasonTraceStore`,
  `M4FocusedSaveReplayHarness`, and read-model builders.
- `MapGrid`, `LocationStore`, `ReservationLedger`, `JobCoreStore`,
  M3 ordinary-life stores, named random streams, and Worker read-model builders
  remain earlier owner surfaces and must not be bypassed.
- Lamps own simulation rule fields; visuals are projections.
- Chronicle evidence rows are structured facts and versioned knowledge, not
  hidden probability truth or UI-only prose.
- Crisis progress and director recovery windows are explicit serializable
  state.
- Focused M4 save/replay does not approve public save compatibility.
- Worker projections are read-only and must carry basis metadata and hashes.

  contradiction handling, confirmed-rule gates, dissemination scope, and no
  UI-only case authority.

  inheritance/visibility basis, compliance reasons, and bounded candidate
  reads.
- Crisis tests prove explicit state transitions, low-risk evidence before
  escalation, resolution paths, terminal reasons, and no coroutine/Promise
  progress.
- Director tests prove bounded incident/recovery candidates, structured
  rejection reasons, cooldowns, recovery windows, named random streams, and no
  direct owner-fact mutation.
- Save/replay tests prove focused checkpoint validation, derived rebuild before
  resumed ticks, uninterrupted-vs-resume hash equality, stream restore, and
  structured divergence output.
- Worker parity tests prove browser Worker and Node headless run the same M4
  command stream with matching authoritative hashes and read-only projection
  hashes.
- Invariant tests fail on M0-M3 regression, reservation leaks, stale offers,
  negative resources, unbounded queues, hash divergence, lamp dirty backlog
  growth, evidence drift, obligation leaks, invalid crisis transitions,
  director recovery-window violation, public protocol/save/schema drift, UI
  mutation, and M5 task/content creation.

## Performance implications

Normal M4 ticks must not allocate objects, arrays, closures, or strings in
loops proportional to actors, map cells, lamps, evidence rows, obligations,
town rules, crisis transitions, director candidates, jobs, reservations, paths,
or read-model rows.

Required M4 metrics include lamp dirty queue peak/final backlog, lamp group
updates, active gap count, evidence row count, evidence support candidate
visits, confirmed-rule count, dissemination dirty backlog, obligation due and
violation counts, town-rule compliance candidate visits, crisis transition
counts, active/recovered crisis count, director pressure sample count, recovery
window count, rejection reasons, ReasonTrace capacity use, save bytes,
load/rebuild time, Worker projection bytes, checkpoint hashes, and M0-M3
regression results.

Existing benchmark policy remains unchanged: key benchmark P95 regression over
10 percent requires explanation, and over 20 percent blocks merge by default.
M4 must add measurement, not threshold weakening.

## Rollback / migration

Before downstream implementation, rollback is deleting
`coordination/decisions/ADR-0009.md`, reverting the M4 scenario/roadmap/report
files and proposed WM-0062 through WM-0071 task packets, and restoring project
state to M3 closed / M4 planning not started.

After implementation starts, migration is incremental:

1. Keep M0-M3 owner stores, fixed tick, named streams, Worker authority,
   save/replay discipline, and benchmark baselines intact.
2. Add M4 owner stores only in their owning tasks.
3. Add exact dirty queues and owner-version fields before derived indexes or
   read models can influence choices.
4. Add focused tests and metrics for each owner surface before integrating the
   scenario.
5. Treat public Worker protocol, public save/schema, dependency, package
   boundary, UI authority, real-time/random authority, hidden shared state,
   global scan, or M5 need as a blocker.

If a downstream task violates this gate, rollback is reverting that task
branch. Do not merge temporary global scans, unversioned caches, UI-authority
fixes, direct director mutation, persisted projections, public contract drift,
or M5 scaffolding as bridges.
