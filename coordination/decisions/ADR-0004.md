# ADR-0004: M1 Worker snapshot and read-model encoding

- Status: Proposed
- Date: 2026-06-24
- Owners: systems-architect
- Related tasks: WM-0017, WM-0028, WM-0029, WM-0030

## Context

M1 needs browser Worker and Node headless execution to expose the same
authoritative simulation state through read-only projections. The immediate
consumers are render snapshots for Pixi, UI detail panels for React, structured
metrics, and the parity harness that compares browser Worker and Node runs.

The existing Worker protocol already defines versioned envelopes, session ids,
sequence streams, render snapshots, UI detail requests, metrics samples,
backpressure, and stale sequence rejection. ADR-0004 proposes the M1 encoding
contract that downstream implementation tasks must satisfy. It does not accept
or implement the decision, and it does not grant UI, Pixi, React, or Electron
write authority over simulation state.

Failure modes this proposal must prevent:

- UI-owned read-model state becoming a second authoritative world copy.
- Renderer or React code reading mutable `sim-core` stores directly.
- Partial or stale snapshots being merged into current UI state.
- Slow main-thread rendering forcing unbounded Worker queues.
- Snapshot, UI detail, or metrics code depending on SharedArrayBuffer or a
  deployment isolation decision that M1 has not made.
- Browser Worker and Node headless producing matching world hashes but different
  read-model hashes because encoding order, versioning, or stale result handling
  is underspecified.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered options

- Full world JSON per tick: easy to inspect, but rejected because it would
  allocate heavily, expose too much authoritative structure to UI, encourage
  UI-side corrections, and make headless/Worker parity depend on object property
  order.
- Renderer or React reads `sim-core` stores directly: rejected. It violates the
  Worker authority boundary, bypasses protocol validation, and cannot run in
  Node headless and browser Worker through the same public read-model contract.
- Diff-only render patches with no complete snapshot stream: lower bandwidth in
  steady state, but rejected as the M1 base contract because missed or stale
  patches require complex recovery and make render correctness depend on
  UI-owned mutable state. Deltas may be introduced later only as derived,
  versioned optimizations with a complete snapshot recovery path.
- SharedArrayBuffer triple-buffering as the required transport: rejected for M1.
  It may be a future optimization after deployment and security review, but
  ADR-0004 must work with ordinary Transferable ArrayBuffers and structured
  messages.
- Transferable SoA render snapshots plus bounded UI detail/read-model messages:
  proposed. The Worker or Node runtime builds read-only projections from
  authoritative stores, transfers complete snapshot buffers when needed, and
  records metrics and hashes using a canonical lane order.

## Decision

Use versioned read-model envelopes for M1 render snapshots, UI detail results,
metrics samples, and parity hash records. The Simulation Worker or Node headless
runtime remains the only authority for world mutation. UI and renderer packages
consume projections and may discard stale projections, but they may not patch,
repair, or commit world state.

### Versioning

Every read-model envelope keeps the existing Worker protocol header fields:
`protocolVersion`, `schemaVersion`, `sessionId`, and top-level `sequence`.
ADR-0004 additionally proposes stream-specific version fields:

| Field                         | Initial value | Applies to                                               |
| ----------------------------- | ------------- | -------------------------------------------------------- |
| `renderSnapshotSchemaVersion` | `1`           | `RenderSnapshot` header and lane meanings                |
| `uiReadModelSchemaVersion`    | `1`           | `UiDelta` summaries and `RequestUiDetail` responses      |
| `metricsSchemaVersion`        | `1`           | `MetricsSample` field names, units, and counter meaning  |
| `readModelHashVersion`        | `1`           | Canonical snapshot/detail hash traversal order and scope |

Changing lane order, units, enum domains, id encoding, or hash traversal scope
requires the relevant stream-specific version to change. Incompatible changes
also require the top-level `schemaVersion` to change. M1 does not support
cross-version hot connections; the Worker rejects unsupported versions at the
boundary with structured reasons.

### Render snapshot encoding

`RenderSnapshot` is a complete render projection for one authoritative tick. It
is not a save format, not a UI mutation surface, and not a complete gameplay
state dump.

The snapshot header contains:

- `renderSnapshotSchemaVersion`
- `snapshotSequence`
- `tick`
- `lastAppliedCommandSequence`
- `entityCount`
- `laneCount`
- `snapshotBytes`
- `worldHash`
- `readModelHash`

`worldHash` is the authoritative simulation hash for the same tick.
`readModelHash` is the canonical hash of the encoded render projection only.
Timing and transport metrics are excluded from both hashes.

The M1 render lanes are structure-of-arrays buffers in a stable lane order:

| Lane                | Shape                      | Notes                                                                          |
| ------------------- | -------------------------- | ------------------------------------------------------------------------------ |
| `entityIndex`       | `Uint32Array(entityCount)` | Entity id index, paired with generation.                                       |
| `entityGeneration`  | `Uint32Array(entityCount)` | Generation validation data; no object references cross the boundary.           |
| `xQ16`              | `Int32Array(entityCount)`  | Fixed-point render x coordinate.                                               |
| `yQ16`              | `Int32Array(entityCount)`  | Fixed-point render y coordinate.                                               |
| `layer`             | `Int16Array(entityCount)`  | Render ordering layer, not authority for map membership.                       |
| `spriteDef`         | `Uint32Array(entityCount)` | Compiled catalog index.                                                        |
| `animationState`    | `Uint16Array(entityCount)` | Compiled animation state enum.                                                 |
| `animationFrameQ16` | `Uint32Array(entityCount)` | Deterministic fixed-point animation progress when simulation-owned.            |
| `facing`            | `Uint8Array(entityCount)`  | Direction enum.                                                                |
| `renderFlags`       | `Uint32Array(entityCount)` | Visibility, selection eligibility, alert markers, and non-authoritative flags. |
| `statusBadgeDef`    | `Uint32Array(entityCount)` | Optional compiled status badge; zero means none.                               |
| `appearanceVariant` | `Uint16Array(entityCount)` | Deterministic visual variant index.                                            |

The Worker may reuse internal staging buffers, but once a buffer is posted as a
Transferable it is no longer read or written by the Worker. The main thread only
accepts complete snapshots whose `sessionId` matches and whose
`snapshotSequence` is greater than the last accepted snapshot for that session.
It must never merge lanes from two snapshot sequences or read a buffer that is
still being written.

Render snapshot ordering is deterministic: entities are encoded in stable
ascending `EntityId(index,generation)` order after filtering by
simulation-owned render visibility. UI selection, hover state, camera position,
and local renderer interpolation are not part of the authoritative snapshot
hash.

### UI summaries and detail behavior

Normal `UiDelta` messages carry bounded summaries keyed by authoritative ids and
basis metadata:

- `uiReadModelSchemaVersion`
- `uiDeltaSequence`
- `basisSnapshotSequence`
- `basisTick`
- `basisWorldHash`
- bounded summary payloads for alerts, selected entity summaries, work/job
  summaries, ReasonTrace excerpts, and metrics panel labels

`RequestUiDetail` is an explicit read-only request. The request includes:

- `requestId`
- `requestSequence`
- `targetKind`
- `targetId`
- target generation or version when the target has one
- `basisSnapshotSequence`
- `basisTick`
- requested detail level and maximum result budget

The response is a `UiDelta` detail entry that echoes `requestId`, target
identity, basis metadata, and either a bounded detail payload or a structured
rejection reason. Detail payloads may include full pawn thought summaries,
recent bounded ReasonTrace data, job/reservation explanations, path/version
diagnostics, and map-cell inspection data. They may not include writable store
references, mutation callbacks, unbounded world scans, or save/load authority.

The UI discards a detail response when any of these are true:

- `sessionId` does not match the active session.
- `requestId` is no longer the latest request for the active panel target.
- `basisSnapshotSequence` is older than the accepted panel target basis.
- target generation or owner-store version no longer matches the selected
  target.

The Worker also rejects stale or invalid detail requests before emitting a
payload when target generation, map/navigation version, job version, or
reservation ledger version no longer matches the request basis. Stale detail
results use structured reasons such as `StaleReadModelBasis`,
`UnknownReadModelTarget`, or `ReadModelBudgetExceeded`; they are never repaired
by UI-local mutation.

### Sequence handling and stale result rejection

ADR-0004 keeps separate monotonic streams:

- top-level Worker output `sequence` for every Simulation-to-Main envelope
- `snapshotSequence` for render snapshots
- `uiDeltaSequence` for UI summary/detail messages
- `metricsSequence` for metrics samples
- main-thread `requestSequence` for `RequestUiDetail`
- main-thread command sequence for `PlayerCommandBatch` and lifecycle messages

The Worker records the highest processed main-thread sequence per session and
rejects stale main-to-Worker messages. The main thread records the highest
accepted Worker output sequence, snapshot sequence, UI delta sequence, and
metrics sequence per session and discards older messages.

Any asynchronous simulation helper result that feeds a read model must carry the
input versions it was computed against. The Simulation Worker or Node headless
runtime validates those versions before publishing the result. Stale async
results are counted and rejected inside the authoritative runtime; UI must not
resolve them by applying old data to current read-model state.

### Backpressure

The snapshot channel is coalescible and bounded. When the main thread or test
harness cannot consume render snapshots fast enough, the Worker may overwrite or
drop unread `RenderSnapshot` outputs and keep only the latest complete snapshot
for the session. This does not roll back simulation ticks.

Reliable message families are not dropped for render backpressure:
`CommandResult`, `AlertBatch`, `SaveReady`, and `FatalSimulationError`.
`UiDelta` summaries and `MetricsSample` messages may be coalesced under
explicit per-stream limits, but request-scoped UI detail responses must either
return a bounded result or a structured rejection reason.

ADR-0004 does not require SharedArrayBuffer, Atomics, cross-origin isolation, or
deployment changes. Those may be evaluated later as transport optimizations only
after metrics show Transferable snapshots are insufficient.

### Metrics fields

`MetricsSample` uses integer counters and explicit units. Non-deterministic
timings are diagnostic data and are excluded from world and read-model hashes.

Required M1 fields:

| Field                                | Unit / type  | Notes                                                        |
| ------------------------------------ | ------------ | ------------------------------------------------------------ |
| `metricsSchemaVersion`               | integer      | Metrics payload version.                                     |
| `metricsSequence`                    | integer      | Monotonic metrics stream sequence.                           |
| `tick`                               | integer      | Authoritative tick at sample boundary.                       |
| `worldHash`                          | integer/hash | Authoritative hash for this tick or latest sampled tick.     |
| `latestSnapshotSequence`             | integer      | Latest produced render snapshot sequence.                    |
| `latestAcceptedCommandSequence`      | integer      | Latest command sequence committed by simulation.             |
| `renderEntityCount`                  | count        | Entity rows encoded in latest snapshot.                      |
| `renderSnapshotBytes`                | bytes        | Total transferable bytes for latest snapshot.                |
| `snapshotsProduced`                  | count        | Produced since session start or metrics reset.               |
| `snapshotsDroppedForBackpressure`    | count        | Coalesced or overwritten before consumption.                 |
| `uiDeltasProduced`                   | count        | UI summary/detail outputs produced.                          |
| `uiDeltasCoalesced`                  | count        | UI summaries coalesced by backpressure policy.               |
| `detailRequestsRejectedStale`        | count        | Stale UI detail requests rejected.                           |
| `asyncReadModelResultsRejectedStale` | count        | Stale helper results rejected before publication.            |
| `workerInputQueueDepth`              | count        | Pending validated main-to-Worker envelopes, when observable. |
| `workerOutputQueueDepth`             | count        | Pending Worker-to-main envelopes, when observable.           |
| `snapshotEncodeTimeUs`               | microseconds | Diagnostic timing, not deterministic parity input.           |
| `postMessageLatencyUs`               | microseconds | Diagnostic timing, may be measured by adapter or harness.    |
| `maxSnapshotLatencyTicks`            | ticks        | Difference between latest tick and latest accepted snapshot. |

Later benchmark tasks may add system-specific metrics, but new fields must be
versioned and must not become a hidden state channel for UI mutation.

### Explicit non-decisions

- ADR-0004 does not implement protocol types, validators, Worker adapters,
  renderer integration, or UI panels.
- ADR-0004 does not choose a persistent save encoding. ADR-0005 owns save
  container semantics.
- ADR-0004 does not require SharedArrayBuffer or deployment isolation changes.
- ADR-0004 does not choose compression, binary codec packages, or new runtime
  dependencies.
- ADR-0004 does not make render snapshots authoritative for save/load, replay,
  work selection, pathing, reservations, or job execution.

## Consequences

### Positive

- UI and renderer code receive predictable read-only projections while the
  Worker/headless runtime remains authoritative.
- Complete render snapshots provide a recovery point after dropped frames,
  avoiding UI-owned world repair logic.
- Canonical lane order and read-model hashes give WM-0028 a concrete parity
  target beyond the authoritative world hash.
- Backpressure behavior is observable through metrics instead of being hidden
  in browser queues.

### Negative

- Snapshot producers must maintain stable lane order, version fields, and hash
  traversal tests before UI features are visible.
- Complete snapshots may transfer more data than future delta schemes.
- UI detail panels must tolerate stale rejections and request again from a fresh
  basis instead of locally patching old details.

### Risks and mitigations

- Risk: render snapshots become too large. Mitigation: metrics record byte size,
  entity count, encode time, latency, and dropped snapshot count; later tasks may
  add versioned deltas only with complete snapshot recovery.
- Risk: metrics timing differences are mistaken for determinism failures.
  Mitigation: timing fields are explicitly excluded from parity hashes; parity
  tests compare deterministic counters separately from diagnostic timings.
- Risk: UI detail data expands into unbounded world inspection. Mitigation:
  requests include target identity, basis metadata, detail level, and maximum
  result budget; Worker returns structured rejection reasons when the budget or
  basis is invalid.
- Risk: stale async helper results leak into UI. Mitigation: every helper result
  used by read models carries owner-store/map/navigation/job/reservation input
  versions and is rejected by the authoritative runtime before publication.

## Contracts introduced

- Simulation Worker and Node headless are the only producers of authoritative
  read-model projections. UI, Pixi, React, and Electron remain read-only
  consumers.
- Render snapshots are complete per `snapshotSequence`; the main thread accepts
  only newer complete snapshots for the active session.
- Render snapshot buffers are transferred or cloned as immutable projection
  data. The Worker must not read or write a transferred buffer after posting it.
- Canonical snapshot hashing walks header fields and lanes in the order defined
  by this ADR. It does not depend on object key order, local UI state, or
  diagnostic timings.
- UI detail responses are request-scoped, bounded, read-only, and tied to a
  basis snapshot/tick plus target generation or owner-store version.
- Stale main messages, stale snapshots, stale UI detail responses, and stale
  async read-model helper results are rejected or discarded. They are not merged
  into current read-model state.
- Backpressure may drop or coalesce render snapshots, UI summaries, and metrics,
  but not command results, alerts, save responses, or fatal errors.
- Metrics include snapshot size, dropped snapshot count, latency, queue depth,
  stale rejection counters, and latest hash/sequence fields.
- SharedArrayBuffer is optional future optimization only; Transferable
  ArrayBuffers and structured envelopes are sufficient for the M1 contract.

- Protocol validation tests reject unsupported `protocolVersion`,
  `schemaVersion`, `renderSnapshotSchemaVersion`, `uiReadModelSchemaVersion`,
  `metricsSchemaVersion`, stale session ids, stale main sequences, and malformed
  snapshot lane lengths.
- Render snapshot tests verify stable entity ordering, lane count/length
  agreement, no object references in lanes, complete-snapshot replacement, and
  rejection of older `snapshotSequence` values.
- UI detail tests verify request id echoing, target generation/version checks,
  budget enforcement, structured stale reasons, and UI discard of out-of-date
  detail results.
- Backpressure tests prove render snapshots and coalescible summaries/metrics
  can be dropped without dropping `CommandResult`, `AlertBatch`, `SaveReady`, or
  `FatalSimulationError`.
- Worker and Node parity tests in WM-0028 run the same M1 hauling/building seed,
  content hash, and command stream through Node headless and browser Worker,
  then compare periodic authoritative `worldHash` values.
- The same parity harness compares canonical `readModelHash` values for render
  snapshots emitted at deterministic ticks.
- The same parity harness issues a fixed `RequestUiDetail` script at fixed
  ticks and compares canonical UI detail hashes or the same structured stale
  rejection reasons across Node and Worker.
- Metrics shape tests verify required fields and units. Deterministic counters
  such as counts, bytes, drops, stale rejections, and latest sequences are
  asserted exactly where the harness controls backpressure; diagnostic timings
  are asserted only for presence and sane ranges.

## Performance implications

- Snapshot production must not allocate per entity in hot loops. Producers use
  typed-array lanes, reusable staging capacity, and explicit growth points.
- Normal ticks must not build UI detail payloads unless requested. Requested
  detail payloads are bounded by target, level, and maximum result budget.
- Render snapshots may be emitted at a deterministic simulation cadence chosen
  by implementation tasks; producing every 30 TPS tick is not required if
  metrics show a lower cadence satisfies rendering and parity tests.
- Snapshot benchmarks must report `renderEntityCount`, `renderSnapshotBytes`,
  `snapshotEncodeTimeUs`, `postMessageLatencyUs`, `snapshotsProduced`,
  `snapshotsDroppedForBackpressure`, queue depths, and stale rejection counters.
- Backpressure must be bounded by stream. No unbounded array of unread snapshots,
  detail results, or metrics samples may grow with wall-clock delay.
- Transferable ArrayBuffers are the M1 baseline for large lanes. Structured
  clone may be used only for small headers and bounded detail payloads.
- SharedArrayBuffer, compression, or external binary codecs require later
  reviewed decisions if metrics prove the baseline insufficient.

## Rollback / migration

Before acceptance, rollback is deleting `coordination/decisions/ADR-0004.md`
and removing the ADR index link. No runtime code, deployed protocol, save file,
or UI feature depends on this Proposed ADR.

If accepted before implementation, migration is from the M0 Worker spike
snapshot notes to versioned M1 read-model contracts:

- Add shared protocol types and validators for the versioned render snapshot,
  UI read-model, metrics, and hash envelopes.
- Implement Node headless and browser Worker read-model builders from the same
  authoritative store inputs.
- Add canonical hash traversal helpers in testkit or protocol test utilities.
- Update renderer/UI consumers to discard stale read models rather than patching
  local state into authority.
- Add backpressure counters before enabling renderer-visible snapshot dropping.

If an implementation task based on this ADR fails review, rollback is reverting
that task branch and keeping ADR-0004 Proposed or superseding it with a narrower
proposal. Do not merge UI-owned correction logic, hidden shared read-model
caches, unbounded snapshot queues, or a SharedArrayBuffer deployment requirement
as temporary bridges.

After an accepted Worker protocol version ships inside M1, changing the
snapshot/read-model encoding requires a new versioned migration path and parity
tests that prove old unsupported sessions fail closed with structured reasons.
