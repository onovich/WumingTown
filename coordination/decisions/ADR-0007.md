# ADR-0007: M2 work logistics ownership and contract gate

- Status: Proposed
- Date: 2026-06-24
- Owners: systems-architect
- Related tasks: WM-0032, WM-0033, WM-0034, WM-0035, WM-0036,
  WM-0037, WM-0038, WM-0039, WM-0040, WM-0041, WM-0042, WM-0043

## Context

M2 composes the M1 hauling/building foundation into the
`m2.work_logistics.lantern_yard.v1` vertical slice: 20 workers, source stacks,
depot storage, four lantern boundary build sites, 100 versioned path requests,
midrun save/load/resume parity, Browser Worker parity, and long-run benchmark
invariants.

This ADR is the M2 architecture gate required before implementation tasks can
run. It records state ownership, derived-index boundaries, cache/version rules,
public contract gates, migration, rollback, test, and performance implications.
It does not implement Worker protocol, save format, schema, package boundary,
dependencies, runtime behavior, or M3.

Failure modes this gate must prevent:

- UI, Pixi, React, Electron, protocol, or read-model code mutating world state.
- Hidden shared state through singleton stores, module-level mutable caches, or
  UI-owned simulation repair.
- Pawn thinking or cleanup scanning the whole map, all entities, all stacks, all
  work rows, or all reservations during normal ticks.
- WorkOffer rows, storage indexes, path caches, read models, or availability
  projections becoming authoritative facts.
- Unversioned caches surviving owner-store, map, reservation, save/load, or
  Worker snapshot changes.
- Save/replay or Worker parity work expanding into a public compatibility,
  protocol, or schema promise without a separate gate.
- Job/build/order progress being stored in Promise, Generator, coroutine,
  closure, async completion order, or UI state.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Alternatives

- Monolithic `LogisticsCoordinator` as the owner of work, inventory, storage,
  reservations, paths, build orders, and read models: rejected. It centralizes
  behavior quickly, but it hides ownership, encourages broad scans, makes
  rollback coarse, and becomes a second world store beside the existing owner
  stores.
- UI/read-model driven planning: rejected. UI and Worker snapshots can help
  explain work, but using them to reserve jobs or repair divergence moves
  authority out of Simulation Worker/Node headless and breaks replay parity.
- Persist derived indexes and path/read-model caches to speed load: rejected for
  M2. These surfaces are valid only for the owner-store versions they were built
  from. Persisting them as authority would create hidden shared state and
  migration pressure before a public save promise exists.
- Let each M2 implementation task adjust Worker protocol, save sections,
  schemas, or package boundaries as needed: rejected. Contract drift must be
  visible before implementation. Public protocol/save/schema/dependency/package
  changes require a separate reviewed gate.
- Existing M1 owner stores plus M2 owner extensions, versioned derived indexes,
  bounded Top-K selection, and focused save/replay/Worker parity gates:
  proposed. Each mutable fact has one owner; derived indexes are rebuildable;
  public contracts stay frozen unless explicitly gated.

## Decision

Keep Simulation Worker or Node headless as the only authoritative world writer.
M2 implementation tasks must preserve the M1 `WorldState` owner-store model and
may extend owner stores only inside their task scopes. UI, Pixi, React,
Electron, protocol adapters, and read-model consumers remain read-only.

### Derived index and cache boundary

All M2 derived surfaces are either versioned or scratch:

- WorkOffer rows record owner target id, owner kind, owner-store version,
  reachability/version basis when used, and the row's own update sequence.
- Storage logistics rows record storage owner version, stack store version,
  reservation ledger version, and exact slot/def keys.
- Path and region caches record map, navigation, region, room, and region-graph
  versions. A mismatch discards or rebuilds the cache.
- Read models record snapshot/detail schema versions, sequence, basis tick,
  world hash or owner version, and target generation/version.
- ReasonTrace is bounded diagnostic evidence. It is not a job cursor, save
  authority, reservation owner, or unbounded queue.

Unversioned derived caches are forbidden. Persisted or cross-tick caches that
cannot name their owner versions must be deleted and rebuilt from owner stores.
Load-time full scans are allowed only for validation and rebuild before ticks
resume; normal pawn thinking, cleanup, work selection, reservation acquisition,
and path selection must use indexes and bounded candidates.

### Public contract gates

WM-0034 does not approve any public contract change. Downstream M2 tasks must
block and request a separate architecture gate before doing any of these:

- Add or change public Worker protocol message families, envelope versions,
  snapshot schemas, UI detail schemas, or metrics schemas.
- Add or change public save container format, public compatibility promises, or
  cross-version migration behavior.
- Add or change data schema/mod schema, content compiler schema, or external
  untrusted data validation contract.
- Change package boundaries, cross-package exports, app/runtime authority, or
  public import paths.
- Add runtime dependencies, codec packages, compression packages, Worker
  transport dependencies, or platform-specific storage dependencies.
- Move authoritative simulation mutation into UI, Pixi, React, Electron,
  preload, protocol adapters, read-model consumers, or platform storage code.

Focused M2 harnesses may add internal test fixtures, local owner-store
subversions, and scenario-specific diagnostics only inside their task scopes and
reports. If those changes become public protocol/save/schema/package contracts,
the task is blocked until a separate gate is reviewed.

### Downstream implementation block conditions

Downstream M2 implementation must remain proposed or be blocked when any of
these conditions apply:

- The task cannot name the owner store for a mutable fact before implementation.
- A derived row/cache would become the only source of truth for work, quantity,
  reservation, build progress, path, save, or read-model state.
- A normal tick would need a full-map/all-entity/all-stack/all-offer/all-job
  scan for thinking, cleanup, repair, path selection, or reservation.
- A cache, path result, WorkOffer row, read model, or save payload cannot carry
  owner versions or be rebuilt from owners.
- The implementation needs a public Worker protocol, save format, schema,
  dependency, or package-boundary change not separately approved.
- Job/order progress would depend on Promise, Generator, coroutine, closure,
  async completion order, UI state, real time, or platform timing.
- Save/load cannot rebuild derived indexes before resumed ticks or cannot prove
  uninterrupted vs resume hash parity for the M2 scenario.
- Worker and Node headless cannot consume the same command stream and compare
  authoritative hashes without UI repair.
- Required performance counters cannot report candidate caps, path budgets,
  stale rejects, reservation cleanup, queue backlog, and hash checkpoints.

## Consequences

### Positive

- M2 implementers get a concrete owner table before touching pathing, work
  selection, reservations, storage, build orders, save/replay, or Worker parity.
- Derived indexes can be optimized without becoming hidden persistent state.
- Public protocol/save/schema/dependency/package changes cannot slip into
  vertical-slice work as incidental implementation details.
- Reviewers can reject global scans, unversioned caches, and UI-owned
  simulation with explicit downstream block language.

### Negative

- Implementation tasks must name owner versions and rebuild paths before visible
  behavior is complete.
- Focused M2 harness work may need extra diagnostics to prove parity without
  relying on public protocol or save compatibility changes.
- Some future optimizations, such as persisted caches or Worker transport
  changes, are delayed until metrics justify a separate gate.

### Risks and mitigations

- Risk: storage or build-order owner stores expand beyond the vertical slice.
  Mitigation: WM-0038 and WM-0039 may add only the owner lanes required by the
  M2 scenario and must keep broad economy/content/catalog work out of scope.
- Risk: rejecting public protocol/save/schema drift blocks implementation late.
  Mitigation: each task report must state whether it touched those surfaces; if
  needed, block early and request a separate architecture task.
- Risk: rebuilding indexes on load hides expensive work. Mitigation: WM-0040
  and WM-0042 must report rebuild time, backlog, and hash checkpoints.
- Risk: bounded candidate caps skip valid work. Mitigation: cap hits are
  observable ReasonTrace entries and benchmark counters; tasks must prove
  terminal scenario invariants still hold.

## Contracts introduced

- M2 contract gate version: `m2WorkLogisticsArchitectureGateVersion = 1` for
  reports and docs that need to cite this architecture decision.
- M2 work offers are derived rows. The owner stores listed in the ownership
  table own the facts behind a row.
- M2 storage and build/order state must have named owner stores before
  implementation. Storage logistics indexes and WorkOffer rows cannot own
  quantities, capacity, sockets, order activation, or progress.
- M2 path results are version-basis data. They cannot mutate jobs,
  reservations, actors, items, or build sites until the authoritative runtime
  validates the basis at commit.
- M2 save/replay uses focused scenario evidence only. Public save compatibility,
  save container changes, and cross-version migration are non-decisions here.
- M2 Worker read models are read-only projections with versioned basis metadata.
  UI/client layers cannot repair or correct authoritative divergence.
- All derived indexes and caches are versioned, scratch, or rebuilt from owner
  stores. Unversioned caches are forbidden.

## Test implications

ADR-0007 proposal verification:

- `node .agents/skills/wuming-town-agent-workflow/scripts/taskctl.mjs validate`
- `node tools/validate-handoff.mjs`
- `git diff --check`
- `pnpm quality`

Future implementation verification required by this ADR:

- WorkOffer tests cover owner-version mismatch, terminal target removal,
  permission denial, material unavailability, capacity conflict, region
  unreachable, exact path failure, reservation conflict, stable tie-breaks, and
  cap-hit traces without global scans.
- Reservation tests prove all-or-nothing multi-channel acquire, explicit release
  on complete/cancel/fail/load/destroy/invalidation, no lease-as-normal-flow,
  and zero active claims at terminal state.
- Item/storage/hauling tests prove integer conservation across source stacks,
  depot stacks, carried lanes, build buffers, consumed completion audit lanes,
  and cancellation/recovery paths.
- Build/order tests prove serializable order/site/job progress, socket demand,
  material rejection, terminal site cleanup, and structured reasons.
- Path tests prove region prefiltering, bounded Top-K exact A*, stale basis
  rejection before mutation, node-budget evidence, and no long-lived actor path
  ownership.
- Save/replay tests prove save at tick 6000, load at 6001 in the resume run,
  derived rebuild before resume, equal hashes to uninterrupted replay, and
  structured divergence reports.
- Worker/headless parity tests prove the same command stream yields matching
  authoritative hashes and read-only projection hashes or matching structured
  stale rejections.

## Performance implications

- Normal M2 ticks must not allocate objects, arrays, closures, or strings in
  loops proportional to actors, offers, stacks, cells, reservations, jobs, or
  path candidates.
- Normal M2 ticks must not use full-world scans, unbounded sort, regex, string
  concatenation, `map`, `filter`, `reduce`, or `flatMap` in hot loops.
- Required M2 counters include visited offers, scored offers, candidate cap
  hits, Top-K path candidates, exact A* node visits, stale path rejects,
  reservation transaction attempts, accepted/rejected claims, cleanup releases,
  active final claims, dirty storage backlog, WorkOffer backlog, path backlog,
  read-model queue depth, snapshot bytes, rebuild time, and hash checkpoints.
- Scenario gates must preserve the existing performance policy: key benchmark
  P95 regression over 10 percent requires explanation, and over 20 percent
  blocks merge by default.
- Load-time rebuild work is allowed at the save boundary only when it reports
  rebuild budget, remaining backlog, invariant validation, and first resumed
  hash comparison.

## Migration

If accepted before implementation, migration from M1 to M2 is incremental:

1. Keep M1 `WorldState`, owner stores, explicit job drivers,
   `ReservationLedger`, `WorkOfferIndex`, versioned path requests, focused
   save/replay harness, and Worker/read-model authority boundaries.
2. Extend owner stores only in the task that owns the surface:
   WM-0038 for storage/hauling, WM-0039 for build/order scaffold, WM-0040 for
   focused save/replay harness evidence, and WM-0041 for focused parity
   evidence.
3. Add or update derived indexes after owner-store mutations expose exact dirty
   keys and version fields.
4. Add tests that prove derived rows/caches rebuild from owner stores before
   exposing M2 scenario runs or Worker parity gates.
5. Treat any public Worker protocol/save/schema/dependency/package-boundary
   need as a blocker and create a separate architecture task before continuing.

Changing a version basis, owner-store scale, save section, or public protocol
after M2 fixtures exist requires fixture migration or fail-closed behavior with
tests and a separate reviewed decision when the change is public.

## Rollback

Before acceptance, rollback is reverting `coordination/decisions/ADR-0007.md`,
the small doc links/notes that cite it, `coordination/tasks/WM-0034.json`, and
`coordination/reports/WM-0034.md`. No runtime state, dependency, save file,
protocol, or package boundary depends on this Proposed ADR.

If a downstream implementation task fails review because it violates this gate,
rollback is reverting that task branch. Do not merge temporary global scans,
unversioned caches, persisted derived indexes, UI-authoritative fixes,
Promise-based job progress, stale path commits, or public protocol/save/schema
drift as bridges.

If this gate is too broad or too narrow after review, supersede it with a
narrower ADR before implementation resumes. Downstream M2 tasks stay proposed
or blocked until the replacement gate is verified.
