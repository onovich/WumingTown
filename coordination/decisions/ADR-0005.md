# ADR-0005: M1 save container v1

- Status: Proposed
- Date: 2026-06-24
- Owners: systems-architect
- Related tasks: WM-0016, WM-0025, WM-0028

## Context

M1 needs a save container that can capture the authoritative Simulation Worker
or Node headless world, load it back, rebuild derived state, and resume the next
tick deterministically. The same container must support the first M1
save/replay parity harness, including map chunks, entity stores, job driver
state, reservation claims, random stream positions, and enough command log tail
to diagnose divergence.

This ADR is a Proposed architecture decision only. It does not implement
serializers, select an external codec, add runtime dependencies, or make a save
compatibility promise beyond M1. It narrows the contract that downstream tasks
must satisfy before any loader can resume simulation from external data.

Failure modes this proposal must prevent:

- Loading untrusted external save bytes directly into authoritative `WorldState`.
- Treating derived indexes, caches, room data, path caches, or read models as
  authoritative persistent state.
- Resuming a tick before all references, indexes, caches, and invariants have
  been rebuilt and validated.
- Persisting job execution as Promise, Generator, coroutine, closure, or UI
  state instead of explicit serializable fields.
- Accepting old or future save shapes through scattered normal-load conditionals
  rather than versioned migration functions.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered options

- Full object graph JSON: easy to inspect and debug, but rejected for M1 because
  it would preserve accidental aliases, hidden shared state, object identity,
  derived caches, and UI-friendly shapes that do not match authoritative
  structure-of-arrays stores.
- Per-store ad hoc files: simple for each subsystem owner, but rejected because
  load order, migration order, checksum coverage, command-log context, and
  whole-save rollback become unclear.
- Persist read-model snapshots only: rejected. Read models are lossy,
  UI-facing, and non-authoritative; using them for load would move simulation
  authority out of the Worker/headless runtime.
- Append-only command log with no state checkpoint: deterministic in principle,
  but rejected for M1 vertical tests because replay cost grows with history and
  load cannot validate map/entity/job/reservation state at the checkpoint.
- Versioned container with authoritative sections and derived-state rebuild:
  proposed. Store only owner-state sections plus command-log tail, validate all
  external data at boundaries, migrate by section, rebuild indexes/caches, then
  validate invariants before the first resumed tick.

## Decision

Use a single Save Container v1 envelope for M1 checkpoint saves. The envelope is
owned by the simulation runtime and persistence boundary; `sim-core` owns the
authoritative section semantics, while platform storage only reads and writes
opaque bytes through the `SaveStore` port.

The proposed envelope has:

- `magic`: Wuming Town save identifier.
- `formatVersion`: `1` for Save Container v1.
- `writerBuildId`: diagnostic build string, not trusted for compatibility.
- `createdTick`: authoritative tick at checkpoint creation.
- `sectionDirectoryVersion`: version of section names, lengths, and checksums.
- `sections`: ordered directory entries for the sections below.

Every persisted section has an explicit section version field. Missing,
duplicate, unknown-required, or future section versions fail validation unless a
reviewed migration function exists for that section. Optional compression or
binary block encoding may be introduced later inside section payloads, but this
ADR does not choose a codec or dependency.

### Container sections

`MapChunks`

- Required version field: `mapChunksVersion`.
- Per-chunk version field: `chunkSchemaVersion`.
- Contains authoritative map grid lanes, chunk coordinates, chunk dimensions,
  terrain/walk cell state, local dirty-version counters needed to rebuild region
  and room state, and cross-reference data that validates against owner stores.
- Does not persist renderer tiles, Pixi objects, UI selections, path caches,
  entity-location ownership, room caches, region graphs, or query acceleration
  caches as authoritative state.

`EntityStores`

- Required version field: `entityStoresVersion`.
- Store version fields: `entityRegistryVersion` plus one explicit version per
  owner store included in M1, such as location, item stack, pawn, job core, and
  low-frequency metadata stores.
- Contains generation-checked entity registry state, active/free lists,
  authoritative component lanes, sparse arena payloads that are owner state, and
  stable ids needed by jobs, reservations, map references, and command replay.
- Does not persist read models, inspector state, object references, or
  unversioned derived availability counters.

`JobsReservations`

- Required version field: `jobsReservationsVersion`.
- Sub-version fields: `jobCoreVersion`, `jobDriverStateVersion`, and
  `reservationLedgerVersion`.
- Contains explicit serializable job status, driver step state, progress
  counters, structured failure reason codes, owner pawn/job references, active
  reservation claims, reservation channels, amounts, interaction cells,
  lease-expiry ticks used only for recovery, and all-or-nothing transaction
  recovery metadata required by M1.
- Does not persist Promise chains, coroutine positions, closures, UI job labels,
  temporary path requests, or unbounded reason traces as execution authority.

`RandomStreams`

- Required version field: `randomStreamsVersion`.
- Contains named deterministic random stream ids, seed material, stream
  counters/positions, and any substream allocation counters required to resume
  M1 with the same future random draws.
- Does not persist `Math.random()` state, wall-clock time, browser timing, or
  ambient platform entropy.

`CommandLogTail`

- Required version field: `commandLogTailVersion`.
- Contains the checkpoint base tick, command schema version, content manifest
  hash used by the tail, accepted player/system commands since the last replay
  checkpoint horizon, periodic world hashes retained for diagnostics, and the
  next command sequence number.
- Does not promise a complete chronicle or cross-major replay contract. It is
  enough for M1 save-interrupt-load-resume parity and divergence reports.

Other sections from the broader save document, such as full content manifest,
world summary, chronicle, obligations, or story state, may be added only by
later reviewed tasks with their own version fields. This ADR does not require
them for the M1 vertical save container.

### Load order

The M1 loader must treat all external save data as untrusted and load through a
validated intermediate representation before mutating authoritative state:

1. Read opaque bytes from `SaveStore`.
2. Validate envelope magic, `formatVersion`, section directory, byte lengths,
   required section names, duplicate sections, and checksums.
3. Parse each section as `unknown` external data and validate its required
   version fields, scalar ranges, array lengths, enum domains, id domains, and
   declared counts before constructing owner stores.
4. Run reviewed N-to-N+1 migration functions by section when the section
   version is older than the current M1 loader version. Future versions fail
   closed.
5. Construct an empty `WorldState` and load `EntityStores` owner state with
   generation validation.
6. Load `MapChunks` and resolve every map/entity reference through
   `EntityStores`.
7. Load `JobsReservations` and resolve every job, pawn, target, quantity, cell,
   and reservation owner reference through the authoritative stores.
8. Load `RandomStreams` and `CommandLogTail`; verify tick, command sequence,
   content hash, and stream positions are consistent with the checkpoint.
9. Rebuild all derived indexes and caches from owner stores, including spatial
   indexes, occupancy indexes, room/region/navigation caches, work-offer
   indexes, reservation availability projections, and read-model dirty state.
10. Validate invariants before the first resumed tick: entity generations,
    single location membership, map bounds, non-negative quantities, reservation
    ownership, job-driver state legality, random stream availability, and command
    sequence continuity.
11. Resume only at `createdTick + 1` after validation succeeds. Any validation
    failure returns structured load failure reasons and must not partially
    mutate a live world.

## Consequences

### Positive

- Save/load is aligned with Worker/headless authority and ADR-0003 owner stores.
- Derived indexes and caches stay rebuildable rather than becoming hidden shared
  persistent state.
- Version fields are local to the sections that own migration pressure.
- M1 replay diagnostics can include the command tail and random stream positions
  needed to explain divergent hashes.

### Negative

- Loader implementation must do explicit boundary validation before any useful
  game state is visible.
- Load time includes index/cache rebuild and invariant validation, including
  allowed full-state scans at load boundaries.
- Section-level migration tests are required before accepting older M1 section
  versions.

### Risks and mitigations

- Risk: downstream tasks persist too much derived data for faster loading.
  Mitigation: only owner state may be authoritative; derived sections, if later
  introduced, must carry input versions and be discardable on mismatch.
- Risk: section versions drift independently and make compatibility unclear.
  Mitigation: the section directory records each version and the loader fails
  closed without a reviewed section migration path.
- Risk: external saves attack memory or CPU through huge declared sizes.
  Mitigation: validation must check section byte length, declared counts, map
  bounds, string/id lengths, and per-section M1 caps before allocation.
- Risk: command-log tail is mistaken for a full save-compatibility promise.
  Mitigation: this ADR limits the tail to M1 parity diagnostics and explicitly
  excludes compatibility promises beyond M1.

## Contracts introduced

- Save Container v1 is Proposed for M1 only. It does not promise compatibility
  with future milestones, public release saves, or unreviewed historical builds.
- External save bytes are untrusted. They must be parsed as `unknown`, validated
  at the persistence/simulation boundary, and rejected with structured reasons
  on any unsupported or unsafe shape.
- Required M1 sections are `MapChunks`, `EntityStores`, `JobsReservations`,
  `RandomStreams`, and `CommandLogTail`.
- Each required section has an explicit required version field:
  `mapChunksVersion`, `entityStoresVersion`, `jobsReservationsVersion`,
  `randomStreamsVersion`, and `commandLogTailVersion`.
- `MapChunks` and `EntityStores` contain authoritative owner state; indexes,
  caches, read models, and renderer state are rebuilt or discarded.
- `JobsReservations` contains explicit serializable job and ledger state;
  coroutine, Promise, Generator, closure, and UI execution positions are
  forbidden.
- `RandomStreams` contains deterministic seed/counter state only; platform time,
  browser timing, and ambient randomness are forbidden.
- `CommandLogTail` is a bounded diagnostic and parity section, not a complete
  chronicle or cross-version replay guarantee.
- Load must rebuild indexes/caches and validate invariants before the first
  resumed tick.
- Migration functions are section-owned, pure N-to-N+1 transforms with tests.
  Old-version conditionals must not be scattered through normal load or runtime
  store code.

Future implementation verification required by this ADR:

- Save fixture validation rejects missing versions, duplicate sections, unknown
  required sections, future versions, invalid lengths, out-of-range ids, bad
  generations, invalid enum values, and checksum mismatch.
- Save-load-resume tests prove that rebuilding indexes/caches before
  `createdTick + 1` produces the same world hash as uninterrupted replay.
- Migration tests cover each supported section version with pure N-to-N+1
  transforms and no old-version branches in normal runtime code.
- Security tests cover hostile external saves with excessive declared counts,
  invalid byte lengths, invalid references, malformed command tails, and
  inconsistent random stream positions.
- Worker/headless parity tests in WM-0028 run the same M1 hauling/building
  command stream through Node and browser Worker save/load paths and compare
  periodic authoritative hashes.

## Performance implications

- Normal ticks must not read or update save container structures. Save/load is a
  boundary operation outside the hot tick path.
- Save benchmarks must report total byte size, per-section byte size, section
  encode/decode time, validation time, migration time, index/cache rebuild time,
  and first-resumed-tick hash comparison.
- Load-time full scans are allowed only for validation and rebuild before the
  first resumed tick. Normal pawn thinking, work discovery, pathing,
  reservations, and cleanup still require indexes and bounded candidates.
- Section payloads should preserve store-friendly numeric block layout where
  downstream implementation can do so without adding dependencies.
- Optional chunk compression or binary codecs must be isolated behind section
  versioning and require a later dependency/security review if an external
  package is introduced.

## Rollback / migration

Before acceptance, rollback is deleting `coordination/decisions/ADR-0005.md`
and removing its ADR index link. No runtime state or external save depends on
this Proposed ADR.

If accepted before implementation, migration is from the current no-persistent-
M1-save state to versioned owner-state sections:

- Add container writer/loader tests before exposing any save UI.
- Implement section validators before section constructors mutate `WorldState`.
- Add section migrations only as pure N-to-N+1 transforms with fixture tests.
- Rebuild derived indexes and caches after load until they prove their input
  versions match; unversioned caches are discarded.

If an M1 implementation task fails review before a save compatibility promise
exists, rollback is reverting that task branch and keeping ADR-0005 Proposed or
superseding it with a narrower proposal. Do not merge an unsafe loader, hidden
shared cache, global scan in normal ticks, or UI-authored save correction as a
temporary bridge.

After M1 save files exist, rollback must preserve the previous valid save slot
where the platform supports it. Downgrading a newer section version back to v1
is not promised unless a later accepted ADR defines an exporter and tests.
