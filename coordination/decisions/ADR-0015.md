# ADR-0015: Public deterministic playable Worker advance helper

- Status: Proposed
- Date: 2026-06-30
- Owners: systems-architect
- Related tasks: WM-0150, WM-0152, WM-0157, WM-0158, WM-0159

## Context

WM-0157 exposed a public browser Simulation Worker session, and WM-0158 exposed
the public `UiDeltaPayload.playable?: PlayableProjectionV1` read model. WM-0152
can now send real WM-0150 lamp/build commands and render authoritative
projection data, but the scenario does not progress after accepted commands
unless the Worker receives the internal WM-0150 advance convention.

Current facts:

- `BrowserSimulationWorkerSession.setSpeed` and `.pause` only send protocol
  control messages. In the current Worker they update speed/paused state and
  emit `CommandResult` / `MetricsSample`; they do not tick the playable
  runtime or emit progressive `UiDelta` projections.
- The playable runtime advances when `sim-worker` receives a `Noop` command
  whose command id parses through `parsePlayableAdvanceCommandId`.
- `createPlayableAdvanceCommandId` / `parsePlayableAdvanceCommandId` are
  exported by `@wuming-town/sim-core`, but `apps/web` must not import
  `@wuming-town/sim-core`.
- Hardcoding `wm0150.advance.<tick>` in Web would turn an internal convention
  into unreviewed public protocol by accident.

WM-0152 needs a legal public way to request deterministic Worker time advance
without giving Web authority over jobs, pawns, resources, build progress or
blocked reasons.

## Decision drivers

- Preserve Simulation Worker / headless authority.
- Keep package imports root-only and boundary-checkable.
- Avoid a larger browser wall-clock simulation loop unless explicitly needed.
- Keep accepted command and advance ordering deterministic and replayable.
- Keep reliable `CommandResult` delivery separate from projection delivery.
- Avoid public save compatibility or migration promises.
- Keep WM-0152 unblocked without product UI implementation in the precursor.

## Considered options

1. Hardcode `wm0150.advance.<tick>` in `apps/web`.
   Rejected. It would bypass the package boundary and make a magic string an
   app-owned gameplay contract.

2. Let `apps/web` import `@wuming-town/sim-core` for
   `createPlayableAdvanceCommandId`.
   Rejected. It violates ADR-0001/ADR-0013 and gives Web direct access to
   simulation-core public surface outside the reviewed Worker bridge.

3. Make `SetSpeed` / `Pause` start a browser-side ticking loop immediately.
   Deferred. This is the eventual runtime shape for continuous play, but it
   introduces hidden timer ownership, lifecycle/backpressure questions, tab
   throttling behavior, replay pacing questions and worker scheduling policy.
   It is wider than the current WM-0152 unblock.

4. Add a new top-level Worker message such as `AdvanceTicks`.
   Deferred. It would be explicit, but it changes protocol message families and
   requires a larger schema decision. The current Worker already has a
   deterministic command-batch advance path.

5. Expose a public root `@wuming-town/sim-worker` helper/session API that sends
   the existing deterministic playable advance command through
   `PlayerCommandBatch`.
   Chosen. This keeps the convention owned by `sim-worker`, hides the command
   id format from Web, uses the existing reliable command/result path, and
   preserves deterministic ordering.

## Decision

`@wuming-town/sim-worker` may expose a public root helper or session method for
the WM-0150 playable slice, with an implementation-equivalent shape such as:

```ts
advancePlayableWorkerScenarioToTick(
  session: BrowserSimulationWorkerSession,
  targetTick: number,
): MainToSimulationMessage;
```

The exact name may vary, but the public contract must:

- be exported from the root of `@wuming-town/sim-worker`;
- send a normal `PlayerCommandBatch` to the Worker;
- hide the internal advance command-id format from Web consumers;
- accept an explicit logical `targetTick`, not a wall-clock duration;
- advance only the authoritative Worker/headless runtime;
- return the posted `MainToSimulationMessage` for tests and diagnostics;
- preserve `CommandResult` reliability and subsequent `UiDeltaPayload.playable`
  projection delivery;
- require explicit session construction and destruction through the existing
  browser session API.

The helper is a deterministic pump, not UI authority. Web may choose when to
request the next logical tick for presentation, but the only authoritative
state changes come from the Worker processing the public helper's batch in
message sequence order. Web must not store or synthesize tick advancement, pawn
motion, job progress, resources, build completion or blocked reasons.

`SetSpeed` / `Pause` semantics remain unchanged for this precursor. They do not
imply autonomous browser ticking until a later ADR explicitly defines timer
ownership, throttling behavior, lifecycle and replay implications.

## Deterministic clock stance

The public advance surface uses explicit integer simulation ticks. It must not
call `Date.now()`, `performance.now()`, `setInterval`, `requestAnimationFrame`
or any real-time source inside `sim-core` or Worker authority. If a Web caller
uses presentation time to decide when to request the next tick, that is an
input scheduling policy; the Worker still receives a deterministic ordered
command stream containing explicit target ticks.

The Worker may ignore or no-op requests at or below the current playable tick,
matching current runtime behavior, but this behavior must be documented and
covered by tests.

## Save/load stance

No public save compatibility, migration or in-progress playable command
persistence is introduced. Advance requests are transient deterministic inputs
for the active session. If a later task persists playable command or advance
state, it must add versioned focused save sections, fail-closed validation and
fixture or migration evidence.

## Verification

Implementation must prove:

- `apps/web` imports only package roots and does not import `@wuming-town/sim-core`.
- Web does not hardcode `wm0150.advance.` or parse internal advance strings.
- The public root helper/session API advances the playable Worker from tick 0
  to a requested tick and emits updated `UiDeltaPayload.playable`.
- Accepted lamp/build commands followed by public advance requests progress
  through job/pawn/build states under Worker authority.
- `CommandResult`, `SaveReady` and fatal/error reliable messages remain
  separately delivered.
- A real Chromium module Worker smoke uses the public root helper/session path.

## Rollback / migration

Before WM-0152 consumes the helper, rollback is removing the root helper/session
method and tests.

After WM-0152 consumes it, rollback must return WM-0152 to a blocked state or
to an explicitly labeled reviewed playback harness. It must not add Web-owned
runtime ticking, deep imports, magic command ids, unversioned caches or public
save compatibility as a fallback.
