# ADR-0011: M8 i18n architecture and locale settings contract

- Status: Proposed
- Date: 2026-06-28
- Owners: systems-architect
- Related tasks: WM-0113, WM-0114, WM-0115, WM-0116, WM-0118, WM-0120

## Context

M8 must turn the current product-gate harness into a player-facing Web and
Windows experience with `zh-CN` and `en` localization. WM-0111 made
localization a required M8 gate: Chinese browser/system language defaults to
`zh-CN`, non-Chinese defaults to `en`, settings allow manual override, missing
translations fail tests, and developer diagnostics may remain English only when
isolated from the default player UI.

This ADR defines architecture and boundaries only. It does not implement
runtime localization, translation resources, save schema, telemetry, accounts,
release/store/signing work or public compatibility claims.

Failure modes this decision must prevent:

- UI, Electron or settings code becoming authoritative for simulation state.
- Locale preference leaking into save files or Worker/headless authority.
- Chinese users defaulting to English because fallback was treated as locale
  detection.
- Missing translations becoming warnings instead of product gates.
- Broad Electron preload APIs being added for a simple language preference.
- Developer diagnostics appearing as the default player experience.

## Decision Drivers

- Correctness and Worker/headless authority.
- Web/Windows portability.
- M8 product UI and accessibility gates.
- Security and Electron least privilege.
- Save compatibility and rollback cost.
- Testing clarity for downstream client/content/QA tasks.

## Considered Options

1. UI-local string switch with no architecture contract.
   - Rejected. It is quick, but it leaves locale ownership, fallback behavior,
     Electron storage and diagnostics boundaries to each implementation task.

2. Store language choice inside Save Container or simulation state.
   - Rejected. Locale is a presentation preference. Putting it in saves creates
     schema and interoperability pressure unrelated to world authority.

3. Electron main process owns all language detection and persistence through a
   generic preload bridge.
   - Rejected. It would expand renderer authority for a narrow setting and
     risks exposing filesystem or arbitrary IPC surfaces.

4. App-shell locale preference with complete catalogs and read-only simulation
   projections.
   - Proposed. The app shell owns `LocalePreferenceV1`; Web and Windows resolve
     locale from manual preference plus browser/system candidates; Simulation
     Worker and headless emit structured keys and reason codes; UI renders
     localized projections.

## Decision

Adopt the M8 i18n contract in
[`docs/05_tech/12_m8_i18n_locale_contract.md`](../../docs/05_tech/12_m8_i18n_locale_contract.md).

### Locale Ownership

The supported player locale ids are `zh-CN` and `en`.

Locale preference is owned by the application shell:

```ts
type LocaleId = "zh-CN" | "en";
type LocaleSource = "system" | "manual";

interface LocalePreferenceV1 {
  readonly version: 1;
  readonly source: LocaleSource;
  readonly manualLocale?: LocaleId;
}
```

`resolvedLocale` is derived at runtime and is not authoritative persisted
state.

Simulation Worker and Node headless remain authoritative for world state and
emit locale-neutral structured facts: keys, reason codes, ids, integer values
and named parameters. UI and renderer layers consume read-only projections and
render localized text.

### Detection

When no valid manual override exists:

- any browser/system candidate whose primary BCP 47 language is `zh` resolves
  to `zh-CN`;
- all non-Chinese, empty, malformed or unsupported candidates resolve to `en`.

This includes `zh-Hant`, `zh-TW` and `zh-HK` resolving to `zh-CN` in M8 until a
Traditional Chinese locale is accepted.

### Persistence

Web stores only `LocalePreferenceV1` in origin-scoped browser preference
storage. Windows Electron should use the same renderer origin preference
storage unless packaged evidence proves a narrow host preference bridge is
required.

Any future Windows bridge must expose only typed locale detection/preference
methods and must update the preload allowlist plus desktop-shell e2e audit. It
must not expose generic `fs`, `shell`, arbitrary IPC, local paths, accounts,
network upload or telemetry.

Locale preference is not part of save files, save export/import, diagnostic
packages, Simulation Worker state, public release metadata or account state.

### Completeness And Fallback

Player-visible `ui.*`, `content.*` and `reason.*` keys require complete
`zh-CN` and `en` entries with matching interpolation schemas.

Runtime fallback is defensive:

```text
active locale -> en -> development missing-key marker
```

Any player-visible fallback for covered keys fails the quality gate. English
fallback does not override Chinese default detection.

Developer diagnostics may remain English-only only when isolated behind dev or
debug surfaces outside the default player UI.

## Consequences

### Positive

- Downstream implementation can add localization without touching simulation
  authority or save schema.
- Chinese default behavior is explicit and testable.
- Missing translations become a hard gate instead of visual cleanup.
- Electron authority remains minimal and auditable.
- Responsive and accessibility tasks get clear bilingual and pseudo-localized
  test targets.

### Negative

- WM-0114 must add real validation infrastructure before WM-0115 can move
  strings confidently.
- UI surfaces must separate player copy from diagnostics instead of relying on
  existing English harness labels.
- Windows persistence evidence may require e2e restart coverage even if no
  preload bridge is added.

## Risks And Mitigations

- Risk: translation fallback hides missing Chinese text.
  Mitigation: missing-key counters and completeness checks fail quality for
  player-visible keys.
- Risk: diagnostics leak back into default launch.
  Mitigation: default player e2e checks must assert diagnostics harness labels
  are absent unless explicitly opened.
- Risk: Electron storage durability differs between dev and packaged builds.
  Mitigation: WM-0114 or WM-0116 must provide packaged desktop evidence; any
  host bridge is typed and narrow.
- Risk: localization lookup adds render churn.
  Mitigation: cache resolved dictionaries by locale/version in the app shell;
  do not run localization inside simulation ticks.
- Risk: future locales need different plural or text-direction behavior.
  Mitigation: a future ADR can add locale ids, direction metadata or a reviewed
  formatter dependency without changing Worker authority.

## Contracts Introduced

- M8 i18n contract version: `m8I18nLocaleContractVersion = 1`.
- Supported player locales: `zh-CN` and `en`.
- Locale preference owner: application shell, not Simulation Worker/headless,
  not save state and not UI authority over world facts.
- Detection rule: any primary `zh` candidate resolves to `zh-CN`; everything
  else resolves to `en` unless manually overridden.
- Web persistence: origin-scoped preference storage containing only
  `LocalePreferenceV1`.
- Windows persistence: renderer origin preference storage by default; only a
  narrow typed preload preference bridge if evidence requires it.
- Player-visible localization namespaces require complete `zh-CN` and `en`
  entries.
- Developer diagnostics may remain English-only only when isolated from default
  player UI.
- Pseudo-localization is a dev/test layout surface, not a supported player
  locale.

## Performance Implications

Locale resolution and dictionary selection are application-shell operations.
They must not run inside authoritative simulation ticks.

Translation lookup may run during UI render, but implementations should cache
resolved locale dictionaries by locale and resource version. UI components
should avoid per-tick string construction from Worker outputs; Worker
projections should carry stable ids, reason codes and parameters.

Completeness checks, hardcoded-string scans and pseudo-localization expansion
are build/test operations, not normal gameplay hot paths.

No benchmark thresholds are changed by this ADR.

## Migration

Migration from the current English-heavy harness proceeds through downstream
M8 tasks:

1. WM-0114 adds locale resolution, preference storage and missing-key gates.
2. WM-0115 moves player-visible strings into complete `zh-CN` and `en`
   resources and records remaining English diagnostics.
3. WM-0116 exposes the player settings language selector.
4. WM-0118 and WM-0120 validate bilingual responsive and accessibility gates.

There is no save migration. Existing saves and focused scenario fixtures remain
world-state artifacts and do not gain locale fields.

If a stored preference key changes before public release, the implementation
may ignore or remove the old app-shell preference key and fall back to system
mode. No save compatibility promise is involved.

## Rollback

Before downstream implementation, rollback is deleting this ADR, deleting
`docs/05_tech/12_m8_i18n_locale_contract.md`, reverting the ADR index entry and
leaving WM-0114 through WM-0120 proposed or blocked.

After downstream implementation begins, rollback must not add UI authority,
save locale fields or broad Electron APIs as a bridge. Revert the owning task
branch or supersede this ADR with a narrower reviewed decision.

Because no runtime code or save schema is introduced by WM-0113, this ADR has
no data migration rollback.
