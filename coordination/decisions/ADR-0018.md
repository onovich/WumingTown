# ADR-0018: Autonomous-town projection v2 and explicit speed multipliers

- Status: Proposed
- Date: 2026-07-11
- Owners: systems-architect
- Related tasks: WM-0169, WM-0170, WM-0171, WM-0172, WM-0173, WM-0174, WM-0175
- Supersedes: only the PR-2 projection and speed portions of ADR-0017; all
  GameSession authority, fixed-tick, compatibility and save boundaries remain
  in force

## Context

PR-1 is independently verified, integrated and done. It proves one
authoritative GameSession, eight same-basis residents, a real Worker-owned
30-TPS clock for 600 seconds, schema-3 GameSession projection v1, and
product-consumed moving/working frames. PR-2 must now show eight residents
choosing and carrying out different forms of life and work over one full
36,000-tick day, with paths, targets, progress and structured explanations
visible in the real product.

The v1 contract is intentionally a PR-1 minimum. It does not represent all PR-2
activity states (`claiming`, `failed`, `interrupted`), a stable work kind,
target cells without an entity, a bounded route and cursor, schedule/capability
decision basis, or a read-only suggested inspection entry for an idle/blocked/
failed reason. Its Web read model contains older fixed lamp/build job kinds and
optional path fields that the Worker projection cannot populate losslessly.
Filling these fields from UI guesses would make the client authoritative.

The current speed surface was also audited directly. `SetSpeedPayload.speed` is
`0 | 1 | 2 | 3`; `GameSessionWorkerHost` multiplies the unchanged three ticks
per 100-ms quantum by that value and reports 0/30/60/90 effective TPS. The value
`3` is therefore an actual multiplier, not a level enum that can silently be
relabelled as 4x. WM-0169's reviewed PR-2 requirement is pause plus 1x/2x/4x,
so changing the label alone would be a breaking semantic lie.

WM-0169 is planning-only. It may approve the future contract and exact writer,
but it does not modify protocol, Worker, Web, core speed or save code.

## Decision drivers

- Keep Simulation Worker/headless as the only GameSession authority.
- Preserve fixed 30-TPS integer simulation and deterministic explicit stepping.
- Make every PR-2 product gate representable without debug payloads or
  client-synthesized state.
- Keep actor thinking indexed, capped and version-basis checked.
- Preserve same-basis RenderSnapshot/UiDelta validation and fail closed on
  unknown or malformed data.
- Avoid introducing PR-3 work-priority, direct-job, selection-command or
  building-command surfaces.
- Keep GameSession save/load unsupported and public compatibility unapproved.
- Give one future task exclusive ownership of the protocol migration.

## Considered options

### 1. Reuse projection v1 and infer missing state in Web

Rejected. A client cannot infer route, target, capability denial, interruption
or a useful structured reason from position/activity alone. The result could
look alive while authority had already completed or failed.

### 2. Add prose/debug fields to v1 without a version change

Rejected. This changes v1 semantics in place, cannot be validated exhaustively,
mixes localization with authority and recreates the evidence false positive
forbidden by the roadmap.

### 3. Add a separate autonomous-life message family

Rejected. RenderSnapshot and UiDelta already provide the correct latest-wins
and coherent read boundary. Another family would duplicate basis, ordering and
backpressure rules.

### 4. Treat speed value 3 as an ordinal whose label becomes 4x

Rejected. Current production code uses the value as a multiplier and advances
nine ticks per quantum. Relabelling it would break observability and parity.

### 5. Add value 4 to schema 3 while retaining the ambiguous `speed` field

Rejected. An old schema-3 peer could accept an envelope whose payload it cannot
validate, and the contract would still not distinguish a level from a
multiplier.

### 6. Move to schema 4, projection v2 and an explicit multiplier

Chosen. Keep protocolVersion 1 and existing message families, add only the
fields required by PR-2, and migrate the Worker and consumers on the reviewed
serial path.

## Decision

WM-0172 is the sole future writer for envelope schema 4, GameSession projection
version 2 and the related protocol/Worker/core-speed files named by its task
packet. No other PR-2 task may edit `packages/sim-protocol/**`.

### Transport and negotiation

- `protocolVersion` remains 1 and no message family is added.
- The envelope `schemaVersion` becomes 4.
- The default PR-2 route requests exactly
  `{ kind: "game_session", version: 2 }`; `Ready` echoes the exact contract.
- Projection v2 is mandatory on every applicable render/UI publication after
  negotiation. Wrong session, wrong version, malformed lanes or incoherent
  basis closes the browser session and never falls back to a fixture.
- Reliable command, alert, save-result and fatal messages remain ordered and
  non-droppable; render/UI snapshots remain latest-wins.

### Minimum v2 additions

V2 retains v1 entity identity, Q16 positions, hashes, map dimensions,
owner-version basis, resource summaries and selection-detail ownership. It adds
only facts needed by a PR-2 exit gate:

1. Resident and job state includes exactly `idle`, `claiming`, `moving`,
   `working`, `blocked`, `completed`, `failed` and `interrupted`.
2. `workKind` distinguishes `eat`, `rest`, `sleep`, `haul`, `lamp_patrol`,
   `lamp_refill`, `treatment` and `wait`. It is descriptive authority, not a
   command to create that work.
3. A target is an optional entity ref plus optional authoritative target cell;
   at least one must exist for non-idle work.
4. A bounded route window carries stable cell indices, current cursor, total
   step count and navigation/version basis. It supports presentation of the
   current path, not long-lived client pathfinding.
5. Integer/Q16 progress, step-entered tick and state-entered tick let clients
   show real change and let evidence detect a one-frame lifecycle.
6. Decision basis adds need, schedule, capability, WorkOffer, path,
   reservation and job owner versions needed to reject stale explanations.
7. Structured reason v2 carries a code, enumerated source, optional subject and
   target, ordered scalar parameters, and one non-authoritative navigation
   suggestion: `none`, `inspect_resident`, `inspect_target`,
   `inspect_resource`, `inspect_schedule` or `inspect_capability`.
8. Selected-resident detail adds the active schedule window, shift kind,
   capability summary and highest-priority need. These are read-only facts.

No v2 field contains localized prose, a direct-job command, work priority,
blueprint/build placement, command availability, player selection authority or
save bytes. Those remain PR-3 or later concerns.

### Explicit speed contract

Schema 4 names the public value `speedMultiplier` rather than overloading the
old `speed` name. `Pause { paused: boolean }` remains the separate lifecycle
control; zero is an effective rate while paused, not a selectable speed.

| Product state |                             Multiplier | Effective target | 100-ms integer quantum |
| ------------- | -------------------------------------: | ---------------: | ---------------------: |
| paused        | retained 1/2/4 multiplier, effective 0 |            0 TPS |                0 ticks |
| normal        |                                      1 |           30 TPS |                3 ticks |
| fast          |                                      2 |           60 TPS |                6 ticks |
| very fast     |                                      4 |          120 TPS |               12 ticks |

V2 `SetSpeed` accepts only `speedMultiplier: 1 | 2 | 4`; it never uses zero as
a pause alias. Pause preserves the requested multiplier and changes only the
effective rate. SetSpeed changes future scheduling only. Wall-time debt and
bounded catch-up stay in the Worker host; `sim-core` sees only integer ticks.
WM-0170 and WM-0171 must not change speed semantics:
their deterministic behavior is authored in ticks, and changing the cross-
package contract before the sole-writer task would create an unreviewable
half-migration.

### Compatibility and staged migration

- Schema-3 and schema-4 peers never hot-connect or down-convert. A mismatch
  fails before session activation.
- WM-0172 switches the default product negotiation atomically to schema 4/v2.
  It may retain projection-v1 negotiation only in explicit test/diagnostics
  entrypoints; the default product module graph must not import or request it.
  A v1 bridge accepts its historical 0/1/2/3 speed semantics, may not claim
  PR-2 evidence and may not accept a 4x request.
- A v2 session accepts only multipliers 1/2/4; values 0 and 3 fail closed with a
  structured unsupported-speed reason.
- WM-0172 owns the minimum Web transport/negotiation cutover needed to keep
  exact main runnable on v2; WM-0173 consumes v2 and adds the full Pixi/HUD
  behavior presentation and real pause/1x/2x/4x controls.
- Any new/unknown schema or projection version fails closed. There is no
  permissive “newer version” acceptance.
- Rollback reverts Worker and Web consumers together to the last reviewed
  schema-3/v1 build. Mixed schema operation is never a rollback mode.

## File ownership

WM-0172 exclusively owns the exact protocol surfaces listed in its task,
including constants, envelope/payload types and validators, v2 projection and
validation, Web read-model types, tests and package exports. It also owns the
focused Worker host, scheduler, mapper, browser-session validation, the minimal
Web session-negotiation boundary and the minimal core speed-type files required
for an honest 4x mapping.

WM-0170/WM-0171 are protocol-neutral. WM-0173 consumes package public roots and
may not edit protocol files. WM-0174/WM-0175 are evidence/acceptance-only.

## Save, determinism and security boundaries

GameSession save/load remains unsupported and fail-closed. V2 fields are
snapshot/read-model data, not a public save container. This ADR does not approve
serialization compatibility, platform persistence, cloud save or migration.

The decision does not add a runtime dependency, DOM access in sim-core,
Electron/IPC/network surface, telemetry, account, release or distribution
action. Determinism remains fixed 30 TPS, integer/Q16 state, seeded streams,
stable iteration and explicit serializable job/route state.

## Consequences

### Positive

- Every PR-2 behavior gate has a lossless Worker-owned representation.
- The product can distinguish what a resident is doing, where it is going,
  why it stopped and which existing inspection surface may help.
- 4x is an honest multiplier with testable 120-TPS scheduling.
- One writer prevents protocol/client/core-speed drift.

### Negative

- Schema 4 is a coordinated Worker/client migration and must remain serial.
- V1 compatibility becomes explicitly test-only and cannot satisfy PR-2.
- Route windows and richer reasons increase projection size, requiring caps and
  backpressure metrics.

## Verification required from WM-0172+

- Exact v2 Init/Ready negotiation and malformed/newer/wrong-basis fail closure.
- V1 bridge quarantine and default-route v2 enforcement.
- Separate pause plus speeds 1x/2x/4x produces effective 0/30/60/120 TPS without
  exposing wall time to sim-core; v2 rejects speed multipliers 0 and 3.
- Route windows and reason parameter counts are bounded; projection generation
  has measured size/allocation budgets.
- Worker/headless hashes match for the same seed and command stream.
- No protocol field or UI surface can directly create a job, reservation,
  path, resource or building order.
