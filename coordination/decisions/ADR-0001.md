# ADR-0001: Monorepo package boundaries

- Status: Accepted
- Date: 2026-06-23
- Owners: systems-architect
- Related tasks: WM-0002

## Context

M0 needs a TypeScript monorepo that makes the approved architecture executable before gameplay or runtime protocol work begins. The package list is fixed by `docs/05_tech/01_technical_architecture.md`, and the task requires enforceable dependency direction.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered options

- Single package with folders: simplest bootstrap, but package boundaries would depend on reviewer discipline.
- pnpm workspaces with TypeScript paths only: good editor experience, but deep imports and reversed dependencies could still slip in.
- pnpm workspaces plus a boundary guard: slightly more setup, but validates the architecture graph from manifests and source imports.

## Decision

Use a pnpm workspace with these workspaces:

- `apps/web`
- `apps/desktop-electron`
- `packages/foundation`
- `packages/content-schema`
- `packages/content-compiler`
- `packages/sim-core`
- `packages/sim-protocol`
- `packages/sim-worker`
- `packages/renderer-pixi`
- `packages/ui-react`
- `packages/persistence`
- `packages/platform`
- `packages/testkit`
- `packages/benchmarks`
- `tools/content-cli`
- `tools/headless-runner`

Each workspace exposes only `./src/index.ts` during bootstrap. Internal dependency versions use `workspace:0.0.0`, and each manifest declares `wumingBoundary.allowedInternalDependencies`. `tools/boundary-check.mjs` parses TypeScript import/export declarations and rejects unknown packages, deep imports, undeclared internal dependencies, and edges outside the allowed list.

## Consequences

### Positive

- Package edges are reviewable in manifests and enforceable from the root.
- `sim-core` begins isolated from UI, Pixi, Electron, Node file system, and true time imports.
- Later tasks can add real APIs without changing the workspace topology.

### Negative

- Every new package edge requires manifest and boundary updates.
- The initial skeleton contains smoke APIs solely to exercise public imports.

### Risks and mitigations

- Risk: boundary rules drift from architecture docs. Mitigation: keep the expected workspace list in the guard and update this ADR when approved package ownership changes.
- Risk: a future dynamic import bypasses static scanning. Mitigation: add AST coverage for that case before approving dynamic internal imports.

## Contracts introduced

- Public package imports must use `@wuming-town/<package>` only.
- Cross-package deep imports are forbidden.
- `sim-core` may depend only on `foundation` and compiled content contracts at this stage.
- Apps and platform packages remain outside simulation authority.

## Verification

- `pnpm boundaries:check`
- `pnpm typecheck`
- `pnpm lint`
- `pnpm test`

## Rollback / migration

Remove the workspace package manifests and revert root configs before any save format or runtime state exists. Later topology changes require a superseding ADR.
