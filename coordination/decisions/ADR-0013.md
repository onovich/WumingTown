# ADR-0013: Public browser Simulation Worker session bridge

- Status: Proposed
- Date: 2026-06-30
- Owners: systems-architect
- Related tasks: WM-0150, WM-0151, WM-0152, WM-0157

## Context

WM-0152 needs the Web shell to issue WM-0150 playable commands to the
authoritative Simulation Worker. The protocol and Worker runtime can already
validate and execute `PrioritizeLampWork` and `QueueSimpleBuild`, but `apps/web`
has no reviewed public path to construct or communicate with that Worker.

Current facts:

- `apps/web/package.json` does not declare or allow `@wuming-town/sim-worker`.
- `@wuming-town/sim-worker` exports only its package root.
- `packages/sim-worker/src/browser-worker-entry.ts` exists, but it is an
  internal file.
- `tools/boundary-check.mjs` rejects cross-package subpath and source-file
  imports, so `@wuming-town/sim-worker/browser-worker-entry` would still be an
  illegal shape unless the boundary policy itself changes.
- `apps/web/src/shell-bootstrap.ts` currently uses the WM-0151 reviewed
  projection harness, not a live Worker session.

The blocker must be solved without UI-owned world mutation, shell-local
authority, hidden shared state, unversioned caches or direct internal imports.

## Decision drivers

- Preserve Simulation Worker / headless authority.
- Keep package imports root-only and boundary-checkable.
- Give `apps/web` the smallest truthful path to send protocol messages and
  receive authoritative Worker output.
- Avoid moving `sim-worker` and `sim-core` ownership into generic platform
  ports unless a broader platform runtime decision is needed.
- Keep save/load promises explicit and fail-closed.
- Keep later Electron or platform wrapping possible without changing the
  command protocol again.

## Considered options

1. Deep-import `packages/sim-worker/src/browser-worker-entry.ts` from Web.
   Rejected. It violates package boundaries, bypasses the public contract, and
   makes Worker construction depend on internal file layout.

2. Keep the WM-0151 reviewed projection harness for WM-0152.
   Rejected. It is a useful presentation harness, but it is not Worker
   authority and cannot satisfy command acceptance, job ownership or replay
   requirements.

3. Add a platform-owned adapter in `@wuming-town/platform`.
   Deferred. It can be useful later for Electron/Web host normalization, but it
   would make `platform` depend on the simulation Worker stack and change the
   meaning of platform ports. That is wider than the current blocker.

4. Add a new `@wuming-town/sim-session` package.
   Deferred. It may become attractive if multiple clients need a shared session
   client independent of Worker implementation, but it adds workspace topology
   and boundary-checker updates before the first Web unblock.

5. Expose a browser session client from the root of `@wuming-town/sim-worker`
   and explicitly allow `apps/web -> @wuming-town/sim-worker`.
   Chosen. It keeps Worker ownership with the Worker package, keeps app imports
   package-root-only, and gives WM-0152 a narrow reviewed bridge without
   changing protocol semantics.

## Decision

`@wuming-town/sim-worker` may expose a public browser session API from its root
entrypoint. The API may internally construct or provide the URL for the existing
browser Worker entry, but consumers must import only `@wuming-town/sim-worker`.

`apps/web` may add `@wuming-town/sim-worker` as a workspace dependency and as an
allowed internal dependency, solely to consume that root-exported browser
session API. `apps/web` must not depend on `@wuming-town/sim-core` and must not
deep-import any `packages/sim-worker/src/*` file.

The public session client is a transport boundary, not world authority. It may:

- start or connect a browser module Worker;
- send validated `MainToSimulationMessage` envelopes;
- provide typed helpers for `InitSession`, `LoadSession`,
  `PlayerCommandBatch`, `RequestUiDetail`, `RequestSave`, `SetSpeed`, `Pause`
  and `Shutdown`;
- publish authoritative `SimulationToMainMessage` events to Web consumers;
- preserve reliable `CommandResult`, `SaveReady` and fatal/error events without
  render backpressure drops.

It must not:

- mutate simulation stores outside the Worker;
- hold authoritative world, job, pawn, resource or build state in Web;
- synthesize command acceptance or blocked reasons;
- provide a hidden global singleton session;
- expose internal Worker source paths as app imports;
- add public save compatibility or migration promises.

## Consequences

### Positive

- WM-0152 can resume through a real Worker session instead of the WM-0151
  projection harness.
- Package boundaries remain mechanically enforceable by the current root-import
  guard.
- Worker-specific browser construction stays owned by `sim-worker`.
- Future platform or Electron wrappers can wrap the same root API later.

### Negative

- `apps/web` gains a direct dependency on `@wuming-town/sim-worker`.
- Web builds will bundle the Worker runtime path when the session client is
  used.
- The browser client API becomes a public surface that needs compatibility
  review when changed.

### Risks and mitigations

- Risk: Web treats the session client's latest messages as authoritative local
  state.
  Mitigation: the API contract must name it as transport/read-model delivery
  only; WM-0152 must render Worker projections and command results, not mutate
  state.
- Risk: reliable command results are dropped with render snapshots.
  Mitigation: the client must separate reliable messages from any lossy
  projection/backpressure behavior.
- Risk: a root API hides a global Worker singleton.
  Mitigation: require explicit session construction and explicit destroy.
- Risk: save/load is implied by `RequestSave` support.
  Mitigation: expose capability/status and fail-closed unsupported save/load
  reasons; do not persist playable command state in Web storage until a
  versioned save task lands.

## Contracts introduced

- Public import shape for Web: `import { ... } from "@wuming-town/sim-worker"`.
- `packages/sim-worker/src/browser-worker-entry.ts` remains internal.
- `apps/web` may depend on `@wuming-town/sim-worker`; it may not depend on
  `@wuming-town/sim-core`.
- The browser session client owns transport lifecycle only.
- Simulation Worker / headless runtime remains the only authority for commands,
  jobs, pawns, resources, build progress, blocked reasons and world hashes.
- Save/load stance: no public save compatibility promise. In-progress playable
  command persistence remains unsupported unless a later task adds versioned
  focused save sections and migration evidence.

## Verification

The precursor implementation must run:

- `corepack pnpm typecheck`
- `corepack pnpm exec vitest run packages/sim-worker/src/playable-command-worker.test.ts packages/sim-protocol/src/protocol-validation.test.ts`
- `node tools/test-runner.mjs e2e --filter worker-smoke`
- `corepack pnpm boundaries:check`
- `node tools/validate-handoff.mjs`
- `node .agents/skills/wuming-town-agent-workflow/scripts/taskctl.mjs validate`
- `node .agents/skills/wuming-town-agent-workflow/scripts/taskctl.mjs status`
- `git diff --check`
- `corepack pnpm quality`

Worker smoke coverage must prove a real Chromium module Worker can be reached
through the public root session API and can return `Ready`, `CommandResult`,
`RenderSnapshot` or `UiDelta` for the WM-0150 playable scenario.

## Rollback / migration

Before WM-0152 consumes the bridge, rollback is removing the public session API
and the `apps/web -> @wuming-town/sim-worker` dependency edge.

After WM-0152 consumes the bridge, rollback must revert Web to a blocked state
or to the explicitly labeled WM-0151 projection harness. It must not add
UI-owned authority, deep imports, magic command strings, local job state,
unversioned caches or public save compatibility as a fallback.

If a later platform-owned adapter supersedes this decision, it should wrap the
same root `sim-worker` API first, then remove the direct Web dependency only
after boundary checks and Web/Electron parity tests pass.
