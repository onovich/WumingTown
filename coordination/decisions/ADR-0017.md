# ADR-0017: Integrated authoritative GameSession runtime

- Status: Proposed
- Date: 2026-07-10
- Owners: systems-architect
- Related tasks: WM-0162, WM-0163, WM-0164, WM-0165, WM-0166, WM-0167, WM-0168

## Context

WM-0161 reset product progress around one continuously running authoritative
town. The repository already has strong M0-M8 evidence: deterministic stores,
indexed work selection, explicit jobs, reservation ledgers, focused scenario
runners, Worker parity tests, focused save/replay harnesses, Web shell
infrastructure and platform gates. The default product path still does not
compose those assets into one long-lived game session. It starts from a static
product-gate read model, overlays a focused playable command slice, and advances
that slice through explicit drain helpers rather than a normal Worker-owned
game clock.

PR-1 needs a reviewed architecture before implementation. This ADR decides the
authoritative runtime shape, the minimum versioned projection contract and the
smallest proposed task DAG. WM-0162 does not implement `GameSession` or the
protocol decision, change save formats, claim or promote PR-1 implementation
tasks, or create PR-2+ tasks. It authorizes WM-0164 to implement the projection
contract and schema bump described below after WM-0163 is complete.

The detailed inventory and execution gates are in
`docs/05_tech/12_integrated_gamesession_architecture.md`.

## Decision Drivers

- Preserve Simulation Worker / headless authority.
- Keep `sim-core` free of DOM, React, Pixi, Electron, Node filesystem and real
  clock dependencies.
- Use fixed 30 TPS, stable iteration, seeded random streams and integer or
  fixed-point state.
- Reuse owner stores and indexed work surfaces instead of adding a new hidden
  runtime.
- Stop treating scenario runners, static fixtures or UI adapters as product
  authority.
- Make resident positions, town summaries and requested selection detail
  representable by one reviewed, versioned Worker projection contract.
- Make save/load boundaries explicit without introducing public compatibility.
- Keep PR-1 implementable as a small DAG with at most three concurrent
  write-heavy tasks.

## Considered Options

1. Keep the current `catalogVersion` Worker modes and add more scenario ids.
   Rejected. Those modes rerun focused scenario summaries for each projection
   and do not create one long-lived session with shared owner stores, command
   tail, pause/speed semantics or save boundaries.

2. Extend the WM-0150 playable command slice into the default product.
   Rejected. The slice is useful proof for command/result/projection plumbing,
   but it has two focused pawns, fixed targets and narrow command chains. It
   would preserve the product false positive instead of integrating M1-M8
   stores.

3. Let Web own a `GameSession` facade and synchronize with Worker output.
   Rejected. That makes UI state authoritative, creates hidden shared state and
   breaks Worker/headless parity. React and Pixi may keep camera, hover,
   selection and locale state only.

4. Build a new runtime package before PR-1.
   Deferred. A future `sim-session` package may be justified after the first
   product session proves its boundary. PR-1 should first use `sim-core`
   owner-store composition and the existing `sim-worker` package.

5. Create one `GameSessionRuntime` in `sim-core`, hosted by Worker/headless,
   initialized by scenario definitions, and projected as read-only render/UI
   models.
   Chosen. This preserves deterministic owner stores, keeps UI thin and gives
   PR-1 a testable path to continuous play.

6. Keep schema v2 and place PR-1 facts in the existing optional hash/string
   fields or `PlayableProjectionV1`.
   Rejected. `RenderSnapshotPayload` has no entity-position lanes and the
   focused playable read model cannot represent the integrated resident,
   resource, day/night, alert and selection-detail contract. Treating missing
   fields as an implementation discovery would make WM-0164 and WM-0165
   mutually blocked.

## Decision

Create one authoritative `GameSessionRuntime` concept for PR-1. The runtime is
an explicit long-lived state object in `sim-core` with no singleton access,
global caches, real-time source or UI dependency. Simulation Worker and Node
headless own instances. React, Pixi, Electron and Web adapters send commands
and render read-only projections only.

### Runtime Ownership

`GameSessionRuntime` owns:

- session id, content/scenario identity, tick, requested speed, pause state and
  deterministic command ordering;
- seeded random streams and canonical world hash inputs;
- map, entity, location, spatial, item, storage, job, reservation, work-offer
  and path basis owner stores;
- the PR-1 minimum town stores for residents, needs, day/night, food/materials,
  lamp/build targets and structured reasons;
- dirty projection queues and metrics derived from owner-store changes.

It must not own:

- wall-clock scheduling;
- renderer interpolation state;
- React/HUD selection, hover, camera, locale or settings;
- platform storage bytes;
- derived work/path/read-model caches without owner versions and rebuild rules.

### Lifecycle

The lifecycle is:

```text
uninitialized -> initializing -> running <-> paused
running/paused -> saving -> running/paused
running/paused -> stopping -> stopped
any state -> fatal
```

`LoadSession` is an initialization source, not a second runtime. Until a
versioned PR-1 save snapshot exists, `LoadSession` for `GameSession` must fail
closed or remain unsupported; it must not load shell fixture state as
authoritative gameplay.

### Tick Semantics

`sim-core` exposes deterministic integer tick advancement. One tick performs a
stable phase order:

1. apply accepted commands for this tick;
2. process owner-store dirty keys and derived indexes with bounded caps;
3. select or advance work through indexed WorkOffer, Top-K pathing,
   reservations and explicit job drivers;
4. advance needs, environment, lamp/build state and PR-1 scenario goals;
5. release terminal reservations and record structured reasons;
6. update dirty projection keys, metrics and canonical hash input.

No phase may scan all pawns against all work, all entities or all cells. Any
new selector must document candidate caps and exact owner-store indexes.

### Worker Scheduling

Simulation Worker owns normal product scheduling. It may use host timers or
message-loop scheduling outside `sim-core`, but the simulation only sees
integer tick requests. Speed maps to an effective target of `0`, `30`, `60` or
`90` ticks per wall second. `Pause` makes effective speed `0` while preserving
the requested speed. Snapshot delivery is latest-wins for render/UI projections
and reliable for `CommandResult`, `SaveReady`, alerts and fatal errors.

Drain helpers from WM-0160 remain legal for tests, command waits and tools.
They are not the normal game clock for default product play.

### Headless Parity

Worker and Node headless share the same `GameSessionRuntime.advanceTicks`
surface. Headless tests drive explicit tick counts; Worker continuous
scheduling drives the same surface. Given the same seed, scenario definition and
command stream, Worker and headless must produce matching checkpoint hashes and
read-model hashes.

### Scenario Initialization

`ScenarioDefinition` describes initial content, seed, map, resident starts,
resources, facilities, goals and optional scripted setup. It does not own tick
progress, jobs, reservations, resident position, save state or product
projection. Existing M1-M8 scenario runners become initializer references,
regression harnesses or test-only evidence; they do not remain mutually
exclusive product runtimes.

### Product Projection And Interpolation

PR-1 projections are read-only products of the same session:

- `RenderSnapshot.gameSession` carries `GameSessionRenderProjectionV1`: an
  explicit projection version, coherent tick/snapshot/world/content/map basis,
  map dimensions, the previous snapshot sequence and bounded render entities.
  Each render entity has an `EntityId(index,generation)`, kind, render def,
  Q16 world position, facing, animation state and numeric flags. Resident,
  resource, structure, lamp and build-site positions are therefore
  representable without Web-owned coordinates.
- `UiDelta.gameSession` carries `GameSessionUiProjectionV1` on the same basis:
  pause/requested-speed state; day index, tick-of-day, phase and daylight Q16;
  at least the PR-1 resident summaries; food/wood/stone/lamp-oil counts; job
  markers; structured alerts; and the latest requested resident/resource/
  structure selection detail. Labels are localized by the client from def ids,
  state enums, reason codes and ordered scalar parameters.
- Presentation interpolation may estimate screen positions between two
  authoritative snapshots, but it must never update resident position, job
  state, build progress, resources or reasons in authority.

The envelope protocol remains `protocolVersion = 1`; WM-0164 must bump
`schemaVersion` from 2 to 3 and add
`GAME_SESSION_PROJECTION_VERSION = 1`. No new message family is added.
`InitSession` gains an optional explicit projection request
`{ kind: "game_session", version: 1 }`; the PR-1 product route must send it.
`Ready` must echo the exact accepted contract before the client enters running
state. The nested GameSession projection fields are optional only for same-build
legacy M0-M8 regression sessions that did not request this contract; after a
GameSession request is accepted they are mandatory on every applicable render
and UI publication.

Compatibility is deliberately narrow and fail-closed:

- schema-v2 or any other envelope schema is rejected by a schema-v3 Worker or
  client; there is no mixed-build hot connection or down-conversion;
- an unknown projection version is rejected during `InitSession` with a
  structured unsupported-schema reason;
- a missing/mismatched `Ready` contract, malformed GameSession projection,
  wrong session id, or incoherent tick/snapshot/hash basis moves the browser
  session to fatal/closed state and never falls back to static fixtures;
- stale lower-sequence render snapshots may be dropped under the existing
  latest-wins rule, but a newer malformed or contract-incompatible message may
  not be silently skipped;
- legacy scenario modes may omit the projection request and keep their focused
  same-build payloads for regression tests, but they cannot satisfy or back the
  default PR-1 product route.

WM-0164 is the sole writer for the listed `packages/sim-protocol` projection
files and their tests. WM-0163, WM-0165 and WM-0166 may consume only the package
public root and must not modify wire-protocol files. WM-0165 has one narrow
consumer-completion exception recorded below for the existing Web presentation
read model; it does not modify a message, schema, projection, validator or
package export.

#### WM-0165 presentation consumer completion

ADR-0017 already requires authoritative render rows and selection detail to
represent `resource` without a client alias. The older public
`WorldEntityKind` presentation enum predates that decision and cannot express
the approved kind. WM-0165 may therefore add only the member `"resource"` to
`packages/sim-protocol/src/web-read-model.ts`, map the existing GameSession
render kind to it one-to-one, and complete the exhaustive Pixi marker/selection
and localized HUD consumers.

This is an additive presentation contract correction, not a Simulation Worker
protocol change: `protocolVersion`, `schemaVersion`,
`GAME_SESSION_PROJECTION_VERSION`, message families, projection payloads,
validators, authority ownership and public exports remain unchanged. It adds
no dependency and authorizes no resource count, trend, job, reservation or
selection-detail synthesis. A broader semantic-kind field, another entity kind
unrelated to PR-1, or any wire change requires a separate reviewed decision.

The default Web route must stop using `WEB_PRODUCT_GATE_READ_MODEL` as gameplay
truth. That fixture may remain in tests, diagnostics or explicitly labeled
historical gates.

### Save/Load Boundary

PR-1 records the authoritative boundary but does not create a public save
compatibility promise. A future `GameSessionSaveSnapshot` must include owner
store snapshots, random stream positions, current tick, command tail, content
manifest, session metadata and rebuild diagnostics. Derived WorkOffer rows,
path caches, read models, render snapshots, Pixi interpolation and UI selection
do not enter authority. Load validates in scratch state, rebuilds derived
surfaces, verifies invariants and only then resumes ticks.

### WM-0166 review repair addendum (2026-07-11)

The first WM-0166 review found that PR-1 does not yet satisfy this ADR's exit
gates. The browser Worker evidence stopped at tick >=9; the separate 18000-tick
run used headless explicit stepping and therefore did not prove 600 wall-clock
seconds of Worker-owned scheduling. The current minimal job also writes
`moving` and then `working` inside tick 0 and completes by tick 3, so the
unchanged 3-tick publication quantum observes idle only.

The approved repair is strictly serial and does not change the protocol,
schema, projection version or save stance:

1. WM-0167, owned by simulation-engineer, is the only ready product writer. It
   repairs the explicit integer-tick JobCore lifecycle so both `moving` and
   `working` persist for at least two consecutive coherent publication pairs.
   It may harden bounded Worker-host wall-time debt/catch-up, but must keep the
   100 ms / 3-tick speed-1 quantum and must not expose a real clock to
   `sim-core`.
2. WM-0168, owned by qa-performance, remains proposed until WM-0167 is done.
   It changes evidence only and must prove a real Chromium module Worker for
   at least 600000 ms and 18000 authoritative ticks, cadence/backpressure, the
   repaired lifecycle publications and the passive default Web route.
3. WM-0166 remains blocked and depends on WM-0168. It may resume only after
   both repair tasks are independently verified, integrated and done; then it
   reruns every required check and requests a fresh independent review.

No fake timer, in-process Worker, headless tick loop, UI advance/wait/drain
helper or reduced threshold may substitute for WM-0168's wall-clock evidence.

## Consequences

### Positive

- Product play has one authority instead of a static fixture plus focused
  overlays.
- Existing deterministic stores and save/replay evidence remain useful.
- Worker scheduling, headless tests and product projections converge on one
  runtime.
- Static product-gate data is quarantined without deleting historical evidence.

### Negative

- Scenario runners can no longer be treated as product runtimes.
- Worker scheduling and projection code need sharper backpressure and lifecycle
  handling.
- The default Web shell must adapt to a session read model before PR-1 can exit.
- Some M4-M8 content remains regression/test-only until later PR phases.

## Migration

1. WM-0163 creates the `sim-core` runtime scaffold, PR-1 initializer and
   headless parity tests.
2. WM-0164 implements the approved schema-v3 GameSession projection contract
   as the only `sim-protocol` writer, then hosts the runtime in `sim-worker`
   with continuous scheduling and fail-closed projection validation.
3. WM-0165, only after WM-0164, routes the default Web shell to that exact
   Worker projection and quarantines static fixtures to diagnostics/tests.
4. WM-0167 repairs the lifecycle/scheduler gaps found by the first WM-0166
   review; WM-0168 independently records real Worker/default Web evidence.
5. WM-0166 resumes only after WM-0168 is done, reruns the integrated exit gates
   and records remaining unconnected historical assets before PR-2 planning.

The reviewed critical path is strictly serial:
`WM-0162 -> WM-0163 -> WM-0164 -> WM-0165 -> WM-0167 -> WM-0168 -> WM-0166
resume`. WM-0164 may make only the protocol/schema change approved in this ADR.
Any additional message family, projection version, command contract or save
mutation requires another reviewed architecture gate.

## Rollback

Before implementation, rollback is deleting this ADR, the WM-0162 planning
document and the proposed WM-0163..WM-0166 task packets.

After WM-0163, rollback reverts the `GameSessionRuntime` scaffold and restores
existing scenario runners as regression-only code. After WM-0164 or WM-0165,
rollback returns the default product route to a clearly labeled blocked/static
state and reverts the Worker and Web protocol consumers together to the last
schema-v2 build. Mixed schema-v2/schema-v3 components are not a rollback mode.
Rollback must not use UI-owned authority, magic command strings, unversioned
caches, global scans, public save compatibility or the WM-0160 drain helper as
normal product time.

## Performance Implications

`GameSessionRuntime` is a hot simulation surface. PR-1 must keep per-tick work
bounded by dirty keys, indexed candidates and explicit caps. Projection builders
may allocate per publication, but hot tick phases must avoid per-entity arrays,
closures and strings. Worker snapshot backpressure may drop stale render
snapshots, but reliable command/save/fatal messages must never be dropped.

Metrics must at least report tick cost, command backlog, projection backlog,
snapshot drops, WorkOffer visits, exact path requests, reservation conflicts,
job terminal counts, resource conservation deltas and hash checkpoints.
