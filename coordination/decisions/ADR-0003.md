# ADR-0003: M1 entity/store memory layout

- Status: Proposed
- Date: 2026-06-24
- Owners: systems-architect
- Related tasks: WM-0015

## Context

M1 needs a deterministic simulation kernel for map storage, entity location,
item stacks, jobs, reservations, and index cleanup. The kernel must run with
the same authoritative state in Node headless and the browser Simulation
Worker. UI, Pixi, React, and Electron must consume read models only.

WM-0007 already introduced generation-checked `EntityId`, bounded numeric
component storage, stable ascending iteration, and deferred structural command
commit buffers. ADR-0003 proposes the M1 ownership and memory layout that
future tasks build on top of that foundation. This ADR does not accept or
implement the decision; it is a Proposed decision for independent review.

The proposal must prevent these failure modes before implementation begins:

- UI-owned simulation state or renderer-local corrections to world state.
- Hidden shared state through singleton stores, ambient caches, or module-level
  mutable registries.
- Global scans for pawn thinking, work discovery, routine cleanup, or cache
  repair.
- Unversioned caches that can survive map, entity, job, reservation, or save
  migrations without explicit invalidation.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered options

- Object graph per entity: easy to prototype and inspect, but rejected for M1
  because map cells, item stacks, jobs, and reservations would become mutable
  object references with unclear ownership, high allocation pressure, fragile
  save traversal, and hidden aliasing between systems.
- Sparse `Map<EntityId, Component>` stores for all domains: simple ownership
  at small scale, but rejected for hot M1 state because hash-map iteration and
  boxed component objects make stable order, memory footprint, and per-tick
  allocation harder to control. Sparse arenas remain acceptable for low-frequency
  metadata when they expose handles and version fields.
- UI/read-model mirror as a writable cache: rejected. It would violate Worker
  authority, create divergent Node/headless and browser behavior, and make
  rollback through save/replay impossible.
- Shared singleton service stores: rejected. A singleton `ReservationService`,
  `LocationCache`, or `ItemIndex` hides dependencies from tests, makes state
  reset unclear, and encourages cross-package deep imports.
- Authoritative SoA owner stores plus explicit derived indexes: proposed. Each
  mutable gameplay fact has one owning store in `sim-core`; derived indexes are
  local, versioned, and rebuilt or invalidated from owner-store mutations.

## Decision

Use an explicit `WorldState` aggregate in `sim-core` as the only authoritative
owner of M1 simulation data. `WorldState` is created by the headless runner or
Simulation Worker and passed to systems explicitly. It is not a module-level
singleton, and it is not imported or mutated by UI, Pixi, React, Electron, or
protocol packages.

High-frequency state uses structure-of-arrays numeric storage: typed arrays,
active bitmaps, generation validation, stable ascending iteration, and fixed
capacity or explicit growth points. Low-frequency state may use arenas or pools
only when handles, generations, version fields, and cleanup hooks are explicit.

All authoritative writes go through the simulation tick/structural commit
surface. Derived indexes may be updated in the same deterministic commit phase
or marked dirty for budgeted rebuild, but they must never become a second owner
of the same gameplay fact.

### M1 authoritative owners

| State surface       | Authoritative owner                                                                                                                                                                                                                                                                                          | Derived/non-owner surfaces                                                                                                                                                                                          |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Map location        | `LocationStore` owns each entity's membership: none, map cell/chunk, or container/slot. It validates `EntityId(index,generation)` and enforces exactly one map or container membership.                                                                                                                      | Map chunk occupancy, spatial indexes, room/region membership, renderer positions, inspector selections, and path targets are derived. They must be updated from location mutations or rebuilt from `LocationStore`. |
| Item quantity       | `ItemStackStore` owns stack `DefIndex`, quantity, quality/status flags needed by M1, and the entity/container that holds the stack through `LocationStore`. Quantity is an integer lane with bounds checks; no floating accumulation.                                                                        | Storage supply/demand indexes, WorkOffer candidates, read models, and reservation availability counters are derived. A reservation may hold a claim against quantity, but it does not own the total quantity.       |
| Job core            | `JobCoreStore` owns pawn/job binding, job id, job kind, target references, status, driver state key, progress counters, failure reason code, and interruption policy lanes. Driver implementation must be an explicit serializable state machine built on these fields.                                      | WorkOffer indexes, UI job labels, logs, and reservation owner metadata reference jobs but cannot own job status or execution position. Promise, Generator, coroutine, or closure execution positions are forbidden. |
| Reservation ledger  | `ReservationLedger` owns active claims for entity, cell, quantity, interaction-spot, and capacity channels. It records owner entity, job id, channel, amount, created tick, lease-expiry tick for recovery, and structured conflict reason codes. Transactions acquire all claims or leave no partial state. | Item availability indexes, WorkOffer filtering, UI explanations, and path candidates read ledger projections. They cannot mutate claims directly and cannot infer ownership from cached availability alone.         |
| Index cleanup hooks | `LifecycleCleanupCoordinator` owns deterministic cleanup dispatch for movement, transfer, despawn, destroy, job cancel/fail/complete, reservation release, and save/load rebuild. Each index owns its own removal/dirty queues and exposes hook handlers registered during `WorldState` construction.        | Opportunistic global cache repair, JavaScript finalizers, UI unmount cleanup, or save-loader side effects are forbidden cleanup owners.                                                                             |

### Explicit non-decisions

- ADR-0003 does not choose final save container byte layout. ADR-0005 owns the
  persistent container and migration format.
- ADR-0003 does not choose final Worker snapshot encoding. ADR-0004 owns the
  read-model protocol.
- ADR-0003 does not finalize deterministic numeric scales or tick phase names.
  ADR-0006 owns the full numeric and phase contract. Until then, M1 owner
  stores must stay integer-only and phase-friendly.
- ADR-0003 does not implement map, item, job, reservation, pathing, or UI code.

## Consequences

### Positive

- Every mutable gameplay fact has one reviewable owner, making save/replay,
  rollback, and Worker/headless parity easier to test.
- Derived indexes can be optimized aggressively without becoming hidden shared
  state because rebuild inputs and invalidation versions are explicit.
- UI and renderer packages remain thin consumers of protocol/read-model data.
- WM-0007 generation validation and stable iteration carry forward into M1
  stores instead of being bypassed by object references.

### Negative

- Implementation tasks must define more typed lanes and lifecycle hook surfaces
  up front before feature behavior is visible.
- Debugging requires owner-store inspection tools because read models and
  indexes are intentionally non-authoritative.
- Save/load migration must rebuild indexes after reading owner stores, which
  adds explicit load-time validation work.

### Risks and mitigations

- Risk: derived indexes drift from owner stores. Mitigation: every owner-store
  mutation must publish a hook event or dirty range, and tests must cover
  movement, transfer, despawn, destroy, cancel, and load rebuild.
- Risk: transaction code duplicates quantity/location authority. Mitigation:
  reservation transactions may reserve claims but must commit quantity/location
  changes through the owning stores.
- Risk: cleanup hooks become another hidden global bus. Mitigation:
  `LifecycleCleanupCoordinator` is constructed inside `WorldState`, receives
  explicit registered handlers, and is reset with the world.
- Risk: migration pressure encourages unversioned caches. Mitigation: every
  derived index and cache records input versions or is declared scratch and
  rebuilt from owner stores after load.

## Contracts introduced

- `sim-core` owns authoritative M1 world state. `sim-worker` and headless runner
  may host it; UI, Pixi, React, Electron, protocol, and persistence packages may
  not mutate it.
- `WorldState` and all owner stores are passed explicitly. Module-level mutable
  world stores and hidden singleton caches are forbidden.
- Entity identity remains `EntityId(index,generation)`. Persistent and
  cross-tick references validate generation before reads or writes.
- A live entity has at most one authoritative location membership. Container
  ownership and map-cell ownership are mutually exclusive.
- Item quantities are integer authoritative lanes owned by `ItemStackStore`;
  derived availability and reservation counters cannot exceed or replace the
  owner quantity.
- Job execution position is serialized through `JobCoreStore` and future driver
  state fields, never through Promise, Generator, coroutine, closure, or UI
  state.
- Reservation acquire/release is ledger-owned and all-or-nothing. Failed
  acquisition returns structured reason codes rather than partial claims.
- Lifecycle cleanup is hook-driven and deterministic. Routine cleanup may not
  scan all entities, all map cells, all item stacks, or all jobs.
- Global scans are allowed only in explicit load validation, debug tooling, or
  benchmark fixtures. They must not run during normal ticks, pawn thinking,
  work-offer lookup, reservation acquisition, or movement cleanup.
- Derived indexes and caches must carry input version fields or be rebuilt from
  owner stores. Unversioned caches are forbidden.

## Performance implications

- Hot owner stores use typed arrays and active bitmaps rather than per-entity
  objects.
- Normal ticks may not allocate objects, arrays, closures, or strings in loops
  proportional to entity, cell, item, job, or reservation count.
- Normal ticks may not use unbounded `sort`, `map`, `filter`, `reduce`,
  `flatMap`, regular expressions, or string concatenation in hot loops.
- Work discovery and cleanup must use spatial/work/reservation indexes,
  dirty queues, and Top-K candidate caps instead of global scans.
- Large rebuilds are local invalidations or budgeted rebuild queues with
  observable backlog metrics.
- Indexes and async results must include input versions; stale results are
  rejected instead of patched in place.
- Benchmarks must report elapsed time, call counts, processed entity/cell
  counts, queue backlog, and allocation-sensitive counters where tooling allows.

## Rollback / migration

Before acceptance, rollback is deleting `coordination/decisions/ADR-0003.md`
and removing the ADR index entry. No runtime state or save file depends on this
proposed ADR.

If accepted before implementation, migration is from the WM-0007 foundation to
domain-specific M1 owner stores:

- Keep `EntityRegistry` generation validation and stable iteration as the
  identity foundation.
- Add map, location, item, job, and reservation stores behind public
  `sim-core` exports only.
- Convert temporary tests or fixtures through explicit construction helpers
  rather than UI-owned or singleton state.
- Rebuild all derived indexes from owner stores after load until ADR-0005
  defines persistent save sections.

If an implementation task based on this ADR fails review before a save
compatibility promise exists, rollback is reverting that task branch and keeping
ADR-0003 as Proposed or superseding it with a narrower proposal. Do not merge
temporary global scans, hidden shared stores, or UI-owned authority as a
rollback bridge.

After ADR-0005 introduces versioned save sections, migration away from this
layout requires a superseding ADR and N-to-N+1 migration tests. Old-version
conditionals must stay in migration functions, not scattered through normal
owner-store reads or index updates.
