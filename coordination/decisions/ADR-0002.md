# ADR-0002: Pinned M0 tool versions

- Status: Accepted
- Date: 2026-06-23
- Owners: systems-architect
- Related tasks: WM-0002

## Context

WM-0002 must turn the approved stack into reproducible quality gates. Versions were verified against npm registry metadata on 2026-06-23 and are pinned exactly where installed.

## Decision drivers

- Correctness and determinism
- Performance and scale
- AI/human development efficiency
- Web/Windows portability
- Save and mod compatibility
- Security and rollback cost

## Considered options

- Floating major ranges: convenient upgrades, but violates the dependency security policy.
- Pin only TypeScript and pnpm: smaller lockfile, but lint/test behavior could drift.
- Pin all installed M0 tools exactly: stronger reproducibility with small maintenance overhead.

## Decision

Pin the installed M0 toolchain as follows:

| Tool                     |     Version | Location                                      |
| ------------------------ | ----------: | --------------------------------------------- |
| Node.js                  | 24 LTS line | `engines.node`, CI `node-version: "24"`       |
| pnpm                     |      11.8.0 | `packageManager`, `engines.pnpm`, CI Corepack |
| TypeScript               |       5.9.3 | root `devDependencies`                        |
| Vite                     |       8.1.0 | root `devDependencies`                        |
| Vitest                   |       4.1.9 | root `devDependencies`                        |
| ESLint                   |      10.5.0 | root `devDependencies`                        |
| `@eslint/js`             |      10.0.1 | root `devDependencies`                        |
| `typescript-eslint`      |      8.62.0 | root `devDependencies`                        |
| Prettier                 |       3.8.4 | root `devDependencies`                        |
| `eslint-config-prettier` |      10.1.8 | root `devDependencies`                        |
| `@types/node`            |     24.13.2 | root `devDependencies`                        |

The architecture-selected runtime frameworks are not installed in WM-0002 because no WM-0002 package consumes them yet. Registry-resolvable candidates recorded for later tasks are React `19.2.7`, React DOM `19.2.7`, PixiJS `8.19.0`, Electron `42.4.1`, and `@vitejs/plugin-react` `6.0.3`. The future task that first imports each runtime framework must add it exactly, update the lockfile, and extend this ADR or create a superseding ADR with dependency review evidence.

## Consequences

### Positive

- Root gates are reproducible from `pnpm-lock.yaml`.
- CI and local setup use the same pnpm line.
- Runtime packages with heavier install/security footprint are not introduced before they are used.

### Negative

- CI uses the Node 24 LTS line rather than a patch-level Node binary pin.
- Future UI/Electron tasks must perform their own dependency security review before first import.

### Risks and mitigations

- Risk: Node patch drift changes behavior. Mitigation: keep source code platform-neutral and add patch-level pinning when packaging/runtime smoke tasks begin.
- Risk: toolchain upgrades pile up. Mitigation: batch upgrades through dependency policy and rerun all root gates.

## Contracts introduced

- External npm dependency versions must be exact; no `latest`, `*`, `^`, or `~`.
- Internal workspace dependencies use `workspace:0.0.0`.
- Root gates are `typecheck`, `lint`, `format:check`, `test`, `content:validate`, and `boundaries:check`.

## Verification

- `pnpm install --frozen-lockfile`
- `pnpm typecheck`
- `pnpm lint`
- `pnpm format:check`
- `pnpm test`
- `pnpm content:validate`
- `pnpm boundaries:check`

## Rollback / migration

Revert root package/config files and regenerate the lockfile before any product runtime state depends on this toolchain.
